# MDM-blog 项目规则

## 4. 独立项目系列文章规范

### 4.1 创作风格

独立项目系列文章需采用「技术叙事 + 深度讲解」的创作风格，既要有真实项目经历的故事性，又要有技术干货的实用性，章节过渡要顺滑，且文章内容不允许完全一致的重复，同时文章字数无封顶。

| 风格要素     | 具体要求                                  |
| -------- | ------------------------------------- |
| **生动比喻** | 每个技术概念都要有通俗易懂的类比，如将注册中心比作物业管理处、网关比作门神 |
| **架构图示** | 复杂架构需配合 ASCII 图或表格说明，如请求流程图、组件关系图     |
| **代码示例** | 核心技术点必须提供真实可运行的代码片段                   |
| **概念详解** | 专业术语首次出现时必须解释，不能只堆砌名词                 |
| **表格对照** | 优先使用表格进行概念对比，如家庭类比 vs 技术概念对照表         |

```
### 核心组件详解
每个核心组件都需要包含：
1. **为什么需要它？** - 问题背景 + 通俗比喻
2. **它是什么？** - 技术原理解释
3. **怎么用？** - 核心配置或代码示例
4. **注意事项** - 常见问题和最佳实践
```

```
### 模块类型参考
| 模块类型 | 示例 | 展示内容 |

| CRUD 型 | 文章、问答、吐槽 | 基本增删改查、关联查询 |

## 项目亮点与踩坑实录
### 问题解决类
每个问题采用以下结构：
- **问题现象**：实际遇到的问题描述
- **根因分析**：为什么会出现这个问题
- **解决方案**：具体配置或代码（必须包含）
- **经验总结**：从这个坑里学到了什么

### 技术对比类
- 方案 A vs 方案 B 的优劣对比（表格形式）
- 为什么最终选择了当前方案

---

*文章结尾感悟，从工程视角总结项目价值*
```

### 4.3 禁止事项

| 禁止      | 说明             |
| ------- | -------------- |
| 堆砌技术名词  | 每个技术名词都要有解释或比喻 |
| 只有配置无代码 | 核心功能必须展示关键代码   |
| 只有代码无解释 | 代码必须有注释和上下文说明  |
