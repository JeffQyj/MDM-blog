2024 年 5 月，我完成了黑马苍穹外卖项目的学习。这是继青橙商城、十次方社区平台、传智健康之后的首个 Spring Boot 企业级实战项目，也是第一次接触完整的外卖平台业务逻辑。苍穹外卖是一款面向中小型餐饮企业的前后端分离外卖订购系统，管理端负责员工、菜品、套餐、订单、数据统计等运营管理，用户端则提供微信小程序点餐体验。通过这个项目，我把 SSM 阶段的 CRUD 能力升级为「带着业务思维写代码」

## 项目背景

苍穹外卖定位为**餐饮企业的私域外卖管理系统**，与美团、饿了么等聚合平台不同，它强调品牌自营，帮助商家建立自己的外卖渠道。系统的核心价值在于：

- **订单履约闭环**：从用户点餐、支付、商家接单、配送到完成的全链路数字化
- **运营数据可视化**：营业额、订单量、热销菜品等经营指标一目了然
- **员工精细化管理**：基于角色的权限控制，让不同岗位的员工各司其职

### 双端架构设计

项目采用\*\*管理端（PC 后台）+ 用户端（微信小程序）\*\*双端架构：

| 端   | 使用角色        | 核心功能                |
| --- | ----------- | ------------------- |
| 管理端 | 商家管理员、厨师、骑手 | 员工管理、菜品管理、订单处理、数据统计 |
| 用户端 | 消费者         | 微信登录、浏览菜品、下单支付、订单跟踪 |

:::note
苍穹外卖的双端架构和传智健康（双端：后台+微信端）思路一脉相承——前后端分离设计，共用同一套后端服务。但业务复杂度差异明显：传智健康是「健康管理」，侧重预约和档案；苍穹外卖是「外卖平台」，核心是交易链路和实时性。两者对比着看，能更深理解「不同业务场景如何影响技术实现」。
:::

## 技术架构

### 整体架构

苍穹外卖采用**单体架构（Monolithic Architecture）**，所有业务模块（用户服务、订单服务、菜品服务、员工管理、文件上传、微信登录等）统一部署于同一 Spring Boot 应用中。这种架构虽不具备微服务的弹性伸缩能力，但极大降低了初学者的学习门槛——无需面对服务发现、分布式事务、链路追踪等复杂概念，可以专注于业务逻辑本身。

```
┌─────────────────────────────────────────────────────────────────┐
│                         用户层                                    │
├─────────────────────────────┬───────────────────────────────────┤

└─────────────────────────────┴───────────────────────────────────┘
```

### 为什么选择单体架构？

**与传智健康的架构对比**

| 对比维度 | 单体架构（苍穹外卖）   | SOA + Dubbo（传智健康） |
| ---- | ------------ | ----------------- |
| 部署方式 | 单一 War/Jar 包 | 多服务独立部署           |
| 服务拆分 | 模块化（代码层面拆分）  | 服务化（进程级别拆分）       |
| 复杂度  | 低，适合学习阶段     | 高，需要服务治理          |
| 扩展性  | 整体扩展         | 按需扩展特定服务          |
| 适用场景 | 中小型项目、团队规模小  | 中大型项目、团队协作多       |

苍穹外卖选择单体架构的理由很实际：

1. **学习曲线要平稳**：这是第一次系统学习 Spring Boot，直接上微服务容易邯郸学步。单体架构先把 Spring Boot 的核心能力（自动配置、 starter 依赖、内嵌容器）吃透，再考虑分布式是更合理的路径。
2. **业务规模不需要拆分**：苍穹外卖只有 6 大模块、十几张表，单体架构完全 hold 得住。
3. **调试方便**：单体应用一个断点全链路追查，比 Dubbo 的 RPC 调用链追踪简单太多。

:::note
回过头看，单体架构不是「落后」的代名词。很多中小企业的主营业务系统，上线十年都没拆分过，一样跑得稳稳当当。架构要匹配业务阶段，而不是一味追新。
:::

:::note
sky-common 模块的设计很值得学习。工具类（UUID 生成、日期格式化）、常量类（状态码、提示信息）、上下文（基于 ThreadLocal 存储当前用户 ID）、统一返回结果（Result<T>）全部抽离出来作为公共组件。好处是：如果以后要做新项目，直接把这个模块 Copy 过去，改改包名就能用。这种「积累可复用组件」的思维，是工程化的第一步。

## 业务模块实战

### 模块一：员工管理 — JWT 鉴权 + ThreadLocal

#### 业务场景

管理员和员工需要登录后台管理系统。登录成功后，后续请求需要携带 Token，服务器验证 Token 有效性后才知道「是谁在操作」。

#### JWT 认证流程

```
用户登录
    │
    ▼
┌─────────────────────────────────────────┐
│  1. 验证用户名密码（MD5 加密存储）           │
│  2. 生成 JWT Token（包含 userId、empId）  │
│  3. 返回 Token 给前端                      │
└─────────────────────────────────────────┘
    │
    ▼
后续请求携带 Header：Authorization: Bearer xxx
    │
    ▼
┌─────────────────────────────────────────┐
│  1. JWT 拦截器拦截请求                     │
│  2. 解析 Token，提取 userId              │
│  3. 存入 ThreadLocal（BaseContext）       │
│  4. 放行请求                             │
└─────────────────────────────────────────┘
    │
    ▼
Service 层通过 BaseContext.getCurrentId() 获取当前用户 ID
```

#### 核心代码

**JWT 工具类**：

```java
public class JwtUtil {
    private static final String SECRET_KEY = "itheima";  // 签名密钥
    private static final long EXPIRATION = 12 * 3600 * 1000L;  // 12小时有效期

    public static String createJWT(String userId) {
        return Jwts.builder()
                .setSubject(userId)
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
                .signWith(SignatureAlgorithm.HS256, SECRET_KEY)
                .compact();
    }

    public static Claims parseJWT(String token) {
        return Jwts.parser()
                .setSigningKey(SECRET_KEY)
                .parseClaimsJws(token)
                .getBody();
    }
}
```

**登录接口**：

```java
@PostMapping("/login")
public Result<EmployeeLoginVO> login(@RequestBody EmployeeLoginDTO loginDTO) {
    // 1. 验证用户名密码
    Employee employee = employeeService.login(loginDTO);

    // 2. 登录成功，生成 JWT
    String token = JwtUtil.createJWT(employee.getId().toString());

    // 3. 封装返回结果
    EmployeeLoginVO loginVO = EmployeeLoginVO.builder()
            .id(employee.getId())
            .name(employee.getName())
            .token(token)
            .build();

    return Result.success(loginVO);
}
```

**JWT 拦截器**：

```java
@Component
@Slf4j
public class JwtTokenInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        // 1. 获取 Token
        String token = request.getHeader("Authorization");

        // 2. 验证 Token
        if (token == null || !token.startsWith("Bearer ")) {
            response.setStatus(401);
            return false;
        }

        try {
            String tokenBody = token.substring(7);  // 去掉 "Bearer " 前缀
            Claims claims = JwtUtil.parseJWT(tokenBody);
            String userId = claims.getSubject();

            // 3. 存入 ThreadLocal
            BaseContext.setCurrentId(Long.valueOf(userId));

            return true;
        } catch (Exception e) {
            log.error("JWT 解析失败: {}", e.getMessage());
            response.setStatus(401);
            return false;
        }
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        // 请求完成后清空 ThreadLocal，防止内存泄漏
        BaseContext.removeCurrentId();
    }
}
```

**ThreadLocal 上下文工具**：

```java
public class BaseContext {
    public static ThreadLocal<Long> threadLocal = new ThreadLocal<>();

    public static void setCurrentId(Long id) {
        threadLocal.set(id);
    }

    public static Long getCurrentId() {
        return threadLocal.get();
    }

    public static void removeCurrentId() {
        threadLocal.remove();
    }
}
```

| 组件          | 作用                             |
| ----------- | ------------------------------ |
| JWT Token   | 无状态认证，Token 包含用户身份信息，服务端无需存储会话 |
| ThreadLocal | 每个请求线程独享的「更衣室」，存储当前登录用户 ID     |
| MD5 加密      | 密码存储加密，即使数据库泄露也无法直接看到明文密码      |
| 拦截器         | 统一拦截需要认证的请求，验证 Token 并解析用户信息   |

:::tip
JWT 的核心思想是「把身份信息存在 Token 里，服务端无需存储」。就像一张电子工牌——卡片里已经印好了员工编号、姓名、部门，门禁系统读一下就知道是谁、解没解授权。但工牌丢了谁都能用，所以 JWT 要设置合理的过期时间（苍穹外卖设为 12 小时）。
:::

:::note
ThreadLocal 是这个模块最需要理解的细节。每个请求进来，Tomcat 从线程池分配一个线程处理，ThreadLocal 就是这个线程的「更衣室」——只能当前线程访问，其他线程看不到。登录时把用户 ID 存进去，后续 Service 层通过 `BaseContext.getCurrentId()` 获取，全程不用显式传递参数。但请求结束一定要 `remove()`，否则线程归还池后数据还在，可能造成内存泄漏。
:::

### 模块二：菜品管理 — Redis 缓存 + 阿里云 OSS

#### 业务场景

菜品是外卖平台的核心商品，用户浏览菜品是高并发读操作。如果每次访问都查数据库，数据库压力巨大。解决方案是：把热门菜品列表缓存到 Redis，同时使用阿里云 OSS 存储菜品图片。

#### Redis 缓存策略

**什么数据适合缓存？**

| 数据类型 | 缓存价值 | 原因           |
| ---- | ---- | ------------ |
| 分类列表 | 高    | 变动少，所有菜品页都用到 |
| 热门菜品 | 高    | 读多写少，缓存命中率极高 |
| 营业状态 | 高    | 每次下单都要检查     |
| 菜品详情 | 中    | 变动频繁，缓存可能过期  |

**缓存实现**：

```java
@Service
@Slf4j
public class DishServiceImpl implements DishService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private static final String CACHE_KEY_PREFIX = "dish_";

    @Override
    public List<DishVO> list(Long categoryId) {
        String key = CACHE_KEY_PREFIX + categoryId;

        // 1. 先查缓存
        List<DishVO> cacheList = (List<DishVO>) redisTemplate.opsForValue().get(key);
        if (cacheList != null) {
            log.info("命中缓存: {}", key);
            return cacheList;
        }

        // 2. 缓存未命中，查数据库
        List<DishVO> dishList = dishMapper.listByCategoryId(categoryId);

        // 3. 写入缓存，设置 60 分钟过期
        redisTemplate.opsForValue().set(key, dishList, 60, TimeUnit.MINUTES);
        log.info("写入缓存: {}", key);

        return dishList;
    }

    @Override
    public void save(DishDTO dishDTO) {
        // 保存到数据库
        dishMapper.save(dishDTO);

        // 清空该分类的缓存（数据变更，缓存失效）
        redisTemplate.delete(CACHE_KEY_PREFIX + dishDTO.getCategoryId());
        log.info("清理缓存: {}", CACHE_KEY_PREFIX + dishDTO.getCategoryId());
    }
}
```

| 缓存问题 | 苍穹外卖的解决方案              |
| ---- | ---------------------- |
| 缓存穿透 | 设置空值缓存，防止恶意查询不存在的数据    |
| 缓存雪崩 | 设置随机过期时间，避免大量 key 同时过期 |
| 缓存击穿 | 使用互斥锁，只允许一个请求查数据库      |

#### 阿里云 OSS 文件上传

**场景**：菜品图片需要存储，如果存在应用服务器本地磁盘，分布式部署时会出现「这台服务器有、那台没有」的问题。

```java
@Component
@ConfigurationProperties(prefix = "sky.alioss")
public class AliOssUtil {

    private String endpoint;
    private String accessKeyId;
    private String accessKeySecret;
    private String bucketName;

    public String upload(byte[] fileBytes, String originalFilename) {
        // 1. 创建 OSS 客户端
        OSS ossClient = new OSSClientBuilder().build(endpoint, accessKeyId, accessKeySecret);

        // 2. 生成唯一文件名：时间戳 + 随机数 + 扩展名
        String extension = originalFilename.substring(originalFilename.lastIndexOf("."));
        String objectName = UUID.randomUUID().toString() + extension;

        // 3. 上传到 OSS
        ossClient.putObject(bucketName, objectName, new ByteArrayInputStream(fileBytes));

        // 4. 返回访问 URL
        String url = "https://" + bucketName + "." + endpoint + "/" + objectName;

        ossClient.shutdown();
        return url;
    }
}
```

| OSS vs 本地存储 | OSS 的优势               |
| ----------- | --------------------- |
| 本地存储        | 依赖应用服务器磁盘，扩容麻烦        |
| OSS         | 云存储，弹性扩容，CDN 加速，全球可访问 |

:::tip
苍穹外卖的图片上传走了阿里云 OSS，传智健康的图片存储则用了 FastDFS 自建集群。两种方案各有优劣：OSS 开箱即用、可靠性高，但需要付费；FastDFS 完全自主可控、零成本，但需要自己运维。对于学习阶段来说，两个都接触一遍，才能理解「自建 vs 云服务」的设计权衡。
:::

### 模块三：订单管理 — 状态机 + 事务控制

#### 业务场景

订单是外卖平台最核心的交易链路，涉及用户下单、支付、商家接单、骑手配送、用户收货等多个状态。每一个状态变更都需要严谨的数据一致性保障。

#### 订单状态流转

```
待支付 ──▶ 已支付 ──▶ 已接单 ──▶ 派送中 ──▶ 已完成
    │           │
    │           ├── 商家拒单 ──▶ 已取消
    │
    └── 超时未支付 ────────────▶ 已取消
    │
    └── 用户取消 ──────────────▶ 已取消
```

**简化理解**：想象一个包裹的旅程——用户下单（待支付）→ 支付成功（已支付）→ 商家打包好（已接单）→ 骑手取走（派送中）→ 用户签收（已完成）。如果中途出了岔子（超时/拒单/取消），包裹就退回去了（已取消）。

| 状态码 | 状态名称 | 触发条件 | 状态释义 |
|-------|---------|---------|---------|
| pending | 待支付 | 用户创建订单 | 「我选好了，等我付钱」 |
| paid | 已支付 | 微信支付成功 | 「钱已到账，商家请接单」 |
| accepted | 已接单 | 商家点击接单 | 「商家：收到，马上准备」 |
| delivered | 派送中 | 骑手取餐出发 | 「骑手：出发啦，预计 XX 分钟」 |
| completed | 已完成 | 用户确认收货 | 「用户：收到，好评！」 |
| cancelled | 已取消 | 超时/用户/商家取消 | 「订单终止，各回各家」 |

#### 核心代码

**创建订单（需要事务保证）**：

```java
@Override
@Transactional  // 开启事务：任一操作失败，全部回滚
public OrderSubmitVO submit(OrderSubmitDTO submitDTO) {
    // 1. 校验购物车是否为空
    List<ShoppingCart> carts = shoppingCartService.getCartByUserId(getCurrentUserId());
    if (carts == null || carts.isEmpty()) {
        throw new OrderBusinessException("购物车不能为空");
    }

    // 2. 校验地址是否有效
    AddressBook addressBook = addressBookMapper.getById(submitDTO.getAddressBookId());
    if (addressBook == null) {
        throw new OrderBusinessException("收货地址不存在");
    }

    // 3. 构造订单数据
    Order order = Order.builder()
            .userId(getCurrentUserId())
            .addressBookId(submitDTO.getAddressBookId())
            .orderTime(LocalDateTime.now())
            .status(1)  // 待支付
            .payMethod(1)  // 微信支付
            .totalAmount(orderAmount)
            .phone(addressBook.getPhone())
            .address(addressBook.getDetail())
            .consignee(addressBook.getConsignee())
            .build();

    // 4. 保存订单主表
    orderMapper.save(order);

    // 5. 保存订单明细（一个订单包含多个菜品）
    for (ShoppingCart cart : carts) {
        OrderDetail detail = OrderDetail.builder()
                .orderId(order.getId())
                .dishId(cart.getDishId())
                .setmealId(cart.getSetmealId())
                .name(cart.getName())
                .image(cart.getImage())
                .amount(cart.getAmount())
                .number(cart.getNumber())
                .build();
        orderDetailMapper.save(detail);
    }

    // 6. 清空购物车
    shoppingCartService.cleanCart(getCurrentUserId());

    // 7. 返回订单号
    return OrderSubmitVO.builder()
            .orderNumber(order.getNumber())
            .orderAmount(order.getTotalAmount())
            .build();
}
```

| 事务控制要点         | 说明                                                |
| -------------- | ------------------------------------------------- |
| @Transactional | 加在 Service 方法上，保证方法内所有数据库操作在同一个事务中                |
| rollbackFor    | 默认只对 RuntimeException 回滚，需指定 `Exception` 才能捕获所有异常 |
| isolation      | 订单创建用 `READ_COMMITTED`，避免脏读但允许不可重复读               |
| propagation    | 远程调用（微信支付）用 `REQUIRES_NEW`，独立开启事务                 |

:::tip
订单创建最怕「库存超卖」——多个用户同时下单，最后发现库存不够分。苍穹外卖用的是「乐观锁」思路：下单时检查库存是否足够，不够直接抛异常，而不是先扣库存再回滚。乐观锁适合「读多写少」的场景；如果并发量特别大，可能需要分布式锁（如 Redis SETNX）来做悲观锁。
:::

### 模块四：实时通信 — WebSocket 推送

#### 业务场景

外卖平台对实时性要求极高：新订单来了，商家需要**立刻**知道，不能刷新页面；骑手送餐途中，用户需要知道**餐到哪了**。传统方案是「轮询」（每隔几秒请求一次），但浪费资源且有延迟。WebSocket 实现了**服务器主动推送**，双向实时通信。

#### WebSocket 原理

**HTTP vs WebSocket**：

| 对比维度 | HTTP 轮询   | WebSocket  |
| ---- | --------- | ---------- |
| 连接方式 | 每次请求新建连接  | 建立一次，长连接保持 |
| 方向   | 只能是客户端发请求 | 双方都可以发消息   |
| 延迟   | 取决于轮询间隔   | 实时推送，无延迟   |
| 资源消耗 | 频繁建连，消耗大  | 一次建连，省资源   |

**苍穹外卖的 WebSocket 场景**：

| 场景     | 推送方向      | 内容               |
| ------ | --------- | ---------------- |
| 来单提醒   | 服务器 → 商家端 | 有新订单，需要接单        |
| 催单提醒   | 服务器 → 商家端 | 用户在催单，请尽快处理      |
| 订单状态变更 | 服务器 → 用户端 | 骑手已取餐，预计 20 分钟到达 |

#### 核心代码

**苍穹外卖 WebSocket 通信流程**：

**通俗理解**：把 WebSocket 想象成**对讲机**。商家和用户各拿一台对讲机，连接到服务器这个「中控台」：
- **建立连接**：商家打开对讲机，按下通话键（建立 WebSocket 连接），告诉中控台「我是商家，已上线」
- **服务器保存**：「中控台」在白板上记下「商家连接了，他的频道是 shop」
- **主动推送**：有新订单时，中控台直接对着商家的频道喊「有新单！」（服务器主动发消息）
- **响应处理**：商家对着用户频道喊「我接单了！」（商家端处理后推送给用户）

**为什么不用 HTTP 轮询？**

| 方案 | 原理 | 比喻 |
|-----|------|-----|
| HTTP 轮询 | 客户端每隔 3 秒问一次「有新单吗？」 | 送水师傅每隔 3 分钟敲门问「要不要水」 |
| WebSocket | 连接建立后，服务器有新单直接推过去 | 装了门铃，有人按你直接知道，不用一直去门口看 |

**WebSocket 服务端（核心逻辑）**：

```java
@Component
@Slf4j
// @ServerEndpoint("/ws/{userId}") 表示这个类是一个 WebSocket 服务端
// {userId} 是路径参数，类似 @PathVariable，允许动态的用户 ID
// 示例：ws://localhost:8080/ws/shop 或 ws://localhost:8080/ws/user_123
@ServerEndpoint("/ws/{userId}")
public class WebSocketServer {

    /**
     * sessionMap：存储所有 WebSocket 连接
     * key = userId（用户标识），value = Session（与该用户的连接）
     * 用 ConcurrentHashMap 保证多线程安全（多个请求同时建立/关闭连接）
     */
    private static Map<String, Session> sessionMap = new ConcurrentHashMap<>();

    /**
     * @OnOpen：客户端成功建立连接时触发
     * 就像「有人按了门铃」——连接成功建立
     *
     * @param session  当前用户的连接会话，包含发送消息的方法
     * @param userId   从 URL 路径提取的用户 ID（@PathParam）
     */
    @OnOpen
    public void onOpen(Session session, @PathParam("userId") String userId) {
        // 把新连接存入 Map：userId -> Session
        sessionMap.put(userId, session);
        log.info("WebSocket 连接建立: userId = {}, 当前在线数 = {}",
                userId, sessionMap.size());
    }

    /**
     * @OnMessage：收到客户端发来的消息时触发
     * 就像「对讲机里传来声音」——收到了对方的消息
     * 苍穹外卖里这个方法用得不多（主要是服务器推送），但必须有
     *
     * @param message  客户端发送的文本消息
     * @param userId   发送消息的用户 ID
     */
    @OnMessage
    public void onMessage(String message, @PathParam("userId") String userId) {
        log.info("收到消息: userId = {}, message = {}", userId, message);
        // 苍穹外卖里，客户端很少主动发消息，这里可以留空或做简单处理
    }

    /**
     * @OnClose：客户端断开连接时触发
     * 就像「对方挂断了对讲机」——连接关闭
     *
     * @param session  断开的那个连接
     * @param userId   断开的用户 ID
     */
    @OnClose
    public void onClose(Session session, @PathParam("userId") String userId) {
        // 从 Map 中移除断开的连接
        sessionMap.remove(userId);
        log.info("WebSocket 连接关闭: userId = {}, 当前在线数 = {}",
                userId, sessionMap.size());
    }

    // ──────────────────────────────────────────────────────────────
    // 以下是苍穹外卖实际用到的方法：向客户端推送消息
    // ──────────────────────────────────────────────────────────────

    /**
     * 群发消息：给所有在线的客户端发送消息
     * 场景：系统通知、广播消息等
     *
     * @param message 要发送的 JSON 文本消息
     */
    public static void sendToAll(String message) {
        // 遍历所有在线连接，逐一发送
        for (Map.Entry<String, Session> entry : sessionMap.entrySet()) {
            try {
                // getBasicRemote().sendText() 是同步发送，服务器要等客户端确认才返回
                // 适合消息量小的场景，苍穹外卖的订单通知就用这个
                entry.getValue().getBasicRemote().sendText(message);
            } catch (IOException e) {
                // 发送失败通常是连接已断开，记录日志即可
                log.error("WebSocket 群发失败: userId = {}, 错误 = {}",
                        entry.getKey(), e.getMessage());
            }
        }
    }

    /**
     * 单发消息：给指定用户发送消息
     * 场景：商家接单后通知对应用户、骑手状态变更通知等
     * 这是苍穹外卖最常用的推送方法
     *
     * @param userId   目标用户的 ID
     * @param message  要发送的 JSON 文本消息
     */
    public static void sendToUser(String userId, String message) {
        // 从 Map 中找到目标用户的 Session
        Session session = sessionMap.get(userId);
        if (session != null) {
            try {
                session.getBasicRemote().sendText(message);
                log.info("WebSocket 单发成功: userId = {}", userId);
            } catch (IOException e) {
                log.error("WebSocket 单发失败: userId = {}, 错误 = {}",
                        userId, e.getMessage());
            }
        } else {
            // 用户不在线？可能是连接已断开或 ID 错误
            log.warn("WebSocket 单发失败: userId = {} 不在线或未连接", userId);
        }
    }
}
```

**商家接单后推送**：

```java
/**
 * 商家接单接口
 * PUT /admin/order/accept/{orderId}
 *
 * 完整流程：
 * 1. 商家在后台看到新订单，点击「接单」按钮
 * 2. 前端调用此接口，orderId = 123
 * 3. 后端更新订单状态为「已接单」
 * 4. 后端通过 WebSocket 通知对应用户「商家已接单」
 */
@PutMapping("/accept/{orderId}")
public Result accept(@PathVariable Long orderId) {
    // Step 1: 更新订单状态（数据库操作）
    orderService.accept(orderId);

    // Step 2: 查出订单关联的用户 ID（谁下的单）
    Order order = orderService.getById(orderId);

    // Step 3: 构造推送消息（JSON 格式，便于前端解析）
    String message = JSON.toJSONString(Map.of(
            "type", "orderStatus",           // 消息类型：订单状态变更
            "orderId", orderId,             // 订单 ID
            "status", "accepted",            // 新状态：已接单
            "message", "商家已接单，正在准备中"  // 给用户的提示文字
    ));

    // Step 4: 推送消息给用户端
    // 注意：这里用用户的 ID（order.getUserId()）找到对应的 WebSocket 连接
    WebSocketServer.sendToUser(order.getUserId().toString(), message);

    return Result.success();
}
```

| WebSocket 方法 | 触发时机 | 苍穹外卖中的用途 |
|---------------|---------|---------------|
| `onOpen` | 客户端连接成功 | 记录日志，标识用户上线 |
| `onMessage` | 收到客户端消息 | 苍穹外卖中几乎不用（客户端不主动发） |
| `onClose` | 客户端断开连接 | 记录日志，移除连接 |
| `sendToUser` | 服务端主动推送 | **最常用**——来单提醒、接单通知、骑手状态 |
| `sendToAll` | 服务端广播 | 苍穹外卖中未使用（不适合外卖场景） |

:::tip
WebSocket 的核心是**保持连接、双向通信**。苍穹外卖用它做实时推送是恰到好处——新订单来单提醒、商家接单通知、骑手状态更新，这些都是「即时性」要求极高的业务。但要注意：WebSocket 不能穿越 Nginx，需要在 Nginx 配置 `proxy WebSocket` 转发，否则外网用户连不上。另外，苍穹外卖没有做消息持久化——假设内网环境稳定，如果消息丢了，用户只能刷新页面重新获取订单状态。
:::

### 模块五：定时任务 — 超时订单自动取消

#### 业务场景

用户下单后如果没有在规定时间（如 15 分钟）内支付，订单应该自动取消，释放锁定的库存。Spring Task 提供了轻量级的定时任务能力，无需额外依赖。

#### 定时任务实现

```java
@Component
@Slf4j
public class OrderTask {

    @Autowired
    private OrderMapper orderMapper;

    @Autowired
    private OrderService orderService;

    /**
     * 处理超时未支付订单
     * 每天凌晨 1 点执行一次
     */
    @Scheduled(cron = "0 0 1 * * ?")
    public void processTimeoutOrder() {
        log.info("开始处理超时未支付订单...");

        // 1. 查询 15 分钟前创建但仍未支付的订单
        LocalDateTime timeoutTime = LocalDateTime.now().minusMinutes(15);
        List<Order> timeoutOrders = orderMapper.findTimeoutOrders(timeoutTime);

        // 2. 逐个取消订单
        for (Order order : timeoutOrders) {
            try {
                orderService.cancel(order.getId(), "超时未支付，自动取消");
                log.info("订单 {} 已自动取消", order.getNumber());
            } catch (Exception e) {
                log.error("订单 {} 取消失败: {}", order.getNumber(), e.getMessage());
            }
        }

        log.info("超时订单处理完成，共处理 {} 个", timeoutOrders.size());
    }

    /**
     * 每天凌晨 2 点执行一次
     * 将 7 天前的订单归档，减少主表数据量
     */
    @Scheduled(cron = "0 0 2 * * ?")
    public void processOrderArchive() {
        log.info("开始归档历史订单...");

        LocalDateTime archiveTime = LocalDateTime.now().minusDays(7);
        int count = orderMapper.archiveOrders(archiveTime);

        log.info("订单归档完成，共归档 {} 条记录", count);
    }
}
```

| 定时任务场景 | 苍穹外卖的实现                 |
| ------ | ----------------------- |
| 超时订单取消 | 每小时执行一次，检查 15 分钟前未支付的订单 |
| 订单归档   | 每天凌晨 2 点，将 7 天前的订单移入归档表 |
| 营业数据统计 | 每天凌晨 0 点，统计昨日营业额、订单量    |

:::note
定时任务的坑主要在「重复执行」和「执行时间过长」。苍穹外卖用的是 Spring Task + @Scheduled 注解，简单轻量。但如果是分布式部署（多台服务器），定时任务会在每台机器都执行一遍，需要通过分布式锁（如 Redis SETNX）来保证只有一台机器执行。传智健康的 Quartz 天然支持集群模式，Spring Task 更轻量但没有集群支持，需要自己加锁。
:::

### 模块六：数据统计 — Apache POI 报表导出

#### 业务场景

商家需要导出运营数据报表（如 Excel 格式的营业额统计），用于财务对账或经营分析。Apache POI 是 Java 操作 Office 文档的标准库，支持读写 Excel 文件。

#### 核心代码

**导出运营数据报表**：

```java
@GetMapping("/export")
public void exportReport(HttpServletResponse response) throws Exception {
    // 1. 查询近 30 天运营数据
    LocalDateTime startDate = LocalDateTime.now().minusDays(30);
    TurnoverReportVO report = reportService.getTurnoverReport(startDate);

    // 2. 生成 Excel
    HSSFWorkbook workbook = new HSSFWorkbook();

    row1.createCell(0).setCellValue("营业额（元）");
    row1.createCell(1).setCellValue(report.getTotalTurnover());

    row1.createCell(0).setCellValue("有效订单数");

    row1 = sheet1.createRow(3);

    HSSFSheet sheet2 = workbook.createSheet("每日明细");

    HSSFRow headerRow = sheet2.createRow(0);
    for (int i = 0; i < headers.length; i++) {
        headerRow.createCell(i).setCellValue(headers[i]);
    }

    List<DailyReport> dailyReports = report.getDailyReports();
    for (int i = 0; i < dailyReports.size(); i++) {
        DailyReport daily = dailyReports.get(i);
        HSSFRow row = sheet2.createRow(i + 1);
        row.createCell(0).setCellValue(daily.getDate());
        row.createCell(1).setCellValue(daily.getTurnover());
        row.createCell(2).setCellValue(daily.getOrderCount());
        row.createCell(3).setCellValue(daily.getUnitPrice());
    }

    // 3. 输出到浏览器
    String filename = "苍穹外卖_运营报表_" + LocalDate.now() + ".xls";
    response.setContentType("application/vnd.ms-excel");
    response.setHeader("Content-Disposition", "attachment;filename=" + filename);
    workbook.write(response.getOutputStream());
    workbook.close();
}
```

| POI 核心类      | 作用                        |
| ------------ | ------------------------- |
| HSSFWorkbook | 操作 Excel 97-2003 格式（.xls） |
| XSSFWorkbook | 操作 Excel 2007+ 格式（.xlsx）  |
| HSSFSheet    | 代表一个工作表                   |
| HSSFRow      | 代表一行                      |
| HSSFCell     | 代表一个单元格                   |

:::tip
POI 导出 Excel 的核心流程：创建 Workbook → 创建 Sheet → 创建 Row → 创建 Cell → 填充数据 → 输出流。整个过程就是「填格子」——先画好表格结构，再往每个格子里填内容。苍穹外卖的报表导出功能，让商家能离线分析经营数据，不用一直盯着后台看。

## 项目亮点与踩坑实录

### 亮点一：AOP 公共字段自动填充

**问题痛点**：订单表、菜品表、员工表都有 `createTime`、`updateTime`、`createUser`、`updateUser` 字段，每次增删改都要手动 set，代码冗余且容易遗漏。

**解决方案**：自定义注解 + AOP 切面，拦截 Mapper 层方法，自动填充这些字段。

**通俗理解**：想象你是医院的**挂号员**——每个病人（数据）进科室前，都要填一张表。如果让每个病人自己填，经常漏填、填错。但如果你在挂号台（切面）统一帮他们填好（自动填充），既规范又不费事。

**苍穹外卖 AOP 自动填充流程**：

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│                              AOP 自动填充工作流程                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘

  Step 1: 定义注解
  ┌─────────────────────────────────────┐
  │  @AutoFill(OperationType.INSERT)    │  ← 注解打在 Mapper 方法上
  │  void insert(Order order);         │    告诉切面：「这个方法需要填充字段」
  └─────────────────────────────────────┘
                │
                ▼
  Step 2: 切面拦截（@Before）
  ┌─────────────────────────────────────────────────────────────────┐
  │  @Before("autoFillPointCut()")     │  ← 切入点匹配所有带 @AutoFill 的方法
  │  public void autoFill(JoinPoint) {  │    在方法执行「之前」自动执行
  │       // ...                        │
  │  }                                  │
  └─────────────────────────────────────────────────────────────────┘
                │
                ▼
  Step 3: 获取注解信息
  ┌─────────────────────────────────────────────────────────────────┐
  │  1. 取出方法上的 @AutoFill 注解                                  │

                │
                ▼
  Step 4: 获取当前用户 ID
  ┌─────────────────────────────────────────────────────────────────┐
  │  Long currentId = BaseContext.getCurrentId();                   │
  │                                                                  │
  │  BaseContext 是什么？                                            │
  │  它是一个 ThreadLocal 工具类，存储当前请求的用户 ID                 │
  │  为什么不用参数传递？因为注解在 Mapper 层，参数已经很长了           │

  Step 5: 填充字段（反射调用 setter）

  │                                                                  │
  │  反射机制：通过「方法名」找到实体类的 setter 方法并调用              │

  ┌─────────────────────────────────────────────────────────────────┐
```

**什么场景需要自动填充？**

| 场景 | INSERT | UPDATE |
|-----|--------|--------|
| 新增员工 | 需要填充 createTime、createUser | — |
| 修改员工信息 | — | 需要填充 updateTime、updateUser |
| 新增菜品 | 需要填充 createTime、createUser | — |
| 修改菜品价格 | — | 需要填充 updateTime、updateUser |
| 创建订单 | 需要填充 createTime、createUser | — |
| 订单支付 | — | 需要填充 updateTime、updateUser |

**Step 1：定义注解**：

```java
/**
 * 自定义注解：标记需要自动填充公共字段的方法
 *
 * @Target(ElementType.METHOD) 表示这个注解只能用在方法上（不能用在类或字段上）
 * @Retention(RetentionPolicy.RUNTIME) 表示这个注解在运行时仍然存在（反射能读取到）
 *
 * 注解就像「贴纸」——贴在方法上，告诉 AOP 切面：「这个方法执行前来找我」
 */
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AutoFill {
    /**
     * 指定操作类型：INSERT（新增）还是 UPDATE（修改）
     * 不同类型填充的字段不同
     */
    OperationType value();
}
```

```java
/**
 * 枚举类：定义操作类型
 * 为什么用枚举而不是 String？因为枚举更规范，IDE 有提示，不容易拼错
 */
public enum OperationType {
    /**
     * INSERT：新增操作，需要填充创建时间、创建人、更新时间、更新人
     */
    INSERT,

    /**
     * UPDATE：修改操作，只需要填充更新时间、更新人
     */
    UPDATE
}
```

**Step 2：Mapper 接口使用注解**：

```java
/**
 * 订单 Mapper 接口
 * 关键：每个方法上都加了 @AutoFill 注解
 */
@Mapper
public interface OrderMapper {

    /**
     * 新增订单 → 填充 createTime、createUser、updateTime、updateUser
     */
    @AutoFill(OperationType.INSERT)
    void insert(Order order);

    /**
     * 更新订单状态 → 只填充 updateTime、updateUser
     */
    @AutoFill(OperationType.UPDATE)
    void update(Order order);

    /**
     * 根据 ID 查询 → 不需要填充（只读操作）
     */
    Order findById(Long id);
}
```

**Step 3：切面类实现**：

```java
/**
 * AOP 切面类：自动填充公共字段
 *
 * @Aspect 标识这是一个切面类（AOP 的核心概念）
 * @Component 把这个类交给 Spring 管理
 */
@Aspect
@Component
@Slf4j
public class AutoFillAspect {

    /**
     * 切入点表达式解析：
     * execution(* com.sky.mapper.*.*(..))
     *   ↑ 返回值任意   ↑ mapper 包下  ↑ 任意类  ↑ 任意方法  ↑ 任意参数
     *
     * && @annotation(com.sky.annotation.AutoFill)
     *   ↑ 并且 ↑ 方法上有 @AutoFill 注解
     *
     * 整体意思：com.sky.mapper 包下所有带 @AutoFill 注解的方法
     */
    @Pointcut("execution(* com.sky.mapper.*.*(..)) && @annotation(com.sky.annotation.AutoFill)")
    public void autoFillPointCut() {
        // 这是一个空方法，仅用于承载切入点表达式
        // 方法名就是切入点名称，供 @Before 等引用
    }

    /**
     * 前置通知：在目标方法执行之前执行
     *
     * @param joinPoint 连接点——被拦截的方法
     *                  可以理解为「当前执行的这个方法」的一个引用
     *                  通过它可以获取：方法签名、方法参数、目标对象等
     */
    @Before("autoFillPointCut()")
    public void autoFill(JoinPoint joinPoint) {
        // ─────────────────────────────────────────────────────────
        // Step 1: 获取方法上的注解类型（INSERT 还是 UPDATE）
        // ─────────────────────────────────────────────────────────
        // joinPoint.getSignature() 获取方法签名（方法名、返回类型、参数类型）
        // 强转为 MethodSignature（这是 Spring 提供的，专门用于获取方法上的注解）
        MethodSignature signature = (MethodSignature) joinPoint.getSignature();

        // 获取方法上的 @AutoFill 注解
        AutoFill autoFillAnnotation = signature.getMethod().getAnnotation(AutoFill.class);

        // 取出注解的值：INSERT 还是 UPDATE
        OperationType operationType = autoFillAnnotation.value();
        log.info("AOP 自动填充开始: 方法={}, 操作类型={}",
                signature.getName(), operationType);

        // ─────────────────────────────────────────────────────────
        // Step 2: 获取方法参数（实体对象）
        // ─────────────────────────────────────────────────────────
        // joinPoint.getArgs() 获取方法参数数组
        // 通常第一个参数就是实体对象（如 Order、Dish、Employee）
        Object[] args = joinPoint.getArgs();
        if (args == null || args.length == 0) {
            // 没有参数，直接返回（不应该发生）
            log.warn("AOP 自动填充失败: 方法没有参数");
            return;
        }

        // 取出第一个参数，假设它是实体对象
        // 注意：苍穹外卖约定实体对象作为第一个参数
        Object entity = args[0];
        log.debug("AOP 自动填充: 实体类型={}", entity.getClass().getName());

        // ─────────────────────────────────────────────────────────
        // Step 3: 准备填充数据
        // ─────────────────────────────────────────────────────────
        // 获取当前时间（所有记录都用这个时间）
        LocalDateTime now = LocalDateTime.now();

        // 获取当前登录用户的 ID
        // BaseContext 是基于 ThreadLocal 的上下文工具类
        // 存储了当前请求的用户 ID（在 JWT 拦截器中设置的）
        Long currentId = BaseContext.getCurrentId();

        // ─────────────────────────────────────────────────────────
        // Step 4: 通过反射调用 setter 方法填充字段
        // ─────────────────────────────────────────────────────────
        // 反射机制：通过方法名找到对应的 setter 并调用
        // 例如：setCreateTime(LocalDateTime.class) → order.setCreateTime(now)
        try {
            // ── INSERT 操作：填充 4 个字段 ──
            if (operationType == OperationType.INSERT) {
                // 反射获取 setCreateTime(LocalDateTime) 方法
                Method setCreateTime = entity.getClass().getMethod(
                        "setCreateTime", LocalDateTime.class);
                // 反射调用：order.setCreateTime(now)
                setCreateTime.invoke(entity, now);
                log.debug("填充 createTime: {}", now);

                // 反射获取 setCreateUser(Long) 方法
                Method setCreateUser = entity.getClass().getMethod(
                        "setCreateUser", Long.class);
                // 反射调用：order.setCreateUser(currentId)
                setCreateUser.invoke(entity, currentId);
                log.debug("填充 createUser: {}", currentId);
            }

            // ── UPDATE 操作：填充 2 个字段 ──
            // 注意：INSERT 也要填充更新时间，所以放外面
            {
                // 反射获取 setUpdateTime(LocalDateTime) 方法
                Method setUpdateTime = entity.getClass().getMethod(
                        "setUpdateTime", LocalDateTime.class);
                setUpdateTime.invoke(entity, now);
                log.debug("填充 updateTime: {}", now);

                // 反射获取 setUpdateUser(Long) 方法
                Method setUpdateUser = entity.getClass().getMethod(
                        "setUpdateUser", Long.class);
                setUpdateUser.invoke(entity, currentId);
                log.debug("填充 updateUser: {}", currentId);
            }

            log.info("AOP 自动填充完成: {}", entity);

        } catch (NoSuchMethodException e) {
            // 如果实体类没有对应的 setter 方法（比如某些 VO 类），记录警告
            log.error("AOP 自动填充失败: 实体类 {} 缺少必要的 setter 方法",
                    entity.getClass().getName());
        } catch (IllegalAccessException | InvocationTargetException e) {
            // 反射调用失败（通常是权限问题）
            log.error("AOP 自动填充失败: 反射调用异常 - {}", e.getMessage());
        }
    }
}
```

| 对比 | 手动填充 | AOP 自动填充 |
|-----|---------|-------------|
| 代码量 | 每个 Mapper 方法都要写 4 行 set | 只需加一个注解 `@AutoFill(INSERT/UPDATE)` |
| 遗漏风险 | 高，容易忘写某个字段 | 低，拦截器统一处理，不会漏 |
| 可维护性 | 差，改时间格式要改多处 | 好，改动只在切面类一处 |
| 性能影响 | 无 | 极小（只是多了几步反射调用） |

**为什么用反射而不是直接调用？**

| 方式 | 示例 | 优缺点 |
|-----|------|--------|
| 直接调用 | `order.setCreateTime(now)` | 简单，但每个类都要写，不灵活 |
| 反射调用 | `method.invoke(entity, now)` | 复杂，但一套代码适配所有实体类 |

苍穹外卖有 `Order`、`Dish`、`Employee`、`Setmeal` 等多个实体类，如果每个类都单独写 set，代码会非常冗余。反射的好处是：**写一套代码，适配所有类**。切面类只关心「有没有 setCreateTime 方法」，不关心是哪个类。

**经验总结**：AOP 的核心价值是「把重复的代码抽离出来，统一处理」。苍穹外卖的这个设计很巧妙——注解打在 Mapper 接口上，切面拦截自动填充，全程对业务代码零侵入。开发新功能时，只需要：

1. 在 Mapper 方法上加 `@AutoFill(INSERT)` 或 `@AutoFill(UPDATE)`
2. 确保实体类有对应的 setter 方法

就这么简单，公共字段自动填充这件事就搞定了。

### 亮点二：统一异常处理

**问题痛点**：如果每个 Controller 都 try-catch，代码会非常臃肿。而且返回给前端的错误格式不统一，前端不好处理。

**解决方案**：全局异常处理器，统一捕获所有异常，返回标准化的错误响应。

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    // 处理业务异常
    @ExceptionHandler(OrderBusinessException.class)
    public Result handleOrderBusinessException(OrderBusinessException e) {
        log.error("订单业务异常: {}", e.getMessage());
        return Result.error(e.getMessage());
    }

    // 处理自定义业务异常
    @ExceptionHandler(BusinessException.class)
    public Result handleBusinessException(BusinessException e) {
        log.error("业务异常: {}", e.getMessage());
        return Result.error(e.getMessage());
    }

    // 处理所有未知异常
    @ExceptionHandler(Exception.class)
    public Result handleException(Exception e) {
        log.error("系统异常: {}", e.getMessage(), e);
        return Result.error("系统繁忙，请稍后再试");
    }
}
```

**前端收到的响应格式**：

```json
// 成功响应
{
    "code": 1,
    "msg": "操作成功",
    "data": { ... }
}

// 失败响应
{
    "code": 0,
    "msg": "订单不存在",
    "data": null
}
```

| 异常处理策略                  | 说明                       |
| ----------------------- | ------------------------ |
| 业务异常（BusinessException） | 业务逻辑校验失败，如「余额不足」「库存不足」   |
| 系统异常（Exception）         | 代码 bug、网络问题等，捕获后返回「系统繁忙」 |
| 日志记录                    | 所有异常都要打日志，方便排查问题         |

**经验总结**：全局异常处理器是 Spring Boot 的标配设计。一个 `@RestControllerAdvice` 类搞定所有 Controller 的异常处理，代码清爽，响应统一。前端只需要判断 `code` 是 1 还是 0，就知道请求是否成功。

### 踩坑实录：微信支付回调验签失败

**问题现象**：用户支付成功后，微信服务器的回调通知始终验签失败，订单状态无法更新。

**根因分析**：

- 微信支付回调验签需要使用**微信支付密钥**（不是 AppSecret）
- 回调 URL 必须是 HTTPS，且不能带端口（微信支付要求）
- 验签时要把 `request body` 里的 XML 数据全部参与签名

**解决方案**：

```java
@PostMapping("/pay-notify")
public void payNotify(HttpServletRequest request) throws Exception {
    // 1. 读取回调 XML 数据（必须用 InputStream，不能先 getParameter）
    String body = IOUtils.toString(request.getInputStream(), "UTF-8");

    // 2. 解析 XML
    Map<String, String> params = new HashMap<>();
    Document document = DocumentHelper.parseText(body);
    Element root = document.getRootElement();
    for (Element element : root.elements()) {
        params.put(element.getName(), element.getTextTrim());
    }

    // 3. 验签
    String signFromWx = params.get("sign");
    params.remove("sign");  // 参与签名要去掉 sign 字段

    // 按字典序拼接参数
    String signStr = params.entrySet().stream()
            .sorted(Map.Entry.comparingByKey())
            .map(e -> e.getKey() + "=" + e.getValue())
            .collect(Collectors.joining("&");

    // 加上商户密钥
    String signCheck = "MD5 encode Utf-8 of everything&key=" + wxPayKey + " and UTF-8 keystripped";
    String mySign = DigestUtils.md5Hex(signCheck).toUpperCase();

    if (!mySign.equals(signFromWx)) {
        log.error("验签失败");
        return;
    }

    // 4. 验签成功，处理订单
    String orderNumber = params.get("out_trade_no");
    orderService.updatePaidStatus(orderNumber);

    log.info("订单 {} 支付成功", orderNumber);
}
```

| 微信支付坑点   | 解决方法                                                   |
| -------- | ------------------------------------------------------ |
| 密钥用错     | 用微信支付密钥（APIv3 密钥），不是 AppSecret                         |
| 回调数据获取   | 用 `request.getInputStream()` 读取原始 XML，不能先 getParameter |
| HTTPS 要求 | 回调 URL 必须支持 HTTPS，不带端口                                 |
| 签名算法     | 微信支付 v2 用 MD5，v3 用 AES                                 |

**经验总结**：第三方支付接口的坑，一半在签名验签，一半在幂等处理。回调可能因为网络问题重复推送，所以「先查订单状态，已支付就直接返回成功」，是防止重复更新的标准做法。

### 设计思想收获

1. **分层架构的价值**：Controller → Service → Mapper 的分层，让每个层各司其职。Controller 处理请求/响应，Service 处理业务逻辑，Mapper 处理数据库访问。分层的目的是「单一职责」——每个类只做一件事。
2. **缓存是银弹，但不是万能**：Redis 缓存能极大提升读性能，但「缓存一致性」是另一个问题。苍穹外卖选择了「写操作时删除缓存」而不是「更新缓存」，简化了一致性处理。
3. **实时性 vs 一致性**：WebSocket 能做到实时推送，但消息可能丢失（网络问题）。苍穹外卖没有做消息持久化，假设内网环境足够稳定。这是「合适场景用合适方案」的体现。
4. **异常处理要统一**：全局异常处理器 + 统一返回格式，让前后端交互变得简单。这种「约定大于配置」的思想，是工程化的基础。

### 与传智健康的技术演进

| 对比维度  | 传智健康（2024.04）    | 苍穹外卖（2024.05）        |
| ----- | ---------------- | -------------------- |
| 架构    | SOA + Dubbo（分布式） | 单体架构                 |
| 持久层   | 原生 MyBatis       | MyBatis-Plus         |
| 缓存    | redis            | Redis + Spring Cache |
| 实时通信  | 无                | WebSocket            |
| 文件存储  | FastDFS（自建）      | 阿里云 OSS（云服务）         |
| 第三方集成 | 微信开发             | 微信登录 + 微信支付          |
| 定时任务  | Quartz           | Spring Task          |
| 报表    | ECharts 可视化      | Apache POI 导出        |

**从传智健康到苍穹外卖**：技术栈从分布式回退到单体，但业务复杂度升级了。苍穹外卖的外卖交易链路（下单→支付→接单→配送→完成）比传智健康的体检预约链路（预约→到店→检查→报告）更复杂，对实时性要求也更高。两个项目对比着看，能更深刻理解「架构要匹配业务场景」。

:::note
苍穹外卖的学习，让我真正理解了「带着业务思维写代码」是什么意思。之前写 CRUD，就是「把数据存进去、取出来」；现在要考虑：用户下单后库存怎么扣？支付失败订单怎么处理？商家拒单后怎么通知用户？这些业务逻辑，才是一个工程师的核心价值。
:::

*苍穹外卖项目完成于 2024 年 5 月，是第一次完整接触外卖平台的交易链路。从传智健康的健康管理，到苍穹外卖的外卖平台，两条完全不同的业务线，却都用到了 Spring Boot + MyBatis-Plus + Redis 的技术组合。技术是相通的，业务是多样的——这是苍穹外卖教会我的最重要一课。*
