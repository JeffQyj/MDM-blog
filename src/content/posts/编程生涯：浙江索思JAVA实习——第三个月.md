## 引言：第三个月，我开始「贡献」了

如果说第一个月的关键词是「适应」，第二个月的关键词是「参与」，那么第三个月的关键词应该是「贡献」。

前两个月，我更多的是在学习和配合——学习企业流程，配合团队工作。但到了第三个月，我开始真正为企业项目贡献自己的力量：修复BUG、保障环境、完善文档、开发模块。

这种感觉，就像是终于从「跟着打下手」变成了「能够独立完成任务」。

这篇文章，我就来聊聊第三个月到底经历了什么。

## 第九周：BUG修复的规范化

### 从「协助」到「独立」

还记得第二个月第一次协助定位BUG时，我还有些手忙脚乱——不知道该从哪里入手，不知道该记录什么信息，一切都要在 mentor 的指导下完成。

第三个月一开始，mentor 给了我一个任务：独立修复一个测试反馈的简单BUG。

这次我需要独立完成整个流程：

### BUG修复记录

**问题编号**：BUG-20250415-001

**问题描述**：用户管理接口在批量查询时，偶发性返回空数据

**复现步骤**：

1. 进入用户管理页面
2. 选择「批量查询」模式
3. 输入多个用户ID（建议3-5个）
4. 点击查询，约第2-3次出现

**排查过程**：

1. 检查接口返回数据 → 数据正常返回
2. 检查前端请求逻辑 → 发现请求参数拼接有问题
3. 定位根因 → 批量ID用逗号分隔时，末尾可能有空格

**解决方案**：

```java
// 修复前
String userIds = String.join(",", userIdList);

// 修复后（trim处理）
String userIds = userIdList.stream()
    .map(String::trim)
    .collect(Collectors.joining(","));
```

**回归测试**：通过 ✓

### 企业BUG修复流程的精髓

这次独立修复让我深刻理解了企业BUG修复的标准化流程：

| 阶段 | 核心动作   | 关键点               |
| -- | ------ | ----------------- |
| 上报 | 记录问题现象 | 必填：问题描述、复现步骤、影响范围 |
| 定位 | 排查问题根因 | 从表象出发，逐步追溯，勿盲目猜测  |
| 修复 | 编写修复代码 | 遵循代码规范，必要时补充注释    |
| 回归 | 验证修复效果 | 确认问题已解决，原有功能不受影响  |

mentor 看过我的修复方案后说：「流程是对的，但下次注意——修复完成后先在本地跑通，再提交代码。」

这句话让我意识到，企业开发中「验证」是修复的最后一道防线，而不是可选项。

:::note
第九周的核心感受：BUG修复不是「改代码就行」，而是「发现→定位→修复→验证」的完整闭环。
:::

## 第十周：测试环境的「幕后」

### 第一次协助运维工作

第十周，我第一次协助运维人员保障测试环境的稳定运行。

之前对「运维」的理解比较模糊，觉得就是「保证服务器不宕机」。真正接触后才发现，运维的核心是**保障服务可用性、快速定位问题、预防潜在风险**。

这次任务我协助做了三件事：

**1. 监控环境运行状态**

不只是 `docker ps` 看看容器在不在运行，更重要的是检查资源使用是否正常：

```bash
# 查看容器资源使用情况
docker stats

# 检查容器健康状态
docker inspect --format='{{.State.Health}}' [container_id]
```

**2. 分析日志定位异常**

当测试人员反馈某个接口偶发性超时，我协助排查。关键是从日志中找到规律：

```bash
# 实时查看容器日志
docker logs -f [container_id] --since "10m"

# 过滤错误信息
docker logs [container_id] | grep ERROR

# 查看日志时间戳，对比问题发生时间点
docker logs [container_id] --timestamps | grep "2025-04-28 14:"
```

通过日志分析，我发现是某个定时任务在整点运行时占用了过多连接资源，导致接口响应变慢。这个发现让正式运维人员能够快速定位根因。

**3. 反馈异常情况**

遇到不熟悉的错误时，我的任务是把现象描述清楚，而不是自己硬扛。mentor 说过：「实习生汇报问题，最重要的是说清楚『现象+时间点+影响范围』，让运维人员能快速复现和判断。」

:::note
第十周的核心感受：运维不是「盯着屏幕看服务器」，而是通过监控、日志、分析这套组合拳，主动发现和预防问题。
:::

### 测试环境与开发环境的区别

这周我还第一次深入理解了「测试环境」和「开发环境」的区别：

| 维度  | 开发环境   | 测试环境   |
| --- | ------ | ------ |
| 使用者 | 开发人员   | 测试人员   |
| 稳定性 | 频繁变更   | 相对稳定   |
| 数据  | 模拟数据为主 | 接近真实数据 |
| 目的  | 快速迭代   | 全面验证   |

之前我总觉得「环境」只是个跑代码的地方，没什么好关注的。但现在我理解了，环境是「让正确的人、在正确的地方、验证正确的功能」的基础保障。

:::note
第十周的核心感受：测试环境是企业质量保障的重要一环，理解环境管理的思路，比会敲几条命令更重要。
:::

## 第十一周：文档的「最后一公里」

### 接口文档的「回头看」

第十一周，我接到任务：完善之前编写的接口文档，补充异常处理说明和使用注意事项。

一开始我觉得这工作挺简单的——之前的文档框架已经有了，补充内容而已。但真正做起来才发现，文档的「最后一公里」往往是最难的。

举个例子，之前我写的接口文档是这样的：

```yaml
# 修复前的文档
POST /api/user/batch-query
说明：批量查询用户信息
参数：userIds（用户ID列表，逗号分隔）
返回：用户信息列表
```

修复后变成了这样：

```yaml
# 修复后的文档
POST /api/user/batch-query
说明：批量查询用户信息
参数：
  - userIds：用户ID列表，逗号分隔，支持1-100个ID
  - 注意：单个ID长度不超过32位，超出会返回参数错误
异常：
  - 400：参数格式错误（如ID包含非法字符）
  - 404：用户不存在（返回空列表，不报错）
  - 500：服务器内部错误
注意事项：
  - 批量查询有频率限制，每分钟最多调用10次
  - 建议使用 POST 而非 GET，避免参数长度限制
```

mentor 看过后说：「文档不是给自己看的，是给调用方看的。要假设调用方完全不了解你的系统，站在他们的角度思考『会有什么疑问』。」

### 测试用例的补充

这周我还协助测试人员补充了简单的测试用例。

还记得第二个月第一次接触测试时，我觉得测试只是「点点按钮」。但现在我明白了——测试用例的编写，本质上是对业务场景的「穷举式覆盖」：

```markdown
## 接口测试用例补充

### 接口：POST /api/user/batch-query

**用例5**：频率限制测试
- 前置：已调用10次接口
- 输入：第11次请求
- 预期：返回频率限制提示，状态码429

**用例6**：空列表测试
- 输入：userIds为空
- 预期：返回空列表，状态码200

**用例7**：超长ID列表测试
- 输入：101个用户ID
- 预期：返回参数错误，状态码400
```

这些边界情况的考虑，让我意识到测试人员的工作远比我想象的复杂和专业。

:::note
第十一周的核心感受：文档和测试，都是在「降低理解成本」。好的文档让调用方少走弯路，好的测试让发布前发现问题。
:::

## 第十二周：模块开发的初体验

### 第一个「真正」的开发任务

第十二周，mentor 给了我一个任务：协助开发一个简单功能模块的辅助工作。

这次不是「整理文档」，不是「协助排查」，而是真正的「编写代码」。

任务是为用户管理模块增加一个「批量导出」功能的辅助开发。说「辅助」，是因为核心逻辑由正式开发人员完成，但我需要负责参数校验、异常处理、导出格式适配等部分。

```java
/**
 * 批量导出用户信息
 * @param request 导出请求（包含用户ID列表、导出格式等）
 * @return 导出结果
 */
@PostMapping("/export")
public Result<ExportVO> batchExport(BatchExportRequest request) {
    // 1. 参数校验
    if (CollectionUtils.isEmpty(request.getUserIds())) {
        return Result.error("用户ID列表不能为空");
    }
    if (request.getUserIds().size() > MAX_EXPORT_COUNT) {
        return Result.error("单次导出不超过" + MAX_EXPORT_COUNT + "条");
    }

    // 2. 查询数据
    List<User> users = userService.queryByIds(request.getUserIds());

    // 3. 格式适配
    ExportData data = convertToExportData(users, request.getFormat());

    // 4. 返回结果
    return Result.success(new ExportVO(data.getFileId(), data.getFileName()));
}
```

当我第一次看到自己写的代码被合并到开发分支，并且最终会在测试环境运行时，心情很复杂——有兴奋，有紧张，也有一种「终于做了一件『真正的开发工作』」的成就感。

### 每日站会的变化

这周的每日站会，我第一次能够清晰地汇报自己的工作进度：

> 「昨天完成了批量导出接口的参数校验和格式适配，今天计划完成异常处理部分的开发，目前没有阻碍。」

mentor 在会后跟我说：「你现在汇报越来越清晰了，不仅说进度，还说问题。这就是进步。」

:::note
第十二周的核心感受：真正的开发工作，需要对『做什么、怎么做、会有什么坑』都有清晰的认知。
:::

## 第三个月总结

### 技术能力提升

| 能力项   | 第二月末   | 第三月末          |
| ----- | ------ | ------------- |
| BUG处理 | 只能协助定位 | 能够独立修复并验证     |
| 环境部署  | 未接触    | 掌握Docker基础操作  |
| 接口文档  | 掌握基础编写 | 能够补充完整异常和注意事项 |
| 测试配合  | 了解测试流程 | 能够补充边界测试用例    |
| 模块开发  | 未接触    | 能够完成辅助模块开发    |

### 心态转变

第三个月最大的转变，是从「完成任务」到「思考质量」。

前两个月，mentor 给我布置任务，我想着的是「怎么完成」。但第三个月，我开始想「怎么做好」——不只是交差了事，而是考虑这段代码后续会不会有人维护、这个接口调用方会不会有疑问、这个场景会不会有遗漏。

mentor 说过一句话让我印象很深：「在企业里，代码写出来是要给别人看的，也是给自己留的。今天的『差不多』，可能就是明天的『大坑』。」

### 不足与改进

当然，第三个月也暴露了一些问题：

- **技术深度仍然有限**：能够完成辅助开发，但核心逻辑还是依赖前辈
- **系统思维有待加强**：有时候只盯着自己负责的部分，缺乏全局视角
- **表达可以更精准**：汇报时偶尔还是会漏掉关键信息

***

*第三个月结束了，开始真正「贡献」了。旅程还在继续。*
