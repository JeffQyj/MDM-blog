## 站在终点的回望

2025年12月31日，又一年走到尽头。

如果要用一个词形容我的2025年，大概是「沉淀」。

2022年，我学会了写代码；2023年，我理解了工程思维；2024年，我完成了从「学习者」到「开发者」的蜕变；而2025年，我开始真正理解什么叫「知其然，更要知其所以然」。

这一年，不再是盲目追逐新技术的喧嚣，而是静下心来，把走过的路重新走一遍，把学过的知识重新串起来。

:::note
这篇文章不求面面俱到，只想记录那些真正见证了我成长的瞬间。
:::

## 技术之路：从知识到能力的沉淀

如果说2024年是「开发者元年」，那2025年就是「能力沉淀年」。

### 职场初体验：18周的蜕变

**浙江索思JAVA实习**（2月-6月），是我2025年最重要的一段经历。

18周，从正月十六到六月中旬，我以实习生的身份走进了一家真正的科技公司。没有课程设计的模拟，没有练手项目的演练，只有真实的企业、真实的项目、真实的交付压力。

**第一个月的关键词是「适应」**。

入职第一天，mentor没给我安排具体任务，只是说了一句：「明天九点准时到就行。」当时我还以为这是客套话，后来才明白——**准时是企业最基本的信用**。

第一周的主要任务是熟悉开发环境：IDEA、MySQL、Git、Jira。在学校只学过`git add`、`git commit`、`git push`三个命令，而公司里使用的是完整的Git Flow分支策略。一开始看到那么多流程是懵的，但实际操作几次后就慢慢理解了——分支策略的目的是保证代码质量和可追溯性。

**第二个月的关键词是「参与」**。

我开始真正参与到开发流程中：参与代码检视、协助接口联调、配合BUG定位。第一次参与code review时，mentor鼓励我说：「code review是很好的学习机会，你看别人的代码，能学到好的写法，也能看到一些容易踩的坑。」

我仔细阅读了前辈们提交的代码，发现了几个让我印象深刻的点：

- 核心业务逻辑都有注释说明意图，而不只是描述「做什么」
- 很多地方用了自定义异常类，而不是直接throw Exception
- 变量命名要顾全上下文，不能只顾自己方便

**第三个月的关键词是「贡献」**。

我开始真正为企业项目贡献自己的力量：独立修复BUG、保障测试环境、完善接口文档、开发辅助模块。

第一次独立负责BUG修复时，mentor只给了一个提示：「这次你来独立完成，有问题可以讨论，但修复方案要你自己出。」那次BUG修复让我深刻理解了企业BUG修复的标准化流程：上报→定位→修复→回归，四步缺一不可。

**第四个月的关键词是「沉淀」**。

我开始把前三个月学到的内容串联起来，形成自己的知识体系。独立进行代码检视、BUG独立闭环、资料归档、版本更新协助——这些工作让我真正理解了什么叫「独立负责」。

mentor说过一句话让我印象很深：「经验不是经历，经历加上反思才是经验。只有沉淀下来的东西，才是真正属于你的。」

:::note
18周的实习教会我的，不只是Git分支策略、接口文档编写、BUG修复流程这些技术细节，更重要的是——**从学生到职场人的心态转变**。主动汇报、交付意识、可维护性比炫技更重要——这些道理，只有真正进入职场才能深刻理解。
:::

### 全栈实战：DM点餐系统的完整交付

**DM点餐系统**（7月-8月），是我第一个真正独立完成的全栈项目。

这是一款面向中小型餐厅的堂食扫码点餐系统：小程序端用uni-app开发，后台管理用Next.js构建，后端直接用Next.js API Routes，数据库则是Supabase（PostgreSQL + Realtime）。

技术选型时，我对比了三种方案：

| 方案 | 小程序端 | 后台管理 | 后端 | 数据库 | 结果 |
| --- | --- | --- | --- | --- | --- |
| 方案A | 原生小程序 | React管理后台 | Node.js | MySQL | 工作量大 |
| 方案B | uni-app | Vue3管理后台 | Spring Boot | PostgreSQL | 熟悉但冗余 |
| 方案C | uni-app | Next.js | Next.js API | Supabase | **简洁高效** |

最终选择方案C，理由很简单：**堂食点餐的核心是「快速落地」，不需要复杂的微服务架构**。

这个项目让我真正理解了什么叫「端到端」的全栈开发——从前端小程序到后端API再到数据库，从业务逻辑到用户体验，每一个环节都要自己把握。

项目中有两个让我印象最深的技术难点：

**一是数据库行锁解决并发超卖**。

想象一个场景：只剩最后一份菜，100个人同时下单。如果没有锁，两个人可能都下单成功，但厨房只能做一份。

```sql
-- PostgreSQL 函数：创建订单并预扣库存（原子操作）
CREATE OR REPLACE FUNCTION create_order_with_stock(
    p_table_id UUID,
    p_items JSONB
)
RETURNS TABLE(id UUID, order_no VARCHAR(20)) AS $$
DECLARE
    v_order_id UUID;
    v_order_no VARCHAR(20);
    v_item JSONB;
    v_dish_id UUID;
    v_quantity INT;
    v_current_stock INT;
BEGIN
    -- 查询库存时加行锁，防止并发超卖
    SELECT stock INTO v_current_stock
    FROM dishes
    WHERE id = v_dish_id
    FOR UPDATE;  -- 关键：行锁

    IF v_current_stock < v_quantity THEN
        RAISE EXCEPTION '库存不足';
    END IF;

    -- 预扣库存
    UPDATE dishes SET stock = stock - v_quantity WHERE id = v_dish_id;
    ...
END;
$$ LANGUAGE plpgsql;
```

**二是Supabase Realtime实时订阅**。

顾客下单后，后台页面无需刷新，自动弹出新订单提醒——这依赖Supabase的Realtime能力。原理是监听PostgreSQL的WAL日志，当有新INSERT时，通过WebSocket推送给所有订阅的客户端。

:::note
DM点餐系统还有很多可以完善的地方，但核心的点餐流程已经跑通。这是一个「MVP（最小可行产品）」该有的样子——先跑通核心链路，再逐步迭代。工程思维不是「一步到位」，而是「小步快跑」。
:::

### 底层原理：操作系统与计算机组成

**操作系统与计算机组成原理**（10月），是我2025年最「硬核」的学习经历。

学习操作系统的起因很直接：当我在IDE里轻点运行按钮，那个小小的程序究竟是怎么跑起来的？操作系统到底是什么？

这个问题困扰了我很久。Java代码能跑，Python脚本能跑，但我从没认真想过它们在计算机内部到底是怎么执行的。

学习之后，我的认知彻底被颠覆了：

**以前我以为的加法**：`1 + 1 = 2` ✓

**实际上计算机的加法**：
```
把两个数字从内存加载到CPU → CPU内部的ALU进行运算 → 结果写回内存
```

这种认知差距，让我觉得自己像个「会开车但不懂发动机的人」——能跑，但不懂原理。

操作系统五大核心功能模块的框架，让我对「软件与硬件的关系」有了全新的理解：

| 功能模块 | 核心职责 | 与开发的关联 |
| --- | --- | --- |
| 进程管理 | 创建、调度、销毁进程 | 并发编程、多线程 |
| 内存管理 | 分配/回收内存、虚拟内存 | JVM调优、OOM排查 |
| 文件系统 | 管理存储空间、文件权限 | 文件读写、路径处理 |
| 设备管理 | 管理硬件设备、驱动 | 外设交互、IO操作 |
| 用户接口 | 提供CLI/GUI给用户 | 系统调用、Shell |

而计算机组成原理，则让我理解了代码从编写到执行，背后到底发生了什么：

```
你写的代码 (e.g., Java)
        ↓ 编译
字节码文件 (.class)
        ↓ 类加载器
JVM虚拟机
        ↓ 解释/编译
机器指令
        ↓
CPU执行
        ↓
硬件操作（内存读写、I/O设备）
```

:::note
理解操作系统和计算机组成原理之后，我看待代码的眼光变了。以前写Java程序，我只关心对象和方法；现在我会想到JVM的堆内存管理、垃圾回收、线程调度。操作系统不是一门孤立的学问，而是理解整个计算机世界的钥匙。
:::

### 工程方法论：软件工程的学习

**软件工程**（10月），解答了我在代码之外无数的困惑。

课程里老师打了个比方，我一下子记住了：

> **软件工程，就是教你「怎么系统地做软件」的学问。如果写代码是「烹饪」，那软件工程就是「开餐厅」——你得考虑选址、装修、菜单设计、人员管理、食品安全、财务流水……光会炒菜可不行。**

软件工程要解决的，是软件开发的三大核心问题：

| 问题 | 表现 | 类比 |
| --- | --- | --- |
| **软件危机** | 项目延期、超预算、bug多、维护困难 | 餐厅开业一拖再拖、装修超支、菜谱难吃、服务差 |
| **质量与效率的矛盾** | 赶工就出bug，不赶工就延期 | 想快速上菜就容易出事故，慢慢做客人又等不及 |
| **需求变化** | 改着改着就面目全非 | 客人吃着吃着要换菜，吃着吃着要换厨师 |

学完软件工程后，我最大的改变是：**开始用「系统思维」去看待软件开发**。

| 维度 | 以前的我 | 现在的我 |
| --- | --- | --- |
| 写代码前 | 直接开写 | 先画用例图，理清需求 |
| 设计时 | 边写边想 | 先画架构图，再动手 |
| 写代码时 | 差不多就行 | 考虑边界，加注释 |
| 写完代码 | 直接提交 | 写单元测试，检查边界 |
| 交付时 | 能跑就行 | 检查文档是否齐全 |

### AI时代的编程：从PTA到SDD

**从古法编程到SDD**（10月），是我对AI时代编程范式的深刻反思。

2025年，AI编程工具开始普及。回顾我的编程学习历程，正好见证了编程方式的演变：

| 阶段 | 核心特征 | 我的状态 |
| --- | --- | --- |
| PTA纯手敲 | 没有代码提示，全靠自己 | 格式地狱，细节折磨 |
| CV工程师 | 搜索复制粘贴 | 解决当前问题，但不理解原理 |
| Tab工程师 | AI补全代码 | 效率提升，但审不出问题 |
| Vibe Coding | 描述需求，AI实现 | 适合小工具，不适合复杂工程 |
| SDD | 规格文档驱动开发 | AI按规格执行，不会跑偏 |

**Vibe Coding很爽，动动嘴就能做个小工具。但如果不懂技术**：

- 你可能描述不清楚需求，AI理解偏差
- 你可能看不出AI代码里的问题
- 你可能在关键地方被AI带偏

**而如果懂技术**：

- 你知道怎么精准描述需求
- 你能看出代码里的潜在问题
- 你知道什么时候该信AI，什么时候该信自己

> **懂技术的人，才能更好地驾驭AI。**

:::note
AI时代，「会用AI」已经不是优势，「能驾驭AI」才是。那些基础扎实、理解原理的人，才能在AI时代真正发挥创造力。
:::

## 生活碎片：那些不能被遗忘的瞬间

技术之外，2025年还有一些值得记录的时刻。

### 长沙四天四夜

五月，和三个朋友从浙江和广东出发，在长沙汇聚。

出发前各种意外——火车停运、航班延误、天气变化。本以为这趟旅行要泡汤，但最终大家还是在5月23日那天安全抵达。

四个人围坐在一起吃火锅时，心里突然有一种很奇妙的感觉——**从广东到浙江，从浙江到长沙，四个人，四个不同的起点，历经波折之后，终于在同一个城市、同一家店里相遇**。

那天晚上 citywalk，没有目的地，没有路线，就那么漫无目的地走着。不知不觉，竟然走回了酒店——掏出手机一看：**40367步**。

四人行的最后一天，我在朋友圈写道：

> "从两人提起，到三人确定，再到四人同行。再从四人游，变为三人行，最后两人结伴。世无不散宴席，唯存永恒记忆。"

**有些风景，必须亲眼所见才能体会。照片再美，也比不上站在现场那一刻的心潮澎湃。**

### 夜爬，得见大罗

七月的最后一天，凌晨3:46，一个念头突然席卷了我的脑海——夜爬，看日出。

没有任何预兆，没有任何计划。骑上共享单车，只带了一把"过关"伞，就这么出发了。

黑漆漆的山路像一张巨口，轻易地吞噬了我的信心。没有手电筒，只有手机照明，远处的不知名鸟叫让寂静更加彻底。

**害怕是理所当然的。**

但来都来了——没有自我鼓励加油的场景，没有想起任何人的教诲。只是单纯地想着：来都来了。

当我终于登顶鲳鱼尖，俯瞰脚下群山时，脑子里突然冒出一句：

> "偶然中的偶然，得见大罗。"

那一刻我自己都惊了。这句话从脑子里蹦出来的时候，我甚至觉得它不像是我写的。

**偶然中的偶然——**
- 偶然，是凌晨3:46突然冒出的夜爬念头
- 偶然，是小红书恰好推了这个地点
- 偶然，是凌晨四点半晨练的大爷让我知道路上有人
- 偶然，是遇见大叔告诉我还有多远

这些偶然串联在一起，构成一个必然的结果：**我登顶了，我看到了日出，我做到了。**

### 第一次独立拆机清灰

十一月，我决定自己动手给笔记本清灰。

朋友劝我：「花那个钱，避免了拆坏的可能性，不是挺值的吗？」

说实话，那一刻挺犹豫的。但最终，还是那股不服输的劲占了上风——越是说「不行」，就越想证明「我可以」。

于是开始行动：问AI、搜教程、查资料、把教程截图保存到手机、对照着图文一个步骤一个步骤地看……

**四十分钟。**

从拆完、清灰、到组装、开机，一共只用了四十分钟。

开机的那一刻，风扇转动的声音明显平稳了许多。跑了一个项目，温度也比之前低了好几度。

那一刻的感觉，不是「哇我好厉害」，而是「**原来我也可以**」。

这次拆机清灰验证了一个道理：**别人可以，那我也可以。**

不服输、倔强，这些性格特质，让我在很多事情上都不愿意认输。很多事看起来很难，但只要愿意去尝试、去准备、去坚持，就没那么难。

### AI浪潮中的FOMO与哲学解药

十二月，在被AI时代的FOMO焦虑裹挟之后，我尝试从马克思主义哲学中寻找答案。

当我用辩证唯物主义的观点来看待AI时代的技术焦虑时，很多让我夜不能寐的问题，突然变得不那么可怕了：

- **从"全面联系"看FOMO**：联系是普遍的，任何个体都不可能参与所有的联系。选择性的"错过"，本身就是一种智慧。
- **从"发展观点"看技术焦虑**：任何技术的发展都是曲折前进的。今天炙手可热的技术，可能明年就被淘汰。
- **从"全面观点"看自我定位**：在"学AI"vs"不学AI"之间，存在大量的中间地带。

> **当别人在讨论某个新技术时，我不再急于判断"要不要学"，而是先问"它是什么"、"它从何而来"、"它要到哪里去"——这三个问题，恰好也是哲学的基本追问。**

## 遗憾与庆幸

如果要挑一个遗憾，大概是：这一年的自己，有时候还是太追求「深度」了。

操作系统、计算机组成原理、软件工程——这些底层知识确实重要，但学习它们花费的时间也远超预期。有时候会想，这些时间如果用来做项目，是不是更「实用」？

但比起遗憾，更多的是庆幸。

庆幸自己在实习的喧嚣之后，还能静下心来学习底层原理。庆幸遇到了浙江索思的mentor，让我在18周里完成了从学生到职场人的心态转变。庆幸遇到了DM点餐系统这个项目，让「全栈」不再是一个概念，而是真正跑通的产品。庆幸每一段学习经历——无论是PTA的格式地狱，还是SDD的规格驱动，都在塑造着一个更完整的自己。

:::note
2025年的我，比2024年更沉稳了一点。但学得越多，越觉得自己无知。这不是谦虚，而是事实。
:::
