## 故事的起点：一栋楼与一排别墅

2024年1月，正式进入微服务相关内容的学习。学到微服务架构那一章时，老师打了个比方，我一下子就记住了：

> **单体架构就像一栋大楼，所有住户（模块）共享同一栋楼的电梯、管道、停车场。好处是管理简单，坏处是一旦某层着火，整栋楼都得遭殃。微服务架构呢，就像一片别墅区，每户人家独立门牌、独立水电表，但又要通过小区大门统一管理。**

这个比喻太准确了。我们写的那些「增删改查」项目，用户模块、订单模块、商品模块全堆在一个SpringBoot应用里，编译一次、部署一件，有种「一荣俱荣、一损俱损」的悲壮感。

课程用的是**黑马商城**实战项目，先跑通单体版，再拆成微服务。我就这样一头扎进了分布式系统的世界。

## 为什么需要微服务？先聊聊单体架构的痛

### 当项目变成「巨无霸」

课程里展示了一个典型场景：黑马商城单体版有用户、商品、订单、支付等多个模块，听起来不多。但当你把所有功能全塞进一个应用时，会发生什么？

| 问题场景 | 单体架构的尴尬 |
|--------|--------------|
| 代码冲突 | 10个人同时改一个模块，Git合并天天打架 |
| 部署噩梦 | 改了一行营销代码，要停机重新部署整个系统 |
| 扩展困难 | 订单系统压力大，只能扩展整个应用，浪费资源 |
| 技术绑架 | 老团队想用新技术？整个重构吧您 |

我当时写过一个课设项目，才3个模块，已经被这种「牵一发动全身」的体验折磨得够呛。课程老师说得对：**单体架构不是不能用，而是业务复杂度上去之后，它会成为制约你成长的天花板。**

### 微服务架构登场

微服务（Micro Services）本质上是一种架构风格，核心思想是**「大拆小」**：把复杂业务拆成若干独立部署、独立运行的子服务，每个服务只关注自己的核心职责。

| 对比维度 | 单体架构 | 微服务架构 |
|---------|---------|-----------|
| 代码组织 | 单一代码仓库 | 按业务边界拆分，多个代码仓库 |
| 部署方式 | 整体打包部署 | 每个服务独立部署 |
| 扩展方式 | 复制整个应用 | 按需扩展单个服务 |
| 技术栈 | 统一技术栈 | 可异构，不同服务用不同技术 |
| 故障隔离 | 一个模块挂，全局崩溃 | 单个服务挂，不影响其他服务 |

但老师也泼了盆冷水：**微服务不是银弹，拆得不好会比单体还难受。** 这句话我后来深有体会。

## SpringCloudAlibaba：国内微服务落地的最优解

### 为什么是SpringCloudAlibaba？

课程选型时，老师直接跳过了Netflix那一套（Eureka、Hystrix），原因是**Netflix宣布停止维护这些组件**。作为2023年后入局的学习者，继续学停更的技术属实49年入国军。

最终选择的是**SpringCloud Alibaba**（后面简称SCA），它是Spring Cloud的国产化增强方案，核心组件包括：

| 组件 | 定位 | 通俗类比 |
|------|------|---------|
| Nacos | 注册中心+配置中心 | 小区物业管理处，既管住户信息（注册）又管规则制度（配置） |
| Sentinel | 服务限流降级 | 景区门口的排队系统，控制人流防止踩踏 |
| Seata | 分布式事务 | 不同店铺间的统一收银台，保证钱货两清 |
| OpenFeign | 服务远程调用 | 业主委员会帮你联系隔壁小区物业办事 |
| Gateway | API网关 | 小区大门保安，所有访客必经的安检口 |

接下来，我逐个聊聊每个组件的学习心得。

## Nacos：物业管理处式服务治理

### 什么是注册中心？

课程里讲了个场景：黑马商城拆成用户服务、订单服务、商品服务后，每个服务都部署在不同的机器上。这时候有个问题：**服务A想调服务B，它怎么知道B在哪台机器上？**

这就需要**注册中心**了。我把它理解成**小区物业管理处**：

- 每个住户入住时要**登记**（服务注册）：告诉物业「我是302业主，有事找我」
- 物业有一个**住户名单**（服务列表）：记录着所有住户的地址和联系方式
- 住户搬家时要**注销**（服务下线）：通知物业「我不住了，把我划掉」
- 住户想找某个服务时，**先去物业查名单**（服务发现）：问「谁是做菜品配送的？」

Nacos就是这样一个物业管理处，它既负责「注册登记」，又负责「信息查询」，还支持「规则变更」。

### 服务注册与发现实战

来看一段我当时写的配置，注释里记满了自己的理解：

```java
// user-service 启动类
@SpringBootApplication
@EnableFeignClients  // 开启OpenFeign远程调用
public class UserServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(UserServiceApplication.class, args);
    }

```

```yaml
# application.yml 配置
spring:
  application:
    name: user-service  # 服务名称，就是你在Nacos里的"门牌号"
  cloud:
    nacos:
      server-addr: 127.0.0.1:8848  # Nacos服务器地址

      # 分组，类似于同一小区的不同"楼栋"

// order-service 调用用户服务的FeignClient

public interface UserClient {

    // 类似打电话给物业管理处说"帮我联系302业主"
    @GetMapping("/{id}")
    User findById(@PathVariable Long id);

```

### 配置中心：规则的统一管理

Nacos还有一个核心能力：**配置中心**。课程里举了个例子：假设黑马商城有开发环境、测试环境、生产环境三套配置，以前你要改一个数据库密码，得登录三台服务器改三次。

有了Nacos配置中心，**规则配置文件存在Nacos服务器上，各服务启动时去拉取**。改一次，三端生效。

```yaml
# Nacos上的 shared-data.yaml 配置文件
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/mall?useSSL=false
    username: root
    password: ${DB_PASSWORD}  # 支持占位符，环境变量注入
  redis:
    host: ${REDIS_HOST:localhost}  # 有默认值的环境变量
```

```java
// order-service 拉取配置
@RestController
@RefreshScope  // 关键注解！加上这个，配置变更后Spring自动刷新
public class OrderController {

    // 这儿的值会随着Nacos配置变更自动刷新，不用重启服务
    @Value("${discount.rate:1.0}")
    private Double discountRate;

    @GetMapping("/order/{id}")
    public Order getOrder(@PathVariable Long id) {
        // 折扣率从配置中心获取，灵活可调
        order.setDiscount(order.getPrice() * discountRate);
        return order;
    }
}
```

:::note
**踩坑记录**：当时我漏写了 `@RefreshScope` 注解，改了Nacos配置后服务愣是没生效，查了半天才发现是这个原因。配置中心虽好，但每个需要动态刷新配置的Bean都得加这个注解。
:::

### 配置分组与共享配置

课程中提到的另一个实用技巧是**配置分组**和**共享配置**：

| 概念 | 作用 | 使用场景 |
|------|------|---------|
| **命名空间（Namespace）** | 环境隔离 | dev/test/prod 环境配置分离 |
| **分组（Group）** | 业务隔离 | 不同业务线的公共配置 |
| **共享配置（shared-data）** | 多个服务共用一份配置 | 数据库连接、Redis等公共配置 |

```yaml
# 共享配置示例：让多个服务共享同一份数据库配置
spring:
  cloud:
    nacos:
      config:
        # 共享配置文件列表（这些配置会被所有服务加载）
        shared-dataids:
          - shared-db.yaml      # 数据库配置
          - shared-redis.yaml   # Redis配置
        refreshable-dataids:
          - shared-db.yaml       # 支持动态刷新的配置
```

```yaml
# shared-db.yaml 内容示例
# 这个文件放在Nacos配置中心，所有服务都能引用
spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/mall
    username: root
    password: ${DB_PASSWORD}
```

## OpenFeign：让服务调用像打电话一样简单

### 为什么需要OpenFeign？

课程里讲服务间通信时，提到了两种方式：

1. **RestTemplate**：用HTTP客户端自己拼URL、序列化参数
2. **OpenFeign**：声明式接口，像调用本地方法一样调远程服务

```java
// RestTemplate方式：像自己动手写信
String url = "http://user-service/user/" + userId;
User user = restTemplate.getForObject(url, User.class);

// OpenFeign方式：像打电话给物业让它们帮你联系
@FeignClient(name = "user-service")
interface UserClient {
    @GetMapping("/user/{id}")
    User findById(@PathVariable Long id);
}
// 调用时直接用，跟本地方法一样
User user = userClient.findById(123L);
```

我当时的感受是：**OpenFeign就是那种用一次就回不去的东西。** 代码清晰程度完全不在一个Level。

## Gateway：小区大门，统一的安检口

### 网关是什么？

课程里Gateway那一章，老师是这样介绍的：

> **假设黑马商城有10个微服务，每个服务都要做登录验证、权限校验、跨域处理。如果每个服务都写一遍，就像10个小区各自建了一个门卫室，重复建设还不好管理。网关就是统一建一个大门，所有人进出都得过我这关。**

这个比喻太生动了。**Gateway就是微服务架构的「统一入口」，所有外部请求先到网关，网关完成鉴权、路由、限流等通用处理后，再转发到具体服务。**

### Gateway路由配置实战

当时我写的第一个Gateway配置是这样的：

```yaml
spring:
  cloud:
    gateway:
      # 路由规则：类似于小区门禁系统配置
      routes:
        # 用户服务路由
        - id: user-service
          uri: lb://user-service  # lb=loadbalance，会从Nacos获取地址
          predicates:               # 匹配条件：路径以 /user 开头的请求
            - Path=/user/**
          filters:                 # 过滤器：在转发前后的处理
            - StripPrefix=1        # 去掉第一层路径前缀

        # 商品服务路由
        - id: product-service
          uri: lb://product-service
          predicates:
            - Path=/product/**
          filters:
            - StripPrefix=1

        # 订单服务路由
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/order/**
          filters:
            - StripPrefix=1
```

### 过滤器链：网关的「安检流程」

Gateway最核心的概念是**过滤器链**。每个请求经过网关时，会被一系列过滤器处理，然后再转发出去。过滤器分两种：

| 类型 | 作用范围 | 示例 |
|------|---------|------|
| **全局过滤器** | 所有路由生效 | 统一鉴权、日志、跨域处理 |
| **路由过滤器** | 指定路由生效 | 路径重写、请求限流 |

```java
// 自定义全局过滤器：模拟门禁刷卡
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 获取请求路径
        String path = exchange.getRequest().getURI().getPath();

        // 2. 放行登录接口，不用鉴权
        if (path.contains("/user/login") || path.contains("/user/register")) {
            return chain.filter(exchange);
        }

        // 3. 其他接口检查Token
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (StringUtils.isBlank(token)) {
            // 没带Token，返回401未授权
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // 4. Token验证通过，放行
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        // 返回优先级，数字越小越先执行
        return -100;
    }
}
```

:::note

## Sentinel：服务限流与降级的大脑

### 为什么要限流？

课程里讲Sentinel时，先抛出了一个场景：

> **「双十一」零点秒杀时，100万人同时涌入下单系统，服务器扛不住，直接宕机。这不只是技术问题，还可能引发系统性风险——超卖、数据库崩溃、资金损失。**

**Sentinel的作用就是「在洪水来临前建好堤坝」**：它监控服务的QPS（每秒请求数），当某个服务的请求量超过阈值时，**拒绝多余请求**（限流），或者**引导请求走降级逻辑**（降级），保证核心服务不宕机。

### Sentinel的流控策略

课程里讲了三种流控效果，对应不同场景：

| 流控效果 | 适用场景 | 类比 |
|---------|---------|------|
| **直接拒绝** | 适合处理瞬时高峰 | 景区满员后直接拦在门外 |
| **Warm Up** | 适合处理预热型流量 | 运动员热身，逐步加量 |
| **排队等待** | 适合处理瞬时高峰 | 银行叫号，排队办理 |

### Sentinel的熔断策略

除了限流，Sentinel还有**熔断器**功能，用于保护系统不被异常拖垮：

| 熔断策略 | 触发条件 | 类比 |
|---------|---------|------|
| **慢调用比例** | 平均响应时间超过阈值 | 餐厅上菜太慢，等不及了直接走人 |
| **异常比例** | 请求异常比例超过阈值 | 一半的菜都是坏的，不吃了 |
| **异常数** | 异常数量超过阈值 | 连续踩坑3次，以后再也不来了 |

    rule.setMinRequestAmount(5);    // 最小请求数（达到这个量才统计）
    rule.setStatIntervalMs(30000);  // 统计时间窗口30秒
    rule.setSlowRatioThreshold(0.5); // 慢调用比例阈值

## 分布式事务：跨越服务的数据一致性挑战

### 为什么分布式事务这么难？

这是整个微服务学习过程中，**我花费时间最多的章节**。课程从CAP定理讲起，然后引入BASE理论，最后落地到Seata的三种事务模式。

在展开讲解之前，先抛出课程里的一个经典场景：**用户下单购买商品，需要同时扣减库存、创建订单、扣减余额。这三个操作分散在三个不同的微服务里，每个服务都有自己的数据库。如果一笔订单卡在「库存已扣但余额不足」这一步，数据就会出现不一致。**

这就是**分布式事务**要解决的问题——**如何在跨服务、跨数据库的场景下，保证数据的一致性。**

### CAP定理：分布式系统的不可能三角

课程里用一张图和一个比喻来讲解CAP定理，我记忆特别深刻：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            CAP 定理图示                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                        ┌───────────────┐                                   │
│                        │   Consistency │                                   │
│                        │    一致性 C    │                                   │
│                        └───────┬───────┘                                   │
│                                │                                            │
│                     ┌──────────┼──────────┐                              │
│                     │          │          │                              │
│                     │          │          │                              │
│              ┌──────▼─────┐┌────▼────┐┌────▼─────┐                        │
│              │  不可同时   ││ 不可同时 ││  不可同时 │                        │
│              │  满足CA   ││  满足CP ││   满足AP │                        │
│              └───────────┘└─────────┘└──────────┘                        │
│                                                                             │
│   📌 P（分区容错）是分布式系统必须满足的                                       │
│   📌 所以实际是在 C（一致性）和 A（可用性）之间做权衡                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

| 特性 | 含义 | 举例 |
|------|------|------|
| **C (Consistency)** | 数据一致性 | 转账后两边余额必须相等 |
| **A (Availability)** | 可用性 | 服务挂了也要返回响应（哪怕是错误响应） |
| **P (Partition)** | 分区容错 | 网络故障时系统仍能工作 |

> **CAP定理**：分布式系统最多同时满足C、A、P中的两个。对于跨网络的微服务来说，P（分区容错）是必须满足的，所以实际上是在C（一致性）和A（可用性）之间做权衡。

课程里打了个比方：**你和你女朋友吵架后发消息道歉，你希望对方「立刻原谅你」还是「最终会原谅你」？前者是强一致性，后者是最终一致性。**

### BASE理论：现实可行的妥协方案

CAP定理告诉我们：分布式系统无法同时满足强一致性和高可用性。那有没有一种**兼顾两者**的折中方案？

**BASE理论**就是对这个问题的一种回答。它的名字来源于三个核心概念的首字母：

| BASE概念 | 全称 | 含义 |
|----------|------|------|
| **B** | **Basically Available** | 基本可用，允许系统在故障时降级服务 |
| **S** | **Soft State** | 软状态，允许数据在一段时间内不一致 |
| **E** | **Eventually Consistent** | 最终一致性，数据最终会达到一致状态 |

课程里老师用了一个很生活的比喻：

> **BASE理论就像网购发货。快递发出后，状态是「运输中」（软状态），各个中转站扫描后状态会「逐渐更新」，最终到达你手上时状态变成「已签收」（最终一致）。整个过程你不需要实时知道精确位置，但最终一定能拿到货。**

**BASE vs CAP的关系**：

| 对比维度 | CAP定理 | BASE理论 |
|---------|---------|---------|
| 追求目标 | 强一致性 | 最终一致性 |
| 可用性 | 可能牺牲 | 基本可用 |
| 一致性时效 | 实时一致 | 允许延迟 |
| 适用场景 | 金融系统、扣款 | 社交Feed、订单状态 |

理解了BASE理论，就理解了为什么大多数互联网系统选择**最终一致性**而非强一致性——因为强一致性的代价太大了。

### Seata AT模式：Automatic Transaction Mode

AT模式是Seata的**核心模式**，全称是 **Automatic Transaction Mode**（自动事务模式）。它的核心原理是**利用undo log自动生成反向SQL进行回滚，对业务代码几乎零侵入**。

为什么叫「自动」？因为框架会自动帮你生成回滚SQL，你只需要正常写业务代码即可。

**AT模式的工作流程**：

```
1. TM 发起全局事务，TC 生成 XID（全局事务ID）
2. 各 RM 执行本地事务，Seata 自动记录 undo log（前置镜像 + 后置镜像）
3. TM 提交/回滚全局事务
4. TC 通知所有 RM 提交/回滚
5. 如果回滚，RM 根据 undo log 自动执行反向 SQL
```

**AT模式实战代码**：

```java
// ========== TM（事务管理器）端的代码 ==========
// 订单服务：下单时扣库存
// ⚠️ 注意：只有 @GlobalTransactional 标注的方法所在的微服务是 TM（事务管理器）
// 被远程调用的服务（user-service/product-service）不需要加此注解
@Service
public class OrderService {

    @GlobalTransactional  // 标记分布式事务的起点（TM角色）
    public void createOrder(OrderParam param) {
        // 1. 创建订单记录
        Order order = new Order();
        order.setUserId(param.getUserId());
        order.setProductId(param.getProductId());
        order.setCount(param.getCount());
        order.setStatus("CREATE");
        orderMapper.insert(order);

        // 2. 调用库存服务扣减库存（远程调用）
        // Seata会自动跟踪这个远程调用，将其纳入同一事务
        productClient.deductStock(param.getProductId(), param.getCount());

        // 3. 调用账户服务扣减余额
        accountClient.deductBalance(param.getUserId(), param.getTotalPrice());

        // 4. 更新订单状态
        order.setStatus("PAID");
        orderMapper.updateById(order);

        // 如果任意一步抛异常，Seata会执行undo log回滚
    }
}
```

```java
// ========== RM（资源管理器）端的代码 ==========
// 商品服务：扣减库存的接口
// ⚠️ 注意：RM（资源管理器）不需要加 @GlobalTransactional
// Seata通过代理数据源自动拦截SQL执行
@RestController
public class ProductController {

    @PutMapping("/product/{id}/stock")
    public void deductStock(@PathVariable Long id, @RequestParam Integer count) {
        // Seata会为这个方法生成反向SQL的记录
        // 执行前记录当前状态（前置镜像）
        // 执行后记录当前状态（后置镜像）
        // 执行失败后自动回滚到前置镜像的状态

    }
}
```

```java
// ProductService 中的实际业务逻辑（Seata自动代理）
@Service
public class ProductService {

    @Transactional  // 本地事务，正常使用
    public void reduceStock(Long productId, Integer count) {
        // 1. 查询当前库存（Seata记录前置镜像）
        Product product = productMapper.selectById(productId);

        // 2. 扣减库存
        product.setStock(product.getStock() - count);
        productMapper.updateById(product);

        // 3. Seata记录后置镜像，全局回滚时执行反向SQL
        // UPDATE product SET stock = stock + count WHERE id = ?
    }
}
```

:::

**AT模式的适用场景**：大多数业务场景，特别是对代码侵入性有要求、但又需要事务支持的情况。

### Seata TCC模式：Try-Confirm-Cancel Mode

TCC模式是Seata的**另一重要模式**，全称是 **Try-Confirm-Cancel Mode**。与AT模式不同，TCC模式**需要业务代码手动实现三个阶段的逻辑**。

课程里老师打了个比方：

> **TCC就像订酒店。Try阶段是打电话预订并冻结房间（预扣库存），Confirm阶段是入住并确认消费（真正扣减），Cancel阶段是取消预订释放房间（回滚冻结）。**

TCC模式的三个阶段：

| 阶段 | 含义 | 类比 |
|------|------|------|
| **Try** | 预留资源，冻结部分库存 | 预订房间，冻结房间 |
| **Confirm** | 确认执行，使用预留资源 | 入住，确认消费 |
| **Cancel** | 取消操作，释放预留资源 | 取消预订，释放房间 |

```java
// ========== TCC 模式的库存服务实现 ==========

// 1. 定义TCC接口
// Try阶段：预留资源
@LocalTCC
public interface StockTccService {

    // TCC的三个阶段方法
    @TwoPhaseBusiness(name = "deductStock")
    boolean tryDeductStock(
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );

    // Confirm阶段：确认扣减
    boolean confirmStock(BusinessActionContext context);

    // Cancel阶段：回滚释放
    boolean cancelStock(BusinessActionContext context);
}
```

```java
// 2. TCC接口实现类
@Component
public class StockTccServiceImpl implements StockTccService {

    @Autowired
    private StockMapper stockMapper;

    // ========== Try阶段：预留资源 ==========
    @Override
    public boolean tryDeductStock(Long productId, Integer count) {
        // 1. 查询当前库存
        Stock stock = stockMapper.selectByProductId(productId);
        if (stock.getAvailableStock() < count) {
            // 库存不足，预留失败
            throw new RuntimeException("库存不足");
        }

        // 2. 冻结库存（可用库存减少，冻结库存增加）
        // 注意：这里只是冻结，并没有真正扣减
        stock.setAvailableStock(stock.getAvailableStock() - count);
        stock.setFrozenStock(stock.getFrozenStock() + count);
        stockMapper.updateById(stock);

        // 3. 返回true表示Try阶段成功
        return true;
    }

    // ========== Confirm阶段：确认执行 ==========
    @Override
    public boolean confirmStock(BusinessActionContext context) {
        // 获取Try阶段预留的资源信息
        Long productId = Long.parseLong(context.getActionContext("productId").toString());
        Integer count = Integer.parseInt(context.getActionContext("count").toString());

        // 确认扣减：冻结库存减少，实际库存减少
        Stock stock = stockMapper.selectByProductId(productId);
        stock.setFrozenStock(stock.getFrozenStock() - count);
        stock.setTotalStock(stock.getTotalStock() - count);
        stockMapper.updateById(stock);

        return true;
    }

    // ========== Cancel阶段：回滚释放 ==========
    @Override
    public boolean cancelStock(BusinessActionContext context) {
        // 获取Try阶段预留的资源信息
        Long productId = Long.parseLong(context.getActionContext("productId").toString());
        Integer count = Integer.parseInt(context.getActionContext("count").toString());

        // 释放冻结：可用库存恢复，冻结库存减少
        Stock stock = stockMapper.selectByProductId(productId);
        stock.setAvailableStock(stock.getAvailableStock() + count);
        stock.setFrozenStock(stock.getFrozenStock() - count);
        stockMapper.updateById(stock);

        return true;
    }
}
```

```java
// 3. 订单服务调用TCC接口
@Service
public class OrderService {

    @Autowired
    private StockTccService stockTccService;

    @GlobalTransactional
    public void createOrder(OrderParam param) {
        // 1. 调用TCC接口预留库存资源
        stockTccService.tryDeductStock(param.getProductId(), param.getCount());

        // 2. 其他业务逻辑...
    }
}
```

**TCC模式的适用场景**：
- 对数据一致性要求高、需要手动控制资源预留的场景
- 需要跨多个不支持Seata数据源代理的异构系统的场景
- 对性能要求高、不能接受AT模式全局锁的场景

### Seata Saga模式：长事务的最优解

Saga模式是Seata的**第三种模式**，它是一种**基于事件驱动的长事务解决方案**。与AT/TCC不同，**Saga不提供自动回滚**，而是要求每个业务操作都有对应的**补偿操作**。

课程里老师举了个很形象的例子：

> **Saga就像接力赛。每个选手（服务）跑完自己的那一棒后，下一个选手接着跑。如果某一棒跑砸了，不是回头重新跑，而是由专门的「补偿员」把所有选手再叫回来，一个一个退回去。**

Saga模式的核心思想是：**正向操作 + 补偿操作 = 事务完整**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Saga 模式执行流程                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   正向执行（Forward）：                      补偿回滚（Backward）：         │
│                                                                             │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐                               │
│   │ 步骤1   │───▶│ 步骤2   │───▶│ 步骤3   │───▶ 成功                      │
│   │ 创建订单 │    │ 扣库存  │    │ 扣余额  │                               │
│   └────┬────┘    └────┬────┘    └────┬────┘                               │
│        │              │              │                                     │
│        │              │              │ 某步骤失败                          │
│        │              │              ▼                                     │
│        │              │         ┌─────────┐                               │
│        │              │◀────────│ 补偿3   │                               │
│        │              │   取消扣余额  └─────────┘                          │
│        │              ▼                                      │              │
│        │         ┌─────────┐                               │              │
│        │◀────────│ 补偿2   │                               │              │
│        │   回滚库存  └─────────┘                               │              │
│        ▼                                                           │              │
│   ┌─────────┐                                                     │              │
│   │ 补偿1   │◀────────────────────────────────────────────────────┘              │
│   │ 取消订单 │         回滚订单                                                │
│   └─────────┘                                                                │
│                                                                             │
│   📌 Saga不保证原子性，但保证最终一致性                                       │
│   📌 每个正向操作都必须有对应的补偿操作                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Saga模式的代码实现**：

```java
// ========== Saga 模式实现：转账服务 ==========

// 1. 定义转账的Saga状态机（状态机编排方式）
// 通常使用状态机描述文件来定义，这里用代码模拟

// 转账步骤1：扣减发起方账户余额
@Component
public class AccountService {

    @Transactional
    public void debit(String accountId, BigDecimal amount) {
        // 扣减余额
        Account account = accountMapper.selectByAccountId(accountId);
        account.setBalance(account.getBalance().subtract(amount));
        accountMapper.updateById(account);
    }

    // 补偿操作1：回滚扣减（加回余额）
    @Transactional
    public void undoDebit(String accountId, BigDecimal amount) {
        Account account = accountMapper.selectByAccountId(accountId);
        account.setBalance(account.getBalance().add(amount));
        accountMapper.updateById(account);
    }
}

// 转账步骤2：增加接收方账户余额
@Component
public class AccountService2 {

    @Transactional
    public void credit(String accountId, BigDecimal amount) {
        // 增加余额
        Account account = accountMapper.selectByAccountId(accountId);
        account.setBalance(account.getBalance().add(amount));
        accountMapper.updateById(account);
    }

    // 补偿操作2：回滚增加（扣减余额）
    @Transactional
    public void undoCredit(String accountId, BigDecimal amount) {
        Account account = accountMapper.selectByAccountId(accountId);
        account.setBalance(account.getBalance().subtract(amount));
        accountMapper.updateById(account);
    }
}
```

```java
// 2. Saga状态机编排器（简化版）
@Service
public class TransferSaga {

    @Autowired
    private AccountService accountService;

    @Autowired
    private AccountService2 accountService2;

    @Autowired
    private TransferRecordMapper transferRecordMapper;

    /**
     * 转账 Saga 流程
     * 正向：扣A账户 -> 加B账户
     * 补偿：加A账户 <- 扣B账户
     */
    public void transfer(String fromAccount, String toAccount, BigDecimal amount) {
        String transactionId = UUID.randomUUID().toString();

        try {
            // 步骤1：扣减发起方账户
            accountService.debit(fromAccount, amount);

            // 步骤2：增加接收方账户
            accountService2.credit(toAccount, amount);

            // 记录成功状态
            transferRecordMapper.insert(new TransferRecord(transactionId, "SUCCESS"));

        } catch (Exception e) {
            // 补偿流程：回滚所有操作
            compensate(fromAccount, toAccount, amount, transactionId);
            throw new RuntimeException("转账失败，已自动回滚", e);
        }
    }

    /**
     * 补偿操作：反向执行
     */
    private void compensate(String fromAccount, String toAccount, BigDecimal amount, String txnId) {
        try {
            // 补偿步骤2：扣减接收方账户（回滚步骤2）
            accountService2.undoCredit(toAccount, amount);
        } catch (Exception e) {
            // 记录补偿失败，需要人工介入
            log.error("补偿步骤2失败，transactionId={}", txnId, e);
        }

        try {
            // 补偿步骤1：回滚发起方账户（回滚步骤1）
            accountService.undoDebit(fromAccount, amount);
        } catch (Exception e) {
            log.error("补偿步骤1失败，transactionId={}", txnId, e);
        }

        // 记录补偿状态
        transferRecordMapper.insert(new TransferRecord(txnId, "COMPENSATED"));
    }
}
```

```java
// 3. 转账控制器
@RestController
public class TransferController {

    @Autowired
    private TransferSaga transferSaga;

    @PostMapping("/transfer")
    public Result<Void> transfer(@RequestBody TransferParam param) {
        try {
            transferSaga.transfer(param.getFromAccount(), param.getToAccount(), param.getAmount());
            return Result.success(null);
        } catch (Exception e) {
            return Result.error("转账失败：" + e.getMessage());
        }
    }
}
```

:::note
**Saga模式的注意事项**：
1. **不保证原子性**：Saga不提供自动回滚，每个步骤的补偿需要业务自己实现
2. **需要人工兜底**：如果补偿操作也失败了，需要人工介入处理
3. **适合长事务**：跨多个系统、耗时较长的业务流程，如金融交易、订单处理等
4. **无全局锁**：性能高，但数据一致性较低
:::

### 三种分布式事务模式对比

讲完了三种模式，最后用一张表做总结：

| 模式 | 全称 | 原理 | 侵入性 | 性能 | 回滚方式 | 适用场景 |
|------|------|------|--------|------|---------|---------|
| **AT模式** | Automatic Transaction | 自动生成反向SQL回滚 | **低**（几乎无侵入） | 中 | 自动回滚 | 大多数业务场景 |
| **TCC模式** | Try-Confirm-Cancel | 手动预留+确认+回滚 | **高**（需业务实现） | 高 | 手动回滚 | 对一致性要求高 |
| **Saga模式** | Saga | 正向操作+补偿操作 | **中**（需实现补偿） | 高 | 补偿回滚 | 长事务、跨系统 |

课程里老师给了一个选型建议：

> - **业务简单、想快速上线**：选AT模式，几乎零侵入
> - **对一致性要求极高**：选TCC模式，性能也好
> - **涉及多个异构系统**：选Saga模式，最灵活

## 链路追踪：服务网格的「记录仪」

### 为什么需要链路追踪？

课程里描述了一个经典场景：**用户反馈订单页面打不开，你得从网关一路查到订单服务、库存服务、用户服务... 几十个服务调用链，哪个环节出问题都不知道。**

**链路追踪就是给每次请求分配一个TraceId，这个Id会随着请求在各个服务间流转，每个服务都把自己的处理记录到这个Id下。最后串起来，就是一次请求的完整生命轨迹。**

### Sleuth+Zipkin实战

```yaml
# 在各个服务中添加依赖和配置
spring:
  zipkin:
    base-url: http://localhost:9411  # Zipkin服务器地址
    discovery-client-enabled: false   # 不使用服务发现，直连
  sleuth:
    sampler:
      probability: 1.0  # 采样率，1.0表示100%采样（测试环境）
```

```java
// 启动类上添加EnableZipkin注解开启链路追踪
@SpringBootApplication
@EnableZipkin  // 启用 Sleuth + Zipkin
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

配置完成后，打开Zipkin的可视化界面，就能看到类似这样的调用链：

```
createOrder [248ms]
├── validateToken [5ms]
├── queryUser [23ms]
│   └── queryUserCache [8ms]
├── queryProduct [45ms]
│   └── queryProductDB [42ms]
└── createOrderRecord [12ms]
```

### SkyWalking：更强大的链路追踪

课程用的是 Sleuth+Zipkin，但如果想用更强大的工具，**SkyWalking** 是一个不错的选择：

| 特性 | Sleuth+Zipkin | SkyWalking |
|------|--------------|-----------|
| **部署复杂度** | 低（轻量级） | 中（需要单独部署OAP服务） |
| **UI界面** | 基础 | 强大（支持拓扑图、性能分析） |
| **agent接入** | 手动埋点 | Java Agent自动埋点，零代码侵入 |
| **语言支持** | 主要Java | 多语言（Java、Go、Python等） |
| **学习成本** | 低 | 中 |

SkyWalking 通过 Java Agent 自动接入，不需要在代码中手动添加 `@EnableZipkin` 等注解，更适合生产环境使用。

## 踩坑实录：那些让我半夜排查的问题

### 坑一：服务注册上了，但调用报错"No instances available"

**问题现象**：

Nacos控制台明明能看到服务注册成功了，但OpenFeign调用时一直报错：

```
feign.RetryableException: No instances available for xxx-service
```

**根因分析**：

检查了半天，发现是**服务名大小写不匹配**。我在 `@FeignClient(name = "User-Service")` 里写了大写，但服务注册时用的是 `user-service`（小写）。

**解决方案**：

```java
// 统一使用小写服务名，或者仔细核对大小写
@FeignClient(name = "user-service")
interface UserClient {
    // ...
}
```

:::warning
**经验教训**：服务名一定要统一规范，建议全小写，避免不必要的排查时间。
:::

### 坑二：Gateway路由转发后404

**问题现象**：

配置好Gateway路由后，访问 `/user/1` 一直返回404，但直接访问后端服务是正常的。

**根因分析**：

这是**路径转发的问题**。Gateway默认会把原始路径一起转发，比如 `/user/1` 转发到 `lb://user-service/user/1`，但我的后端服务只暴露了 `/1`，没有 `/user/1`。

**解决方案**：

使用 `StripPrefix` 过滤器去掉前缀：

```yaml
filters:
  - StripPrefix=1  # 去掉第一层路径
```

### 坑三：分布式事务不回滚

**问题现象**：

模拟了一个异常，Seata AT模式愣是没回滚，数据还是写进去了。

**根因分析**：

检查了半天，发现是**Seata代理数据源的问题**。如果没有正确配置Seata的数据源代理，它就跟踪不到你的SQL执行。

**解决方案**：

```yaml
# Seata AT模式需要排除默认数据源，使用Seata代理
spring:
  datasource:
    druid:
      # 排除自动配置，使用Seata的DataSourceProxy
      exclude: com.alibaba.druid.spring.boot.autoconfigure.DruidDataSourceAutoConfigure
  cloud:
    alibaba:
      seata:
        tx-service-group: my-test-group
```

### 坑四：Nacos健康检查误判

**问题现象**：

服务明明正常运行，但Nacos控制台显示服务不健康，最后被下线了。

**根因分析**：

这是**心跳检测超时**的问题。如果服务处理请求耗时较长（比如数据库查询慢），而Nacos心跳间隔是5秒，可能被误判为宕机。

**解决方案**：

```yaml
spring:
  cloud:
    nacos:
      discovery:
        # 心跳间隔，默认5秒
        heart-beat-interval: 5000
        # 心跳超时时间，默认15秒
        heart-beat-timeout: 15000
        # 多少个心跳周期后删除实例，默认3
        heart-beat-cancel: 3
```

### 坑五：Feign超时配置不生效

**问题现象**：

配置了超时10秒，但请求超过3秒就报错了。

**根因分析**：

这是**超时配置的继承问题**。OpenFeign默认超时是60秒，如果Ribbon也配置了超时，要确保两边配置一致。

**解决方案**：

```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 10000
        readTimeout: 10000

# 同时确保Ribbon超时配置一致
ribbon:
  ConnectTimeout: 10000
  ReadTimeout: 10000
```

### 坑六：Gateway跨域配置失效

**问题现象**：

前端调用网关时报跨域错误，但配置了CorsWebFilter。

**根因分析**：

可能是**过滤器顺序**的问题。CorsWebFilter需要在其他过滤器之前执行。

**解决方案**：

```java
@Configuration
public class CorsConfig {

    @Bean
    public CorsWebFilter corsWebFilter() {
        CorsConfiguration config = new CorsConfiguration();
        config.addAllowedOriginPattern("*");
        config.addAllowedHeader("*");
        config.addAllowedMethod("*");
        config.setAllowCredentials(true);

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource(
            new PathPatternParser());
        source.registerCorsConfiguration("/**", config);

        // 设置更高的优先级
        return new CorsWebFilter(source) {
            @Override
            public int getOrder() {
                return -100;  // 优先级高于Gateway过滤器链
            }
        };
    }
}
```

### 坑七：Sentinel限流规则重启后丢失

**问题现象**：

在Sentinel Dashboard配置的限流规则，重启服务后全没了。

**根因分析**：

默认情况下Sentinel规则是存在内存里的，**没有持久化**。

**解决方案**：

将规则推送到Nacos配置中心：

```yaml
spring:
  cloud:
    sentinel:
      # 推送规则到Nacos
      datasource:
        ds:
          nacos:
            server-addr: ${NACOS_SERVER_ADDR:127.0.0.1:8848}
            data-id: sentinel-flow-rules
            group-id: SENTINEL_GROUP
            data-type: json
            rule-type: flow
```

## 总结：微服务不是银弹，但它是必经之路

2024年1月的这段学习经历，让我对微服务从「懵懂」到「入门」。回头看，有几点体会想分享：

**1. 微服务是手段，不是目的**
拆分的目的是让系统更可控、更易扩展。如果业务复杂度不高，单体反而更简单高效。老师说得好：**「能用单体解决的场景，不要强行微服务。」**

**2. 每个组件都要理解「为什么需要它」**
学Nacos要知道服务发现问题是怎么来的，学Gateway要知道统一鉴权有多重要，学Sentinel要知道高并发下系统是怎么崩的。**带着问题学组件，比孤立地背概念有效得多。**

**3. 实战项目是最好的催化剂**
黑马商城项目贯穿整个课程，每个组件都对应真实的业务场景。没有这个项目，很多概念只能停留在「知道但不会用」的层面。

**4. 分布式事务是最大的难点**
CAP、BASE、AT、TCC、Saga... 这些概念需要反复咀嚼才能理解。课程里讲得清楚，但真正搞懂还是要靠踩坑。建议多动手模拟故障场景，看看Seata怎么自动回滚的。

:::tip
微服务这条路，才刚起步。后面还有Service Mesh、云原生、Kubernetes... 每一座山都比前一座高。但只要基础打得牢，一步一步来，总会到的。
:::
