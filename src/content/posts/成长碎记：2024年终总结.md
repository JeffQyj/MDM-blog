## 站在终点的回望

2024年12月31日，又一年走到尽头。

如果要用一个词形容我的2024年，大概是「蜕变」。

2022年，我学会了写代码；2023年，我理解了工程思维；而2024年，我终于完成了从「学习者」到「开发者」的转变。这一年，不再是被动接受知识的输入，而是主动解决问题的输出。

:::note
这篇文章不求面面俱到，只想记录那些真正见证了我成长的瞬间。
:::

## 技术之路：从分布式到全栈驾驭

如果说2023年是「理解工程」，那2024年就是「驾驭工程」。

### 工程化觉醒：Spring Boot自动配置与NoSQL入门

一月的**从XML配置地狱到SpringBoot自动配置**，让我彻底理解了什么叫「约定大于配置」。

曾经的我，在SSM框架里写过无数的XML配置：数据源、事务管理器、Mapper扫描、视图解析器……每一行配置都是「显式」的，但也是「沉重」的。Spring Boot的自动配置机制，让我第一次感受到什么叫「零配置」——只需要一个starter，框架就能自动推断出你需要什么，并帮你配置好一切。

但「自动」不等于「黑盒」。学习自动配置的原理，让我明白了「条件注解」的魔法——`@ConditionalOnClass`、`@ConditionalOnProperty`、`@ConditionalOnMissingBean`……这些注解，才是自动配置的灵魂。

而**NoSQL数据库理论入门**系列，则让我打开了另一扇门。

Redis的五大数据类型，不仅仅是String、Hash、List、Set、Sorted Set的区别，而是五种截然不同的建模思维。缓存过期策略、持久化机制、主从复制……Redis不是银弹，但它确实是解决「高频读取、低频修改」场景的利器。

MongoDB的文档模型，则让我第一次理解了什么叫「 schemaless」。传统关系型数据库里，我要先设计表结构再存数据；而MongoDB里，我可以随心所欲地存储任意结构的文档。但「自由」不等于「随意」——文档设计的好坏，直接影响查询效率和存储空间。

:::note
NoSQL不是关系型数据库的替代品，而是互补品。关系型擅长「结构稳定、事务要求高」的场景；NoSQL擅长「灵活扩展、高并发读写」的场景。选对数据库，比用好数据库更重要。
:::

### 分布式架构：SOA与微服务的分野

**青橙商城**（2月）和**十次方社区**（3月），让我真正见识了大型系统的架构之美。

**青橙商城**采用的是传统SOA架构，通过Dubbo实现分布式调用，Zookeeper作为注册中心。我第一次理解了什么叫「垂直分库」——将业务拆分为八个独立的数据库，每个库负责自己的领域。这不是简单的多数据库配置，而是对业务边界的深刻理解。

而**十次方社区**则带来了Spring Cloud微服务的完整生态：Eureka注册中心、Zuul网关、Feign声明式调用、Hystrix熔断器。我第一次意识到，微服务不是简单的技术选型，而是一整套「治理」思路——服务如何被发现、如何被调用、调用失败了怎么办、超时了怎么办。

| 对比维度 | SOA架构（青橙） | 微服务架构（十次方）       |
| ---- | --------- | ---------------- |
| 通信方式 | RPC远程过程调用 | HTTP RESTful API |
| 注册中心 | Zookeeper | Eureka           |
| 熔断机制 | Dubbo内置   | Hystrix          |
| 部署方式 | 单体打包      | 容器化Docker        |

:::note
两种架构没有优劣之分，只有适用场景之别。SOA适合业务复杂、调用频繁的内部系统；微服务适合快速迭代、团队协作的互联网产品。
:::

### 微服务进阶：Spring Cloud Alibaba生态

一月的**从单体到微服务**系列，让我对微服务架构有了更体系化的理解。

Spring Cloud Alibaba并不是简单的「另一个微服务方案」，而是一套更适合国内云原生环境的解决方案。Nacos作为注册中心和配置中心，让我第一次理解了「配置中心」的价值——不需要重启服务，就能动态修改配置；Nacos的命名空间隔离，让我学会了如何在一套环境里管理多套配置。

OpenFeign的声明式调用，让跨服务调用变得像调用本地方法一样简单。而Gateway网关则让我明白了「统一入口」的意义——鉴权、限流、路由，全部在网关层完成，后端服务只需要专注于业务逻辑。

Sentinel的限流和降级，让我理解了什么叫「防御性编程」——不是等系统崩溃了再补救，而是提前做好保护措施。Seata的分布式事务，则是分布式架构中最棘手的难题之一，AT模式的「自动补偿」机制，让我第一次看到了解决分布式事务的希望。

:::note
微服务架构的核心挑战，不在于技术本身，而在于「分布式」带来的复杂性：服务发现、负载均衡、熔断降级、分布式事务……每一个都是独立的课题。
:::

### 从传统到工程化：SSM到Spring Boot

**传智健康**（4月）和**苍穹外卖**（5月），让我完成了从「能用」到「好用」的跨越。

传智健康用的是经典SSM+Freemarker，让我重新回顾了XML配置的「艺术」——那是理解Spring底层原理的必经之路。而苍穹外卖则全面拥抱Spring Boot+MyBatis-Plus，「约定大于配置」的理念让我第一次感受到什么叫「把时间花在刀刃上」。

苍穹外卖是第一个让我真正理解「业务思维」的项目。之前写代码，想的是「这个功能怎么实现」；苍穹外卖让我开始想「这个功能背后的业务逻辑是什么」。订单状态机、库存扣减策略、缓存穿透问题……每一个细节都是工程思维的体现。

### 框架二次开发：从使用者到贡献者

**帝可得**（6月）和**蝶梦**（7月），让我完成了从「消费者」到「生产者」的蜕变。

帝可得基于RuoYi-Vue3，我第一次深入源码，理解了代码生成器的魔法——只需要设计好数据库表结构，就能自动生成增删改查的完整代码。这让我开始思考：如何改造这个框架，让它更适合自己的业务？

而**蝶梦后台管理系统**，则是我独立完成的第一个真正意义上的「二次开发」项目。从需求分析到架构设计，从前端到后端，从开发到部署——全部由我一个人完成。最终，这份毕业设计获得了「优秀」的评定。

> **优秀，从来不是终点，而是「我可以做得更好」的证明。**

### 全栈探索：跨平台与全栈框架

**微信小程序**（8月）和**Next.js与Nuxt.js**（9月），让我开始探索前端跨平台开发的边界。

八月的**微信小程序原生语法学习**，让我第一次深入一个「类前端但又不是传统前端」的领域。

微信小程序的开发模式很特别：WXML不是HTML，WXSS不是CSS，JS也不是完全的JavaScript。生命周期函数、组件系统、setData的同步与异步……每一个细节都在提醒我：这是微信生态的专属开发范式。

但更重要的是，我理解了什么叫「平台限制下的开发」。微信小程序的API只能调用微信提供的功能，组件只能使用小程序支持的标签，样式只能使用WXSS的规则……在限制中找解决方案，才是真正的技术能力。

而九月的**Next.js与Nuxt.js——同门师兄弟的巅峰对决**，则让我对全栈框架有了全新的认知。

Next.js（React生态）和Nuxt.js（Vue生态），表面上只是「把React/Vue从纯前端扩展到全栈」，但实际上带来了截然不同的开发体验。SSR服务端渲染让首屏加载更快，SSG静态生成让页面可以部署在CDN上，ISR增量静态再生则在性能和实时性之间找到了平衡点。

| 对比维度 | Next.js               | Nuxt.js             |
| ---- | --------------------- | ------------------- |
| 生态   | React                 | Vue                 |
| 渲染模式 | SSR/SSG/ISR/CSR       | SSR/SSG/ISR         |
| 路由   | 文件系统路由                | 文件系统路由+约定路由         |
| 状态管理 | Context/Redux/Zustand | Pinia/Vuex          |
| API层 | Route Handlers        | Server Routes/Nitro |

那一刻我明白了：**全栈不只是「前端写后端」，而是「用统一的思维，构建前后端一体的应用」。**

:::note
从前端的微信小程序到全栈的Next.js/Nuxt.js，这一年的前端探索让我意识到：前端的边界，远比想象的要宽广。跨平台开发、全栈能力、性能优化……每一个方向都值得深耕。
:::

### 企业实训：从校园到职场的桥梁

**工匠工坊**（10月-12月），是一段从uniapp框架入门到团队协作开发的实战之旅。

这是江苏一道云科技发展有限公司的工程师走进校园带来的13周企业实训项目。与学校课程完全不同，企业导师没有照本宣科，带来的是真实的企业开发案例和实际在用的技术栈。

**三个阶段的核心收获**：

**第一阶段：框架入门**
uniapp的「配置先行」理念让我重新理解了前端开发的思路。pages.json是整个应用的「宪法」，tabBar的「五不要」是无数次踩坑后的经验总结。flex布局和rpx单位是跨平台适配的必备知识。

**第二阶段：独立实战**
独立完成「每日打卡」小程序，让我彻底意识到：「学过了」和「做出来了」之间，隔着一道认知偏差的鸿沟。本地存储的同步与异步问题、日期对象的时间陷阱、页面间数据传递……这些问题只有在真正动手时才会遇到。

**第三阶段：团队协作**
这是第一次真正体验「企业级开发流程」。Git分支管理、接口文档约定、Code Review、每日站会……当代码不再只属于自己，当每个人都是平等的贡献者，沟通能力和协作意识变得与技术能力一样重要。

## 遗憾与庆幸

如果要挑一个遗憾，大概是：这一年的自己，有时候还是太追求「量」了。

六个企业级项目加上工匠工坊的全部阶段，平均每月都有新的任务。回头看，有些项目的深度挖掘不够，有些技术点只是「会用」而非「理解」。数量上来了，质量却参差不齐。

但比起遗憾，更多的是庆幸。

庆幸自己没有被「技术焦虑」吞噬，而是一个项目一个项目地啃了下来。庆幸遇到了毕设导师，让我在蝶梦项目中实现了真正的独立开发。庆幸遇到了工匠工坊的企业导师，让我提前体验了企业开发的流程。庆幸每一个debug的深夜都没有白费，它们都在塑造着一个更专业的我。

:::note
2024年的我，比2023年更成熟了一点。但学得越多，越觉得自己无知。这不是谦虚，而是事实。
:::
