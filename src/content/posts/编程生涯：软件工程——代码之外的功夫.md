## 故事的起点：为什么我的代码总是「能用但不好用」？

2025年9月，大三开学。

学期伊始，软件工程这门课赫然出现在了课表里。说实话，一开始我对这门课没抱太大期望——「不就是教人怎么写文档、怎么画图的课吗？」彼时的我，已经能熟练写出用户模块、订单模块、支付模块，自认为「实战经验」还算丰富。

但课上老师提了一个问题，直接把我问懵了：

> **「你们写的代码，有多少是给别人维护的？别人能看懂吗？」**

我沉默了三秒。

然后我发现——我的代码，别说给别人维护了，就是让自己三个月后再看，大概率也得懵。

这就是我软件工程学习之路的起点。

## 软件工程是什么？

### 书本上的定义

翻开软件工程的教材，第一章往往是这样写的：

> **软件工程（Software Engineering）是一门研究用工程化方法构建和维护有效、实用、高质量软件的学科。**

听起来很正确，但也很空洞。

「工程化方法」是什么？「构建」和「维护」怎么做？「有效、实用、高质量」又怎么衡量？

老师打了个比方，我一下子记住了：

> **软件工程，就是教你「怎么系统地做软件」的学问。如果写代码是「烹饪」，那软件工程就是「开餐厅」——你得考虑选址、装修、菜单设计、人员管理、食品安全、财务流水……光会炒菜可不行。**

这个比喻太生动了。我们之前写的那些项目，就像「家庭厨房」——自己吃，做完就行。但如果是要「开餐厅」，那要考虑的东西可就多了。

### 软件工程的核心问题

软件工程要解决的，是软件开发的三大核心问题：

| 问题 | 表现 | 类比 |
|------|------|------|
| **软件危机** | 项目延期、超预算、bug多、维护困难 | 餐厅开业一拖再做、装修超支、菜谱难吃、服务差 |
| **质量与效率的矛盾** | 赶工就出bug，不赶工就延期 | 想快速上菜就容易出事故，慢慢做客人又等不及 |
| **需求变化** | 改着改着就面目全非 | 客人吃着吃着要换菜，吃着吃着要换厨师 |

软件工程的各种方法论、模型、工具，都是为了解决这些问题。

## 软件生命周期：软件也有生老病死

### 四阶段循环

软件工程告诉我们：软件也有自己的「生命周期」。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           软件生命周期四阶段                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│      ┌─────────────┐                                                       │
│      │   需求分析   │ ←─ 软件要「做什么」                                   │
│      └──────┬──────┘                                                       │
│             │                                                                │
│             ▼                                                                │
│      ┌─────────────┐                                                       │
│      │     设计     │ ←─ 软件要「怎么做」                                   │
│      └──────┬──────┘                                                       │
│             │                                                                │
│             ▼                                                                │
│      ┌─────────────┐                                                       │
│      │    编码     │ ←─ 把设计「实现出来」                                  │
│      └──────┬──────┘                                                       │
│             │                                                                │
│             ▼                                                                │
│      ┌─────────────┐                                                       │
│      │     测试     │ ←─ 验证软件「做得对不对」                              │
│      └──────┬──────┘                                                       │
│             │                                                                │
│             │  ────────────────── 维护阶段 ──────────────────               │
│             │        │                    │                    │            │
│             ▼        ▼                    ▼                    ▼            │
│      ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐              │
│      │  纠错性   │ │  适应性   │ │  完善性   │ │  预防性   │              │
│      │  维护     │ │  维护     │ │  维护     │ │  维护     │              │
│      └───────────┘ └───────────┘ └───────────┘ └───────────┘              │
│                                                                             │
│   📌 需求分析和设计是「上游」，上游出问题，下游代价成倍增加                   │
│   📌 维护阶段占软件生命周期的大部分时间（约60%-80%）                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 我的理解：为什么维护这么重要？

课程里说，维护阶段占软件生命周期的大部分时间。一开始我不理解——代码写完了，不就没了吗？维护什么？

老师说了一个数据：**一个典型的软件系统，80%的成本花在维护上。**

然后他举了个例子：

> **你写的「图书管理系统」交给图书馆老师用了。三个月后，图书馆要换一批新书，要更新书库目录——你得改代码吧？一年后，图书馆要增加「预约功能」——你得改代码吧？两年后，Java版本升级了，你的代码跑不起来了——你得改代码吧？**

每次「改代码」，都是在维护。而这些维护工作，往往比开发新功能更费时费力——因为你得先理解原来的代码，才能改。万一原来的代码写得很烂（通常是），那维护简直是噩梦。

:::note
**课程设计时的感悟**：大二做课设，我从来不写注释，变量名都是`a`、`b`、`c`。结果验收的时候，老师让我讲解代码，我自己都愣了半天。这大概就是「自己写的代码自己都看不懂」的最佳写照。
:::

## 开发模型：软件是怎么「做出来」的？

### 瀑布模型：老老实实按顺序来

课程里第一个讲的是**瀑布模型**（Waterfall Model）。这个名字很形象——像瀑布一样，从上往下，一级一级流下去。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              瀑布模型                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌──────────┐                                                             │
│   │ 需求分析  │                                                             │
│   └────┬─────┘                                                             │
│        │  ──── 文档驱动 ────                                                │
│        ▼                                                                    │
│   ┌──────────┐                                                             │
│   │   设计    │                                                             │
│   └────┬─────┘                                                             │
│        │  ──── 文档驱动 ────                                                │
│        ▼                                                                    │
│   ┌──────────┐                                                             │
│   │   编码    │                                                             │
│   └────┬─────┘                                                             │
│        │  ──── 文档驱动 ────                                                │
│        ▼                                                                    │
│   ┌──────────┐                                                             │
│   │   测试    │                                                             │
│   └────┬─────┘                                                             │
│        │  ──── 文档驱动 ────                                                │
│        ▼                                                                    │
│   ┌──────────┐                                                             │
│   │   运维    │                                                             │
│   └──────────┘                                                             │
│                                                                             │
│   📌 每个阶段「必须」完成后才能进入下一阶段                                  │
│   📌 文档是阶段之间的「契约」                                              │
│   📌 优点：阶段清晰、易于管理；缺点：不适应需求变化                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

瀑布模型的优点是**阶段清晰、易于管理**。每个阶段做什么、交付什么文档，都明明白白。

但它的缺点也很明显：**不适应需求变化**。

> **想象你按瀑布模型开发一个点餐系统，需求分析做了两个月，设计做了两个月，编码做了两个月……结果到测试阶段，用户说「我们不想做点餐系统了，想做外卖平台」。这时候，你前面半年的工作基本白费。**

### 敏捷开发：拥抱变化，快速迭代

正因为瀑布模型的「不灵活」，后来出现了**敏捷开发**（Agile）。

敏捷的核心思想是四个字：**「拥抱变化」**。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            敏捷开发流程                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   Product Backlog (产品待办列表)                                            │
│          │                                                                   │
│          ▼                                                                   │
│   ┌─────────────┐     ┌─────────────┐     ┌─────────────┐                 │
│   │  Sprint 1   │     │  Sprint 2   │     │  Sprint N   │                 │
│   │  迭代1      │ ──▶ │  迭代2      │ ──▶ │  迭代N      │                 │
│   └──────┬──────┘     └──────┬──────┘     └──────┬──────┘                 │
│          │                    │                    │                        │
│          ▼                    ▼                    ▼                        │
│   ┌─────────────┐     ┌─────────────┐     ┌─────────────┐                 │
│   │  增量交付   │     │  增量交付   │     │  增量交付   │                 │
│   │  可用软件   │     │  可用软件   │     │  可用软件   │                 │
│   └─────────────┘     └─────────────┘     └─────────────┘                 │
│                                                                             │
│   📌 每个Sprint（冲刺）通常1-4周                                            │
│   📌 每个Sprint结束都要交付「可工作的软件」                                 │
│   📌 根据用户反馈调整Product Backlog                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

敏捷开发的代表是**Scrum**和**XP（极限编程）**。

课程里用了一个比喻：

> **瀑布模型像是「建一栋大楼」——设计图纸全部完成后再施工。敏捷开发像是「做一顿饭」——先做一道上一道，边做边调整口味。**

### 两种模型的对比

| 维度 | 瀑布模型 | 敏捷开发 |
|------|---------|---------|
| 适用场景 | 需求明确、变化少 | 需求不确定、变化多 |
| 文档 | 重视文档 | 重视可工作的软件 |
| 客户参与 | 初期确定，后期少参与 | 全程参与，持续反馈 |
| 风险 | 后期才发现问题，风险大 | 早期交付，风险分散 |
| 灵活性 | 低 | 高 |

:::note
**课程设计时的感悟**：我们做课设基本都用的瀑布模型——先写需求文档，再写设计文档，然后编码，最后测试。但实际做的时候，往往是「写文档给老师看，代码自己随便写」。结果就是文档和代码两张皮，文档写的是「用户模块」，代码里连个注释都没有。
:::

## 需求分析：搞清楚「做什么」比「怎么做」更重要

### 需求分析的三个层次

课程里把需求分为三个层次：

| 层次 | 内容 | 类比 |
|------|------|------|
| **业务需求** | 组织为什么要做这个系统 | 「餐厅要赚钱」 |
| **用户需求** | 用户想用系统做什么 | 「客人要点餐、要排队」 |
| **系统需求** | 系统要实现哪些功能 | 「系统要有菜单展示、订单管理、支付功能」 |

我之前写项目，从来不区分这些。上來就开始想「数据库怎么设计」「接口怎么写」。

老师说：**这是典型的「技术思维」，而不是「业务思维」。**

正确的顺序应该是：

1. 先搞清楚「业务要解决什么问题」
2. 再搞清楚「用户在使用场景中需要什么」
3. 最后才是「系统怎么实现」

### 需求分析的利器：UML用例图

课程里讲了UML（统一建模语言），其中最实用的就是**用例图**（Use Case Diagram）。

用例图的核心是三个元素：**参与者（Actor）**、**用例（Use Case）**、**系统边界**。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           点餐系统用例图                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│    ┌─────────────────────────── 系统边界 ────────────────────────────┐     │
│    │                                                               │     │
│    │   ┌──────────────┐      ┌──────────────┐                     │     │
│    │   │  浏览菜单     │      │   下单       │                     │     │
│    │   └──────────────┘      └──────┬───────┘                     │     │
│    │          │                      │                            │     │
│    │          │                      │                            │     │
│    │   ┌──────┴───────┐      ┌──────┴───────┐                     │     │
│    │   │   查看菜品详情 │      │   取消订单   │                     │     │
│    │   └──────────────┘      └──────────────┘                     │     │
│    │                                                               │     │
│    └───────────────────────────────────────────────────────────────┘     │
│                   │                    │                               │
│                   │                    │                               │
│              ┌────┴────┐          ┌────┴────┐                          │
│              │  顾客   │          │  商家   │                          │
│              └─────────┘          └─────────┘                          │
│                                                                             │
│   📌 参与者（Actor）是小人图标，表示与系统交互的人或事物                    │
│   📌 用例是椭圆图标，表示系统提供的功能                                    │
│   📌 系统边界是方框，框住所有用例                                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

当时课程要求我们画一个系统的用例图。我选的是「在线考试系统」。

一开始我画出来的用例图只有三个用例：「登录」「考试」「交卷」。老师说：「太简单了，你把系统想成只有三个功能的黑盒。」

后来在老师的引导下，我才发现一个考试系统要考虑这么多：

- 考生要「注册」「登录」「修改密码」
- 考生要「选择试卷」「开始考试」「答题」「保存草稿」「交卷」
- 考生要「查看成绩」「查看错题」
- 管理员要「创建试卷」「添加题目」「管理考生」

光是「答题」一个用例，背后就有「单选题」「多选题」「判断题」「填空题」「简答题」等不同题型要考虑。

**这就是需求分析的价值——在写代码之前，把「要做什么」想清楚。**

### 需求文档怎么写？

课程里给了需求文档的模板，其中最重要的两个部分是：

**1. 功能性需求（What）：系统要做什么**

```markdown
### 用例1：用户登录

**描述**：允许已注册用户登录系统

**参与者**：已注册用户

**前置条件**：用户已注册且未登录

**基本流程**：
1. 用户输入用户名和密码
2. 系统验证凭证
3. 验证成功，用户登录系统

**备选流程**：
- 2a. 用户名不存在 → 提示「用户不存在」
- 2b. 密码错误 → 提示「密码错误」，错误次数+1
- 2c. 错误次数>=5 → 提示「账号锁定，请15分钟后再试」

**后置条件**：用户成功登录，跳转到首页
```

**2. 非功能性需求（How Well）：系统要做到什么程度**

| 类型 | 示例 | 衡量标准 |
|------|------|---------|
| **性能需求** | 系统响应要快 | 页面加载不超过3秒 |
| **可靠性需求** | 系统要稳定 | 可用性99.9% |
| **安全性需求** | 数据要安全 | 密码加密存储，传输加密 |
| **易用性需求** | 要好上手 | 新用户培训不超过2小时 |

:::note
**课程设计时的感悟**：之前写课设，需求文档都是「借鉴」网上的模板，随便改改交上去。现在才知道，需求文档是团队沟通的契约，写不清楚，后续开发就是鸡同鸭讲。
:::

## 系统设计：把「做什么」变成「怎么做」

### 架构设计：高层蓝图

需求分析回答的是「做什么」，系统设计回答的是「怎么做」。

系统设计分为几个层次：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           系统设计层次                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                         ┌───────────────┐                                   │
│                         │   架构设计   │ ←─ 最高层，决定技术选型            │
│                         └───────┬───────┘                                   │
│                                 │                                            │
│                         ┌───────▼───────┐                                   │
│                         │   详细设计   │ ←─ 具体到类、接口、算法             │
│                         └───────────────┘                                   │
│                                                                             │
│   架构设计关注「整体」，详细设计关注「局部」                                 │
│   架构设计决定「能不能扩展」，详细设计决定「好不好维护」                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

架构设计要考虑的问题包括：

- 系统分成哪几个模块
- 模块之间怎么通信
- 用什么技术栈
- 怎么保证性能、安全、可扩展

课程里以一个「校园外卖系统」为例，画了一张架构图：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        校园外卖系统架构图                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                          用户端 App / 小程序                                 │
│                               │                                              │
│                               ▼                                              │
│                    ┌───────────────────┐                                   │
│                    │     API 网关      │ ←─ 统一入口，鉴权、限流            │
│                    └─────────┬─────────┘                                   │
│                              │                                               │
│         ┌────────────────────┼────────────────────┐                          │
│         │                    │                    │                         │
│         ▼                    ▼                    ▼                         │
│   ┌───────────┐        ┌───────────┐        ┌───────────┐                  │
│   │  用户服务  │        │  商家服务  │        │  订单服务  │                  │
│   └─────┬─────┘        └─────┬─────┘        └─────┬─────┘                  │
│         │                    │                    │                         │
│         └────────────────────┼────────────────────┘                          │
│                              │                                               │
│                              ▼                                               │
│                    ┌───────────────────┐                                   │
│                    │     MySQL         │ ←─ 主数据库                       │
│                    └───────────────────┘                                   │
│                                                                             │
│                              │                                               │
│                              ▼                                               │
│                    ┌───────────────────┐                                   │
```

### 数据库设计：数据的「户型图」

如果说架构设计是「楼盘规划」，那数据库设计就是「户型图」。

课程里讲了数据库设计的三大范式（NF）：

| 范式 | 要求 | 类比 |
|------|------|------|
| **1NF** | 字段不可再分 | 每个房间有固定功能，不能「厨房卧室一体化」 |
| **2NF** | 消除部分依赖 | 每个房间只属于一个「套间」，不出现「主卧在A套间又属于B套间」 |
| **3NF** | 消除传递依赖 | 房间的直接归属要明确，不能「通过套间间接属于楼盘」 |

我之前设计数据库，从来不考虑什么范式。看到一个需求，恨不得把所有字段都塞进一张表里。

结果就是：数据冗余严重、更新一条信息要改好几处、关系混乱难以维护。

后来课上学了范式，才知道**「规范化」**的好处：

```sql
-- 错误的设计：把所有信息塞进一张表
-- 用户表
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    phone VARCHAR(20),
    province_name VARCHAR(50),    -- 省名
    city_name VARCHAR(50),         -- 市名
    district_name VARCHAR(50),      -- 区名
    school_name VARCHAR(100),      -- 学校名
    school_address VARCHAR(200),   -- 学校地址
    ...
);
-- 问题：同一个省/市/区/学校名会重复存储，造成冗余

-- 正确的设计：拆分成多张表
-- 用户表
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    phone VARCHAR(20),
    province_id INT,
    city_id INT,
    district_id INT,
    school_id INT,
    FOREIGN KEY (province_id) REFERENCES province(id),
    FOREIGN KEY (city_id) REFERENCES city(id),
    FOREIGN KEY (district_id) REFERENCES district(id),
    FOREIGN KEY (school_id) REFERENCES school(id)
);

-- 地区表
CREATE TABLE province (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);

CREATE TABLE city (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    province_id INT,
    FOREIGN KEY (province_id) REFERENCES province(id)
);
-- ...以此类推
```

规范化的代价是查询时要JOIN多张表，但换来的是**数据一致性和维护性的大幅提升**。

## 测试：软件质量的「守门员」

### 为什么测试这么重要？

课程里讲了一个经典的「软件测试三大定律」：

> **1. 测试是证明软件有bug的手段，而不是证明软件没有bug的手段**
> **2. 测试用例要设计得能发现bug，而不是重复已通过的用例**
> **3. 写测试用例的代价远小于修复线上bug的代价**

我之前写代码，从来不写测试。写完跑一遍，没报错，就提交了。

老师说：**「你跑一遍没报错，只能说明你没有故意制造bug，不能说明你的代码是对的。」**

想想还真是。我写的那些「没报错」的代码，真要仔细测起来，边界条件漏洞百出：

- 输入负数，崩溃了
- 输入空字符串，崩溃了
- 并发访问，数据错了
- 数据库连接超时，没做异常处理……

### 测试的分类

课程里把测试分为几个层次：

| 测试类型 | 测试对象 | 测试目的 | 类比 |
|---------|---------|---------|------|
| **单元测试** | 最小功能单元（函数、方法） | 每个零件是否正常 | 检查螺丝钉是否合格 |
| **集成测试** | 模块之间的交互 | 组装后是否协调 | 检查螺丝钉拧进主板是否匹配 |
| **系统测试** | 整个系统 | 整体功能是否达标 | 检查整台电脑是否能开机 |
| **验收测试** | 用户实际使用场景 | 是否满足用户需求 | 用户试用，确认能办公 |

### 单元测试怎么写？

课程里以Java为例，讲了JUnit单元测试的写法：

```java
// 被测试类：计算器
public class Calculator {
    public int add(int a, int b) {
        return a + b;
    }

    public int divide(int a, int b) {
        if (b == 0) {
            throw new IllegalArgumentException("除数不能为0");
        }
        return a / b;
    }
}
```

```java
// 测试类
public class CalculatorTest {

    @Test
    // 正常情况测试
    public void testAdd() {
        Calculator calculator = new Calculator();
        assertEquals(5, calculator.add(2, 3));  // 期望5，实际2+3
        assertEquals(0, calculator.add(-1, 1));
        assertEquals(-5, calculator.add(-2, -3));
    }

    @Test
    // 边界情况测试
    public void testDivide() {
        Calculator calculator = new Calculator();
        assertEquals(2, calculator.divide(6, 3));
        assertEquals(0, calculator.divide(1, 2));  // 整数除法，向下取整
    }

    @Test(expected = IllegalArgumentException.class)
    // 异常情况测试：期望抛出除零异常
    public void testDivideByZero() {
        Calculator calculator = new Calculator();
        calculator.divide(1, 0);  // 应该抛异常
    }
}
```

写单元测试的时候，要考虑几种情况：

| 情况类型 | 说明 | 示例 |
|---------|------|------|
| **正常情况** | 典型输入，正常返回 | `add(2, 3) = 5` |
| **边界情况** | 输入在边界值附近 | `add(Integer.MAX_VALUE, 1)` |
| **异常情况** | 输入非法，处理是否得当 | `divide(1, 0)` 抛异常 |

:::warning
**课程设计时的感悟**：之前觉得写测试是「浪费时间」，代码写完就完事了。但真正上线出问题的时候，才发现测试的价值。测试不是为了证明代码没问题，而是为了**尽早发现潜在问题**。越早发现，修复成本越低。
:::

## 项目管理：让一群人高效地做软件

### 软件工程师不只是写代码

课程里有一章讲项目管理，让我对「团队协作」有了新的认识。

以前我觉得：软件工程师嘛，就是写代码。一群人坐在一起，各写各的模块，最后拼起来就行了。

但实际做项目才发现，**沟通成本往往比编码成本还高**：

- A写的接口，B说「不是你这样设计的」
- C改了一个公共类，导致D的代码编译不通过
- E花了两周做的功能，发现和F做的功能完全重复
- 项目 deadline 到了，还有三分之一的代码没写完

## 软件工程与我的关系

学完软件工程，我最大的改变是：**开始用「系统思维」去看待软件开发**。

以前的我，写代码是「直线型思维」：

```
想要什么功能 → 直接开写 → 写完就跑 → 跑完就交
```

现在的我，写代码前会多想几步：

- **这个功能，用户真的需要吗？** → 需求分析
- **这个设计，以后能扩展吗？** → 架构设计
- **这个实现，会不会有什么边界情况没考虑到？** → 测试思维
- **这段代码，三个月后我能看懂吗？** → 可维护性

| 维度 | 以前的我 | 现在的我 |
|------|---------|---------|
| 写代码前 | 直接开写 | 先画用例图，理清需求 |
| 设计时 | 边写边想 | 先画架构图，再动手 |
| 写代码时 | 差不多就行 | 考虑边界，加注释 |
| 写完代码 | 直接提交 | 写单元测试，检查边界 |
| 交付时 | 能跑就行 | 检查文档是否齐全 |

## 总结：软件工程是「道」，编程是「术」

2025年10月，学完软件工程这门课，我对「软件工程」这四个字有了更深的理解。

编程能力，决定的是「你能不能把代码写出来」。

软件工程能力，决定的是「你能不能把软件做出来，并且做好」。

这两个能力，缺一不可。

**代码写得好，软件不一定做得好。**

但软件做得好的人，代码一定不会差到哪里去。

因为软件工程教会我的，不只是那些 UML 图、那些文档模板、那些测试用例——

它教会我的，是**「在动手之前先思考，在实现之后验证」**的工程思维。

这种思维，比任何编程语言、任何框架、任何工具，都更重要。

:::tip
软件工程这条路，我才刚入门。后面还有架构设计、DevOps、技术债务管理……每一座山都比前一座高。但只要基础打得牢，一步一步来，总会到的。
:::
