2026年的今天，AI工具已经能帮你写代码、调试、测试甚至部署。但有一个东西是AI无法替代你完成的——**做出技术选型的决策**。

而决策的前提是认知。这篇文章不教你深入任何一项技术，只帮你建立一张足够完整的技术地图。

## 认知，才是AI时代的稀缺资源

我第一次听说"跨端开发"这个词，是在B站刷到的一个技术视频里。那条视频很短，讲的是Flutter的一个介绍，我记得弹幕里有人刷"下次一定"。

后来我又刷到了讲Tauri的视频——"用Rust写桌面应用，包体积才10MB？"——我在心里记下了。

再后来，uni-app、Taro、React Native、Kuikly……一个又一个视频，像碎片一样散落在我的信息流里。

**我对这些技术的了解是碎片化的。并没有系统化的整合认知**

AI可以帮我写代码，但它不会告诉我"这个项目适合用Flutter还是Tauri"。因为AI不知道我的业务场景、团队背景、用户规模。**决策，永远是人做的。**

这篇文章，就是把我从各个视频里碎片化认知到的内容，做一次系统性的整合。希望你读完之后，也能建立一张完整的技术地图。

这就是为什么，在AI时代，认知的广度比深度更值钱。

## 全端技术地图：一张图看清互联网全貌

互联网产品按照运行平台划分，可以分为五大端：

| 端        | 典型入口           | 技术特征                        |
| -------- | -------------- | --------------------------- |
| **Web端** | 浏览器            | H5、React、Vue、Next.js        |
| **PC端**  | 桌面应用           | Electron、Tauri、Qt           |
| **小程序端** | 微信/支付宝/抖音      | uni-app、Taro、Flutter        |
| **移动端**  | App Store/应用市场 | Flutter、React Native、Kotlin |
| **CLI端** | 终端             | Node.js、Bun、Deno            |

下面展开每个端的核心框架认知。

## Web端：浏览器是最通用的入口

Web端是互联网最古老的端，也是你大概率最熟悉的端。

### 主流技术栈（2026）

| 框架          | 定位        | 语言         | 特点                   |
| ----------- | --------- | ---------- | -------------------- |
| **Next.js** | React全栈框架 | TypeScript | SSR/SSG、API Routes   |
| **Nuxt**    | Vue全栈框架   | TypeScript | SSR/SSG、auto-imports |
| **Astro**   | 内容优先静态站   | TypeScript | Island架构、零JS默认       |
| **Remix**   | Web基础理念框架 | TypeScript | 渐进增强、嵌套路由            |

### 认知要点

- **Next.js统治地位**：如果你要做React技术栈的Web应用，Next.js几乎是默认选择。它的Vercel部署生态也很成熟
- **Astro的崛起**：内容型网站（博客、文档、官网）越来越多人选择Astro，性能碾压传统SSR——因为它默认不加载JavaScript
- **SSR vs SSG 的选择**：如果你的内容频繁变化，用SSR（服务器渲染）；如果你的内容几乎不变，用SSG（静态生成）——SSG性能更好，托管也更简单
- **边缘计算普及**：Vercel、Cloudflare Pages推动前端进入边缘时代——你的代码可以部署到离用户最近的服务器上，延迟更低

:::note

**认知行动**：如果你只熟悉Vue或React其中一种，建议花半天时间快速浏览另一种的官方文档，建立"这个东西存在"的认知。你不需要深入，但需要知道它能做什么。

:::

聊完Web端，接下来聊聊pc端。

## PC端：桌面应用的归途

PC端（桌面应用）是最被我忽视的端，因为学校根本没有提及过相关内容。

### 主流技术框架

| 框架           | 内核           | 语言      | 包体积     | 适用场景         |
| ------------ | ------------ | ------- | ------- | ------------ |
| **Electron** | Chromium     | JS/TS   | \~150MB | 复杂交互、Web技术复用 |
| **Tauri**    | Rust WebView | Rust/JS | \~10MB  | 轻量工具、性能敏感    |
| **Qt**       | 原生           | C++     | 视平台     | 复杂原生GUI、专业软件 |

### Tauri为什么值得关注

Electron最大的问题是包体积——一个最简单的Electron应用，打包出来也要150MB起步。而Tauri使用系统原生WebView，包体积可以压缩到10MB以内。

简单理解这个差异：**Electron像是自带厨房的快餐车，走到哪都要拖着整个厨房跑；Tauri则是借用当地的厨房，只把自己的食材和菜谱带过去。**

### 认知要点

- **Electron适合**：需要复杂UI、已有Web技术栈、团队前端背景。典型应用：VS Code、Slack、Postman
- **Tauri适合**：在意包体积、性能敏感、想用现代前端框架。典型应用：工具类产品、轻量级客户端
- **Qt适合**：专业级原生应用、游戏、复杂图形处理。典型应用：Adobe全家桶、AutoCAD、3D建模软件
- **一个趋势**：越来越多的工具类应用从Electron转向Tauri，比如Figma收购前就在用Tauri做轻量化探索

:::note

**认知行动**：下次你在电脑上装了一个"动不动就占1GB内存"的App，去查查它是用什么技术做的。这是一种很好的学习方式。

:::

说完了桌面端，再来看看国内特有的生态——小程序。

## 小程序端：国内特有的流量战场

小程序是国内特有的生态，微信、支付宝、抖音、百度、QQ、钉钉各有自己的小程序平台。

### 主流开发框架对比

| 框架          | 出品方    | 语言        | 跨平台能力               | 生态成熟度 |
| ----------- | ------ | --------- | ------------------- | ----- |
| **uni-app** | DCloud | Vue/JS    | 全平台小程序+H5+App       | 成熟    |
| **Taro**    | 京东     | React/Vue | 小程序+H5+React Native | 成熟    |
| **Flutter** | Google | Dart      | 需单独处理小程序            | 一般    |

### uni-app的核心优势

uni-app最大的特点是**一套代码，十端运行**：

- iOS、Android
- H5（响应式）
- 微信、支付宝、抖音、百度、QQ、钉钉小程序
- 快应用

你可以把它理解为一个"自动翻译员"——你写一套话术，它能自动转写成十种不同平台的方言，而且听起来都很地道。

### 认知要点

- **uni-app适合**：需要快速覆盖国内所有小程序平台、团队熟悉Vue。它的生态很成熟，插件市场丰富
- **Taro适合**：React技术栈、需要同时兼顾小程序和React Native的场景。Taro3.x之后支持编译到React Native
- **Flutter做小程序**：Flutter官方对小程序支持不友好，需要用第三方方案（如uni-app集成Flutter），不是Flutter的主战场
- **原生开发适合**：小程序只作为补充入口、核心业务在App。原生开发能获得最好的平台特性支持

**小程序的本质**：小程序不是替代App的产物，而是App的"轻量版"。它的核心价值是**流量入口**——用户不需要下载，就能体验核心功能。微信、支付宝、抖音这些超级App就是最大的流量分发渠道。

:::note

**认知行动**：在你的认知地图里，记住"uni-app"和"Taro"这两个名字。下次有人问"小程序怎么做多端"，你可以接上话。

:::

小程序解决了"流量入口"的问题，但如果你的产品需要更深度的用户交互，那就需要原生App了。

## 移动端：原生与跨端的永恒博弈

移动端（iOS/Android/鸿蒙）是用户时长最大的端，也是技术选型最复杂的端。

### 2026年跨端框架格局

| 框架               | 引擎         | 语言       | 性能  | 生态  |
| ---------------- | ---------- | -------- | --- | --- |
| **Flutter**      | Skia自绘     | Dart     | 极高  | 成熟  |
| **React Native** | JSI/Fabric | JS/React | 高   | 庞大  |
| **KMP**          | 原生组件       | Kotlin   | 原生级 | 增长中 |
| **Kuikly**       | 原生组件       | Kotlin   | 原生级 | 新兴  |

### Flutter：性能天花板

Flutter的核心优势是**自绘引擎**——不依赖平台原生UI组件，在任何平台渲染出一致的像素级效果。

你可以把Flutter理解为一个"自带画布的画家"——它不借用任何平台的UI组件，而是自己画出一个像素不差的界面。所以无论在iOS还是Android上，效果都是一模一样的。

**Flutter的软肋**：

- 包体积：Release包Android端约10MB起步
- 平台特有能力：调用相机、蓝牙、推送等需要Platform Channel
- Dart语言：前端团队有学习成本
- 鸿蒙支持：目前官方支持进度较慢

### KMP：原生组件复用方案

说完Flutter的自绘方案，再来看看另一种思路——**Kotlin Multiplatform（KMP）**。

KMP的核心思路是**逻辑复用，UI独立**。什么意思呢？

简单来说，KMP允许你用同一套Kotlin代码编写业务逻辑（如网络请求、数据处理、工具类），然后在各个平台上复用。而UI层则由各平台自行实现，保持原生体验。

你可以把它理解为一个**擅长翻译的业务主管**——他把核心业务流程翻译成各平台能理解的语言，但具体怎么展示由各平台的UI团队决定。

**为什么KMP值得关注**：

- Google官方在2024年开始支持Kotlin Multiplatform，2026年已经成为Kotlin生态的重要组成部分
- 与Flutter的"自己画"不同，KMP是"借用平台原生的画"
- 性能接近原生，因为底层就是原生组件
- 适合需要**原生体验 + 逻辑复用**的场景

**KMP vs Flutter的核心区别**：

| 维度 | Flutter | KMP |
|-----|---------|-----|
| **UI方案** | 自绘引擎，完全一致 | 借用原生，各平台有差异 |
| **性能** | 高（自绘优化） | 原生级 |
| **学习成本** | Dart语言 | Kotlin语言 |
| **适用场景** | UI高度定制、一致性要求高 | 逻辑复杂、UI要求原生 |

### Kuikly：腾讯的新棋子

Kuikly是腾讯基于Kotlin MultiPlatform推出的跨端框架，2026年正在快速崛起。

它的核心思路和Flutter不同——不是自己画界面，而是**借用平台原生的UI组件**。相当于一个"翻译官"，把你的Kotlin代码翻译成各平台原生的表达方式。

**核心特点**：

- 编译为平台原生二进制（.aar/.framework/.so），性能接近原生
- SDK极小：Android约300KB，iOS约1.2MB
- 原生UI渲染，无桥接延迟
- 支持Android/iOS/鸿蒙/Web/小程序/macOS

### 鸿蒙：不得不提的变量

在国内做移动端开发，2024年之后鸿蒙是一个**无法回避的话题**。

鸿蒙（HarmonyOS）是华为推出的操作系统，到2026年已经迭代到HarmonyOS Next版本。用户规模突破3亿，成为仅次于Android和iOS的第三大移动操作系统。

对于开发者来说，鸿蒙意味着：

| 维度       | Android/iOS       | 鸿蒙                 |
| -------- | ----------------- | ------------------ |
| **开发语言** | Java/Kotlin、Swift | ArkTS（类TypeScript） |
| **开发框架** | 原生开发、Flutter、RN   | 原生开发、Kuikly        |
| **生态定位** | 成熟生态              | 快速成长中              |

**关键认知**：如果你的产品目标用户在国内，鸿蒙是不可忽视的渠道。目前主流跨端框架对鸿蒙的支持程度参差不齐——Kuikly对鸿蒙的支持最为完善，Flutter官方支持进度较慢，React Native则需要额外适配。

### 认知要点

- **Flutter适合**：新项目、UI高度定制、团队接受Dart成本。
- **React Native适合**：前端团队、已有React技术栈、需要快速交付。
- **KMP/Kuikly适合**：团队有Kotlin背景、需要原生性能+逻辑复用。
- **原生开发适合**：性能敏感、需要平台全部特性、团队完整。

**技术选型的本质**：没有最好的框架，只有最适合当前场景的选择。创业团队用Flutter快速验证，成熟产品可能切回原生追求极致性能，这都是合理的决策。

讲完了面向用户的产品端，还有一类工具是面向开发者的——CLI。

## CLI端：被低估的效率战场

CLI（命令行工具）是开发者的专属端，也是AI辅助编程的主战场。

### 主流运行时对比

| 运行时         | 语言                    | 启动速度 | 生态  | 适用场景       |
| ----------- | --------------------- | ---- | --- | ---------- |
| **Node.js** | JavaScript            | 慢    | 庞大  | 企业应用、npm生态 |
| **Bun**     | JavaScript/TypeScript | 极快   | 新兴  | 性能敏感、现代CLI |
| **Deno**    | JavaScript/TypeScript | 中    | 发展中 | 安全敏感应用     |

### CLI框架选择

| 框架            | 特点             | 适合场景         |
| ------------- | -------------- | ------------ |
| **oclif**     | Node.js官方CLI框架 | 企业级CLI、复杂子命令 |
| **commander** | 轻量级参数解析        | 简单CLI工具      |
| **ink**       | React组件构建CLI   | 交互式CLI界面     |

### Claude Code的启示

2026年最火的CLI工具是Claude Code——一个AI驱动的全流程编程助手。它的架构选择很有意思：

- 选择**Bun**作为运行时（启动速度至关重要）
- 选择**TypeScript**原生支持（减少构建复杂度）
- 选择**分层架构**（服务层、业务层、UI层分离）

这给我们的启示是：CLI工具不需要花哨的界面，但需要一个**高性能的内核**。对于普通用户而言，CLI可能是陌生的；但对于开发者而言，CLI是每天都要打交道的工具。

### 认知要点

- **CLI是AI的主战场**：Claude Code、Cursor、Windsurf这些工具都是CLI或以CLI为核心。AI辅助编程的第一入口是终端，不是图形界面
- **Bun正在崛起**：如果你要开发新CLI工具，Bun是值得考虑的选择——启动速度快、TypeScript原生支持、包体积小
- **交互式CLI**：用React/Ink构建的CLI可以有彩色输出、进度条、交互表单——CLI不一定是黑白文字
- **Node.js依然重要**：虽然Bun快，但npm庞大的生态是Bun暂时无法替代的。企业内部工具、成熟项目，Node.js依然是稳妥选择

:::note

**认知行动**：安装一个CLI工具体验，比如Claude Code或Bun。感受一下现代CLI的设计，理解"原来命令行也可以这么做"。

:::

## 认知地图的使用方法

说了这么多，你的认知地图现在应该是这样的：

```
全端技术地图
├── Web端：Next.js / Nuxt / Astro
├── PC端：Electron / Tauri / Qt
├── 小程序端：uni-app / Taro / Flutter
├── 移动端：Flutter / React Native / KMP / Kuikly
└── CLI端：Bun / Node.js / oclif / commander
```

**这张地图的价值不在于记住，而在于认知：**

1. **技术选型**：你知道有哪些选项，而不是只熟悉手里的那一两个
2. **项目需求**：你能想到"这个是不是可以用Flutter快速覆盖多端"
3. **AI辅助**：你知道自己不知道什么，知道要去问什么

## 结语：认知先行，行动跟上

AI时代，行动的成本越来越低——你让AI写一段Flutter代码，它一定写得比你快。

但**认知的成本从未降低**——你需要见过、想过、踩过坑，才能形成真正的认知。

这篇文章的目的，就是帮你节省"见过"的时间。

你不需要现在就学会Flutter、学会KMP、学会Tauri。你只需要记住：**这些技术存在，这些场景可以用它们。**

剩下的，交给AI，交给你自己的决策。
