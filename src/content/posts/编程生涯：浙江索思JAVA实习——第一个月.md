## 引言：第一个月结束了

一个月前，我还在为「什么都不会」而忐忑；一个月后，我能够独立完成接口文档整理、简单BUG定位、Git分支操作这些基础辅助工作。回头看这段时光，既有初入职场的手忙脚乱，也有逐步适应的踏实感。

这篇文章，我就来聊聊第一个月到底经历了什么、收获了什么。

## 第一周：从学生到职场人的转变

### 入职第一天的印象

2月17日，早上九点。

我准时到达公司，被分配到开发组工位。没有入职培训，没有欢迎仪式，mentor直接给我发了一份内部文档链接：「先把这个看完，有问题问我。」

这种「自学为主」的模式，与学校里的老师手把手教学完全不同。一开始我有些不适应，但后来慢慢理解了——企业讲求效率，每个人的时间都很宝贵，主动自学本身就是一种职场能力。

### 熟悉开发环境

第一周的主要任务是熟悉开发环境：

| 工具/系统 | 用途          | 学习方式          |
| ----- | ----------- | ------------- |
| IDEA  | Java开发主力IDE | 官方文档 + 教程视频   |
| MySQL | 数据库操作       | 内部文档 + 实际练习   |
| Git   | 版本控制        | 教程 + 请教前辈     |
| Jira  | 任务跟踪        | 观察同事使用 + 实际操作 |

其中让我印象最深的是Git。在学校只学过`git add`、`git commit`、`git push`三个命令，而公司里使用的是完整的Git Flow分支策略：

```bash
# 创建功能分支
git checkout -b feature/xxx

# 提交代码
git add .
git commit -m "feat: 添加xxx功能"

# 推送到远程
git push -u origin feature/xxx

# 合并到开发分支（需要通过PR）
git checkout develop
git merge feature/xxx
```

一开始看到这么多流程我是懵的，但实际操作几次后就慢慢理解了——分支策略的目的是保证代码质量和可追溯性。

:::note
第一周的核心感受：职场与校园最大的不同是「没有人会盯着你学」，主动性和自学能力直接决定适应速度。
:::

## 第二周：融入团队的开始

### 每日早会体验

第二周起，我被要求开始参加每日早会。

早会只有15分钟，每个开发人员轮流汇报三件事：

1. 昨天完成了什么
2. 今天计划做什么
3. 遇到了什么阻碍

这种简洁直接的沟通方式让我印象深刻。没有长篇大论的PPT，没有形式主义的汇报，就是纯粹的「同步进度、反馈问题」。

我第一次参加早会时还有些紧张，说话结结巴巴的。mentor鼓励我说：「不用紧张，说清楚三件事就行。这也是练习表达能力的机会。」

### 业务流程初认知

第二周我还第一次接触了公司主营的系统集成类项目整体架构。

说实话，一开始看项目文档时我是懵的——模块多、依赖关系复杂、还有各种我从来没听过的专业术语。

但mentor给了我一个很好的建议：「先不要试图理解全部，先搞清楚你负责的那一块与其他模块的关系。」

按照这个思路，我画了一张简单的模块关系图：

```
┌─────────────┐
│  外部系统   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   网关层    │ ◀── 负责鉴权、路由
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  业务服务层 │ ◀── 我主要接触这层
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   数据层    │
└─────────────┘
```

搞清楚模块关系后，看文档的速度快了很多。

:::note
第二周的核心感受：面对复杂问题时，先从全局把握，再聚焦局部，这是处理大型项目的基本思路。
:::

## 第三周：辅助开发的实践

### 第一次参与代码编写

第三周，我终于开始参与正式的辅助开发工作了。

mentor给我分配了一个简单的任务：协助整理相关功能模块的开发资料，包括梳理需求细节、整理接口文档、编写辅助代码。

虽然只是辅助工作，但当我看到自己的代码被提交到开发分支时，还是有一种莫名的成就感——这是第一次，代码不再只是「作业」，而是真正被团队使用的产出。

### Git分支实操

这周我第一次在 mentor 指导下独立完成Git分支操作：

```bash
# 1. 从开发分支创建功能分支
git checkout -b feature/my-task

# 2. 在功能分支上编写代码
# ... 修改代码 ...

# 3. 提交代码（遵循企业提交规范）
git add .
git commit -m "feat: 辅助整理xxx资料"

# 4. 推送分支
git push -u origin feature/my-task

# 5. 在GitLab上发起Merge Request
# 等待code review ...

# 6. 合并后删除分支
git checkout develop
git pull
git branch -d feature/my-task
```

最让我收获的是code review环节。前辈在review时指出了我代码中不规范的地方，比如注释不够详细、变量命名不够清晰等。这些问题在我自己看来可能察觉不到，但通过review，代码质量确实提升了很多。

### 问题解决的心态转变

这周我还遇到了一个问题，让我印象很深。

当时我负责的一个辅助脚本运行报错，我花了半小时排查还是解决不了。本来想直接去问 mentor，但想起之前他说的「先自主思考、再请教」，我又坚持了一会儿。

最后我发现是一个参数配置的问题——之前没仔细看文档。解决后我立刻把这个坑记录下来，避免下次再犯。

:::note
第三周的核心感受：遇到问题先思考再请教，这种习惯不仅让学习更高效，也让他人更愿意帮助你。
:::

## 第四周：规范与文档

### 接口文档的编写

第四周，我接到任务：协助使用Swagger编写接口文档。

在学校里，我从来没有系统学习过如何编写接口文档，觉得「代码能跑就行」。但在公司里，接口文档是团队协作的基础——文档不规范，联调时就会出现各种沟通成本。

这次我认真学习了Swagger的编写规范：

```java
/**
 * 用户管理接口
 */
@RestController
@RequestMapping("/api/user")
public class UserController {

    /**
     * 用户登录接口
     * @param username 用户名
     * @param password 密码
     * @return 登录结果
     */
    @PostMapping("/login")
    @ApiOperation(value = "用户登录", notes = "根据用户名密码进行登录")
    @ApiImplicitParams({
        @ApiImplicitParam(name = "username", value = "用户名", required = true, dataType = "String"),
        @ApiImplicitParam(name = "password", value = "密码", required = true, dataType = "String")
    })
    public Result<UserVO> login(
        @RequestParam String username,
        @RequestParam String password
    ) {
        // 登录逻辑
        return Result.success(userVO);
    }
}
```

通过这次实践，我深刻理解了接口文档的重要性：好的文档能让调用方快速理解接口用途、参数要求、返回格式，减少大量不必要的沟通。

### 工作日志与周报

这周还第一次写了工作日志和周报。

mentor告诉我，工作日志不是流水账，而是「工作内容 + 遇到的问题 + 解决方案 + 明日计划」的组合。这种记录方式让我能够定期复盘自己的工作，而不是浑浑噩噩地过完一天。

```markdown
## 2025.03.15 工作日志

### 今日完成
- [x] 协助整理用户模块接口文档
- [x] 完成登录接口的Swagger注释
- [x] 参与code review并修复问题

### 遇到的问题
- Swagger文档中参数描述不够清晰，已补充

### 明日计划
- [ ] 继续完善接口文档
- [ ] 协助测试人员进行接口测试
```

:::note
第四周的核心感受：文档能力是开发者的基本功，规范的文档能大大提升团队协作效率。
:::

## 第一个月总结

### 技术能力提升

| 能力项     | 学习前   | 学习后                  |
| ------- | ----- | -------------------- |
| Git版本控制 | 只会三剑客 | 掌握分支策略、code review流程 |
| 开发工具    | 学校教的  | 熟练使用IDEA、MySQL、Jira  |
| 接口文档    | 没概念   | 掌握Swagger编写规范        |
| 问题解决    | 依赖老师  | 先思考再请教               |

### 心态转变

第一个月最大的收获不是技术本身，而是心态的转变：

1. **从「学什么」到「用什么」**：学校是学知识，公司是用知识解决问题
2. **从「被动等」到「主动问」**：职场不会有人盯着你，主动性决定成长速度
3. **从「个人英雄」到「团队协作」**：代码是给团队写的，可读性比炫技更重要

### 不足与改进

当然，第一个月也暴露了一些问题：

- **效率有待提升**：有时候一个问题会卡很久，需要继续提升排查问题的能力
- **沟通需要更主动**：有时候遇到问题会不好意思问，导致进度延迟
- **文档规范意识要加强**：好的文档习惯要尽早就养成

### 下一步目标

进入第二个月后，我希望能够：

1. 独立完成更复杂的辅助开发任务
2. 深入学习业务逻辑，不再只是「打辅助」
3. 建立自己的问题排查思路和工具箱

***

*第一个月结束了，旅程还在继续。*
