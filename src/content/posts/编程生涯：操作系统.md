## 为什么操作系统是门"必修课"？

在正式进入学习之前，我先问了自己一个问题：操作系统和我有什么关系？

答案出乎意料地直接：**操作系统是所有软件的地基**。

无论是你的Java程序、Python脚本，还是运行在服务器上的微服务，它们都跑在操作系统之上。操作系统管理着CPU、内存、磁盘、网络等硬件资源，为应用程序提供统一的抽象接口。

| 层次   | 代表                  | 职责        |
| ---- | ------------------- | --------- |
| 硬件   | CPU、内存、磁盘、网卡        | 物理存在      |
| 操作系统 | Linux、Windows、macOS | 资源管理、接口抽象 |
| 应用程序 | 你的Spring Boot项目     | 业务逻辑      |

一个形象的比喻：

> 如果把计算机比作一家大型工厂，操作系统就是那个精明能干的厂长——它要管理工人（CPU）、仓库（内存）、生产线（磁盘）、物流通道（网络），让整个工厂高效运转。而我们写的程序，就是厂长接到的生产订单。

:::note
理解操作系统能让你写出更优质的代码。比如，知道进程和线程的区别，就能更好地处理并发问题；理解虚拟内存，就知道为什么会发生OOM；懂得磁盘I/O的原理，就能在数据库优化时心中有数。
:::

## 操作系统的核心功能

现代操作系统有五大核心功能模块：

| 功能模块 | 核心职责         | 与开发的关联      |
| ---- | ------------ | ----------- |
| 进程管理 | 创建、调度、销毁进程   | 并发编程、多线程    |
| 内存管理 | 分配/回收内存、虚拟内存 | JVM调优、OOM排查 |
| 文件系统 | 管理存储空间、文件权限  | 文件读写、路径处理   |
| 设备管理 | 管理硬件设备、驱动    | 外设交互、IO操作   |
| 用户接口 | 提供CLI/GUI给用户 | 系统调用、Shell  |

## 进程与线程：程序的"生死簿"

### 进程：程序的一次生命周期

进程（Process）是程序的一次执行过程，是操作系统进行资源分配和调度的基本单位。

你可以这样理解：

> 进程就像是一个正在营业的餐厅。当顾客点了一道菜，厨房就"启动"了做菜这个"程序"。做菜需要用到炉灶（CPU时间）、食材（内存）、锅碗瓢盆（文件句柄）等资源，这些资源由餐厅（操作系统）统一分配。

```c
// Linux下创建新进程的示例
#include <stdio.h>
#include <unistd.h>

int main() {
    pid_t pid = fork();  // fork() 是创建进程的系统调用

    if (pid < 0) {
        // 创建失败
        perror("fork failed");
    } else if (pid == 0) {
        // 子进程：fork()返回0
        printf("我是子进程，PID=%d\n", getpid());
        execlp("ls", "ls", "-la", NULL);  // 执行新程序
    } else {
        // 父进程：fork()返回子进程PID
        printf("我是父进程，子进程PID=%d\n", pid);
        wait(NULL);  // 等待子进程结束
        printf("子进程已退出\n");
    }
    return 0;
}
```

### 进程状态：生命的不同阶段

进程从创建到销毁，经历了多个状态：

```
                    ┌──────────┐
                    │   创建   │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
              ┌─────│   就绪   │←────┐
              │     └────┬─────┘     │
              │          │           │
         调度发生    获得CPU      时间片用完
              │          │           │
              │     ┌────▼─────┐     │
              │     │   运行   │─────┘
              │     └────┬─────┘
              │          │ I/O完成或资源等待
         ┌────▼──────────▼─────┐
         │        阻塞          │
         └─────────┬───────────┘
                   │ I/O完成
                   ▼
              返回就绪状态
```

三个核心状态：

| 状态          | 含义         | 触发条件            |
| ----------- | ---------- | --------------- |
| 就绪（Ready）   | 已准备好，只等CPU | 进程创建、调度选中、时间片用完 |
| 运行（Running） | 正在使用CPU    | 被调度器选中          |
| 阻塞（Blocked） | 等待某些事件     | I/O请求、等待信号、等待锁  |

### 线程：轻量级的执行单位

线程（Thread）是CPU调度的最小单位。同一个进程内的线程共享进程的地址空间、文件描述符、堆等资源，但每个线程有独立的栈和寄存器。

继续餐厅的比喻：

> 如果进程是整个餐厅，线程就是餐厅里的各个员工。他们共享餐厅的厨房、食材、桌椅，但每个人有自己专属的工具包（栈）和记事本（寄存器）。

```java
// Java中创建线程的两种方式
public class ThreadDemo {

    // 方式一：继承Thread类
    static class MyThread extends Thread {
        @Override
        public void run() {
            System.out.println("方式一：继承Thread，线程名=" + getName());
        }
    }

    // 方式二：实现Runnable接口（推荐）
    static class MyRunnable implements Runnable {
        @Override
        public void run() {
            System.out.println("方式二：实现Runnable");
        }
    }

    public static void main(String[] args) {
        // 启动线程一
        MyThread t1 = new MyThread();
        t1.start();

        // 启动线程二
        Thread t2 = new Thread(new MyRunnable(), "Runnable线程");
        t2.start();

        System.out.println("主线程执行完毕");
    }
}
```

### 进程 vs 线程：一张表说清楚

| 对比维度 | 进程           | 线程         |
| ---- | ------------ | ---------- |
| 资源分配 | 进程是资源分配的基本单位 | 线程是调度的基本单位 |
| 地址空间 | 独立地址空间       | 共享进程地址空间   |
| 开销   | 创建/切换开销大     | 创建/切换开销小   |
| 通信   | 需要IPC机制      | 直接读写共享内存   |
| 安全性  | 完全隔离         | 共享带来线程安全问题 |

:::warning
Java中的线程安全问题是典型案例。多个线程同时访问同一个共享变量，如果没有同步机制，就会导致数据不一致。比如经典的"i++多线程不安全"问题：

```java
// 假设i的初始值是0，两个线程同时执行10000次i++
// 最终结果很可能不等于20000，因为i++不是原子操作
public class Counter {
    private int i = 0;

    public void increment() {
        i++;  // 实际是：读取→加1→写入三步操作
    }
}
```

:::

## 进程间通信：进程之间的"对话方式"

进程虽然有独立的地址空间，但有时候也需要互相通信。操作系统提供了多种IPC（Inter-Process Communication）机制：

| 通信方式   | 特点          | 适用场景    | 速度 |
| ------ | ----------- | ------- | -- |
| 管道     | 半双工、面向字节流   | 父子进程通信  | 快  |
| 消息队列   | 消息结构化、可按优先级 | 多进程异步通信 | 中  |
| 共享内存   | 直接读写内存      | 高频数据交换  | 最快 |
| 信号量    | 计数器、用于同步    | 进程同步    | 快  |
| Socket | 可跨主机通信      | 网络通信    | 慢  |

一个通俗的比喻：

> 管道就像对讲机——简单直接，但只能一方说另一方听；消息队列就像快递柜——有编号、取件码，接收方什么时候取都行；共享内存就像两人共用一个笔记本——写完直接能看到，但要注意不要同时写。

```c
// Linux下管道通信示例
#include <stdio.h>
#include <unistd.h>

int main() {
    int pipefd[2];  // pipefd[0]是读端，pipefd[1]是写端

    if (pipe(pipefd) == -1) {
        perror("pipe failed");
        return 1;
    }

    pid_t pid = fork();

    if (pid == 0) {
        // 子进程：关闭读端，写数据
        close(pipefd[0]);
        char msg[] = "Hello from child!";
        write(pipefd[1], msg, sizeof(msg));
        close(pipefd[1]);
    } else {
        // 父进程：关闭写端，读数据
        close(pipefd[1]);
        char buf[100];
        int len = read(pipefd[0], buf, sizeof(buf));
        buf[len] = '\0';
        printf("父进程收到: %s\n", buf);
        close(pipefd[0]);
        wait(NULL);
    }
    return 0;
}
```

## 调度算法：CPU时间的"分配艺术"

当多个进程同时需要CPU时，操作系统必须决定谁先执行——这就是进程调度。

### 调度算法的常见指标

评价一个调度算法好不好，主要看这几个指标：

| 指标     | 含义            | 优化目标 |
| ------ | ------------- | ---- |
| CPU利用率 | CPU忙碌的时间比例    | 越高越好 |
| 吞吐量    | 单位时间内完成的进程数   | 越高越好 |
| 周转时间   | 进程从提交到完成的总时间  | 越短越好 |
| 等待时间   | 进程在就绪队列中等待的时间 | 越短越好 |
| 响应时间   | 从提交到首次响应的时间   | 越短越好 |

### 经典调度算法

#### 1. 先来先服务（FCFS）

最简单的调度算法，按照进程到达的顺序服务。

```c
// FCFS调度模拟
// 假设有三个进程，按到达顺序服务
// P1: 到达时间0，服务时间10
// P2: 到达时间1，服务时间5
// P3: 到达时间2，服务时间3

// 执行顺序：P1 → P2 → P3
// P1周转时间 = 10 - 0 = 10
// P2周转时间 = (10+5) - 1 = 14
// P3周转时间 = (10+5+3) - 2 = 16
// 平均周转时间 = (10+14+16)/3 = 13.33
```

优点：简单、公平
缺点：短作业可能等待很久（"护航效应"）

#### 2. 最短作业优先（SJF）

优先服务最短的作业，平均等待时间最优。

```c
// SJF调度示例
// P1: 服务时间6
// P2: 服务时间3
// P3: 服务时间8
// P4: 服务时间4

// 按服务时间排序：P2(3) → P4(4) → P1(6) → P3(8)
// 如果所有进程同时到达，这就是最优顺序
```

优点：平均等待时间最小
缺点：长作业可能"饿死"

#### 3. 时间片轮转（RR）

每个进程执行一个时间片，然后切换到下一个：

```
时间片大小 = 4

P1[0-4] → P2[4-7] → P3[7-11] → P4[11-14] → P1[14-18] → ...
```

:::note
时间片太大会退化为FCFS，时间片太小会增加上下文切换开销。实际系统中，时间片通常设置为10-100毫秒。
:::

#### 4. 优先级调度

按优先级高低服务，高优先级进程先执行。

| 类型    | 说明          |
| ----- | ----------- |
| 静态优先级 | 进程创建时就确定，不变 |
| 动态优先级 | 会随等待时间等因素调整 |

优先级调度的隐患：**优先级反转**。比如低优先级进程持有锁，高优先级进程等待锁，而中等优先级进程一直在运行，导致低优先级进程无法释放锁。

#### 5. 多级反馈队列（MLFQ）

综合多种算法的优点：

```
队列1（最高优先级）: 时间片=8ms
队列2（中优先级）  : 时间片=16ms
队列3（低优先级）  : 时间片=32ms
```

规则：

1. 新进程进入队列1
2. 如果时间片用完还没完成，下降到下一级队列
3. 低优先级队列用完了才会执行更高级的

这种设计让短作业快速响应，长作业也不会饿死。

## 死锁：进程之间的"僵持局面"

### 什么是死锁？

死锁（Deadlock）是指两个或多个进程在运行过程中，因争夺资源而造成一种互相等待的现象，若无外力干预，它们都将无法推进。

一个经典的死锁场景——哲学家就餐问题：

```
五个哲学家围坐一圈，每两人之间放一根筷子，进餐时需要左右两根筷子。

如果每个哲学家都同时拿起左边的筷子：
- 哲学家1拿起筷子1，等待筷子2
- 哲学家2拿起筷子2，等待筷子3
- ...以此类推
- 每个人都拿着左边，在等右边 → 死锁！
```

### 死锁的四个必要条件

| 条件    | 含义              |
| ----- | --------------- |
| 互斥    | 资源只能被一个进程占用     |
| 占有并等待 | 已持有资源的进程还能请求新资源 |
| 不可抢占  | 已分配的资源不能被强行夺走   |
| 循环等待  | 形成进程-资源的循环链     |

只有四个条件同时满足，才会发生死锁。

### 解决死锁的方法

| 方法    | 思路             | 适用场景     |
| ----- | -------------- | -------- |
| 预防    | 破坏四个条件之一       | 资源少、确定性强 |
| 避免    | 银行家算法动态检查      | 资源可预测    |
| 检测与恢复 | 允许死锁发生，定期检测后恢复 | 不便预防时    |
| 忽略    | 假装没死锁          | 死锁概率极低   |

银行家算法的核心思想：**在分配资源前，先模拟最坏情况，如果所有进程都能完成，就分配；否则拒绝。**

## 内存管理：从物理到虚拟的跨越

### 物理内存的困境

早期的计算机直接使用物理内存，问题显而易见：

> 假设物理内存只有512MB，同时运行三个程序：
>
> - 程序A需要300MB
> - 程序B需要400MB
> - 程序C需要200MB
>
> 结果：A和B加起来就满了，C根本跑不了。但实际上A和B可能并没有同时使用全部内存！

### 虚拟内存：让程序"觉得"内存很大

虚拟内存是现代操作系统的核心机制。它给每个进程"幻觉"——仿佛自己拥有完整的地址空间。

```
虚拟地址空间（每个进程独立）：
0x00000000 ┌──────────────────┐
           │                  │
           │    用户空间       │
           │  (如 0x00004000   │
           │   ~ 0xBFFF0000)   │
           │                  │
           ├──────────────────┤
           │    内核空间       │
0xFFFFFFFF └──────────────────┘

物理内存（所有进程共享）：
0x00000000 ┌──────────────────┐
           │     内核空间     │
           ├──────────────────┤
           │    物理页帧      │
           │  (可被映射到     │
           │   多个进程的     │
           │   虚拟空间)      │
           └──────────────────┘
```

虚拟内存的优势：

| 优势     | 说明                   |
| ------ | -------------------- |
| 进程隔离   | 每个进程有独立地址空间，互不干扰     |
| 内存抽象   | 程序员不需要关心物理地址         |
| 内存利用率  | 按需加载，不用把整个程序加载到内存    |
| 简化内存分配 | 只需分配虚拟地址，实际物理地址由系统管理 |

### 内存分页：高效管理内存

分页（Paging）将虚拟内存和物理内存都划分为固定大小的页（通常是4KB）：

```
虚拟地址组成：
┌────────────────┬────────────┐
│   页号（VPN）   │ 页内偏移   │
└────────────────┴────────────┘
      │                │
      ▼                ▼
   页表查询        直接使用
      │
      ▼
┌────────────────┬────────────┐
│ 物理页号（PPN） │ 页内偏移   │
└────────────────┴────────────┘
```

页表（Page Table）存储虚拟页到物理页的映射关系：

```c
// 简化版页表项结构
struct PageTableEntry {
    unsigned int present     : 1;   // 是否在内存中
    unsigned int frame      : 20;  // 物理页帧号
    unsigned int modified   : 1;   // 是否被修改过（脏位）
    unsigned int referenced : 1;   // 是否被访问过
    unsigned int protection : 3;   // 访问权限
};
```

:::note
页表会带来额外的内存访问开销——访问一个虚拟地址需要先查页表，再访问实际数据。这就是为什么CPU有TLB（Translation Lookaside Buffer）来缓存常用的页表项。
:::

### 页面置换算法：当内存不够时

当物理内存不足时，需要把某些页换出到磁盘，这就是页面置换。

| 算法    | 核心思想       | 优缺点              |
| ----- | ---------- | ---------------- |
| FIFO  | 淘汰最早进入的页   | 简单但性能差（Belady异常） |
| OPR   | 淘汰最远将来访问的页 | 理论最优，但无法实现       |
| LRU   | 淘汰最近最少使用的页 | 近似OPT，但开销大       |
| Clock | 改进的近似LRU   | 用一位"访问位"近似实现     |

```c
// LRU算法思想（实际实现更复杂，这里是简化版）
// 维护一个访问时间戳表：
// 每次访问某页，更新其最近访问时间
// 淘汰时，选择时间戳最小的（最久未使用）

typedef struct {
    int page_num;      // 页号
    int last_access;   // 上次访问时间（可用时间戳模拟）
} PageInfo;
```

### 段页式管理：取长补短

段式管理按程序逻辑划分（代码段、数据段、堆、栈），页式管理按固定大小划分。段页式结合两者优点：

```
逻辑地址：段号 + 段内偏移
           ↓
段表：段号 → (段起始地址, 段长度)
           ↓
段内地址：页号 + 页内偏移
           ↓
页表：页号 → 物理页帧号
           ↓
物理地址：物理页帧号 + 页内偏移
```

## 文件系统：数据的"图书馆"

### 文件系统的层次结构

磁盘上的数据井井有条，背后是一套精密的管理机制：

```
用户程序
   ↓ 系统调用（open/read/write/close）
文件系统（ext4/NTFS/FAT32）
   ↓ 块设备接口
通用块层（bio/request）
   ↓ I/O调度
磁盘驱动
   ↓
物理磁盘
```

### 文件存储方式

| 方式   | 原理        | 优点          | 缺点         |
| ---- | --------- | ----------- | ---------- |
| 连续分配 | 文件占有一串连续块 | 读写快         | 空间浪费、扩展难   |
| 链接分配 | 块间用指针连接   | 空间利用率高      | 随机访问慢、指针易坏 |
| 索引分配 | 建立索引块记录位置 | 随机访问、顺序访问都快 | 大文件索引开销大   |

### inode：文件的"身份证"

Linux使用inode（索引节点）来管理文件元数据：

```c
// inode结构简化版
struct inode {
    unsigned long i_ino;        // inode号（唯一标识）
    unsigned int  i_mode;        // 文件类型和权限
    unsigned int i_nlink;       // 硬链接数
    unsigned long i_size;        // 文件大小（字节）
    unsigned long i_blocks;     // 文件占用的块数
    unsigned long i_atime;      // 最后访问时间
    unsigned long i_mtime;      // 最后修改时间
    unsigned long i_ctime;      // 最后状态改变时间
    unsigned long i_block[15]; // 直接/间接块指针
    // ...
};
```

### 虚拟文件系统（VFS）

Linux支持多种文件系统（ext4、xfs、btrfs、NTFS等），它们通过虚拟文件系统（VFS）提供统一接口：

> VFS就像是一个万能翻译器。无论你说中文、英文还是法文，通过翻译器都能和只懂中文的人沟通。应用程序只需调用标准的open/read/write接口，VFS负责找到对应的文件系统驱动。

## I/O管理：与外设的交互

### I/O控制方式的发展

| 方式      | 原理        | CPU介入 | 适用场景          |
| ------- | --------- | ----- | ------------- |
| 程序控制I/O | CPU轮询设备状态 | 完全介入  | 简单设备、老式系统     |
| 中断驱动I/O | 设备完成后发中断  | 中等介入  | 多数外设          |
| DMA     | 直接内存访问    | 几乎不介入 | 大数据量传输（磁盘、网络） |

DMA（Direct Memory Access）的意义：

> 想象你要把一本书从A房间搬到B房间。方式一是自己搬（一页一页），累死；方式二是让助理搬（DMA），你只需告诉助理"把书搬到B房间"，助理完成后通知你一声，你可以去干别的事。

### I/O缓冲区：减少磁盘访问

操作系统使用缓冲区（Cache）来减少磁盘I/O次数：

```c
// 缓冲区的作用示意
// 无缓冲：每次write都直接写磁盘（很慢）
write(fd, buf, 1);  // 磁盘写入1字节
write(fd, buf, 1);  // 又是一次磁盘写入
write(fd, buf, 1);  ...

// 有缓冲：数据先写到内存缓冲区
write(fd, buf, 1);  // 写入缓冲区
write(fd, buf, 1);  // 继续写缓冲区
write(fd, buf, 1);  // 还是写缓冲区
// 缓冲区满了或调用fsync时才真正写磁盘
```
