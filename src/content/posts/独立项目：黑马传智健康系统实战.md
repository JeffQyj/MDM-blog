2024 年 4 月，我完成了传智健康项目的学习。这是继青橙商城和十次方之后的第三个企业级实战项目，采用了 SOA 分布式架构设计，通过 Dubbo 实现服务治理。相比前两个项目，传智健康补齐了 SSM 生态的更多关键拼图：Freemarker 静态化、Quartz 定时任务、Spring Security 权限控制、Apache POI 报表处理、FastDFS 文件存储，以及微信端的完整开发。

## 项目背景

传智健康管理系统是一款应用于健康管理机构（类似慈铭体检、爱康国宾）的业务系统，核心目标是：

- **工作可视化**：让健康管理师的工作内容清晰可见
- **会员管理专业化**：建立完整的会员健康档案体系
- **健康评估数字化**：用数据驱动健康风险识别
- **健康干预流程化**：从评估到干预全链路数字化

### 双端闭环设计

项目采用**后台管理系统 + 微信用户端**双端架构，覆盖四类核心用户角色：

| 用户角色  | 使用场景               | 系统入口  |
| ----- | ------------------ | ----- |
| 系统管理员 | 平台全量数据管理、权限配置、系统运维 | 后台管理端 |
| 健康管理师 | 会员健康评估、干预方案制定、随访跟踪 | 后台管理端 |
| 会员用户  | 体检预约、报告查询、健康评估     | 微信端   |
| 匿名用户  | 套餐浏览、健康资讯查看        | 微信端   |

:::note
这是第一次接触双端架构设计。微信端和后台管理端虽然面向不同用户，但底层调用的服务是同一套。这种「服务复用」的思想，是 SOA 架构的核心价值之一。在青橙商城项目里也用过类似思路——后台管理端和用户商城端共用商品服务、用户服务。传智健康把这个模式用到了健康管理场景。
:::

### 核心功能模块一览

| 模块类型 | 主要功能              | 技术价值                    |
| ---- | ----------------- | ----------------------- |
| 预约管理 | 检查项、检查组、套餐管理、预约设置 | SSM CRUD + MyBatis 关联查询 |

| 定时任务 | 预约设置批量导入、垃圾图片清理   | Apache POI + Quartz     |

## 技术架构

### 整体架构

项目采用 SOA（Service-Oriented Architecture）分布式服务架构，基于 Dubbo 实现服务治理。前后端分离设计，后端按业务模块拆分，前端分为管理端和微信端。

```
┌─────────────────────────────────────────────────────────────────┐
│                         用户层                                    │
├─────────────────────────────┬───────────────────────────────────┤
│      后台管理系统 (health_backend)    │     微信端 (health_mobile)        │
│           [服务消费方 Consumer]        │        [服务消费方 Consumer]        │
└─────────────────────────────┴───────────────────────────────────┘
                                    │
                                    │ Dubbo RPC 调用
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Zookeeper 注册中心                            │
│                    (服务注册与发现的中介平台)                       │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    │ 服务发布/订阅
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Dubbo 服务提供方                              │
│              [health_service_provider]                         │
│  ┌─────────────┬──────────────┬─────────────┬──────────────┐   │
│  │  用户服务    │  预约服务    │  会员服务    │  报表服务    │   │
│  └─────────────┴──────────────┴─────────────┴──────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 为什么选择 SOA？

**与十次方的分叉：两条分布式演进路线**

在做这个项目之前，我已完成[青橙商城项目](/articles/独立项目：青橙商城分布式电商实战/)（2024年2月），采用的是 SOA + Dubbo + Zookeeper 架构。紧接着又学了[十次方项目](/articles/独立项目：黑马十次方社区平台/)（2024年3月），它用的是 Spring Cloud 微服务架构。传智健康排在十次方之后（2024年4月），但仍然选择了 SOA 架构。

两条路线有什么区别？

| 对比维度   | SOA 分布式（传智健康 / 青橙商城）   | 微服务架构（十次方）                |
| ------ | ---------------------- | ------------------------- |
| 服务拆分粒度 | 按业务模块拆分，服务较粗           | 按业务功能拆分，服务更细              |
| 注册中心   | Zookeeper（CP 原则，强一致性）  | Eureka / Nacos（AP 原则，高可用） |
| 调用方式   | RPC（Dubbo），基于 TCP      | REST（Feign），基于 HTTP       |
| 技术栈    | SSM + Dubbo，Java 技术栈统一 | Spring Cloud 全家桶，支持多语言    |
| 适用场景   | 业务边界清晰的中大型系统           | 快速迭代、团队规模大的互联网产品          |

**为什么传智健康选了 SOA 而不是微服务？**

理由很实际：

1. **业务规模不需要**：传智健康只有 4 类用户角色、6 大业务模块，服务拆分到「用户服务、预约服务、会员服务、报表服务」就够了。如果用微服务拆分出十几二十个服务，反而过度设计。
2. **技术储备还在积累中**：Dubbo + Zookeeper 的组合已经在青橙商城项目中验证过了，驾轻就熟。Spring Cloud 的 Zuul、Hystrix、Config 那些新概念在十次方里刚消化完，需要进行交叉巩固。
3. **学习路径要循序渐进**：先 SOA 搞懂服务治理的核心思想（注册发现、负载均衡、容错），再进阶到 Spring Cloud 理解微服务全生态，这条路更扎实。

:::note
回过头看这个选择是对的。十次方的 Spring Cloud 确实更「潮」，但刚学完 SOA 直接跳过去，容易邯郸学步。先用传智健康把 Dubbo 的服务发布、订阅、调用链路摸透，再去看十次方的 Feign、Ribbon、Hystrix，就能理解它们「殊途同归」的设计思想了。
:::

**单体架构的痛点**：

- 所有代码堆在一个 War 包里，部署一次要重启整个系统
- 预约模块出了 Bug，可能影响会员模块一起挂
- 团队协作时，改同一份代码，Git 合并冲突让人崩溃

| 对比维度 | 单体架构       | SOA 分布式     |
| ---- | ---------- | ----------- |
| 部署方式 | 单一 War 包   | 多 War 包独立部署 |
| 容错性  | 一个模块挂，全站崩溃 | 单个服务挂，不影响其他 |
| 扩展性  | 整体扩展，资源浪费  | 按需扩展特定服务    |
| 团队协作 | 代码冲突频繁     | 分模块开发，独立迭代  |

### 后端技术栈

| 架构层级   | 核心技术选型                                         | 通俗比喻            |
| ------ | ---------------------------------------------- | --------------- |
| 前端层    | Vue.js + Element UI（管理端）HTML5 + Bootstrap（微信端） | 超市的展示货架         |
| 应用框架层  | Spring + Spring MVC + MyBatis（SSM）             | 货架的标准化管理系统      |
| 分布式服务层 | Dubbo（RPC 框架） + Zookeeper                      | 超市的广播系统 + 物业管理处 |
| 安全权限层  | Spring Security + JWT + HTTPS                  | 门禁系统 + 身份证验证    |
| 数据持久层  | MySQL 5.7+（关系型数据库）Redis 6.x（缓存）                | 仓库管理员 + 快递柜     |
| 中间件与工具 | Apache POI、Quartz、RabbitMQFastDFS、Nginx        | 各种专业工具          |
| 第三方服务  | 阿里云短信、微信开放平台                                   | 快递服务 + 会员卡系统    |

:::tip
**Dubbo 是什么？** 可以把它想象成小区物业管理处——它不负责送快递（不参与业务），但没有它，整个小区就乱套了。服务提供方（Provider）到物业处登记注册，服务消费方（Consumer）通过物业处找到目标服务，实现解耦。
:::

:::note
这里 Dubbo 的用法和[青橙商城](/articles/独立项目：青橙商城分布式电商实战/)基本一致——都是用 XML 配置方式暴露服务和引用服务。青橙商城用的是 `dubbo:service` 和 `dubbo:reference` 标签，传智健康也是一模一样的手法。学完青橙再看传智健康，几乎没有学习成本，把精力放在了业务本身。
:::

### Dubbo 核心角色解析

| 角色              | 职责                 | 项目中的位置                         |
| --------------- | ------------------ | ------------------------------ |
| Provider（服务提供方） | 实现具体业务逻辑，向注册中心注册服务 | health\_service\_provider      |
| Consumer（服务消费方） | 从注册中心订阅服务，调用提供方的接口 | health\_backend、health\_mobile |
| Registry（注册中心）  | 服务的注册与发现           | Zookeeper                      |
| Monitor（监控中心）   | 统计服务调用次数、响应时间      | Dubbo Admin                    |

:::note
注册中心这块，十次方用的是 Eureka，而传智健康和青橙商城都用 Zookeeper。两者的核心区别在于：Zookeeper 追求强一致性（CP），Eureka 追求高可用（AP）。放到十次方这种需要高可用的互联网场景，Eureka 更合适；而传智健康的业务场景，Zookeeper 的强一致性完全够用，而且我更熟悉。
:::

### Maven 分模块工程结构

项目采用 Maven 分模块开发，这是 SOA 架构的标准工程组织方式：

| 模块名称                      | 打包方式 | 核心职责                              |
| ------------------------- | ---- | --------------------------------- |
| health\_parent            | pom  | 父工程，锁定依赖版本，聚合子模块                  |
| health\_common            | jar  | 工具类、实体类、统一返回结果、常量                 |
| health\_interface         | jar  | Dubbo 服务接口定义（消费方与提供方的契约）          |
| health\_service\_provider | war  | Dubbo 服务提供方，存放 Service、DAO、Mapper |
| health\_backend           | war  | 后台管理系统（服务消费方）                     |
| health\_mobile            | war  | 微信用户端（服务消费方）                      |

```
health_parent (pom)
    │
    ├── health_common (jar)  ──────────────────────┐
    │                                                │
    ├── health_interface (jar)  ───────────────────┐ │
    │                                             │ │
    │                                             ▼ ▼
    └── health_service_provider (war) ◄────────────┘
    │         │
    │         │  依赖
    │         ▼
    │   health_interface
    │
    ├── health_backend (war) ◄─────────────────────┐
    │         │                                     │
    │         ▼                                     │
    └── health_mobile (war) ◄────────────────────────┘
              │
              ▼
        health_interface
```

:::note
health\_interface 是整个项目的「契约层」——服务提供方和消费方都依赖它，但彼此不直接依赖。这就像 USB 接口标准：不管是谁生产的 U 盘，只要符合 USB 协议，就能插上电脑使用。这个设计在[青橙商城](/articles/独立项目：青橙商城分布式电商实战/)里也有，两个项目的模块划分几乎一模一样。
:::

## 业务模块实战

### 模块一：预约管理 — 检查项与套餐设计

#### 业务场景

预约管理是整个系统的根基模块。一个体检套餐由多个检查组组成，每个检查组包含多个检查项。比如「入职体检套餐」可能包含：

- 外科检查组（身高、体重、视力）
- 内科检查组（血压、心率）
- 化验组（血常规、尿常规）

这是典型的**多对多关联**场景——一个套餐有多个检查组，一个检查组也属于多个套餐。

#### 核心数据结构

```java
// 检查项：最基础的体检单元，如"血压"、"血常规"
public class CheckItem {
    private Integer id;
    private String code;         // 编码，用于系统内部标识
    private String name;         // 检查项名称
    private String sex;          // 适用性别（M/F/A）
    private String remark;       // 注意事项
}

// 检查组：一组相关检查项的集合，如"内科"、"外科"
public class CheckGroup {
    private Integer id;
    private String code;
    private String name;
    private String remark;
    private List<CheckItem> checkItems;  // 一个检查组包含多个检查项
}

// 体检套餐：面向用户的完整体检产品
public class Setmeal {
    private Integer id;
    private String code;
    private String name;
    private Float price;
    private List<CheckGroup> checkGroups;  // 一个套餐包含多个检查组
}
```

#### MyBatis 关联查询

套餐详情查询是典型的嵌套查询场景，涉及三张表的关联：

```xml
<!-- SetmealMapper.xml -->
<resultMap id="SetmealDetailMap" type="com.itheima.pojo.Setmeal">
    <id property="id" column="id"/>
    <result property="name" column="name"/>
    <result property="price" column="price"/>
    <!-- 嵌套查询：套餐 → 检查组 → 检查项 -->
    <collection property="checkGroups" column="id"
                ofType="com.itheima.pojo.CheckGroup"
                select="findCheckGroupsBySetmealId"/>
</resultMap>

<select id="findCheckGroupsBySetmealId" resultMap="CheckGroupMap">
    SELECT cg.* FROM t_checkgroup cg
    INNER JOIN t_setmeal_checkgroup sc ON cg.id = sc.checkgroup_id
    WHERE sc.setmeal_id = #{id}
</select>
```

| 技术要点   | 说明                       |
| ------ | ------------------------ |
| 一对多关联  | 一个套餐对应多个检查组              |
| 多对多关联  | 检查组与检查项的关联表设计            |
| 动态 SQL | 使用 `<if test="">` 处理动态条件 |

:::note
当时写这个查询时，最大的坑是 SQL 的嵌套顺序。一开始搞反了层级关系，查出来的套餐详情少了一半的检查项。教训是：画 ER 图再写 SQL，不要凭感觉。
:::

### 模块二：预约设置 — Apache POI 批量导入

#### 业务场景

健康管理机构每天的预约接待量是有限的，需要提前配置每天的的可预约人数。如果每天手动一条一条录入，工作量巨大。Apache POI 的功能就是让管理员**下载 Excel 模板、填写数据、批量导入**，系统自动解析并写入数据库。

#### Apache POI 是什么？

**一句话理解**：Apache POI 是 Java 操作 Office 文档的瑞士军刀——能读写 Excel（`.xls/.xlsx`）、Word、PowerPoint 文件。本项目主要用到它操作 Excel。

**什么场景用？** 当你需要批量录入数据时——比如导入预约设置、导出统计报表——POI 就是首选工具。

#### 批量导入实现

**Step 1：提供 Excel 模板下载**

管理员点击「下载模板」，系统生成一个标准格式的 Excel 文件：

```java
// OrderSettingsController.java
@GetMapping("/downloadTemplate")
public void downloadTemplate(HttpServletResponse response) throws Exception {
    // 创建一个空的预约设置模板
    HSSFWorkbook workbook = new HSSFWorkbook();
    HSSFSheet sheet = workbook.createSheet("预约设置");

    // 表头
    HSSFRow headerRow = sheet.createRow(0);
    headerRow.createCell(0).setCellValue("日期");
    headerRow.createCell(1).setCellValue("可预约人数");

    // 设置列宽
    sheet.setColumnWidth(0, 20 * 256);
    sheet.setColumnWidth(1, 15 * 256);

    // 输出到浏览器
    response.setContentType("application/vnd.ms-excel");
    response.setHeader("Content-Disposition",
        "attachment;filename=order_settings_template.xls");
    workbook.write(response.getOutputStream());
    workbook.close();
}
```

**Step 2：解析上传的 Excel 文件**

管理员填写完数据后上传，系统读取 Excel 并写入数据库：

```java
// OrderSettingsServiceImpl.java
public void importOrderSettings(MultipartFile file) throws Exception {
    HSSFWorkbook workbook = new HSSFWorkbook(file.getInputStream());
    HSSFSheet sheet = workbook.getSheetAt(0);

    for (int i = 1; i <= sheet.getLastRowNum(); i++) {
        HSSFRow row = sheet.getRow(i);
        if (row == null) continue;

        // 读取日期列（第0列）和可预约人数列（第1列）
        String dateStr = row.getCell(0).getStringCellValue();
        Integer orderNum = (int) row.getCell(1).getNumericCellValue();

        // 解析日期并保存
        SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd");
        Date orderDate = sdf.parse(dateStr);

        OrderSettings orderSettings = new OrderSettings();
        orderSettings.setOrderDate(orderDate);
        orderSettings.setNumber(orderNum);
        orderSettingsDao.save(orderSettings);
    }
    workbook.close();
}
```

| POI 核心类      | 作用                        |
| ------------ | ------------------------- |
| HSSFWorkbook | 操作 Excel 97-2003 格式（.xls） |
| XSSFWorkbook | 操作 Excel 2007+ 格式（.xlsx）  |
| HSSFSheet    | 代表一个工作表                   |
| HSSFRow      | 代表一行                      |
| HSSFCell     | 代表一个单元格                   |

:::tip
POI 的坑：**日期格式**。Excel 单元格的日期在 POI 里读取出来是 `Double` 类型（距离1900年的天数），不是字符串。必须用 `HSSFDateUtil` 转换：`HSSFDateUtil.getJavaDate(cell.getNumericCellValue())`。
:::

### 模块三：微信端 — 套餐列表与 Freemarker 静态化

#### 业务场景

用户打开微信端，首先看到套餐列表，点击某个套餐查看详情。这是典型的 C 端访问场景，高并发下需要优化。

#### Freemarker 页面静态化

套餐详情页面访问量大但内容变动少，采用 Freemarker 生成静态 HTML，避免频繁查询数据库：

```java
// SetmealServiceImpl.java
public class SetmealServiceImpl implements SetmealService {

    @Autowired
    private FreeMarkerConfigurer freeMarkerConfigurer;

    @Autowired
    private Configuration configuration;

    public void generateMobileStaticPage() {
        try {
            // 1. 获取套餐列表数据
            List<Setmeal> setmeals = setmealDao.findAll();

            // 2. 生成套餐列表页（所有套餐的汇总页面）
            Template listTemplate = configuration.getTemplate("m_setmeal.html.ftl");
            Map<String, Object> dataModel = new HashMap<>();
            dataModel.put("setmealList", setmeals);
            listTemplate.process(dataModel,
                new FileWriter("target/m_setmeal.html"));

            // 3. 遍历生成每个套餐详情页（静态 HTML）
            for (Setmeal setmeal : setmeals) {
                generateDetailPage(setmeal);
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    private void generateDetailPage(Setmeal setmeal) throws Exception {
        Map<String, Object> dataModel = new HashMap<>();
        dataModel.put("setmeal", setmealDao.findById(setmeal.getId()));

        Template detailTemplate = configuration.getTemplate("setmeal_detail.html.ftl");
        String fileName = "target/setmeal_" + setmeal.getId() + ".html";
        detailTemplate.process(dataModel, new FileWriter(fileName));
    }
}
```

#### Freemarker 模板示例

### 模块五：定时任务 — Quartz + Redis 垃圾图片清理

#### 业务场景

用户上传套餐图片时，可能多次上传但最终只保存一张。历史图片成为「垃圾」，需要定期清理。

#### Quartz + Redis 解决方案

```java
// 定时任务类
@DisallowConcurrentExecution  // 禁止并发执行
public class CleanGarbageImageJob extends QuartzJobBean {

    @Autowired
    private JedisPool jedisPool;

    @Override
    protected void executeInternal(JobExecutionContext context)
            throws JobExecutionException {
        // 1. 获取数据库中所有正在使用的套餐图片
        List<String> usedImages = setmealDao.findAllImages();

        // 2. 获取 Redis 中缓存的所有图片
        Set<String> cachedImages = jedisPool.getResource().smembers("setmeal:images");

        // 3. 计算差集 = 垃圾图片
        for (String cachedImage : cachedImages) {
            if (!usedImages.contains(cachedImage)) {
                // 4. 删除 FastDFS 上的垃圾图片
                fastDfsClient.deleteFile(cachedImage);
                // 5. 从 Redis 移除
                jedisPool.getResource().srem("setmeal:images", cachedImage);
            }
        }
    }
}
```

| 组件                           | 作用               |
| ---------------------------- | ---------------- |
| Quartz Job                   | 定义定时任务的执行逻辑      |
| @DisallowConcurrentExecution | 确保任务不并发执行，避免数据竞争 |
| Redis Set                    | 存储已使用图片的快速查找     |
| 差集计算                         | 找出不在使用中的垃圾图片     |

### 模块六：文件存储 — FastDFS 分布式文件系统

#### 业务场景

套餐图片、会员头像、体检报告等文件需要集中存储。如果存在应用服务器的本地磁盘，分布式部署时就会出现「这台服务器有、那台没有」的问题。FastDFS 就是来解决这个问题的——一个分布式文件存储系统，所有服务器都从它那里读写文件。

#### FastDFS 核心原理

```
                    ┌─────────────────────────────────┐
                    │         FastDFS 集群            │
┌──────────────┐    │  ┌─────────────────────────┐   │
│   应用服务器   │───▶│  │    Storage Server 1    │   │
│  (上传/下载)  │    │  │  /group1/m00/xxx.jpg   │   │
└──────────────┘    │  └─────────────────────────┘   │
                    │  ┌─────────────────────────┐   │
                    │  │    Storage Server 2    │   │
                    │  │  /group1/m01/yyy.jpg   │   │
                    │  └─────────────────────────┘   │
                    └─────────────────────────────────┘
                              ▲
                              │ Tracker Server
                              │ (调度+管理)
                    ┌─────────────────────────────────┐
                    │       Nginx + fastdfs_nginx_module │
                    └─────────────────────────────────┘
```

| 角色                             | 职责                                             |
| ------------------------------ | ---------------------------------------------- |
| Tracker Server                 | 调度器，负责管理 Storage Server 的注册和心跳，客户端通过它获取可用的存储节点 |
| Storage Server                 | 实际存储文件的节点，按组（group）组织，每个组内文件互为备份               |
| Nginx + fastdfs\_nginx\_module | 提供 HTTP 访问入口，解决 Storage Server 跨组访问问题          |

#### 文件上传流程

```java
// FastDFS 文件上传工具类
public class FastDFSClient {

    // 加载 FastDFS 配置
    private static Properties props = new Properties();

    static {
        props.load(FastDFSClient.class.getClassLoader()
            .getResourceAsStream("fastdfs.conf"));
    }

    // 上传文件
    public String upload(MultipartFile file) throws Exception {
        // 1. 初始化 Tracker 连接
        ClientGlobal.init(props);

        // 2. 创建 Tracker 客户端
        TrackerClient trackerClient = new TrackerClient();
        TrackerServer trackerServer = trackerClient.getConnection();

        // 3. 获取 Storage 客户端
        StorageServer storageServer = trackerClient.getStoreStorage(trackerServer);

        // 4. 创建 Storage 客户端
        StorageClient1 storageClient = new StorageClient1(trackerServer, storageServer);

        // 5. 获取文件扩展名
        String originalFilename = file.getOriginalFilename();
        String ext = originalFilename.substring(
            originalFilename.lastIndexOf(".") + 1
        );

        // 6. 上传文件，返回文件 ID（如 group1/M00/00/00/xxx.jpg）
        String fileId = storageClient.upload_file1(file.getBytes(), ext, null);

        trackerServer.close();
        storageServer.close();

        return fileId;
    }
}
```

| 方法                 | 说明           |
| ------------------ | ------------ |
| `upload_file1()`   | 上传文件，返回文件 ID |
| `download_file1()` | 通过文件 ID 下载   |
| `delete_file1()`   | 通过文件 ID 删除   |

:::tip
**FastDFS vs 七牛云**：传智健康用 FastDFS，是因为课程要求掌握自建文件存储的能力。在[青橙商城](/articles/独立项目：青橙商城分布式电商实战/)里也用过七牛云——它是云服务，上传下载更简单，不需要自己搭集群。FastDFS 的优势是数据完全自主可控，适合企业内网部署；七牛云的优势是开箱即用，适合快速上线。两者各有适用场景。
:::

## 项目亮点

### 亮点一：Dubbo 接口设计分离

**问题现象**：消费方和提供方代码耦合，每次接口改动都要同步修改两边的代码。

**解决方案**：将服务接口抽离到独立的 `health_interface` 模块，作为消费方和提供方的共同依赖。

```java
// health_interface 模块：服务接口定义
public interface OrderService {
    Order findById(Integer id);
    List<Order> findByMemberId(Integer memberId);
    void save(Order order);
}
```

| 方案对比 | 耦合方案        | 接口分离方案            |
| ---- | ----------- | ----------------- |
| 代码依赖 | 提供方和消费方直接依赖 | 共同依赖 interface 模块 |
| 接口改动 | 两边都要改       | 只改接口，实现方自行迭代      |
| 部署耦合 | 必须同时部署      | 可独立部署             |

**经验总结**：接口分离的好处是接口契约固定，实现方和调用方可以独立迭代。就像 USB 接口标准，不管是谁生产的设备，只要符合协议就能用。

### 亮点二：Spring Security 细粒度权限控制

**问题现象**：不同角色的用户（系统管理员、健康管理师、健康咨询师）需要看到不同的菜单。

**解决方案**：基于 Spring Security 的 URL 权限 + 注解权限双重控制。

```java
// SecurityConfig.java
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/api/admin/**").hasRole("ADMIN")
                .antMatchers("/api/health/**").hasAnyRole("ADMIN", "HEALTH_MANAGER")
                .antMatchers("/api/report/**").hasAnyRole("ADMIN", "HEALTH_MANAGER", "CONSULTANT")
                .anyRequest().authenticated()
            .and()
            .httpBasic();
    }
}
```

| 角色    | 权限范围        |
| ----- | ----------- |
| 系统管理员 | 所有权限        |
| 健康管理师 | 健康业务全权（除删除） |
| 健康咨询师 | 查询 + 统计分析   |

:::note
Spring Security 的权限配置和[十次方项目](/articles/独立项目：黑马十次方社区平台/)用的 Shiro 不同。Spring Security 是 Spring 官方安全框架，和 Spring Boot 集成更紧密；Shiro 则是独立的安全框架，配置更灵活。两个都学一遍，对比着理解印象更深刻。
:::

### 亮点三：Freemarker 静态化 vs 十次方 NUXT SSR

传智健康用 Freemarker 做页面静态化，十次方用 NUXT 做服务端渲染（SSR），两者解决的问题类似——**减少重复查询、提升页面加载速度**——但实现思路完全不同：

| 对比维度    | Freemarker 静态化（传智健康） | NUXT SSR（十次方） |
| ------- | -------------------- | ------------- |
| 生成时机    | 提前生成静态 HTML 文件       | 运行时动态渲染       |
| 更新策略    | 数据变化后重新生成            | 每次请求重新渲染      |
| 适用场景    | 数据变化频率低的页面           | 需要实时数据的页面     |
| 服务器压力   | 几乎无压力（纯静态文件）         | 有一定压力（每次渲染）   |
| SEO 友好度 | 非常好（纯 HTML）          | 非常好（服务端渲染）    |

**经验总结**：Freemarker 像印刷报纸——提前印好，发的都是成品；NUXT 像快餐店的现点现做——每次都要厨师做，但能保证最新鲜。两者没有绝对优劣，只有适用场景的差异。

## 项目踩坑实录

### 踩坑一：Zookeeper 会话超时

**问题现象**：服务启动后运行一段时间，偶尔出现 `Session timeout` 错误，服务自动下线。

**根因分析**：

- Zookeeper 客户端心跳间隔默认是 30 秒
- 网络抖动导致心跳丢失，Zookeeper 认为服务宕机
- 服务提供者被从注册中心移除，消费方无法找到服务

**解决方案**：

```xml
<!-- dubbo.properties -->
dubbo.registry.parameters:
    session: 60000        # 会话超时时间调整为 60 秒
    alive: 30000          # 心跳检测间隔
```

**经验总结**：生产环境务必调整 Zookeeper 的超时参数，默认值在网络不稳定时容易出问题。

***

### 踩坑二：Dubbo 序列化失败

**问题现象**：服务调用时抛出 `serialization exception`，数据无法传输。

**根因分析**：

- Dubbo 默认使用 Hessian 序列化
- 传输的对象包含了不支持序列化的字段（如 `HttpSession`）
- 实体类使用了 Lombok 的 `@Data`，但部分字段没有 getter/setter

**解决方案**：

```java
// 确保实体类实现 Serializable
public class Member implements Serializable {
    private static final long serialVersionUID = 1L;
    // 移除任何与 HTTP 相关的对象引用
}
```

| 常见序列化问题          | 解决方法                    |
| ---------------- | ----------------------- |
| 对象包含 HttpSession | 传输对象必须是「纯数据对象」          |
| Lombok 注解问题      | 显式添加 getter/setter      |
| 日期类型不一致          | 统一使用 Long 时间戳或 JDK Date |

**经验总结**：Dubbo 传输的对象必须是「纯数据对象」，不能包含任何容器对象引用。

***

### 踩坑三：Spring Security 注解权限不生效

**问题现象**：配置了 `@PreAuthorize("hasAuthority('CHECKITEM_EDIT')")` 注解，但没有权限的用户仍然可以访问。

**根因分析**：

- Spring Security 和 Spring MVC 的组件扫描冲突
- `web.xml` 中同时使用了 `ContextLoaderListener` 和 `DispatcherServlet`，导致 Spring 容器嵌套
- 权限注解被外层容器的 Spring Security 配置管理，而不是内层的

**解决方案**：

```xml
<!-- web.xml：避免组件扫描冲突 -->
<!-- 只在 ContextLoaderListener 中扫描 service/dao 层 -->
<context-param>
    <param-name>contextConfigLocation</param-name>
    <param-value>classpath:spring-security.xml</param-value>
</context-param>

<!-- DispatcherServlet 只扫描 controller 层 -->
<servlet>
    <servlet-name>springmvc</servlet-name>
    <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
    <init-param>
        <param-name>contextConfigLocation</param-name>
        <param-value>classpath:spring-mvc.xml</param-value>
    </init-param>
    <!-- 禁止默认扫描，避免覆盖 Spring Security 配置 -->
</servlet>
```

| 常见问题                          | 原因            | 解决                           |
| ----------------------------- | ------------- | ---------------------------- |
| 注解权限不生效                       | Spring 容器扫描冲突 | 分层扫描，各管各的                    |
| hasAuthority vs hasPermission | 注解写错方法名       | 用 `hasAuthority`（对应数据库权限字符串） |
| Session 并发登录                  | 没有限制同一用户并发    | 配置 `maximumSessions(1)`      |

**经验总结**：Spring Security 的坑大多数出在配置层面，而不是代码层面。注解不生效先查 `web.xml` 和组件扫描范围，比查代码更有效率。

### 三代项目的技术演进

回顾这条学习路径，三个项目的技术选型有一条清晰的演进主线：

| 项目                                  | 时间线     | 架构                      | 关键技术                                                      |

**从青橙商城到传智健康**：架构一脉相承，但业务复杂度降了一个档次。传智健康的价值在于补齐了 SSM 生态的更多关键拼图：Freemarker 静态化、Apache POI 批量处理、FastDFS 文件存储、Spring Security 权限控制。青橙商城里没覆盖的内容，传智健康全部补上了。

**从十次方到传智健康**：虽然技术栈回退到了 SOA + Dubbo + SSM，但这种「退步」恰恰是巩固的过程。十次方的 Spring Cloud 微服务新概念太多，学得浮。回到传智健康，用熟悉的 Dubbo + Zookeeper 组合把业务吃透，把 SSM 生态的短板补齐，才算真正把 SOA 的核心原理夯实了。

:::note
三个项目走下来，最深的体会是：技术选型没有绝对的对错，只有「当前阶段适不适合」。SOA 和微服务都是手段，不是目的。能解决业务问题、团队能 hold 住的技术栈，就是好的技术栈。
:::
