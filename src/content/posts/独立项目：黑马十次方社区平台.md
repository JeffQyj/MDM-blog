## 服务注册与发现

### 为什么需要注册中心？

想象一个大型小区里有 17 栋楼（微服务），每栋楼有几十户（服务实例）。如果没有「物业管理处」（注册中心），你想给某户送快递（调用服务）：

- 你得记住每户的具体门牌号（IP + 端口）
- 某户搬迁了（服务重启/迁移），你还得更新记录
- 快递员累死，用户体验极差

有了物业管理处（Eureka），情况完全不同了：

- 每户入住时自动到物业登记（服务注册）
- 物业处有一份住户名册（服务列表）
- 快递员只需问物业就能找到目标住户（服务发现）
- 住户搬走时会主动销户（心跳续约 + 故障剔除）

:::tip
**注册中心就像小区物业管理处**：它不负责送快递（不参与业务），但没有它，整个小区就乱套了。
:::

### Eureka Server 配置

Eureka 作为注册中心，采用集群模式部署，避免单点故障。以下是十次方项目中 Eureka Server 的核心配置：

```yaml
# application.yml (Eureka Server)
server:
  port: 8761

spring:
  application:
    name: tensquare-eureka

eureka:
  instance:
    hostname: localhost
    prefer-ip-address: true
  client:
    register-with-eureka: false
    fetch-registry: false
    service-url:
      defaultZone: http://${eureka.instance.hostname}:${server.port}/eureka/
```

:::note
`register-with-eureka: false` 表示当前服务不做为客户端注册到其他 Eureka Server；`fetch-registry: false` 表示不获取注册表。在单节点模式下，这两个配置都必须设为 false。

**为什么 Eureka Server 本身不需要注册？**
因为物业管理处不需要再去另一个物业管理处登记——它本身就是最高管理机构。同理，在单机模式下，Eureka Server 只提供服务，不充当客户端。
:::

### Eureka Client 配置

将业务微服务注册到 Eureka Server，只需在 pom.xml 添加依赖并在 application.yml 中配置：

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>
```

```yaml
# application.yml (业务微服务)
spring:
  application:
    name: tensquare-user

eureka:
  instance:
    prefer-ip-address: true
    instance-id: ${spring.cloud.client.ip-address}:${server.port}
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/
```

主启动类添加 `@EnableEurekaClient` 注解即可完成注册。

:::tip
服务启动后，在 Eureka Dashboard 中可以看到所有注册的服务实例，包括 IP 地址、端口、健康状态等信息。这是微服务治理的基础，就像物业管理处的电子公示牌。
:::

## 服务间通信

### 为什么需要 Feign？

在没有 Feign 之前，微服务之间想「打电话」（调用）：

```java
// 原始方式：手动拼接 HTTP 请求，像用对讲机一样麻烦
String url = "http://tensquare-user/user/" + id;
User user = restTemplate.getForObject(url, User.class);
```

这就像让你记住每个同事的手机号、并且自己拨号一样——号码可能换（服务地址变化）、通话质量没保障（没有熔断）、出了问题难排查。

Feign 来了，就像给每个部门装了一部内线电话：

- 只需知道对方「部门名称」（服务名），不用记号码
- 打电话的方式和打本地电话一样简单（声明式接口）
- 有自动录音备份（熔断降级）
- 话务员自动处理转接（负载均衡）

### Feign 声明式接口

```java
@FeignClient(name = "tensquare-user", fallback = UserFeignFallback.class)
public interface UserClient {

    @GetMapping("/user/{id}")
    User findById(@PathVariable("id") String id);

    @PostMapping("/user/login")
    Map<String, Object> login(@RequestBody User user);
}
```

实现降级逻辑的 Fallback 类：

```java
@Component
public class UserFeignFallback implements UserClient {

    @Override
    public User findById(String id) {
        User user = new User();
        user.setId("0");
        user.setNickname("默认用户");
        return user;
    }

    @Override
    public Map<String, Object> login(User user) {
        Map<String, Object> result = new HashMap<>();
        result.put("success", false);
        result.put("message", "服务降级，当前用户服务不可用");
        return result;
    }
}
```

:::warning
Fallback 类必须实现 Feign 接口，并在 `@FeignClient` 注解中通过 `fallback` 属性指定。当远程服务调用失败时，Hystrix 会自动触发降级逻辑，返回 fallback 的响应。

**这就像公司的总机转接**：当分机占线时（服务不可用），总机自动转接到语音留言（降级处理），而不是让你一直等待或收到忙音。
:::

### 熔断器模式：电路保险丝

Hystrix 的熔断机制就像家里的「保险丝」：

| 概念   | 家庭保险丝        | Hystrix           |
| ---- | ------------ | ----------------- |
| 正常工作 | 电流正常通过       | 请求正常处理            |
| 过载触发 | 电流过大，保险丝发热熔断 | 错误率超过阈值           |
| 熔断状态 | 断开电路，保护电器    | 快速失败，不调用下游        |
| 自动恢复 | 冷却后更换保险丝     | sleepWindow 后尝试半开 |
| 半开状态 | 换好保险丝，测试能否用  | 放行部分请求测试          |
| 恢复成功 | 电路恢复正常       | 关闭熔断器             |

### 启用 Hystrix 熔断

```yaml
feign:
  hystrix:
    enabled: true

hystrix:
  command:
    default:
      execution:
        isolation:
          thread:
            timeoutInMilliseconds: 5000
```

Hystrix 的核心配置参数：

| 参数                                       | 说明         | 默认值    | 家庭比喻          |
| ---------------------------------------- | ---------- | ------ | ------------- |
| timeoutInMilliseconds                    | 超时时间       | 1000ms | 等待电话接通的最长时间   |
| circuitBreaker.requestVolumeThreshold    | 触发熔断的最小请求数 | 20     | 至少打多少通电话才考虑熔断 |
| circuitBreaker.sleepWindowInMilliseconds | 熔断后尝试恢复时间  | 5000ms | 保险丝冷却需要多久     |
| circuitBreaker.errorThresholdPercentage  | 触发熔断的错误比例  | 50%    | 多少比例电话打不通就熔断  |

## API 网关与权限认证

### 网关：小区门神

Zuul 网关就像小区的「门神」：

```
        访客请求
            │
            ▼
    ┌───────────────┐
    │   Zuul 门神   │
    │  ┌─────────┐  │
    │  │1.验身份 │  │  ← JWT 令牌校验
    │  │2.查权限 │  │  ← 角色权限判断
    │  │3.记日志 │  │  ← 请求记录
    │  │4.做路由 │  │  ← 转发到对应楼栋
    │  └─────────┘  │
    └───────────────┘
            │
            ▼
    ┌───────────────┐
    │  目标微服务    │
    │   (某栋楼)     │
    └───────────────┘
```

没有网关的后果：

- 每人随便进，security 全靠「自觉」（每个服务自己验权）
- 访客找不到路（没有统一路由）
- 出事了不知道是谁进来的（没有日志）
- 门卫（每个服务）各管各的，效率低下

### Zuul 网关配置

```yaml
# application.yml
spring:
  application:
    name: tensquare-web

zuul:
  routes:
    tensquare-user:
      path: /user/**
      service-id: tensquare-user
    tensquare-article:
      path: /article/**
      service-id: tensquare-article
  sensitive-headers: Cookie,Set-Cookie,Authorization
```

:::note
`routes` 配置定义了「门神」的指路规则：看到 `/user/**` 的请求，就转发到 tensquare-user 服务。

`sensitive-headers` 配置指定哪些请求头不允许传递到下游服务。默认情况下 Zuul 会过滤掉 Cookie、Set-Cookie、Authorization。如果需要传递这些头，需要显式配置为空（表示不屏蔽任何头）。
:::

### JWT 令牌验证过滤器

在 Zuul 网关层统一处理 JWT 认证：

```java
@Component
public class JwtAuthFilter extends ZuulFilter {

    @Override
    public String filterType() {
        return "pre"; // 前置过滤器，在路由之前执行
    }

    @Override
    public int filterOrder() {
        return 0; // 优先级，数字越小越先执行
    }

    @Override
    public boolean shouldFilter() {
        // 对登录接口放行
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest request = ctx.getRequest();
        return !request.getRequestURI().contains("/login");
    }

    @Override
    public Object run() {
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest request = ctx.getRequest();

        String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            ctx.setSendZuulResponse(false);
            ctx.setResponseStatusCode(401);
            return null;
        }

        String token = authHeader.substring(7);
        try {
            Claims claims = Jwts.parserBuilder()
                    .setSigningKey("tensquare-secret-key".getBytes())
                    .build()
                    .parseClaimsJws(token)
                    .getBody();

            // 将用户信息传递给下游服务
            ctx.addZuulRequestHeader("X-User-Id", claims.getSubject());
            ctx.addZuulRequestHeader("X-User-Roles",
                    claims.get("roles", String.class));

        } catch (Exception e) {
            ctx.setSendZuulResponse(false);
            ctx.setResponseStatusCode(401);
        }
        return null;
    }
}
```

### 什么是 JWT？

**JWT（JSON Web Token）** 就像一张「游乐园通行证」：

| 组成部分          | 游乐园类比   | 说明               |
| ------------- | ------- | ---------------- |
| Header（头部）    | 通行证材质   | 标注类型（JWT）和签名算法   |
| Payload（载荷）   | 通行证上的信息 | 包含用户 ID、角色、过期时间等 |
| Signature（签名） | 防伪水印    | 防止通行证被伪造         |

```java
public class JwtUtil {

    private static final String SECRET_KEY = "tensquare-secret-key";
    private static final long EXPIRATION = 86400000L; // 24小时

    public static String generateToken(String userId, String roles) {
        return Jwts.builder()
                .setSubject(userId)
                .claim("roles", roles)
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
                .signWith(SignatureAlgorithm.HS256, SECRET_KEY.getBytes())
                .compact();
    }

    public static Claims parseToken(String token) {
        return Jwts.parserBuilder()
                .setSigningKey(SECRET_KEY.getBytes())
                .build()
                .parseClaimsJws(token)
                .getBody();
    }

    public static boolean validateToken(String token) {
        try {
            parseToken(token);
            return true;
        } catch (Exception e) {
            return false;
        }
    }
}
```

## 缓存策略设计

### 为什么需要缓存？

**没有缓存的场景**：

```
用户请求文章 → 查询数据库 → 返回结果（耗时 500ms）
用户请求文章 → 查询数据库 → 返回结果（耗时 500ms）
用户请求文章 → 查询数据库 → 返回结果（耗时 500ms）
```

想象餐厅里，顾客每点一道菜都要去「农场现抓鸡」——效率低下，顾客等待时间长。

**有缓存的场景**：

```
用户请求文章 → 查询缓存（命中） → 直接返回（耗时 5ms）
用户请求文章 → 查询缓存（命中） → 直接返回（耗时 5ms）
用户请求文章 → 查询缓存（未命中） → 查询数据库 → 写入缓存 → 返回（耗时 505ms）
```

这就变成了：厨房里有「预制菜」（缓存），大部分需求直接加热就能上桌；没准备的话再去现做，并补货到冰柜。

### Redis 缓存实现

```java
@Service
public class ArticleService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String ARTICLE_KEY = "article:";

    public Article findById(String id) {
        String cacheKey = ARTICLE_KEY + id;
        String cached = redisTemplate.opsForValue().get(cacheKey);

        if (cached != null) {
            return JSON.parseObject(cached, Article.class);
        }

        Article article = articleMapper.selectById(id);
        redisTemplate.opsForValue().set(cacheKey,
                JSON.toJSONString(article),
                1, TimeUnit.HOURS);

        return article;
    }
}
```

:::tip
**缓存 Key 命名规范**：建议采用 `业务名:主键:值` 的格式，如 `article:12345`，便于问题排查和缓存管理。

就像仓库库位编号：`A区-12排-05号`，一看到就知道是什么东西存在哪里。
:::

### 缓存三大经典问题

| 问题       | 形象比喻      | 问题描述                    | 解决思路                    |
| -------- | --------- | ----------------------- | ----------------------- |
| **缓存穿透** | 拿着假钥匙频繁开门 | 查询一个不存在的数据，每次都穿透到数据库    | 布隆过滤器判断 key 是否存在，或缓存空值  |
| **缓存击穿** | 爆款商品被抢光了  | 热点 key 过期瞬间，大量请求同时涌入数据库 | 互斥锁（只有一个线程去查库）或热点数据永不过期 |
| **缓存雪崩** | 仓库突然全部过期  | 大量 key 同时过期，或 Redis 宕机  | 过期时间加随机值，或 Redis 高可用集群  |

## 消息队列异步通信

### 什么是消息队列？

**消息队列**就像高铁站的「自动寄存柜」：

| 场景     | 高铁站寄存柜      | RabbitMQ   |
| ------ | ----------- | ---------- |
| 存行李    | 把行李放进柜子，拿小票 | 生产者发送消息到队列 |
| 取行李    | 拿小票去开柜门     | 消费者从队列取消息  |
| 24小时可取 | 行李会保存一段时间   | 消息持久化      |
| 柜子满了   | 得等有空柜子      | 队列满了生产者阻塞  |
| 多窗口取   | 多个窗口同时取     | 多个消费者并发消费  |

**为什么需要消息队列？**

比如十次方中「发布文章」功能：

**没有 MQ**：同步流程，耗时 2 秒

```
用户点击发布 → 存入文章表(100ms) → 更新搜索索引(500ms) → 发送通知(400ms) → 返回成功(2s)
```

**有 MQ**：异步流程，耗时 100ms

```
用户点击发布 → 存入文章表(100ms) → 发送「待索引」消息到队列 → 立即返回成功(100ms)
                      ↓
              搜索服务异步处理「待索引」消息
                      ↓
              通知服务异步发送「文章更新」消息
```

就像点外卖：店家接单后让你先吃（快速返回），后厨继续做菜、骑手取餐配送都是异步进行的。

### RabbitMQ 消息发送与消费

```java
// 消息生产者
@Service
public class ArticleService {

    @Autowired
    private AmqpTemplate amqpTemplate;

    public void publishArticle(Article article) {
        articleMapper.insert(article);
        // 发送消息通知搜索服务更新索引
        amqpTemplate.convertAndSend("article.exchange",
                "article.publish", article.getId());
    }
}
```

```java
// 消息消费者
@Component
public class SearchListener {

    @RabbitListener(queues = "search.queue")
    public void handleArticlePublish(String articleId) {
        Article article = articleService.findById(articleId);
        searchService.indexArticle(article);
    }
}
```

```yaml
# RabbitMQ 配置
spring:
```

:::warning
**分库分表**是什么？

想象一个超大型小区，有 100 万住户。如果所有住户信息都存在一个「超级档案室」：

- 找一个人要翻很久（查询慢）
- 档案室爆满了（存储不够）
- 管理员忙不过来（并发瓶颈）

分库分表就是：按照一定规则（比如按姓氏字母分区），把档案分成多个档案室：

| 档案室   | 存放范围       |
| ----- | ---------- |
| 档案室 A | 姓氏 A-D 的住户 |
| 档案室 B | 姓氏 E-H 的住户 |
| 档案室 C | 姓氏 I-L 的住户 |
| ...   | ...        |

这样，每个档案室的压力就小了，查询效率大幅提升！
:::

## 业务模块实战

技术架构是骨架，业务代码是血肉。以下从 18 个微服务中选取 6 个最具代表性的模块，展示它们的核心业务逻辑。

### 问答模块：问题发布与标签筛选

问答模块（tensquare\_qa）是十次方中「关系最复杂」的业务——一个问题对应多个标签、一个回答对应一个问题。

**核心实体关系**：

| 实体    | 表名          | 关系说明            |
| ----- | ----------- | --------------- |
| 问题    | tb\_problem | 主表，存储问题标题、内容、状态 |
| 回答    | tb\_reply   | 一问题对多回答（一对多）    |
| 标签    | tb\_label   | 问题与标签的关联表       |
| 问题-标签 | tb\_pl      | 多对多关联表          |

**问题发布的关键代码**：

```java
@Service
public class ProblemService {

    @Autowired
    private ProblemMapper problemMapper;

    @Autowired
    private ProblemLabelMapper problemLabelMapper;

    @Autowired
    private AmqpTemplate amqpTemplate;

    public Problem publish(Problem problem, List<String> labelIds) {
        problem.setCreatetime(new Date());
        problem.set updatetime(new Date());
        problem.setVisits(0L);      // 初始浏览量为0
        problem.setThumbup(0L);      // 初始点赞数为0
        problem.setReply(0L);        // 初始回复数为0
        problem.setState("0");       // 0=待解决，1=已解决

        problemMapper.insert(problem);

        // 建立问题与标签的关联
        if (labelIds != null && labelIds.size() > 0) {
            for (String labelId : labelIds) {
                ProblemLabel pl = new ProblemLabel();
                pl.setProblemId(problem.getId());
                pl.setLabelId(labelId);
                problemLabelMapper.insert(pl);
            }
        }

        // 发布消息：通知搜索服务建立索引
        amqpTemplate.convertAndSend("problem.exchange", "problem.publish", problem.getId());

        return problem;
    }
}
```

**标签筛选查询**：查询「标签为 Java 且状态为待解决」的问题：

```java
public List<Problem> findByLabel(String labelId, String state, int page, int size) {
    // MyBatis Plus 条件构造器
    LambdaQueryWrapper<Problem> wrapper = new LambdaQueryWrapper<>();

    // 关联标签表查询
    wrapper.eq(Problem::getState, state)
           .inSql("id", "SELECT problem_id FROM tb_pl WHERE label_id = '" + labelId + "'")
           .orderByDesc(Problem::getCreatetime);

    Page<Problem> problemPage = new Page<>(page, size);
    IPage<Problem> result = problemMapper.selectPage(problemPage, wrapper);

    return result.getRecords();
}
```

对应的 SQL 本质是：

```sql
SELECT p.* FROM tb_problem p
INNER JOIN tb_pl pl ON p.id = pl.problem_id
WHERE p.state = '0' AND pl.label_id = 'java'
ORDER BY p.createtime DESC
LIMIT 0, 10
```

:::tip
**INNER JOIN vs LEFT JOIN**：

- INNER JOIN：只返回两表有关联的数据——只会返回有标签的问题
- LEFT JOIN：返回左表所有数据，右表没有的显示 NULL——会返回所有问题，包括没标签的

选择哪种 JOIN，取决于业务需求。
:::

```java
@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;

    @Autowired
    private PasswordEncoder passwordEncoder;

    @Autowired
    private JwtUtil jwtUtil;

    public Map<String, Object> register(User user, String code) {

        // 1. 验证码校验（省略，实际从 Redis 获取验证码）
        if (!"1234".equals(code)) {
            result.put("success", false);
            result.put("message", "验证码错误");
            return result;

        User existUser = userMapper.selectByUsername(user.getUsername());
        if (existUser != null) {
            result.put("success", false);
            result.put("message", "用户名已存在");
            return result;
        }

        // 3. 密码加密存储
        user.setPassword(passwordEncoder.encode(user.getPassword()));
        user.setCreatetime(new Date());
        user.setLogintime(new Date());

        userMapper.insert(user);

        // 4. 生成 JWT 令牌
        String token = jwtUtil.generateToken(user.getId(), "user");

        result.put("success", true);
        result.put("message", "注册成功");
        result.put("data", token);
        return result;
    }

    public Map<String, Object> login(String username, String password) {
        Map<String, Object> result = new HashMap<>();

        User user = userMapper.selectByUsername(username);
        if (user == null) {
            result.put("success", false);
            result.put("message", "用户不存在");
            return result;
        }

        if (!passwordEncoder.matches(password, user.getPassword())) {
            result.put("success", false);
            result.put("message", "密码错误");
            return result;
        }

        // 更新登录时间
        user.setLogintime(new Date());
        userMapper.updateById(user);

        String token = jwtUtil.generateToken(user.getId(), "user");
        result.put("success", true);
        result.put("data", token);
        return result;
    }
```

**好友关系管理**：交友微服务的核心是维护「关注」关系表 tb\_friend：

```java
@Service
public class FriendService {

    @Autowired
    private FriendMapper friendMapper;

    @Autowired
    private UserClient userClient;  // Feign 调用用户服务

    public Result addFriend(String userId, String friendId, String type) {
        // type: "1"=喜欢（添加好友），"0"=不喜欢（添加非好友）
        Friend friend = new Friend();
        friend.setUserid(userId);
        friend.setFriendid(friendId);
        friend.setType(type);
        friend.setCreatetime(new Date());

        // 防止重复添加
        Friend exist = friendMapper.selectByUserAndFriend(userId, friendId);
        if (exist != null) {
            return Result.fail("不能重复添加");
        }

        friendMapper.insert(friend);

        // 如果是添加好友，同时更新对方的粉丝数和自己的关注数
        if ("1".equals(type)) {
            userClient.updateFollowCount(friendId, " follower");  // 更新粉丝数
            userClient.updateFollowCount(userId, " following");  // 更新关注数
        }

        return Result.success("添加成功");
    }

    public Result deleteFriend(String userId, String friendId) {
        friendMapper.deleteByUserAndFriend(userId, friendId);

        // 更新统计数据
        userClient.updateFollowCount(friendId, " follower");  // 对方粉丝数 -1
        userClient.updateFollowCount(userId, " following");  // 自己的关注数 -1

        return Result.success("删除成功");
    }
}
```

**文章收藏功能**：

```java
@Service
public class ArticleService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String COLLECTION_KEY = "article:collection:";

    public Result collect(String articleId, String userId) {
        String key = COLLECTION_KEY + articleId;

        // 检查是否已收藏
        Boolean isCollected = redisTemplate.opsForSet().isMember(key, userId);
        if (Boolean.TRUE.equals(isCollected)) {
            // 取消收藏
            redisTemplate.opsForSet().remove(key, userId);
            articleMapper.decrementCollection(articleId);
            return Result.success("已取消收藏");
        } else {
            // 添加收藏
            redisTemplate.opsForSet().add(key, userId);
            articleMapper.incrementCollection(articleId);
            return Result.success("收藏成功");
        }
    }

    public boolean isCollected(String articleId, String userId) {
        String key = COLLECTION_KEY + articleId;
        return Boolean.TRUE.equals(redisTemplate.opsForSet().isMember(key, userId));
    }
}
```

:::note
**为什么要用 Redis Set 存储收藏状态？**

- 查收藏：SISMEMBER 命令，时间复杂度 O(1)，比数据库快
- 去重：Set 自动去重，同一用户不能重复收藏
- 统计：SCARD 命令可以快速获取收藏数

但要注意：Redis 是缓存，最终一致性有延迟。如果对一致性要求高，需要配合数据库表存储真实的收藏记录。
:::

```java
public Page<ArticleDocument> search(String keyword, int page, int size) {
    // 构建查询条件
    NativeSearchQuery query = new NativeSearchQueryBuilder()
            .withQuery(QueryBuilders.multiMatchQuery(keyword,
                    "title",           // 搜索标题
                    "content",         // 搜索内容
                    "labelnames"))     // 搜索标签
            .withPageable(PageRequest.of(page, size))
            .withSort(SortBuilders.scoreSort())          // 按相关性排序
            .withHighlightFields(
                    new HighlightBuilder.Field("title"),
                    new HighlightBuilder.Field("content")
            )
            .build();

    // 执行搜索
    SearchHits<ArticleDocument> hits = elasticsearchTemplate.search(query, ArticleDocument.class);

    // 构建高亮结果
    List<ArticleDocument> results = new ArrayList<>();
    for (SearchHit<ArticleDocument> hit : hits.getSearchHits()) {
        ArticleDocument doc = hit.getContent();

        // 替换高亮片段
        Map<String, List<String>> highlightFields = hit.getHighlightFields();
        if (highlightFields.containsKey("title")) {
            doc.setTitle(highlightFields.get("title").get(0));
        }

        results.add(doc);
    }

    return new PageImpl<>(results, PageRequest.of(page, size), hits.getTotalHits());
}
```

### 活动微服务：报名签到与状态机

活动微服务（tensquare\_gathering）涉及活动发布、报名、签到等功能，是典型的「状态机」场景。

**活动报名**：

```java
@Service
public class GatheringService {

    public Result signUp(String gatheringId, String userId) {
        // 1. 查询活动信息
        Gathering gathering = gatheringMapper.selectById(gatheringId);
        if (gathering == null) {
            return Result.fail("活动不存在");
        }

        // 2. 检查活动状态
        if (!"1".equals(gathering.getState())) {
            return Result.fail("活动未开始报名");
        }

        // 3. 检查是否已满员
        if (gathering.getSignupnumber() >= gathering.getMaxpeople()) {
            return Result.fail("报名已满");
        }

        // 4. 检查是否已报名
        Participation participation = participationMapper.selectByGatheringAndUser(gatheringId, userId);
        if (participation != null) {
            return Result.fail("您已报名此活动");
        }

        // 5. 创建报名记录
        participation = new Participation();
        participation.setGatheringid(gatheringId);
        participation.setUserid(userId);
        participation.setState("0");  // 0=已报名未签到
        participation.setSigntime(null);
        participation.setCreatetime(new Date());

        participationMapper.insert(participation);

        // 6. 更新活动报名人数
        gathering.setSignupnumber(gathering.getSignupnumber() + 1);
        gatheringMapper.updateById(gathering);

        return Result.success("报名成功");
    }

    public Result checkIn(String gatheringId, String userId) {
        Participation participation = participationMapper.selectByGatheringAndUser(gatheringId, userId);
        if (participation == null) {
            return Result.fail("您未报名此活动");
        }

        if (!"0".equals(participation.getState())) {
            return Result.fail("您已签到或已取消");
        }

        // 更新签到状态
        participation.setState("1");  // 1=已签到
        participation.setSigntime(new Date());
        participationMapper.updateById(participation);

        // 更新活动签到人数
        gatheringMapper.incrementCheckinnumber(gatheringId);

        return Result.success("签到成功");
    }
}
```

**活动状态流转**：

```
    创建活动
       │
       ▼
  ┌─────────┐     报名人数已满      ┌──────────┐
  │ 报名中   │ ─────────────────► │ 报名结束  │
  └─────────┘                      └──────────┘
       │                                │
       │     活动开始时间到达            │
       ▼                                ▼
  ┌─────────┐     所有人签到完成      ┌──────────┐
  │ 进行中   │ ─────────────────► │ 已结束   │
  └─────────┘                      └──────────┘
```

:::note
**状态机设计的好处**：

1. **边界清晰**：每个状态只能转到特定的后继状态，不会出现「已签到」直接变回「报名中」的非法流转
2. **易于审计**：活动进行到哪一步一目了然
3. **便于扩展**：新增状态或流转规则只需修改状态转换表
   :::

### 交友微服务：好友推荐与黑名单

交友微服务（tensquare\_friend）是最考验「关系设计」的模块，涉及关注、取关、拉黑、好友申请等。

**好友推荐算法（简单版）**：

```java
public List<User> recommend(String userId, int limit) {
    // 1. 获取用户已关注的好友 ID 列表
    List<String> followingIds = friendMapper.selectFollowingIds(userId);

    // 2. 获取用户已拉黑的好友 ID 列表
    List<String> blackIds = noFriendMapper.selectBlackIds(userId);

    // 3. 找出「朋友的朋友」——可能认识的人
    List<String> friendOfFriends = new ArrayList<>();
    for (String friendId : followingIds) {
        List<String> fof = friendMapper.selectFollowingIds(friendId);
        friendOfFriends.addAll(fof);
    }

    // 4. 排除自己、已关注的人、黑名单
    friendOfFriends.removeAll(followingIds);
    friendOfFriends.removeAll(blackIds);
    friendOfFriends.remove(userId);

    // 5. 统计出现次数（出现越多，推荐优先级越高）
    Map<String, Long> frequency = friendOfFriends.stream()
            .collect(Collectors.groupingBy(id -> id, Collectors.counting()));

    // 6. 按频率排序，返回前 N 个
    return frequency.entrySet().stream()
            .sorted(Map.Entry.<String, Long>comparingByValue().reversed())
            .limit(limit)
            .map(entry -> userClient.findById(entry.getKey()))
            .collect(Collectors.toList());
}
```

**拉黑与取消拉黑**：

```java
public Result block(String userId, String blackUserId) {
    // 双向拉黑：在 tb_friend 和 tb_nofriend 表同时插入记录
    // 1. 移除好友关系（如果存在）
    friendMapper.deleteByUserAndFriend(userId, blackUserId);
    friendMapper.deleteByUserAndFriend(blackUserId, userId);

    // 2. 添加到黑名单
    NoFriend noFriend = new NoFriend();
    noFriend.setUserid(userId);
    noFriend.setFriendid(blackUserId);
    noFriend.setCreatetime(new Date());
    noFriendMapper.insert(noFriend);

    // 3. 更新统计数
    userClient.updateFollowCount(userId, " following", -1);  // 关注数 -1
    userClient.updateFollowCount(blackUserId, " follower", -1);  // 粉丝数 -1

    return Result.success("已拉黑");
}
```

:::tip
**为什么推荐算法是「朋友的朋友」？**

这基于「六度分隔理论」：世界上任意两个人之间，最多只隔 6 个人。你的朋友认识的人，更可能成为你的朋友。

实际生产环境中，推荐系统会更复杂，可能会结合：

- 共同标签（都关注了「Java」话题）
- 地理位置（都在北京）
- 互动行为（给你点赞/评论过的人）

作为课程项目，十次方采用简化的「朋友的朋友」算法，已经能展示核心思路了。
:::

## 踩坑实录

### 1. 服务间调用超时

**问题现象**：在网络波动时，服务间调用频繁触发熔断降级，导致用户体验下降。

**根因分析**：初期配置的 Hystrix 超时时间过短（默认 1000ms），而实际网络延迟加上业务处理时间经常超过这个阈值。

**什么是「超时」？**

就像打电话时：

- 你说「喂？」（发送请求）
- 对方没回应...
- 你等 1 秒（超时时间），觉得不对劲
- 挂掉电话，说「打不通」（触发降级）

问题是：有时候不是对方没接，只是信号不好（网络延迟）或对方正在处理（业务耗时长），1 秒太短了！

**解决方案**：

```yaml
hystrix:
  command:
    default:
      execution:
        isolation:
          thread:
            timeoutInMilliseconds: 5000  # 调整为 5 秒
      circuitBreaker:
        requestVolumeThreshold: 20       # 触发熔断的最小请求数
        sleepWindowInMilliseconds: 5000   # 熔断后 5 秒尝试恢复
        errorThresholdPercentage: 50      # 50% 错误率触发熔断
```

:::tip
超时时间设置并非越长越好。过长的超时会导致线程资源被长时间占用，在高并发场景下可能引发线程池耗尽。就像电话占线时间太长，后面想打电话的人全都得排队等着。
:::

### 2. 数据库连接池耗尽

**问题现象**：在压测过程中，MySQL 连接池迅速耗尽，后续请求堆积，最终导致服务雪崩。

**什么是「数据库连接池」？**

想象餐厅的「等位系统」：

| 概念     | 餐厅类比   | 技术解释         |
| ------ | ------ | ------------ |
| 连接池    | 等位区的椅子 | 预先建立的数据库连接集合 |
| 获取连接   | 顾客坐下等位 | 从池中取一个连接     |
| 释放连接   | 顾客吃完离开 | 把连接返还池中      |
| 连接池耗尽  | 等位区坐满了 | 所有连接都在使用中    |
| 等待获取连接 | 门口排队   | 请求阻塞等待       |

**什么是「慢查询」？**

就像点了一道「佛跳墙」：

- 普通炒饭（下饭馆）：5 分钟上桌（普通查询 5ms）
- 佛跳墙（慢查询）：2 小时才做好（耗时 2 秒以上）

而餐厅厨师（数据库）是固定的，如果佛跳墙占用了太多厨师（连接），其他菜都做不了了。

**慢查询的常见原因**：

1. **没建索引**：在 100 万人的名单里找「张三」，不按姓氏排序，得一个个翻
2. **SQL 写得太烂**：`SELECT * FROM user WHERE name LIKE '%张%'` 全表扫描
3. **数据量太大**：查一年的日志，硬盘 IO 扛不住
4. **关联太多表**：join 了七八张表，像点了个全家桶套餐

**解决方案**：

```yaml
spring:
  datasource:
    druid:
      initial-size: 5          # 餐厅开门时准备 5 把椅子
      min-idle: 5               # 空闲时至少保持 5 把椅子可用
      max-active: 20            # 最多同时接待 20 位顾客
      max-wait: 60000           # 等位超过 60 秒就放弃
      time-between-eviction-runs-millis: 60000  # 每分钟检查一次椅子是否需要清理
```

:::warning
连接池配置需要根据服务器硬件配置和实际并发量来调整。不能盲目追求大连接数，否则反而会加重数据库负担。同时，务必配合慢查询日志进行 SQL 优化。

**优化慢查询的常用手段**：

| 方法     | 做法                   | 类比                |
| ------ | -------------------- | ----------------- |
| 加索引    | 在常用查询字段上建索引          | 按姓氏建档案柜，不用一个个翻    |
| 优化 SQL | 避免 SELECT \*、少用 LIKE | 点菜直接说菜名，别说「来点好吃的」 |
| 分页查询   | LIMIT 分批取            | 全套菜单太长，按类别翻页看     |
| 读写分离   | 主库写、从库读              | 点餐和结账分开，提高效率      |
| 加缓存    | 热点数据放 Redis          | 招牌菜提前备好，不用现做      |
:::

### 3. 配置中心更新不生效

**问题现象**：修改 Git 仓库中的配置文件后，通过 Spring Cloud Bus 触发刷新，但部分微服务配置未生效。

**什么是「配置中心」？**

想象连锁酒店：

| 场景   | 连锁酒店           | Spring Cloud Config |
| ---- | -------------- | ------------------- |
| 集中管理 | 总部统一制定服务标准     | 所有配置存在 Git 仓库       |
| 实时更新 | 总部发通知，各店更新     | 配置变更后 Bus 广播        |
| 配置内容 | 房价、早餐时间、WiFi密码 | 数据库连接、Redis 地址、超时时间 |

**为什么配置更新不生效？**

1. **配置文件命名不规范**：就像酒店连锁店太多，总部通知发错了店
2. **@RefreshScope 漏加**：某些配置「钉死」在内存里，没刷新
3. **消息总线故障**：通知没发出去，部分店没收到

**解决方案**：

```java
// 确保需要刷新的配置类添加 @RefreshScope 注解
@RefreshScope
@Configuration
public class RedisConfig {
    @Value("${spring.redis.host:localhost}")
    private String host;

    @Bean
    public RedisTemplate<String, String> redisTemplate() {
        RedisTemplate<String, String> template = new RedisTemplate<>();
        template.setConnectionFactory(
            new JedisConnectionFactory(new JedisShardInfo(host, 6379))
        );
        return template;
    }
}
```

手动触发配置刷新的接口：

```bash
curl -X POST http://localhost:端口号/actuator/bus-refresh
```

:::note
**哪些配置能热更新？**

| 类型          | 能否热更新  | 说明                  |
| ----------- | ------ | ------------------- |
| @Value 注入的值 | ✅ 可以   | 运行时重新读取             |
| @Bean 创建的对象 | ⚠️ 看情况 | 加上 @RefreshScope 可以 |
| 数据库连接池大小    | ❌ 不行   | 属于「初始化配置」，改了要重启     |
| 线程池大小       | ❌ 不行   | 同上                  |

**配置热更新虽然方便，但并非所有配置都能动态刷新**。涉及到 Bean 初始化的配置需要重启服务才能生效。在生产环境中，重要配置变更仍建议通过滚动更新来处理。
:::
