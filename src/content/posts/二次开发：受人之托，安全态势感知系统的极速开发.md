## 缘起：两件事，一起拜托

2026年3月初，朋友同时拜托了我两件事。

第一件，是LaTeX论文排版——《成长碎记：受人之托，一次跨越困难的LaTeX学习之旅》记录的就是这段经历。

第二件，是一套毕业设计系统——「基于OpenEuler的安全态势感知系统，老师给的选题我自己定了，但具体怎么做我真的不会……」

两件事同时拜托，但有轻重缓急之分——论文明确要用LaTeX排版，系统可以后做。所以，我先学了LaTeX，然后才回过头来做系统。

朋友说他的编码能力实在不行，编程基础基本为零，更别说让他从零实现一个完整的系统了。

关于系统的资料只有一份开题报告。

开题报告我见过，通常只讲「要做什么」和「为什么做」，不会讲「怎么做」和「做成什么样」。换句话说，我不仅要做系统，还要从开题报告的描述里，自己分析出需求，自己决定技术选型，自己设计系统架构。

时间只有不到一周。

:::note
LaTeX那篇文章里我说过：人的成长一定在于解决问题，而解决问题的前提，是承担责任。这次更极端——朋友定了选题，但实现方案全靠我自己想。没有需求文档，没有技术要求，一切从0到1。
:::

## 开局：没有需求文档，只有开题报告

朋友把开题报告发过来，我仔细看了几遍。

说实话，开题报告写得比较笼统，基本上就是：「本课题旨在设计并实现一个面向OpenEuler操作系统的安全态势感知系统，包括系统资源监控、安全威胁检测、自动化响应等核心功能。」

就这么一句话，要我展开成一整套系统。

**我能确定的信息**：

| 信息   | 说明                      |
| ---- | ----------------------- |
| 操作系统 | OpenEuler（国产开源系统）       |
| 目标   | 安全态势感知系统                |
| 方向   | 系统资源监控 + 安全威胁检测 + 自动化响应 |

**我需要自己决定的信息**：

| 信息     | 我需要自己决定             |
| ------ | ------------------- |
| 具体功能模块 | 从「态势感知」四个字里自己拆      |
| 技术选型   | 用什么框架、什么数据库、什么可视化方案 |
| 系统架构   | B/S还是C/S，前后端如何分离    |
| 实现细节   | 每个功能怎么实现            |

这就是最难的地方——**没有标准答案，所有决定都要自己做**。

## 破局：技术选型的思考过程

关于技术选型，我没有犹豫太久——蝶梦项目的经验告诉我，若依是最合适的选择（尤其时间如此紧迫的情况下，不可能从头开发）：

| 对比   | 从零开发   | 使用若依       |
| ---- | ------ | ---------- |
| 开发周期 | 2-3个月  | 1-2周（极限压缩） |
| 权限系统 | 自己实现   | 开箱即用       |
| 监控接口 | 全部手写   | 框架自带       |
| 文档生态 | 依赖自己积累 | 社区完善       |

时间太紧了，我没有试错成本。若依的监控模块和安全框架与这个项目的需求高度契合——「态势感知」需要的基础监控能力，若依原生就有。

### 技术架构

**技术选型理由**：

| 技术                         | 选择理由                |
| -------------------------- | ------------------- |
| RuoYi-Vue3                 | 蝶梦项目用过，上手快，监控模块开箱即用 |
| Vue3 + Element UI          | 组合式API，组件化开发，效率高    |
| ECharts                    | 安全态势需要图表展示，成熟稳定     |
| Spring Boot + MyBatis Plus | Java最熟悉的生态，CRUD效率高  |
| MySQL + Redis              | 若依原生支持，够用           |

**核心功能模块**（我自己从「态势感知」拆出来的）：

| 模块     | 功能      | 说明                  |
| ------ | ------- | ------------------- |
| 安全仪表盘  | 可视化展示   | 安全评分、事件分布、实时监控      |
| 安全事件管理 | 事件记录与查询 | 事件列表、详情查看、状态管理      |
| 安全规则管理 | 检测规则配置  | 规则增删改查、启用/禁用        |
| 智能检测引擎 | 自动检测威胁  | 登录检测、网络检测、文件检测、进程检测 |
| 自动化响应  | 处置动作执行  | IP封禁、进程终止、连接阻断、文件恢复 |
| 动态安全评分 | 量化安全状态  | 0-100分，实时计算         |

规划完这些，我心里总算有了底。开发在本地Windows环境下进行，最后才部署到OpenEuler服务器。

## 开发：从零到一

开发阶段全部在本地Windows环境下完成。说实话，这一块倒没有太大波折——技术栈我熟，若依框架我用过，问题都能解决。

### 拆需求：把一句话展开成一整套系统

开题报告只有一句话：「本课题旨在设计并实现一个面向OpenEuler操作系统的安全态势感知系统。」

就这么一句话，我要把它展开成一整套功能。

这不是简单的事。之前做项目，不管是蝶梦还是帝可得，都有一个明确的「正确答案」——课程告诉你用什么框架、怎么设计、达到什么效果。但这次不一样。

开题报告只讲「要做什么」和「为什么做」，不会讲「怎么做」和「做成什么样」。所以，我不仅要分析需求，还要自己决定：

- **具体功能模块**：安全仪表盘、安全事件管理、安全规则管理、智能检测引擎、自动化响应、动态安全评分
- **技术选型**：RuoYi-Vue3（蝶梦项目用过）、Vue3 + Element UI + ECharts、Spring Boot + MyBatis Plus
- **系统架构**：前端展示层 + 后端服务层 + 安全态势感知模块

这是第一次，我既是产品经理，又是架构师，又是开发工程师。每一步都要自己做决定。

### 写代码：核心功能实现

功能拆解完了，就开始写代码。

四大检测引擎是核心——登录检测、网络检测、文件检测、进程检测。检测逻辑我采用策略模式，每种检测类型都是一种策略，通过策略模式实现检测逻辑的灵活扩展。

动态安全评分系统也是自己设计的——基础分100分，按事件级别扣分，近期事件权重更高，已处理事件影响减半。

前端用ECharts做可视化，安全仪表盘包含安全评分仪表盘、事件分布饼图等核心图表，以及实时事件列表，支持按类型、级别、状态筛选。

来来回回折腾了一周，代码写完了，本地测试也通过了。

:::note
本地开发的过程反而是最顺利的。技术栈熟悉，框架上手快，遇到问题都能解决。但真正的挑战，在后面——部署到OpenEuler虚拟机。
:::

## 攻坚：虚拟机部署

本地开发完成、本地测试通过后，最后一步就是部署到OpenEuler虚拟机。本以为有若依官网的一键部署脚本，应该是最简单的一步——没想到，这反而是最折磨我的一段经历。

### 第一难：包版本冲突

若依官网给了一键部署脚本，本以为跑个脚本就完事了。结果一跑，傻眼了——依赖包版本之间互相冲突，一堆报错，各种版本不兼容。改了一堆配置，降级的降级、升级的升级，好不容易才解决。

### 第二难：若依插件的坑

版本问题刚解决，又发现若依的部署插件本身就有问题——配置文件缺胳膊少腿，很多关键配置都没有，需要自己琢磨着补。

尤其是 `docker-compose.yml` 文件，里面一堆缺失的配置项。我一边搜资料，一边问AI，一边看B站教程，三管齐下，一点一点把配置补全。

### 第三难：镜像源问题

配置补完了，还要配置Docker镜像加速。切换到国内镜像源——没用。再换——还是没用。

试了七八个镜像源，全是龟速或者直接超时。那时候的心态真的快崩了，一度以为是不是网络问题。

最后在一个不起眼的论坛角落里，发现有人提到了**渡渡鸟镜像**，抱着试试看的心态用了——居然成了，速度直接起飞。

### 第四难：部署后页面500

终于部署成功，兴冲冲地打开页面——**500 Internal Server Error**。

笑容瞬间凝固。

又是新一轮的搜索、排查、问AI。最后发现是本地开发环境和服务器环境的配置差异导致的。没办法，代码回滚到本地，重新排查bug，修复后再部署到虚拟机上。

来来回回折腾了三四次。

### 第五难：终于能用了

前前后后折腾了不知道多少次，终于——

页面加载出来了，功能正常了，测试全部通过了。

那一刻，整个人真的是长舒一口气。感觉就像经历了一场九九八十一难的取经路，终于修成正果。

:::tip
本以为最简单的一步，反而踩了最多的坑。有时候你以为的「捷径」，反而是最曲折的路。
:::

## 交付：关关难过，关关过

3月16日晚，系统部署完成，论文也写完了。

当我把系统源码和论文文档发给朋友的时候，整个人突然就松弛下来了。那种感觉就像高考结束走出考场的那一刻——紧绷了一周的神经终于可以放松。

回望总结：

- 3月8号，朋友同时拜托LaTeX论文排版 + 毕业设计系统
- 3月9号，先学LaTeX，完成论文排版（详见《一次跨越困难的LaTeX学习之旅》）
- 3月10号，系统开发 + 论文编写
- 3月16日，系统部署 + 论文完成 + 交付

一周时间，LaTeX加系统，全部搞定。

## 感悟：纸上得来终觉浅，深知此事要躬行

在《虚拟机全认知》那篇文章里，我写道：

> 学习Linux的过程就是不断踩坑、填坑的过程。每个错误都是一次学习机会。

这次的经历，让我对这句话有了更深的理解。

**这次最大的挑战，不是技术本身，而是「没有标准答案」这件事。**

之前做项目，不管是蝶梦还是帝可得，都有一个明确的「正确答案」——课程告诉你用什么框架、怎么设计、达到什么效果。但这次不一样。

开题报告只有一句话，所有功能要我自己拆，所有方案要我自己定，所有技术选型要我自己拍板。

**这是第一次，我既是产品经理，又是架构师，又是开发工程师。**

从需求分析到系统设计，从技术选型到具体实现，每一步都要自己做决定。没有参考答案，没有课程指导，只有自己对自己的选择负责。

这种感觉很不一样，但也正是这种「没有退路」的状态，逼出了我最大的潜力。

:::tip
**「纸上得来终觉浅，绝知此事要躬行」的新理解**：

- **「纸上」是知道**，看教程、读书、看别人做，有参考答案
- **「躬行」是做到**，自己动手、踩坑、解决问题，没有参考答案
- **「绝知」是真正理解**，不是背答案，而是能自己设计答案
  :::

## 写在最后

交付的那一刻，我给朋友发了一条消息：「搞定了。」

朋友回复了一个大大的「牛逼」。

说实话，那一刻真的有一种说不出的成就感。不是因为帮朋友完成了任务，而是因为：

**我逼了自己一把，发现自己真的可以。**

一周时间，LaTeX论文加系统，从开题报告到系统交付，从0到1。这次不只是coding能力的考验，更是一次「独立做决策」能力的锻炼。

LaTeX那篇文章里我说过：人的成长一定在于解决问题，而解决问题的前提，是承担责任。

这次的经历再次验证了这句话。朋友之托，让我有了责任；责任之重，让我挖掘出了自己的潜力。

不逼自己一把，永远不知道自己有多能扛。

***

*受人之托，忠人之事。这是我在2026年3月，收获的又一宝贵一课。*
