## 项目背景

### 为什么选择这个方向

学完帝可得之后，我对若依框架的二次开发有了初步的认知。但帝可得是跟着课程做的，很多东西是「已知答案的练习」。我想做一个真正属于自己的项目——从需求调研、方案设计到代码实现，全流程走一遍。

选择流浪宠物收养这个方向，有几个原因：

1. **业务场景清晰**：宠物管理、领养申请、审核流程、捐款记录，这些都是相对标准的 CRUD 业务，容易上手
2. **有一定的社会价值**：比起做一个「玩具项目」，流浪宠物救助这个主题让我更有动力
3. **技术栈匹配**：若依框架天生适合这种管理后台系统，不需要过度设计

### 项目定位与核心目标

蝶梦管理系统定位是**流浪宠物救助组织的数字化运营平台**，核心解决三类角色的需求：

| 角色 | 核心诉求 | 系统支撑 |
| --- | --- | --- |
| 管理员 | 宠物信息管理、领养审核、数据统计 | 后台管理全功能 |
| 领养用户 | 提交申请、查看进度、了解公益进展 | C 端用户前台（规划中） |
| 运营人员 | 日常业务处理、社区运营 | 帖子管理、评论审核 |

**核心功能模块**：

| 模块 | 主要功能 | 技术价值 |
| --- | --- | --- |
| 首页大盘 | 运营数据统计、捐款榜单、异常预警 | ECharts 数据可视化 |
| 宠物管理 | 宠物信息 CRUD、状态流转、图片上传 | 阿里云 OSS 文件存储 |
| 领养审核 | 申请提交、审核管理、进度跟踪 | 状态机设计 |
| 社区模块 | 帖子管理、评论审核 | 富文本编辑、内容风控 |
| 捐款管理 | 捐款记录、资金公示 | 数据统计与趋势分析 |

:::note
蝶梦项目目前只实现了后台管理端，C 端用户前台还在规划中。这也是项目最明显的不足之一——用户无法自主操作，所有申请都需要管理员代提交。
:::

## 技术架构

### 为什么选择 RuoYi-Vue3

选择若依框架不是拍脑袋决定的。在帝可得项目中，我深刻体验到了「框架二次开发」的效率优势：

| 对比维度 | 从 0 开发 | 使用若依 |
| --- | --- | --- |
| 开发周期 | 2-3 个月（含权限系统） | 1-2 个月（纯业务开发） |
| 权限系统 | 需要自己实现 | 开箱即用 |
| 代码生成 | 全部手写 | 可视化生成 + 业务改造 |
| 文档生态 | 依赖团队积累 | 社区活跃、文档完善 |

但若依不是银弹。它的局限性也很明显：

- **业务定制有天花板**：很多功能是「刚好够用」的程度，想要更好需要魔改
- **学习曲线**：框架封装得太好，有时候反而不知道底层是怎么工作的
- **过度封装**：某些场景下，框架的设计反而成了限制

**最终选择**：在充分评估后，我认为若依的「开箱即用」 + 「可扩展性」能够满足蝶梦项目的需求。

### 后端技术架构

蝶梦后端采用 **Maven 多模块分层设计**，遵循「高内聚、低耦合」原则：


## 核心业务模块实战

### 模块一：宠物信息管理

#### 业务场景分析

宠物管理是整个系统的基础。核心需求包括：

- **多维度检索**：按宠物名字、种类、性别、领养状态筛选
- **全量数据维护**：增删改查、批量导出
- **图片管理**：宠物照片上传、预览（限制 ≤5MB，支持 png/jpg/jpeg）
- **状态管控**：宠物状态流转（可领养 → 领养中 → 已领养）

#### 图片上传功能实现

**前端上传组件**：

```vue
<!-- ImageUpload.vue -->
<template>
    <el-upload
        class="avatar-uploader"
        action="/api/common/upload"
        :headers="{ Authorization: 'Bearer ' + token }"
        :show-file-list="false"

    </el-upload>

<script setup>
/**
 * 图片上传逻辑
 *
 * 核心流程：
 * 1. 用户选择文件，触发 beforeUpload 校验
 * 2. 校验通过，发送请求到后端
 * 3. 后端接收文件，上传到阿里云 OSS

const beforeUpload = (file) => {

        return false
    }

        return false
    }

    return true
}

/**
 * 上传成功回调
 */
const handleUploadSuccess = (response) => {
    // 若依统一响应体格式：{ code: 200, msg: '操作成功', data: 'URL' }
    imageUrl.value = response.data
    emit('update:modelValue', response.data)

```

**后端上传 Controller**：

```java
/**
 * 通用文件上传接口
 *
 * 核心流程：
 * 1. 接收前端上传的文件
 * 2. 校验文件类型和大小
 * 3. 生成唯一文件名（UUID + 原扩展名）
 * 4. 上传到阿里云 OSS
 * 5. 返回文件访问 URL
 */
@RestController
@RequestMapping("/common")
public class CommonController {

    @Autowired
    private OssService ossService;

    /**
     * 文件上传
     */
    @PostMapping("/upload")
    public AjaxResult upload(MultipartFile file) {
        // 1. 校验文件是否为空
        if (file == null || file.isEmpty()) {
            return AjaxResult.error("上传文件不能为空");
        }

        // 2. 校验文件大小（5MB）
        if (file.getSize() > 5 * 1024 * 1024) {
            return AjaxResult.error("文件大小不能超过 5MB");
        }

        // 3. 校验文件类型
        String contentType = file.getContentType();
        if (!Arrays.asList("image/png", "image/jpg", "image/jpeg").contains(contentType)) {
            return AjaxResult.error("只能上传 png/jpg/jpeg 格式的图片");
        }

        // 4. 上传到 OSS
        String url = ossService.uploadFile(file);

        // 5. 返回 URL
        return AjaxResult.success(url);
    }
}
```

**OSS 服务实现**：

```java
/**
 * 阿里云 OSS 文件上传服务
 *
 * 核心配置：
 * - endpoint：OSS 区域节点
 * - accessKeyId/accessKeySecret：访问凭证
 * - bucketName：存储桶名称
 */
@Service
public class OssServiceImpl implements OssService {

    @Value("${oss.endpoint}")
    private String endpoint;

    @Value("${oss.accessKeyId}")
    private String accessKeyId;

    @Value("${oss.accessKeySecret}")
    private String accessKeySecret;

    @Value("${oss.bucketName}")
    private String bucketName;

    @Override
    public String uploadFile(MultipartFile file) {
        // 1. 创建 OSS 客户端
        OSS ossClient = new OSSClientBuilder().build(endpoint, accessKeyId, accessKeySecret);

        try {
            // 2. 生成唯一文件名
            String originalFilename = file.getOriginalFilename();
            String extension = originalFilename.substring(originalFilename.lastIndexOf("."));
            String fileName = UUID.randomUUID().toString().replace("-", "") + extension;

            // 3. 上传文件
            // 存储路径：pets/2024/07/15/xxx.jpg
            String datePath = new SimpleDateFormat("yyyy/MM/dd").format(new Date());
            String objectName = "pets/" + datePath + "/" + fileName;

            ossClient.putObject(bucketName, objectName, file.getInputStream());

            // 4. 返回访问 URL
            return "https://" + bucketName + "." + endpoint + "/" + objectName;

        } finally {
            // 5. 关闭 OSS 客户端（重要！防止连接泄漏）
            if (ossClient != null) {
                ossClient.shutdown();
            }
        }
    }
}
```

#### 状态流转控制

**业务规则**：

```
宠物状态流转规则：
1. 新增宠物 → 默认状态为「可领养」
2. 用户提交领养申请 → 宠物状态变为「领养中」
3. 领养申请审核通过 → 宠物状态变为「已领养」
4. 领养申请审核拒绝 → 宠物状态恢复为「可领养」
```

**领养申请通过时的状态更新**：

```java
/**
 * 审核通过领养申请
 *
 * @param applyId 申请ID
 */
@Override
@Transactional
public void approveApply(Long applyId) {
    // 1. 查询申请信息
    AdoptionApply apply = adoptionApplyMapper.selectById(applyId);
    if (apply == null) {
        throw new ServiceException("申请不存在");
    }

    // 2. 校验申请状态（必须是待审核）
    if (!"0".equals(apply.getApplyStatus())) {
        throw new ServiceException("只能审核待处理的申请");
    }

    // 3. 更新申请状态为「已通过」
    apply.setApplyStatus("1");  // 1=已通过
    apply.setAuditTime(new Date());
    apply.setAuditRemark("审核通过");
    adoptionApplyMapper.updateById(apply);

    // 4. 更新宠物状态为「已领养」（核心业务逻辑）
    petInfoMapper.updateAdoptStatus(apply.getPetId(), "2");  // 2=已领养

    log.info("领养申请 {} 审核通过，宠物 {} 已领养", applyId, apply.getPetId());
}
```

:::note
**状态流转的注意事项**：
1. 状态变更需要事务保证，防止数据不一致
2. 状态变更前需要校验前置状态（如：只有「可领养」才能被申请）
3. 状态变更后可能需要触发其他业务（如：审核通过后更新宠物状态）
:::

### 模块二：领养申请管理

#### 业务场景分析

领养申请是系统的核心业务流程，完整流程如下：

```
用户提交申请 → 管理员审核 → 通过/驳回 → 宠物状态更新
```

**核心功能**：

| 功能 | 说明 |
| --- | --- |
| 申请提交 | 记录宠物ID、用户信息、申请备注 |
| 申请查询 | 按状态筛选（待审核/已通过/未通过） |
| 审核管理 | 查看申请详情，执行审核操作 |
| 进度跟踪 | 记录审核时间、审核人、审核备注 |

#### 数据库表设计

**申请状态机**：

| 状态 | 值 | 说明 | 可执行操作 |
| --- | --- | --- | --- |
| 待审核 | 0 | 等待管理员审核 | 审核通过/审核拒绝 |
| 已通过 | 1 | 申请通过 | 无 |
| 未通过 | 2 | 申请被拒绝 | 无 |

#### 审核功能实现

**审核 Controller**：

```java
/**
 * 领养申请管理
 */
@RestController
@RequestMapping("/adoption/apply")
public class AdoptionApplyController {

    @Autowired
    private IAdoptionApplyService adoptionApplyService;

    /**
     * 分页查询申请列表
     */
    @GetMapping("/list")
    public TableDataInfo list(AdoptionApply apply) {
        startPage();
        List<AdoptionApply> list = adoptionApplyService.selectApplyList(apply);
        return getDataTable(list);
    }

    /**
     * 审核申请
     *
     * @param applyId   申请ID
     * @param status    审核状态（1=通过，2=拒绝）
     * @param auditRemark 审核备注
     */
    @PutMapping("/audit")
    public AjaxResult audit(Long applyId,
                            @RequestParam("status") String status,
                            @RequestParam(value = "auditRemark", required = false) String auditRemark) {
        if ("1".equals(status)) {
            // 审核通过
            adoptionApplyService.approveApply(applyId, auditRemark);
            return AjaxResult.success("审核通过");
        } else if ("2".equals(status)) {
            // 审核拒绝
            adoptionApplyService.rejectApply(applyId, auditRemark);
            return AjaxResult.success("审核已拒绝");
        } else {
            return AjaxResult.error("无效的审核状态");
        }
    }
}
```

**审核通过服务**：

```java
/**
 * 审核通过领养申请
 *
 * 核心流程：
 * 1. 校验申请状态（必须是待审核）
 * 2. 更新申请状态为「已通过」
 * 3. 记录审核信息（审核人、审核时间、审核备注）
 * 4. 更新宠物状态为「已领养」
 * 5. 记录操作日志
 */
@Override
@Transactional(rollbackFor = Exception.class)
public void approveApply(Long applyId, String auditRemark) {
    // 1. 获取当前登录用户
    LoginUser loginUser = getLoginUser();

    // 2. 查询申请
    AdoptionApply apply = adoptionApplyMapper.selectById(applyId);
    if (apply == null) {
        throw new ServiceException("申请不存在");
    }

    // 3. 校验状态（必须是待审核）
    if (!"0".equals(apply.getApplyStatus())) {
        throw new ServiceException("该申请已处理，无法重复审核");
    }

    // 4. 查询宠物信息（校验宠物状态）
    PetInfo pet = petInfoMapper.selectById(apply.getPetId());
    if (pet == null) {
        throw new ServiceException("宠物不存在");
    }

    // 5. 校验宠物状态（必须是「可领养」才能被领养）
    if (!"0".equals(pet.getAdoptStatus())) {
        throw new ServiceException("该宠物当前不可被领养");
    }

    // 6. 更新申请状态
    apply.setApplyStatus("1");  // 已通过
    apply.setAuditBy(loginUser.getUsername());
    apply.setAuditTime(new Date());
    apply.setAuditRemark(auditRemark);
    adoptionApplyMapper.updateById(apply);

    // 7. 更新宠物状态为「已领养」
    pet.setAdoptStatus("2");  // 已领养
    pet.setUpdateBy(loginUser.getUsername());
    pet.setUpdateTime(new Date());
    petInfoMapper.updateById(pet);

    log.info("领养申请 {} 审核通过，宠物 {} 已领养，操作人 {}", applyId, apply.getPetId(), loginUser.getUsername());
}

/**
 * 审核拒绝领养申请
 */
@Override
@Transactional(rollbackFor = Exception.class)
public void rejectApply(Long applyId, String auditRemark) {
    LoginUser loginUser = getLoginUser();

    AdoptionApply apply = adoptionApplyMapper.selectById(applyId);
    if (apply == null) {
        throw new ServiceException("申请不存在");
    }

    if (!"0".equals(apply.getApplyStatus())) {
        throw new ServiceException("该申请已处理，无法重复审核");
    }

    // 更新申请状态为「未通过」
    apply.setApplyStatus("2");  // 未通过
    apply.setAuditBy(loginUser.getUsername());
    apply.setAuditTime(new Date());
    apply.setAuditRemark(auditRemark);
    adoptionApplyMapper.updateById(apply);

    log.info("领养申请 {} 审核拒绝，操作人 {}", applyId, loginUser.getUsername());
}
```

:::note
**审核功能的关键点**：
1. **状态校验**：只有「待审核」状态的申请才能被审核，防止重复操作
2. **数据一致性**：宠物状态更新和申请状态更新要在同一个事务中完成
3. **日志记录**：审核操作需要记录审核人、审核时间、操作内容
:::

### 亮点一：完整的业务闭环设计

蝶梦项目虽然功能不多，但实现了**从宠物信息管理到领养审核的完整业务闭环**：

```
宠物信息录入 → 领养申请提交 → 管理员审核 → 审核结果通知 → 宠物状态更新
```

这个闭环的意义在于：

| 环节 | 价值 |
| --- | --- |
| 宠物信息录入 | 数字化管理宠物档案，替代纸质台账 |
| 申请提交 | 规范领养流程，有据可查 |
| 审核管理 | 保障领养人资质，提升领养成功率 |
| 状态更新 | 数据一致性，实时反映宠物状态 |

### 踩坑实录：图片上传的 OSS 路径问题

#### 问题现象

本地测试图片上传功能正常，但部署到服务器后，图片上传成功，但前端无法展示图片。

#### 根因分析

```java
// 错误写法：返回的 URL 缺少协议前缀
String url = bucketName + "." + endpoint + "/" + objectName;
// 返回：dm-bucket.oss-cn-beijing.aliyuncs.com/pets/2024/07/15/xxx.jpg
// 浏览器无法识别这是什么协议
```

```java
// 正确写法：需要加上 https:// 前缀
String url = "https://" + bucketName + "." + endpoint + "/" + objectName;
// 返回：https://dm-bucket.oss-cn-beijing.aliyuncs.com/pets/2024/07/15/xxx.jpg
```

#### 解决方案

```java
@Override
public String uploadFile(MultipartFile file) {
    OSS ossClient = new OSSClientBuilder().build(endpoint, accessKeyId, accessKeySecret);

    try {
        String originalFilename = file.getOriginalFilename();
        String extension = originalFilename.substring(originalFilename.lastIndexOf("."));
        String fileName = UUID.randomUUID().toString().replace("-", "") + extension;

        String datePath = new SimpleDateFormat("yyyy/MM/dd").format(new Date());
        String objectName = "pets/" + datePath + "/" + fileName;

        ossClient.putObject(bucketName, objectName, file.getInputStream());

        // 修复：加上 https:// 前缀
        return "https://" + bucketName + "." + endpoint + "/" + objectName;

    } finally {
        if (ossClient != null) {
            ossClient.shutdown();
        }
    }
}
```

#### 经验总结

| 经验点 | 说明 |
| --- | --- |
| 本地 vs 服务器环境差异 | 本地可能有特殊配置掩盖了问题 |
| URL 格式要完整 | 完整的 URL 应该包含协议、域名、路径 |
| 测试要覆盖全流程 | 上传成功不代表能访问 |

### 踩坑实录：领养审核的状态并发问题

#### 问题现象

在测试领养审核功能时，发现一个奇怪的 bug：两个管理员同时审核同一个申请，都显示「审核成功」，但宠物状态只更新了一次。

#### 根因分析

```java
// 问题代码：没有做状态校验和并发控制
public void approveApply(Long applyId) {
    AdoptionApply apply = adoptionApplyMapper.selectById(applyId);

    // 问题：没有检查 apply 的当前状态
    // 两个请求同时到达，都能查到 status = "0"（待审核）

    // 两个请求都执行更新
    apply.setApplyStatus("1");
    adoptionApplyMapper.updateById(apply);  // 两次 UPDATE，第二次覆盖第一次

    // 问题：没有检查宠物当前状态
    petInfoMapper.updateAdoptStatus(apply.getPetId(), "2");
}
```

**问题场景**：

```
时间线：
T1: 管理员A 查询申请 → status = "0"（待审核）
T2: 管理员B 查询申请 → status = "0"（待审核）
T3: 管理员A 审核通过 → status = "1"
T4: 管理员B 审核通过 → status = "1"（覆盖了管理员A的结果）
T5: 管理员B 的审核备注被覆盖丢失
```

#### 解决方案

**方案一：数据库唯一索引 + 乐观锁**

```java
// 使用版本号实现乐观锁
apply.setApplyStatus("1");
apply.setVersion(apply.getVersion() + 1);  // 每次更新 version + 1
int updated = adoptionApplyMapper.updateWithOptimisticLock(apply);

if (updated == 0) {
    throw new ServiceException("该申请已被其他管理员处理，请刷新后重试");
}
```

```xml
<!-- Mapper XML：乐观锁更新 -->
<update id="updateWithOptimisticLock">
    UPDATE adoption_apply
    SET apply_status = #{applyStatus},
        version = version + 1
    WHERE apply_id = #{applyId}
      AND version = #{version}  -- 关键：版本号匹配才能更新
</update>
```

**方案二：分布式锁（更严格的并发控制）**

```java
@Override
public void approveApply(Long applyId) {
    // 获取分布式锁（基于 Redis）
    String lockKey = "adoption:apply:lock:" + applyId;
    Boolean lockAcquired = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);

    if (!lockAcquired) {
        throw new ServiceException("该申请正在被处理，请稍后重试");
    }

    try {
        // 业务处理
        doApprove(applyId);
    } finally {
        // 释放锁
        redisTemplate.delete(lockKey);
    }
}
```

**蝶梦项目的选择**：由于项目并发量不高，采用了**方案一（乐观锁）**，实现简单且足够应对实际场景。

#### 经验总结

| 经验点 | 说明 |
| --- | --- |
| 并发问题很难在开发环境复现 | 需要在测试环境模拟多用户操作 |
| 乐观锁适合低并发场景 | 实现简单，不需要额外的中间件 |
| 分布式锁适合高并发场景 | 需要 Redis 等组件，增加复杂度 |
| 状态校验是基础 | 任何状态变更前都要校验当前状态 |

**1. 从「跟着做」到「自己想」**

- 框架的权限模型是通用的，宠物领养的审核流程需要自己改造

蝶梦项目让我第一次完整经历了：
