## 引言：从「我」到「我们」

第二阶段结束时，我已经能够独立完成一个完整的「每日打卡」小程序。从项目规划到代码实现，从调试排查到版本迭代，这个过程让我颇有成就感。

但企业导师在第三阶段第一次课上抛出的问题，让我重新审视自己的能力边界：

> 「独立完成项目是本事，但能和别人配合完成项目，才是真正的职场能力。你们准备好了吗？」

那一刻我才意识到，前两个阶段我一直在和自己较量——和代码较劲、和bug搏斗。而第三阶段的挑战截然不同：我需要学会和「人」打交道。

第三阶段的核心任务是**小组协作开发**——4-5人为一组，分工负责不同模块，最终交付一个完整的项目。这意味着我要面对：代码怎么合并、接口怎么约定、进度怎么同步、风格怎么统一。这些问题，独立开发时根本不会遇到。

## 团队组建：找到你的「开发搭子」

### 分组方案

企业导师在第三阶段开始时给出了分组方案：自由组队，4-5人一组。我们班二十多人，很自然地分成了5个小组。

我所在的小组共5人：

- **我**：负责用户模块、打卡模块及部分后端接口（因为有一定全栈能力，分担了后端开发工作）
- **张同学**：负责首页和搜索模块
- **虞同学**：负责消息通知模块
- **王同学**：负责个人中心模块
- **金同学**：负责核心后端接口开发

:::note
分工原则是「前端+后端」的组合。每个小组配备一名后端同学，负责编写接口文档和提供Mock数据。这种配置模拟了真实企业中的前后端分离开发模式。
:::

### 角色定位

企业导师没有指定「组长」这个角色，而是让我们采用**扁平化协作模式**：

| 角色             | 职责               | 说明        |
| -------------- | ---------------- | --------- |
| **模块负责人**      | 各自负责1-2个模块的开发    | 核心开发者     |
| **接口对接人**      | 统一管理接口文档，协调前后端联调 | 通常由后端同学担任 |
| **代码reviewer** | 每次提交前互相检查代码      | 轮流担任      |

这种模式的好处是：没有「领导」，每个人都是平等的贡献者。但这也对个人主动性提出了更高要求——你不能等着别人分配任务，而要主动推进自己的模块。

### 我的全栈担当

由于我在前两个阶段表现出的学习能力和技术积累，企业导师和组员们默认我能承担更多工作。在分工时，我主动申请了**部分后端接口的开发任务**——这在小组里是独一无二的。

为什么要这样做？一是金同学是后端新手，面对多个接口需求时压力很大；二是打卡模块的业务逻辑我最熟悉，由我来实现后端逻辑可以减少前后端的沟通成本。

于是，我的职责变成了：

- **前端**：用户模块、打卡模块的完整UI和交互
- **后端**：打卡模块的接口实现（习惯CRUD、打卡记录查询、统计数据聚合）

这种「前端做到后端」的全栈体验，让我真正理解了什么叫**前后端分离**：前后端只是分工不同，本质上都是在处理数据和业务逻辑。

## Git版本控制：团队开发的「时光机」

### 为什么需要Git

第二阶段的独立开发，我所有的代码都在本地保存，最多做个备份。但团队开发完全不同——5个人同时在改代码，如果没有版本控制工具，简直是噩梦。

企业导师在第三阶段第一堂课就花了半天时间讲Git：「**Git不只是备份工具，它是团队协作的基石**。」

**Git的核心价值**：

| 价值   | 传统方式的问题                    | Git带来的改变    |
| ---- | -------------------------- | ----------- |
| 版本管理 | 手动命名如`v1_final_real_final` | 提交历史清晰可追溯   |
| 并行开发 | 轮流修改同一个文件                  | 分支并行，互不干扰   |
| 代码合并 | 复制粘贴替换，容易丢失                | 智能合并，保留变更   |
| 权限控制 | 所有人都有全部代码                  | 可以设置分支保护规则  |
| 问题溯源 | 「这行代码是谁改的？」                | 每一行都能追溯到提交者 |

### 团队Git工作流

我们小组采用的工作流是**Git Flow简化版**：

```
master (保护分支)
    └── develop (开发分支)
            ├── feature/用户模块
            ├── feature/首页模块
            ├── feature/消息模块
            └── feature/个人中心
```

**日常开发流程**：

1. **从develop拉取新分支**：`git checkout -b feature/用户模块`
2. **开发并提交**：`git add .` → `git commit -m "feat: 添加用户登录页面"`
3. **推送到远程**：`git push origin feature/用户模块`
4. **发起Pull Request**（简称PR）
5. **其他成员Code Review通过后** → 合并到develop
6. **定期从develop同步最新代码到各自分支**

```bash
# 典型的一次开发流程
# 1. 确保在最新代码基础上开发
git checkout develop
git pull origin develop

# 2. 创建功能分支
git checkout -b feature/打卡模块

# 3. 开发完成后，提交代码
git add .
git commit -m "feat: 实现打卡功能

- 添加打卡页面UI
- 实现本地存储逻辑
- 添加打卡记录查询"

# 4. 推送到远程
git push origin feature/打卡模块

# 5. 在GitHub/Gitee上创建Pull Request
```

### 踩坑实录：合并冲突

**问题现象**：

有一天我兴冲冲地准备提交代码，执行`git push`后却收到了这样的错误：

```
! [rejected] develop -> develop (non-fast-forward)
error: failed to push some refs
hint: Updates were rejected because the tip of your current branch is behind
its remote counterpart. Merge the remote changes before pushing again.
```

**根因分析**：

远程的develop分支已经被其他人合并了新的代码，而我的本地分支还是基于旧的代码。我必须先拉取远程最新代码，合并后再推送。

**解决方案**：

```bash
# 方法一：git pull + merge
git pull origin develop
# 如果有冲突，手动解决冲突后
git add .
git commit -m "merge: 解决冲突"
git push origin feature/打卡模块

# 方法二：git rebase（更干净的历史）
git fetch origin
git rebase origin/develop
# 如果有冲突，解决后
git add .
git rebase --continue
git push origin feature/打卡模块 --force  # rebase后需要force push
```

**经验总结**：

| 操作                  | 适用场景   | 风险               |
| ------------------- | ------ | ---------------- |
| `git pull`          | 日常同步   | 可能产生merge commit |
| `git rebase`        | 保持历史线性 | 不要在已推送的分支上rebase |
| `git merge --no-ff` | 合并功能分支 | 保留完整分支信息         |

:::warning
企业导师特别提醒：**永远不要对master/develop等保护分支执行force push**。一旦force push，其他成员的本地分支可能会被覆盖。在推送前，先确认分支状态。
:::

## 接口联调：从「各自为战」到「无缝对接」

### 接口文档的重要性

第二阶段我用的是本地存储，数据都在客户端。但第三阶段是真实的后端接口，数据要从服务器获取。这带来一个新问题：**前端和后端必须提前约定好接口格式**。

如果后端写好了接口，但前端不知道返回什么字段、什么格式，那就只能「盲写」——前端猜着写，后端改一个字段前端就崩一次。这种协作方式效率极低。

企业导师要求我们在正式开发前，先**编写接口文档**：

```javascript
/**
 * 接口名称：获取用户打卡记录
 * 接口路径：GET /api/checkin/records
 * 请求参数：
 *   - userId: string 用户ID
 *   - month: string 月份，格式YYYY-MM
 * 返回格式：
 *   {
 *     "code": 200,
 *     "msg": "success",
 *     "data": {
 *       "totalDays": 15,
 *       "continueDays": 7,
 *       "records": [
 *         {
 *           "date": "2024-12-01",
 *           "habits": [
 *             { "id": "1", "name": "早起", "done": true },
 *             { "id": "2", "name": "读书", "done": false }
 *           ]
 *         }
 *       ]
 *     }
 *   }
 */
```

**接口文档的核心要素**：

| 要素   | 说明                  | 示例                   |
| ---- | ------------------- | -------------------- |
| 接口名称 | 清晰描述功能              | 获取用户打卡记录             |
| 请求方法 | GET/POST/PUT/DELETE | GET                  |
| 请求路径 | 具体的API路径            | /api/checkin/records |
| 请求参数 | 参数名、类型、说明           | userId: string       |
| 返回格式 | code、msg、data结构     | JSON对象               |
| 错误码  | 可能的错误情况             | 401: 未登录             |

### Mock数据：前端不等后端

在实际开发中，前后端进度往往不一致——后端的接口可能还没写好，但前端已经需要开始开发了。这时候**Mock数据**就派上用场了。

Mock数据的核心思想是：**前端自己造数据，模拟后端接口返回**，等后端接口真正完成后，只需把Mock地址换成真实接口即可。

```javascript
// mock.js - 模拟接口数据
const mockData = {
  '/api/checkin/records': {
    code: 200,
    msg: 'success',
    data: {
      totalDays: 15,
      continueDays: 7,
      records: [
        {
          date: '2024-12-01',
          habits: [
            { id: '1', name: '早起', done: true },
            { id: '2', name: '读书', done: false }
          ]
        },
        {
          date: '2024-12-02',
          habits: [
            { id: '1', name: '早起', done: true },
            { id: '2', name: '读书', done: true }
          ]
        }
      ]
    }
  }
}

export default mockData
```

**使用Mock数据的好处**：

1. **前后端解耦**：前端不依赖后端进度，可独立开发
2. **数据可控**：可以构造各种边界数据（空数据、错误数据）
3. **并行开发**：前后端同时开工，最后联调
4. **便于测试**：可以提前发现数据结构的适配问题

### 联调实战：那些「说不通」的时刻

**问题一：字段名对不上**

接口文档写的是`userId`，但后端实际返回的是`user_id`。前端按文档写的代码，一解析就报错。

**解决方案**：在接口封装层做字段映射

```javascript
// api/checkin.js
import request from './request'

export function getCheckinRecords(params) {
  return request({
    url: '/api/checkin/records',
    method: 'GET',
    params
  }).then(res => {
    // 统一字段名，解决后端字段不统一问题
    if (res.data && res.data.records) {
      res.data.records = res.data.records.map(item => ({
        ...item,
        userId: item.user_id || item.userId  // 兼容两种字段名
      }))
    }
    return res
  })
}
```

**问题二：数据类型不一致**

文档写的是`totalDays`为数字，但后端返回的是字符串`"15"`。前端做计算时`totalDays + 1`变成了字符串拼接`"151"`。

**解决方案**：在数据层做类型转换

```javascript
// 数据解析时强制转换类型
parseRecords(data) {
  return {
    totalDays: Number(data.totalDays) || 0,
    continueDays: Number(data.continueDays) || 0,
    records: data.records || []
  }
}
```

**问题三：分页参数不统一**

不同接口的分页参数不一样，有的用`page`/`size`，有的用`offset`/`limit`，有的甚至用`start`/`count`。

**解决方案**：封装统一的分页方法

```javascript
// utils/pagination.js
export function getPagination(params, config = { page: 'page', size: 'size' }) {
  const { page = 1, size = 10 } = params
  return {
    [config.page]: page,
    [config.size]: size
  }
}
```

:::note
企业导师的一个建议：**接口文档不是一次性定稿的**，它会随着开发深入不断调整。关键是要有「变更通知机制」——后端改了接口字段，一定要第一时间在群里通知前端，否则前端会花大量时间排查「为什么接口调不通」。
:::

## 代码评审：让代码「经得起审视」

### 为什么需要Code Review

独立开发时，代码只要自己能跑就行。但团队开发中，你的代码会被其他人看到、被其他人维护。如果代码写得「只有自己能懂」，那后续维护就会变成灾难。

企业导师在第三阶段引入了\*\*Code Review（代码评审）\*\*制度：每次提交PR到develop前，至少需要一名组员Review通过。

**Code Review的核心价值**：

| 维度       | 作用                |
| -------- | ----------------- |
| **质量把控** | 提前发现bug、逻辑错误、安全问题 |
| **知识共享** | 组员了解彼此的代码，便于后续维护  |
| **风格统一** | 确保团队代码风格一致        |
| **新人培养** | 新成员通过Review学习团队规范 |

### 我的第一次被Review

我还记得第一次提交PR时的紧张感。

那天我完成了「打卡页面」的UI开发，兴冲冲地提交了代码，发到群里请张同学帮忙Review。几分钟后，张同学的反馈来了：

> 「这个打卡按钮的点击事件用的是`@tap`，但在快速连续点击时没有防抖处理，可能导致重复提交。」
>
> 「`todayHabits`这个变量名不太直观，建议改成`todayHabitList`，和`checkedHabitList`区分。」
>
> 「这里用了`uni.getStorageSync`，但如果存储内容很大，同步读取会阻塞主线程。建议改成异步版本。」

三条反馈，每一条都直指问题要害。那一刻我才意识到，**自己写的代码和「合格的代码」之间，还有不小的距离**。

**Review后的修改**：

```javascript
// 修改前：缺少防抖处理
handleCheckin() {
  this.todayChecked = true
  this.saveRecords()
}

// 修改后：添加防抖处理
handleCheckin() {
  // 防止重复点击
  if (this.isChecking) return
  this.isChecking = true

  this.todayChecked = true
  this.saveRecords()

  setTimeout(() => {
    this.isChecking = false
  }, 1500)  // 1.5秒内禁止重复提交
}
```

### 我做Review时的思考

轮到我Review别人的代码时，我也在思考：怎样才是有效的Review？

**有效的Review关注点**：

1. **逻辑正确性**：代码能正确处理各种情况吗？边界条件考虑了吗？
2. **可读性**：变量命名清晰吗？复杂逻辑有注释吗？
3. **性能**：有没有明显的性能问题？如循环嵌套、重复计算。
4. **安全性**：用户输入有校验吗？敏感数据有加密吗？
5. **风格一致**：命名规范、缩进、注释风格与团队统一吗？

**无效的Review**：

- 「这里写法和我不一样，不太好」——应该说明具体哪里不好、为什么不好
- 「这个函数太长了」——应该具体指出哪段逻辑可以抽取
- 「变量名很差」——应该给出更好的命名建议

:::note
企业导师分享的一个观点：**Review不是批评，是协作**。被Review时不要有防御心态，Review别人时也不要过于苛刻。目标是「让代码更好」，而不是「证明自己更厉害」。
:::

## 沟通协作：比代码更重要的事

### 那些「说不清楚」的时刻

团队开发中，技术问题往往不是最难的，最难的是**沟通**。

有一周，虞同学在开发消息通知模块时遇到了问题。他在群里问：「这个消息列表的接口什么时候能好？」金同学回复：「明天。」

第二天，虞同学去找金同学要接口，发现接口确实返回了数据，但字段和文档不一致。虞同学问：「这个字段是什么意思？」金同学说：「哦，我忘了更新文档，这个字段是`create_time`而不是文档里写的`createTime`。」

虞同学花了两小时排查问题，最后发现是字段名拼写错误。这种「文档和实现不一致」的问题，在团队开发中太常见了。

**经验总结**：

| 沟通问题   | 后果              | 解决方案          |
| ------ | --------------- | ------------- |
| 文档不更新  | 前端按文档写，实际对不上    | 约定文档变更的同步机制   |
| 进度不透明  | 别人以为接口好了，实际还在开发 | 每天站会同步进度      |
| 问题描述不清 | 对方无法准确理解问题      | 用截图、日志、复现步骤描述 |
| 需求变更没说 | 开发了一半发现要改       | 建立需求变更的确认流程   |

### 每日站会：让进度「可视化」

企业导师建议我们每天早上开一个简短的「站会」，控制在15分钟内，每人回答三个问题：

1. **昨天做了什么？**
2. **今天要做什么？**
3. **有什么阻碍？**

```markdown
# 2024-12-15 站会记录

## 张同学（首页模块）
- 昨天：完成首页轮播图组件开发
- 今天：开发首页推荐列表
- 阻碍：推荐接口还没好，先用Mock数据

## 虞同学（消息模块）
- 昨天：完成消息列表页UI
- 今天：对接消息接口
- 阻碍：无

## 王同学（个人中心）
- 昨天：完成个人资料页UI
- 今天：开发设置页面
- 阻碍：头像上传接口需要调试

## 金同学（后端）
- 昨天：完成用户接口和打卡接口
- 今天：开发消息相关接口
- 阻碍：无

## 我（用户模块 + 打卡模块 + 部分后端）
- 昨天：完成打卡模块UI初版
- 今天：对接打卡接口
- 阻碍：无
```

这种透明化的沟通方式，让每个人都能了解整体进度，及时发现和解决协作问题。

## 项目收尾：我们的第一个「作品」

### 12月底的最终交付

2024年12月28日，各小组陆续完成了项目开发。12月31日，企业导师组织了一次正式的成果展示。

我们小组展示的是「习惯打卡」小程序，完整实现了：
