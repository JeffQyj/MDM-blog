:::tip
2026年4月，我完成了对 Art Design Pro 的二次开发探索。这次二开没有做出什么「可见的成果」，但收获的「二开哲学」却是无价的。本文记录我从具体实践到抽象提炼的完整思考过程。

## 项目了解：Art Design Pro

Art Design Pro 是一个基于 **Vue 3 + TypeScript** 的中后台管理系统前端解决方案，提供主题定制、菜单布局、权限管理、国际化等企业级功能。

### 技术栈一览

| 层级 | 技术 | 版本/说明 |
| --- | --- | --- |
| 框架 | Vue 3 | 3.5.x + Composition API |
| 语言 | TypeScript | 5.6.x |
| 构建工具 | Vite | 7.x |
| UI 框架 | Element Plus | 2.11.x |
| 状态管理 | Pinia | 3.x（持久化） |
| 路由 | Vue Router | 4.x |
| 样式 | Tailwind CSS + SCSS | 4.x |
| 图表 | ECharts | 6.x |
| 国际化 | vue-i18n | 9.x |
| HTTP | Axios | 1.x |

### 架构模式

| 特性 | 说明 |
| --- | --- |
| 前端 SPA 应用 | Hash 路由模式 |
| 权限控制 | 动态路由 + 菜单权限 |
| 状态持久化 | LocalStorage (pinia-plugin-persistedstate) |
| API 模式 | 前端控制型（菜单从后端获取） |

### 关键入口点

| 文件 | 用途 |
| --- | --- |
| `src/main.ts` | 应用入口，初始化 Store、Router、国际化 |
| `src/router/index.ts` | 路由配置，Hash 模式 |
| `src/store/index.ts` | Pinia 状态管理配置 |
| `src/router/guards/beforeEach.ts` | 路由守卫（权限验证） |
| `src/utils/http/index.ts` | Axios 封装 |

### 请求生命周期

```
用户操作 → Vue Router → beforeEach 守卫
    ↓
检查登录状态 → 未登录跳转登录页
    ↓
获取用户信息 + 菜单数据
    ↓
动态注册路由 → RouteRegistry.register()
    ↓
验证路径权限 → RoutePermissionValidator
    ↓
渲染目标页面
```

## 缘起：不做开发做架构

通常意义的二次开发，是拿一个现成项目往特定业务方向改造——蝶梦如此，安全态势感知系统也是如此。但 Art Design Pro 的这次二开不太一样。

我没有用它做具体的业务系统，而是尝试了三件事：

| 尝试方向 | 目标 | 结果 |
| --- | --- | --- |
| UI 风格增加 | 增加一套高智感、极客风的 UI 风格 | 一次性完成，效果不错 |
| ArtBot 聊天机器人 | 实现项目预留的聊天机器人功能 | 一次性完成，迅速且无误 |
| Vue 到 Nuxt 架构迁移 | 将项目从 Vue 架构升级到 Nuxt | 指令不明确导致失败 |

三个方向，只有第一个和第二个算是「成功」。第三个失败了。

但恰恰是第三个——**那个失败的任务**，让我学到了最多。

:::note
这次二开的核心收获，不在于完成了什么功能，而在于**通过实践验证了我对 AI 协作的认知**。有些道理，必须亲自踩过坑才能真正理解。
:::

## 尝试一：UI 风格定制

第一个尝试是给 Art Design Pro 增加一套新的 UI 风格——高智感、极客风。

这个需求其实很模糊。「高智感」是什么？「极客风」又该怎么定义？但我意识到一件事：**类似这种指明方向的 UI 任务，AI 完全能够一次做到**。

关键在于「指明方向」这四个字。

我把需求描述清楚，AI 给出方案，我看方案提反馈，AI 再调整。来回两三轮，一套全新的 UI 风格就集成到了项目中。代码规范、风格统一、没有 bug。

这让我意识到：AI 做 UI 设计的效率，远比我想象中高。前提是——你得知道你自己想要什么。

:::caution
**AI 擅长执行，不擅长决策。** 看似简单的「指明方向」四个字，背后是产品思维的考验。「高智感」是什么风格？目标用户是谁？要传达什么感觉？这些问题没有标准答案，但必须你自己先想清楚。方向越模糊，AI 返回的结果越难以落地。
:::

## 尝试二：ArtBot 聊天机器人

第二个尝试是实现项目预留的 ArtBot 聊天机器人功能。

这个功能本身并不复杂，本质上是一个对话框 + 一个对接后端 API 的前端交互逻辑。难度在于怎么让 AI 快速理解项目架构，然后写出符合项目规范的代码。

结果是：AI 迅速理解了整个项目的代码组织方式，写出的代码直接就能用，甚至比我预期的还要规范。

这又一次印证了我的判断：**AI 极大加速了功能开发**，前提是你能给 AI 足够清晰的上下文。

## 尝试三：Vue 到 Nuxt 的架构迁移

第三个尝试，也是最难的一个——将 Art Design Pro 从 Vue 架构迁移到 Nuxt。

这个任务失败了。

但失败的原因不是 AI 做不到，而是**我的指令不够明确**。我以为「把 Vue 升级到 Nuxt」已经说得很清楚了，结果 AI 理解成了「Vue 和 Nuxt 的混合架构」，导致方案设计从一开始就错了。

回滚代码，重新来过——但这一次，我学到了比成功更多的东西。

### Git 工作流的实战洗礼

这次失败让我被迫深入使用了 Git 的一系列高级操作：

| Git 操作 | 之前 | 这次之后 |
| --- | --- | --- |
| 提交 PR | 知道概念，没实际操作过 | 完整走了一遍流程 |
| 创建分支 | 只会从 main 拉 | 熟练使用 feature 分支策略 |
| 回滚分支 | `git reset` 偶尔用 | 精准使用 `git revert` 和 `git reset --hard` |
| 清除未跟踪文件 | `git clean` 不知道有啥用 | 用来清理失败的构建产物 |
| git worktree | 第一次听说 | 用来同时开多个分支开发 |

如果不是这次失败，我可能永远不会主动去学这些操作。**塞翁失马，焉知非福**——这句话在 AI 时代依然适用。

### 指令精确度的重要性

这次失败让我重新审视了一个老生常谈的概念：**提示词工程**。

之前看各种文章讲「怎么写提示词」，总觉得道理很简单——说清楚你要什么就行了。但只有亲身经历过「说不清楚导致失败」，才真正理解「说清楚」有多难。

Nuxt 是什么？Vue 是什么？迁移和混合的区别是什么？迁移的范围是什么？这些问题在我发出第一条指令时，脑子里并没有想得很清楚。

AI 并不能替你做决策，它只是把你脑子里的模糊想法翻译成代码。如果你脑子里想法是模糊的，AI 给你做出来的东西一定是错的。

## 真正的二开哲学

说了这么多尝试，终于可以聊正题了——**二开哲学**。

### 从「用 AI」到「驾驭 AI」

这次二开之前，我对 AI 协作的认知停留在「用 AI 写代码」这个层面。AI 是工具，我是使用者，我下达指令，AI 执行。

但这次二开让我意识到：**「用 AI」和「驾驭 AI」是两件完全不同的事**。

「用 AI」的关键是表达能力——你能多清楚地描述你的需求。

「驾驭 AI」的关键是**认知能力**——你自己得先想清楚你要做什么、为什么要这么做、有哪些可能的方案、每个方案的优缺点是什么。

AI 是你认知的放大器，而不是你认知的替代品。你的认知边界，就是 AI 协作的能力边界。

:::warning
**不要把 AI 当作认知的替代品。** 这是我在 AI 协作中最严肃的警告。如果你自己不去深入理解 Vue/Nuxt 的区别，不去搞清楚迁移和混合架构的差异，AI 给你的方案你根本无法判断对错。没有基础的认知，AI 的输出就是盲人骑瞎马——看起来在前进，实际上在迷路。
:::

### 提示词工程、上下文工程、驾驭工程

这次二开让我对「驾驭 AI」这件事有了更系统的理解：

| 概念 | 含义 | 我的理解 |
| --- | --- | --- |
| 提示词工程 | 怎么写好给 AI 的指令 | 是基础，但不充分 |
| 上下文工程 | 怎么给 AI 提供足够的背景信息 | 关键中的关键 |
| 驾驭工程 | 怎么把 AI 的能力发挥到最大 | 目标 |

**提示词工程**解决的是「说什么」的问题。但光说清楚还不够，你还得让 AI 理解你说的东西在说什么——这就是**上下文工程**。

ArtBot 开发之所以顺利，是因为 AI 理解了项目的整体架构。Nuxt 迁移之所以失败，是因为 AI 误解了我的意图，根源在于我没有把「Vue 和 Nuxt 的关系」解释清楚。

而**驾驭工程**是更高层次的概念——它要求你不只是把 AI 当工具用，而是把 AI 真正纳入你的工作流，成为你思考的一部分。

### Trae 使用方法的全面升级

说到驾驭工程，就不得不提 **Trae**——我一直在用的 AI 编程工具。

这次二开让我对 Trae 的使用方法有了全新的理解：

1. **规则体系**：之前创建的规则体系，在这次实践中得到了全面检验和升级
2. **技能体系**：之前积累的技能，在这次二开中被打磨得更加实用
3. **工作流**：之前固化的流程，在这次实践中被优化迭代
4. **认知飞轮**：不断思考→实践→复盘→再思考，这个循环在不断加速

**认知飞轮**这个概念尤其重要。之前我的认知是「慢慢迭代」的，但这次二开让我发现：当你的工作流和 AI 协作深度绑定时，认知迭代的速度会呈指数级提升。

### 理论验证与实践闭环

最后，也是最重要的一点：**这次二开把之前很多「理论上知道但没实践过」的想法，真正落地验证了**。

比如：

- 「上下文比指令更重要」——Nuxt 迁移的失败让我真正理解了这句话
- 「AI 是认知的放大器」——ArtBot 开发的顺利让我验证了这一点
- 「失败是最好的学习」——Git 工作流的学习让我体会到了这一点

没有这次实践，这些道理永远只是「知道」，而不是「理解」。

:::important
**知道和理解之间，隔着一整个实践。**
:::

## 写在最后

回头看这次 Art Design Pro 的二开，我没有做出什么「拿得出手」的系统。UI 风格定制是一次探索，ArtBot 是一次练手，Nuxt 迁移是一次失败的尝试。

但正是这次失败，让我收获了真正的二开哲学：

> **真正的二开，不是用别人的框架做自己的业务。而是在与 AI 的深度协作中，不断升级自己的认知边界。**

这不是什么高深的大道理，但只有亲身经历过的人，才能真正理解它的分量。

---

*二开哲学，不在于「开」什么，而在于通过「开」的过程，成为一个更好的 AI 协作者。*
