## 一切的开端

2023 年 3 月 3 日，大一上学期的那个下午。

老师在课堂上宣布：「以后你们的代码都要用 git 管理，最后推送到远程仓库。」

说完这句话，他便继续往下讲其他内容。

至于什么是 git、为什么要用 git、怎么用 git——一个字都没提。

我当时满脑子都是问号：

- git 是什么？
- 什么是版本控制？
- 版本控制又是什么意思？
- 为什么需要它？

这些概念在当时的我看来，完全是一团迷雾。我甚至不知道该从何查起，因为连搜索都不知道该搜什么关键词。

:::note
回想起来，这种「被扔进深水区自己学游泳」的经历，恰恰是大学编程教育的一个缩影。老师默认你应该会，学生只能硬着头皮自学。
:::

## 被迫的自学

和很多同学一样，我选择了去网上搜教程。

那时候 AI 还没有现在这么强大，给出的答案要么过于笼统，要么直接报错，没有参考价值。无奈之下，我转向了 CSDN 和百度。

搜索结果倒是能看懂一些，但问题在于：**网上的教程大多是给「已经知道 git 是什么」的人看的。**

它们会告诉你「执行这条命令」「add 这个文件」「commit 这个改动」，却很少解释「为什么需要这样做」。

我照着教程一步步操作，却始终处于一种「机械执行」的状态：

```bash
git init
git add .
git commit -m "first commit"
git remote add origin https://github.com/xxx/xxx.git
git push -u origin master
```

这些命令我背得滚瓜烂熟，却不知道自己到底在做什么。

:::note
当时最大的困惑是：为什么需要「暂存区」？为什么不直接 commit？为什么需要 remote？本地不能存吗？
:::

## 那些让我恍然大悟的时刻

自学过程中也有过「原来如此」的时刻。

真正让我理解「版本控制」这个概念的，是一个很笨的类比：

> 写论文时，你是不是会保存多个版本？
> 「论文\_v1.doc」「论文\_v2.doc」「论文\_最终版.doc」「论文\_真的最终版.doc」……

git 做的其实就是这件事，只不过它把这些版本保存在一个叫「仓库」的地方，并且用一种更高效、更清晰的方式管理它们。

后来我才知道，**git 的核心是「记录文件的变更」，而不是「保存文件的副本」**。每一次 commit，其实就是记录「相对于上一个版本，我改了什么」。

这个理解让我豁然开朗。

:::note
如果你也刚开始学 git，不要急着记命令。先理解它的核心思想：git 是一个「变更记录器」，它保存的是「修改」而不是「文件」。
:::

## 踩过的坑

当然，自学嘛，踩坑是必然的。

印象最深的一次，我在修改代码时不小心把整个文件夹删了。那一刻心跳加速——这是我花了一周写的课程作业，没有备份，没有历史记录。

后来才知道，如果当时已经用 git 管理了，一个 `git checkout -- .` 就能找回修改前的版本。

还有一次，我在 `git push` 时填错了远程仓库地址，代码被推进了一个陌生的仓库。慌乱之下，我又搜了半天怎么「撤销 push」，结果越搜越乱，最后只好重新 clone 一份，把本地修改手动合并进去。

这些经历让我深刻体会到：**git 的价值不在于你会敲命令，而在于它能帮你找回「后悔药」。**

## 深入理解：git 的三层架构

当我有了一定的实战经验后，开始好奇：这些命令背后到底发生了什么？

翻阅资料后我发现，git 的核心是一个**三层架构**：

| 区域                        | 说明       | 文件位置            |
| ------------------------- | -------- | --------------- |
| 工作区（Working Directory）    | 你正在编辑的文件 | 项目目录本身          |
| 暂存区（Staging Area / Index） | 即将提交的快照  | `.git/index`    |
| 仓库（Repository）            | 已提交的历史记录 | `.git/objects/` |

### 工作区

就是你的项目文件夹，你正常编辑代码的地方。

### 暂存区

这是 git 最有特色的设计之一。**暂存区是一个「准备提交」的中间地带。**

为什么需要它？想象你要开发一个新功能，修改了 5 个文件，但其中只有 3 个是相关的。这时候你就可以用 `git add` 只暂存那 3 个文件，剩下 2 个先不管。

### 仓库

所有已经提交的历史都保存在这里。每一笔提交都是一个「快照」，记录了项目在某个时间点的完整状态。

## 核心原理：对象模型与 SHA-1 哈希

真正让我对 git 刮目相看的，是理解它的**对象模型**。

git 的所有数据都是「对象」，存储在 `.git/objects/` 目录下。对象类型有四种：

### blob：文件内容

**类比**：快递柜的每个格子，存储的是文件的内容本身。

当我执行 `echo "hello" > a.txt` 然后 `git add a.txt` 时，git 会：

1. 读取文件内容「hello」
2. 计算 SHA-1 哈希值：`echo -n "hello" | sha1sum` → 类似 `8ab686e...`
3. 在 `.git/objects/` 下创建对象文件

**关键点**：blob 只存储内容，不存储文件名！

### tree：目录结构

**类比**：快递柜的管理清单，记录哪个格子放了什么。

tree 对象记录了「目录」的结构，它会指向多个 blob 或子 tree：

```
tree abc123...
├── a.txt          -> blob 8ab686e...
├── README.md      -> blob 9d48f2a...
└── src/           -> tree def456...
    └── main.js    -> blob 7h3k9s1...
```

### commit：提交快照

**类比**：每次快递送达后的签收确认单。

commit 对象包含：

- 指向一个 tree（项目快照）
- 指向上一个 commit（parent，形成链表）
- 作者信息、时间戳、提交消息

```
commit f7a1b2c...
tree abc123...
parent 9e4d8f7...
author 小明 <xiaoming@example.com> 1679900000 +0800
committer 小明 <xiaoming@example.com> 1679900000 +0800

first commit
```

### 对象存储机制

所有对象按 SHA-1 哈希的前两位分成子目录，后 38 位作为文件名：

```
.git/objects/
├── 8a/
│   └── b686eaf4...    # blob 对象
├── ab/
│   └── c1234df...     # tree 对象
└── f7/
    └── a1b2c3de...    # commit 对象
```

**为什么这样设计？** 避免单个目录下文件过多。如果所有对象都放在一个目录，百万个文件会让文件系统崩溃。

## 分支的本质

学 git 必学分支，但很多人不知道：**分支只是一个指向 commit 的指针。**

创建一个新分支：`git branch dev`

这时候 git 做了什么？

1. 在 `.git/refs/heads/` 下创建一个 `dev` 文件
2. 文件内容写入当前 commit 的 SHA-1 哈希值

```
.git/refs/heads/
├── main    -> f7a1b2c...    # 主分支
└── dev     -> f7a1b2c...    # 新分支，指向同一个 commit
```

切换分支：`git checkout dev`

这时候 git 做了什么？

1. 修改 `.git/HEAD` 指向 `refs/heads/dev`
2. 根据 dev 指向的 commit，将工作区恢复成那个时刻的状态

**分支切换为什么这么快？** 因为只改了 HEAD 指针，然后从对象数据库里读取对应文件到工作区。没有复制任何东西。

## 暂存区的秘密

理解了对象模型后，再来看暂存区。

暂存区是 `.git/index` 文件，一个二进制文件。但它的作用很关键：**它是「工作区」到「对象数据库」的桥梁。**

执行 `git add a.txt` 时：

1. git 读取文件内容，创建 blob 对象存入 `.git/objects/`
2. 在 `.git/index` 中记录：`a.txt -> blob哈希值`

执行 `git commit` 时：

1. git 根据 index 生成 tree 对象
2. 创建 commit 对象（包含 tree 指针 + parent 指针）
3. 更新当前分支指针指向新 commit

:::note
**为什么需要暂存区？** 因为它给了你「选择权」。你可以选择哪些文件进入这次提交，哪些先不提交。这在 SVN 里是做不到的——SVN 是全量提交。
:::

## git push 做了什么

远程仓库是另一个 git 仓库（通常在 GitHub、GitLab 服务器上）。

执行 `git push origin main` 时：

1. git 从本地的 main 分支出发，追踪所有祖先 commit
2. 找出本地有但远程没有的对象（commit、tree、blob）
3. 打包这些对象发送给远程

```
本地：C1 — C2 — C3 — C4
远程：C1 — C2

需要传输：C3、C4 以及它们包含的所有对象
```

这就是为什么第一次 push 会比较慢（要上传整个历史），而之后的 push 很快（只上传新增的变更）。

## 分散式 vs 集中式

说到这里，必须提一下 git 的「分布式」特性。

像 SVN 这样的集中式版本控制系统，所有历史都存在中央服务器。如果服务器挂了，历史记录就没了。

而 git 是**每个克隆都是完整的仓库**：

```
你的电脑：完整的项目历史 + 所有 commit
GitHub 服务器：完整的项目历史 + 所有 commit
同学电脑：完整的项目历史 + 所有 commit
```

这意味着：

- 离线也能查看完整历史
- 服务器挂了不会丢失任何数据
- 每个节点都是平等的

## 庆幸与感恩

站在今天回顾，我很庆幸自己在 2023 年就接触并学会了 git。

那时候觉得「git 只是一个工具，会用就行了」。但现在回头看，正是那些「被迫自学」的经历，让我真正理解了版本控制的思想，也为我后来搭建这个博客奠定了基础。

做独立项目时，每次遇到问题，我都会 `git commit` 记录当下的状态。即使改错了，也可以随时回退到任何一个历史版本。

**如果没有工作留痕，没有历史记录，我想要去回顾整理自己的生涯，该是多么困难的一件事啊。**

现在我的每一篇博客、每一个项目，都有完整的 git 提交记录。每一次 `git log`，都像是在和过去的自己对话。

## 写在最后

回想当初学 git 的那段时光，有几点感悟：

| 维度   | 当时的困境           | 现在的理解          |
| ---- | --------------- | -------------- |
| 学习方式 | 照着教程敲命令，不知其所以然  | 先理解核心概念，再学习命令  |
| 问题解决 | 遇到 bug 就去搜，盲目尝试 | 看报错信息，理解原理再动手  |
| 版本控制 | 觉得是多余的操作        | 深刻体会到「后悔药」的价值  |
| 工具认知 | 只知道几个命令         | 理解底层原理才能用得游刃有余 |

git 不仅仅是一个工具，更是一种思维方式。它教会我：**代码要边做边记录，问题要留痕可追溯。**

现在每当我 `git add` 并 `git commit` 时，都会想起那个满脑子问号的下午。正是那段迷茫又坚持的自学时光，让我在今天能够游刃有余地管理自己的项目。

**感谢 git，感谢那段被迫自学却让我受益至今的日子。**
