## 项目背景

青橙商城是一个典型的 B2C（Business-to-Customer）电商平台，运营商将商品发布到网站，会员注册后可以浏览商品、加购、下单、支付，完成完整的线上购物流程。

这个项目涵盖三大核心模块：

| 模块 | 描述 | 技术挑战 |
| -- | -- | -- |
| 后台管理系统 | 商品、订单、用户、营销、系统的全生命周期管理 | 复杂表单处理、数据可视化报表 |
| 前台用户商城 | 面向消费者的商品浏览、搜索、购物车、订单、支付 | SEO 友好、用户体验优化 |
| 秒杀专项模块 | 高并发场景下的限时抢购 | 流量管控、库存扣减、缓存穿透 |

:::note
相比之前做的学习项目，青橙商城的业务复杂度更高。第一次感受到「做项目」与「做练习」的本质区别——业务是复杂的、技术是综合的、工程是协作的。
:::

## 技术架构

### 整体架构

项目采用前后端分离的系统架构，后端基于 SSM + Dubbo 构建 **SOA（Service-Oriented Architecture，面向服务架构）** 分布式服务集群，前端采用 Vue.js 技术栈。

**什么是 SOA？**

SOA 的核心思想是「服务化」——将业务功能拆分成独立、可复用的服务，每个服务只专注做一件事。就像一家分工明确的餐厅：厨师只负责做菜，服务员只负责传菜，收银员只负责收钱，各司其职，通过标准接口协作。

| 对比维度 | 传统单体架构 | SOA 分布式架构 |
| -- | -- | -- |
| 代码组织 | 所有人做一个大项目 | 按业务拆分成独立服务 |
| 部署方式 | 整体部署 | 各服务独立部署 |
| 扩展方式 | 整体扩展（浪费资源） | 按需扩展特定服务 |
| 开发协作 | 多人改同一份代码（冲突多） | 各自开发独立服务（解耦） |
| 故障影响 | 一个 Bug 可能拖垮整个系统 | 单个服务故障不影响其他 |

SOA 的关键特征：
- **服务复用**：商品服务可以被订单、搜索、营销等多个模块调用
- **独立部署**：秒杀模块出问题，不会影响商品浏览
- **统一技术栈**：Dubbo 基于 RPC 通信，Java 技术栈内部调用高效，但跨语言支持有限

## 服务治理：Dubbo 详解

### 为什么需要 Dubbo？

想象一个大型小区里有 8 栋楼（微服务），每栋楼有几十户（服务实例）。如果没有「物业管理处」（注册中心），你想给某户送快递（调用服务）：

- 你得记住每户的具体门牌号（IP + 端口）
- 某户搬迁了（服务重启/迁移），你还得更新记录
- 快递员累死，用户体验极差

有了 Dubbo（基于 Zookeeper），情况完全不同了：

- 每户入住时自动到物业登记（服务注册）
- 物业处有一份住户名册（服务列表）
- 快递员只需问物业就能找到目标住户（服务发现）
- 住户搬走时会主动销户（故障剔除）

:::tip
**Dubbo 就是小区物业管理处**：它不负责送快递（不参与业务），但没有它，整个小区就乱套了。
:::

### Dubbo 负载均衡策略

| 策略 | 原理 | 适用场景 |
| -- | -- | -- |
| random | 随机选择 | 默认策略，适合请求量均匀的场景 |
| roundrobin | 轮询选择 | 适合服务器性能相近的场景 |
| leastactive | 选择活跃数少的 | 适合处理能力有差异的场景 |
| consistenthash | 相同参数请求到同一台 | 适合需要会话保持的场景 |

## 数据库架构：垂直分库设计

### 什么是垂直分库？

想象一个超大型仓库：

- **不分库**：所有货物存在一个巨大仓库，找东西要翻很久
- **垂直分库**：按货物类别分区——电器区、服装区、食品区，每个区独立管理

```
应用层 ───────────────────────────────────────────────
  │        │        │        │        │
  ↓        ↓        ↓        ↓        ↓
┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐
│商品库 ││订单库 ││用户库 ││运营库 ││支付库 │
│ MySQL││ MySQL││ MySQL││ MySQL││ MySQL│
└──────┘└──────┘└──────┘└──────┘└──────┘
```

## 业务模块实战

技术架构是骨架，业务代码是血肉。以下选取最具代表性的核心模块进行深入讲解。

### 商品模块：SPU/SKU 与 ES 全文检索

商品模块（qingcheng_goods）是整个商城的「根基」，涉及商品分类、品牌、规格参数、SPU/SKU 等核心概念。

**核心实体关系**：

| 概念 | 说明 | 示例 |
| -- | -- | -- |
| SPU | Standard Product Unit，标准化产品单元 | iPhone 15 |
| SKU | Stock Keeping Unit，库存量单位 | iPhone 15 黑色 256G |

**商品搜索服务实现**：

```java
@Service
public class GoodsSearchService {

    @Autowired
    private RestHighLevelClient restHighLevelClient;

    public Map<String, Object> search(Map<String, String> searchMap) {
        try {
            // 1. 构建搜索请求
            SearchRequest searchRequest = new SearchRequest("qingcheng_goods");
            SearchSourceBuilder sourceBuilder = new SearchSourceBuilder();

            // 2. 多条件组合查询
            BoolQueryBuilder boolQuery = QueryBuilders.boolQuery();

            // 关键词搜索（商品名、品牌、分类）
            if (StringUtils.isNotBlank(searchMap.get("keywords"))) {
                boolQuery.must(QueryBuilders.matchQuery("name", searchMap.get("keywords")));
            }

            // 分类过滤
            if (StringUtils.isNotBlank(searchMap.get("category"))) {
                boolQuery.filter(QueryBuilders.termQuery("categoryName", searchMap.get("category")));
            }

            // 品牌过滤
            if (StringUtils.isNotBlank(searchMap.get("brand"))) {
                boolQuery.filter(QueryBuilders.termQuery("brandName", searchMap.get("brand")));
            }

            // 价格区间筛选
            if (StringUtils.isNotBlank(searchMap.get("priceMin"))) {
                boolQuery.filter(QueryBuilders.rangeQuery("price").gte(searchMap.get("priceMin")));
            }
            if (StringUtils.isNotBlank(searchMap.get("priceMax"))) {
                boolQuery.filter(QueryBuilders.rangeQuery("price").lte(searchMap.get("priceMax")));
            }

            // 3. 分页
            int page = Integer.parseInt(searchMap.getOrDefault("page", "1"));
            int size = Integer.parseInt(searchMap.getOrDefault("size", "20"));
            sourceBuilder.from((page - 1) * size);
            sourceBuilder.size(size);

            // 4. 高亮
            sourceBuilder.highlighter(new HighlightBuilder()
                    .field("name")
                    .preTags("<em>")
                    .postTags("</em>"));

            sourceBuilder.query(boolQuery);
            searchRequest.source(sourceBuilder);

            // 5. 执行搜索
            SearchResponse response = restHighLevelClient.search(searchRequest, RequestOptions.DEFAULT);

            // 6. 封装结果
            return getSearchResult(response);

        } catch (Exception e) {
            throw new RuntimeException("搜索失败：" + e.getMessage());
        }
    }
}
```

:::tip
**ES 倒排索引机制**：不是像 MySQL 那样遍历找关键词，而是根据词去找文档。就像书的索引页——想找「分布式」出现在哪页，直接查索引，而不是从第一页翻到最后一页。
:::

### 订单模块：状态机与分布式会话

订单模块（qingcheng_order）涉及订单创建、支付、发货、收货等状态流转，是典型的「状态机」场景。

**订单状态流转**：

```
    待支付 ────→ 已支付 ────→ 已发货 ────→ 已完成
       ↑            │
       │            │ 超时/退款
       │            ↓
       └─────── 已取消
```

**订单创建服务**：

```java
@Service
public class OrderServiceImpl implements OrderService {

    @Autowired
    private OrderMapper orderMapper;

    @Autowired
    private CartService cartService;

    @Autowired
    private AmqpTemplate amqpTemplate;

    @Transactional
    public Order createOrder(String userId, String addressId, String[] cartIds) {
        // 1. 查询购物车商品
        List<CartItem> cartItems = cartService.getCheckedItems(userId, cartIds);
        if (cartItems == null || cartItems.isEmpty()) {
            throw new RuntimeException("购物车为空");
        }

        // 2. 计算总金额
        BigDecimal totalMoney = cartItems.stream()
                .map(item -> item.getPrice().multiply(BigDecimal.valueOf(item.getNum())))
                .reduce(BigDecimal.ZERO, BigDecimal::add);

        // 3. 创建订单
        Order order = new Order();
        order.setId(String.valueOf(System.currentTimeMillis()));
        order.setUserId(userId);
        order.setTotalMoney(totalMoney);
        order.setPayMoney(totalMoney);  // 简化处理，暂不考虑优惠
        order.setAddressId(addressId);
        order.setOrderTime(new Date());
        order.setPayStatus("0");  // 0=待支付
        order.set consigneeStatus("0");  // 0=待发货
        order.set consigneeTime(null);
        order.set consigneeStatus("0");  // 0=待收货

        orderMapper.insert(order);

        // 4. 保存订单明细
        for (CartItem item : cartItems) {
            OrderItem orderItem = new OrderItem();
            orderItem.setOrderId(order.getId());
            orderItem.setGoodsId(item.getGoodsId());
            orderItem.setSkuId(item.getSkuId());
            orderItem.setNum(item.getNum());
            orderItem.setPayMoney(item.getPrice().multiply(BigDecimal.valueOf(item.getNum())));
            orderItemMapper.insert(orderItem);
        }

        // 5. 发送消息通知库存服务扣减库存
        amqpTemplate.convertAndSend("order.exchange", "order.create", order.getId());

        // 6. 清空已购买的购物车商品
        cartService.removeItems(userId, cartIds);

        return order;
    }
}
```

### 用户模块：SpringSecurity + CAS 认证

用户模块（qingcheng_user）是整个系统的「身份中枢」，涉及注册、登录、权限管理等。

**CAS（Central Authentication Service，单点登录认证中心）** 解决的核心问题是：用户在一个系统登录后，访问其他相关系统无需重复登录。

**SpringSecurity 核心配置**：

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/api/user/login", "/api/user/register").permitAll()
                .antMatchers("/api/**").authenticated()
                .anyRequest().permitAll()
            .and()
            .formLogin()
                .loginProcessingUrl("/api/user/login")
                .successHandler((request, response, authentication) -> {
                    response.setContentType("application/json;charset=UTF-8");
                    response.getWriter().write("{\"success\":true,\"message\":\"登录成功\"}");
                })
                .failureHandler((request, response, exception) -> {
                    response.setContentType("application/json;charset=UTF-8");
                    response.getWriter().write("{\"success\":false,\"message\":\"登录失败\"}");
                })
            .and()
            .csrf().disable()
            .sessionManagement()
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS);
    }
}
```

**CAS 单点登录流程**：

```
用户访问系统A → 系统A发现未登录 → 跳转到CAS服务器
                                              ↓
                                    用户在CAS登录
                                              ↓
                              CAS返回ticket（临时凭证）
                                              ↓
用户携带ticket → 访问系统A → 系统A验证ticket → 登录成功
                                              ↓
用户访问系统B → 系统B发现未登录 → 跳转到CAS服务器
                   ↓
            CAS发现已登录 → 直接返回ticket → 系统B验证 → 登录成功
```

### 秒杀模块：高并发核心场景

这是整个项目最精华的部分，专门解决电商高并发场景下的核心问题。

#### 流量管控体系

**多层流量过滤**：

```
用户请求
   │
   ▼
┌───────────────┐
│   Nginx 层   │  ← 基础限流（黑白名单）
└───────┬───────┘
        │
        ▼
┌───────────────┐
│  令牌桶限流    │  ← 每秒发放固定数量令牌
└───────┬───────┘
        │
        ▼
┌───────────────┐
│  布隆过滤器    │  ← 过滤无效请求
└───────┬───────┘
        │
        ▼
┌───────────────┐
│ Redis Lua 脚本 │  ← 原子性库存扣减
└───────┬───────┘
        │
        ▼
    订单创建成功
```

**令牌桶限流实现**：

```java
@Component
public class RateLimiter {

    // 每秒发放 100 个令牌
    private RateLimiter rateLimiter = RateLimiter.create(100);

    public boolean tryAcquire() {
        return rateLimiter.tryAcquire();
    }
}

// 在秒杀接口中使用
@GetMapping("/seckill/{goodsId}")
public Result seckill(@PathVariable String goodsId) {
    // 1. 前置限流检查
    if (!rateLimiter.tryAcquire()) {
        return Result.error("活动太火爆，请稍后再试");
    }

    // 2. 检查用户是否已购买
    String userKey = "seckill:user:" + getCurrentUserId() + ":" + goodsId;
    if (Boolean.TRUE.equals(stringRedisTemplate.hasKey(userKey))) {
        return Result.error("您已购买过该商品");
    }

    // 3. 库存扣减
    boolean success = seckillService.decreaseStock(goodsId, getCurrentUserId());
    if (!success) {
        return Result.error("库存不足");
    }

    // 4. 返回成功
    return Result.ok("秒杀成功");
}
```

#### 库存核心设计

**秒杀前：活动开始前，将库存预热到 Redis**：

```java
// 秒杀活动开始前调用

    String stockKey = "seckill:stock:" + goodsId;
    stringRedisTemplate.opsForValue().set(stockKey, stock);
}
```

**秒杀时：使用 Lua 脚本保证原子性扣减**：

```lua
-- seckill_stock.lua
local stock_key = KEYS[1]
local user_key = KEYS[2]

-- 检查用户是否已购买
if redis.call('exists', user_key) == 1 then
    return -1  -- 已购买
end

-- 检查库存
local stock = tonumber(redis.call('get', stock_key))
if stock <= 0 then
    return 0  -- 库存不足
end

-- 原子性扣减库存
redis.call('decr', stock_key)
-- 标记用户已购买（24小时过期）
redis.call('setex', user_key, 86400, 1)
return 1  -- 成功
```

```java
public class SeckillServiceImpl implements SeckillService {

    private String luaScript;

    @PostConstruct
    public void init() {
        luaScript = ResourceUtil.readString("classpath:seckill_stock.lua");
    }

    public boolean decreaseStock(Long goodsId, Long userId) {
        String stockKey = "seckill:stock:" + goodsId;
        String userKey = "seckill:user:" + userId + ":" + goodsId;

        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(luaScript, Long.class),
            Arrays.asList(stockKey, userKey)
        );

        return result == 1;
    }
}
```

:::tip
**Lua 脚本在 Redis 中执行是原子的**——就像银行转账操作，要么全部成功，要么全部失败，不存在中间状态。这是从根本上杜绝超卖问题的关键。
:::

## 缓存策略设计

商城系统的缓存有其特殊性——商品信息变化不频繁但访问量巨大，库存要求强一致但查询频繁，价格和促销随时变化。如何在性能和一致性之间取得平衡，是电商缓存设计的核心难题。

### 商城缓存的特点

| 缓存对象 | 特点 | 缓存策略 |
| -- | -- | -- |
| 商品信息 | 变化不频繁，访问量大 | 缓存时间长，更新时主动失效 |
| 商品价格 | 变化频繁，实时性要求高 | 缓存时间短或不做缓存 |
| 库存数据 | 要求强一致，写操作多 | Redis 缓存 + 数据库双写 |
| 购物车 | 用户私有，数据量小 | Redis Hash 存储 |
| 会话信息 | 用户私有，过期即失效 | Redis Session |

### 商品信息缓存实现

商品详情页是典型的「读多写少」场景，非常适合缓存：

```java
@Service
public class GoodsCacheService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String GOODS_KEY = "goods:detail:";
    private static final int CACHE_EXPIRE = 3600;  // 1小时

    public Goods findById(Long id) {
        String cacheKey = GOODS_KEY + id;
        String cached = redisTemplate.opsForValue().get(cacheKey);

        // 缓存命中
        if (cached != null) {
            return JSON.parseObject(cached, Goods.class);
        }

        // 缓存未命中，查询数据库
        Goods goods = goodsMapper.selectById(id);

        if (goods != null) {
            // 写入缓存
            redisTemplate.opsForValue().set(cacheKey,
                    JSON.toJSONString(goods),
                    CACHE_EXPIRE,
                    TimeUnit.SECONDS);
        }

        return goods;
    }

    // 商品更新时，主动删除缓存（Cache Aside 模式）
    public void updateGoods(Goods goods) {
        goodsMapper.updateById(goods);
        // 删除缓存，下次查询会重新加载
        redisTemplate.delete(GOODS_KEY + goods.getId());
    }
}
```

**Cache Aside 模式**的核心原则是「更新时先删缓存，而不是更新缓存」。为什么？

想象这个场景：
1. 商品原价 100 元，缓存中也是 100 元
2. 管理员改为 80 元，但还没来得及更新缓存
3. 此时用户访问，看到的仍是 100 元（脏数据）

如果采用「先删缓存」策略：
1. 管理员更新价格为 80 元
2. 删除缓存中的商品
3. 用户访问，发现缓存没了，查询数据库得到 80 元

这样用户最多看到「缓存过期」期间的旧数据，不会看到被更新前的脏数据。

### 库存缓存的特殊处理

库存与其他缓存不同——它要求**强一致性**，不能有超卖。但在高并发场景下，如果每次都查数据库，数据库压力巨大。

青橙商城的解决方案是「**数据库 + Redis 双重校验**」：

```java
@Service
public class StockService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String STOCK_KEY = "stock:";

    // 扣除库存（双重校验）
    public boolean decreaseStock(Long skuId, Integer count) {
        // 1. 先检查 Redis 库存是否充足（快速判断）
        String stockStr = redisTemplate.opsForValue().get(STOCK_KEY + skuId);
        if (stockStr == null) {
            // Redis 没有，从数据库加载
            stockStr = String.valueOf(goodsMapper.selectStockBySkuId(skuId));
            redisTemplate.opsForValue().set(STOCK_KEY + skuId, stockStr);
        }

        int stock = Integer.parseInt(stockStr);
        if (stock < count) {
            return false;  // 库存不足
        }

        // 2. 乐观锁更新数据库（保证数据一致性）
        int result = stockMapper.decreaseStockWithVersion(skuId, count);
        if (result > 0) {
            // 数据库更新成功，同步更新 Redis
            redisTemplate.opsForValue().decrement(STOCK_KEY + skuId, count);
            return true;
        }

        return false;
    }
}
```

:::tip
**为什么用「双重校验」？**

Redis 库存只是「预扣」机制，真正的库存以数据库为准。如果 Redis 扣成功了，但数据库更新失败（比如版本冲突），说明有并发问题，此时返回失败让用户重试。这比直接用 Redis 作为唯一库存源要安全得多。
:::

## 踩坑实录

### 1. 分布式会话丢失

**问题现象**：用户登录后刷新页面，忽然又跳转到登录页。明明已经登录成功了，但后续请求却提示未授权。

**什么是「分布式会话」？**

就像小区里的「访客登记」：

| 场景 | 实际情况 | 技术问题 |
| -- | -- | -- |
| 单机模式 | 访客信息写在物业办公室的本子上 | Session 存在应用服务器本地 |
| 分布式模式 | 小区有 8 栋楼，访客可能去任意一栋 | 请求可能被路由到不同服务器 |

**根因分析**：分布式环境下，用户请求可能被路由到不同服务器。服务器 A 登录后把会话存在本地，但下一次请求被路由到服务器 B，B 上没有这个会话数据。

**解决方案**：将会话统一存储到 Redis

```xml
<!-- Spring Session 配置 -->
<bean id="redisHttpSessionConfiguration"
      class="org.springframework.session.data.redis.config.annotation.web.http.RedisHttpSessionConfiguration">
    <property name="maxInactiveIntervalInSeconds" value="3600"/>
</bean>

<bean id="jedisConnectionFactory"
      class="org.springframework.data.redis.connection.jedis.JedisConnectionFactory">
    <property name="hostName" value="192.168.1.100"/>
    <property name="port" value="6379"/>
</bean>
```

```java
// 登录时将会话存入 Redis
@RequestMapping("/login")
public Result login(String username, String password, HttpSession session) {
    User user = userService.login(username, password);
    // 将会话 ID 存入 Redis，设置过期时间
    stringRedisTemplate.opsForValue().set("session:" + session.getId(), user.getId(), 3600, TimeUnit.SECONDS);
    return Result.ok(session.getId());
}
```

:::note
**分布式系统中，状态存储一定要统一**。Java Web 应用的 session 是本地存储的，在分布式环境下必须外置化。
:::

### 2. MySQL 连接池耗尽

**问题现象**：在压测过程中，MySQL 连接池迅速耗尽，后续请求堆积，最终导致服务雪崩。

**根因分析**：在青橙商城中，商品详情页涉及多表关联查询（商品表、分类表、品牌表、SKU表），加上分库后每个服务只有一个数据源，连接池压力尤为突出。

**为什么电商系统更容易遇到连接池问题？**

| 场景 | 对连接池的影响 |
| -- | -- |
| 商品列表页 | 多表 JOIN + 分页，查询时间长 |
| 秒杀活动 | 瞬时大量请求同时访问数据库 |
| 垂直分库 | 每个库独立连接池，总体连接数翻倍 |
| 慢查询 | 一个慢查询占用连接直到查询完成 |

在 8 库架构下，如果每个库连接池最大 20 个，理论最大连接数是 160 个，比单体架构更容易触发连接耗尽。

**解决方案**：

```yaml
spring:
  datasource:
    druid:
      # 连接池大小根据服务器配置调整
      initial-size: 5          # 初始连接数
      min-idle: 5               # 最小空闲连接
      max-active: 20            # 最大活跃连接（根据压测调整）
      max-wait: 3000            # 获取连接超时 3 秒
      # 建议监控指标
      validation-query: SELECT 1  # 检测连接是否有效
      test-while-idle: true      # 空闲时检测
      time-between-eviction-runs-millis: 60000  # 每分钟检查一次
```

**压测调优经验**：

```java
// 通过注解标记慢查询，方便排查
@SlowQuery(threshold = 1000)  // 超过 1 秒的查询记录日志
public List<Goods> searchGoods(Map<String, String> params) {
    // ...
}
```

:::warning
**连接池配置的核心原则**：连接池不是越大越好。对于 4 核 8G 的服务器，20 个连接已经是比较保守的设置。如果压测时频繁超时，说明 SQL 本身有问题，优先优化 SQL 而不是增大连接池。
:::

### 3. Redis 缓存雪崩

**问题现象**：在做秒杀模块压力测试时，发现一个问题——当秒杀活动一开始，大量请求同时涌入，系统很快就开始报错。一开始以为是限流没生效，后来发现是缓存出了问题。

**根因分析**：商品详情的缓存设置了统一的 2 小时过期时间。在压测环境下，所有商品的缓存几乎同时创建、同时过期。当活动开始的瞬间，所有缓存失效，大量请求直接打到数据库，就像「雪崩」一样。

**解决方案**：

**方案一：过期时间随机化**

```java
// 设置缓存时，加上随机过期时间
int expireTime = baseExpireTime + RandomUtils.nextInt(300);
stringRedisTemplate.opsForValue().set(cacheKey, data, expireTime, TimeUnit.SECONDS);
```

**方案二：多级缓存**

```java
// 本地缓存（JVM 内存）+ Redis 分布式缓存
private Map<String, Object> localCache = new ConcurrentHashMap<>();

public Object getData(String key) {
    // 1. 先查本地缓存
    if (localCache.containsKey(key)) {
        return localCache.get(key);
    }

    // 2. 再查 Redis
    Object data = stringRedisTemplate.opsForValue().get(key);
    if (data != null) {
        localCache.put(key, data);  // 回填本地缓存
    }
    return data;
}
```

**方案三：熔断降级**

```java
@HystrixCommand(fallbackMethod = "getDataFallback")
public Object getData(String key) {
    return goodsService.getDataFromDB(key);
}

public Object getDataFallback(String key) {
    return defaultData;  // 降级处理
}
```

:::tip
**缓存是双刃剑**，用好了能极大提升性能，用不好反而会成为系统瓶颈。要充分考虑各种异常场景，提前做好防护。
:::

### 4. 垂直分库后的跨库查询

**问题现象**：需要查询「用户最近购买商品的分类统计」，但用户表在用户库，订单表在订单库，商品分类在商品库，三个库要关联查询。

**解决方案**：

1. **应用层聚合**：在服务层分别查询各库数据，然后在应用层做内存关联

```java
public Map<String, Integer> getUserCategoryStats(String userId) {
    // 1. 查询用户订单（订单库）
    List<Order> orders = orderService.getUserOrders(userId);

    // 2. 查询订单对应的商品（商品库）
    List<Long> goodsIds = orders.stream().map(Order::getGoodsId).collect(toList());
    List<Goods> goodsList = goodsService.getGoodsByIds(goodsIds);

    // 3. 在应用层聚合统计
    Map<String, Integer> categoryStats = new HashMap<>();
    for (Goods goods : goodsList) {
        String categoryName = goods.getCategoryName();
        categoryStats.merge(categoryName, 1, Integer::sum);
    }

    return categoryStats;
}
```

2. **使用 ES 做跨库搜索**：提前将多库数据同步到 ES，通过 ES 的关联查询实现

:::note
**垂直分库的代价**：分库后跨库查询变复杂了，不像单库时可以简单 SQL JOIN。解决方案通常是「服务层聚合」或「ES 搜索」。
:::

踩过了这些坑，终于可以聊聊更深层的问题了——**为什么选择 SOA 而不是微服务？**

## 技术对比：SOA vs 微服务

| 对比维度 | SOA（青橙商城采用） | 微服务（Spring Cloud 方案） |
| -- | -- | -- |
| 服务粒度 | 粗粒度，按业务模块拆分 | 细粒度，按功能点拆分 |
| 服务通信 | Dubbo RPC，同步调用为主 | HTTP/REST，混合同步异步 |
| 数据管理 | 每个服务有独立数据库 | 理论上每个服务独立数据库 |
| 适用场景 | 中大型企业级应用 | 互联网弹性业务 |
| 复杂度 | 中等 | 较高 |

:::note
SOA 和微服务不是非此即彼的选择，而是要根据业务规模、团队能力、成本预算来权衡。青橙商城用 SOA 是合理的——业务稳定、团队较小、追求可控。
:::
