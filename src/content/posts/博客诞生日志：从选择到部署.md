## 为什么选择写博客

最初的想法很简单：**记录学习生活，同时打造个人作品集**。

作为一个 Agent 全栈开发者，我每天都在和 AI 编程打交道。写博客既是整理知识的过程，也是展示个人技术能力的方式。相比于把代码堆在 GitHub 上无人问津，一个持续更新的技术博客显然更能让人印象深刻。

## 为什么选 Fuwari

在挑选主题的过程中，我对比了多个 Astro 博客主题，最终选择 Fuwari 的原因很直接：

- **界面好看**：整体设计简洁现代，亮/暗模式切换自然
- **技术栈熟悉**：Astro + Tailwind CSS + Svelte，正好是我日常使用的技术栈
- **文档清晰**：主题作者提供了详尽的中文二开指南，很多配置项都有清楚说明
- **部署灵活**：支持 Vercel、Netlify、GitHub Pages 等多种部署方式

最终我选择将它部署到 **GitHub Pages**，一方面是免费，另一方面和自己的代码仓库天然结合，push 即部署，流程简洁。

## 二开过程

得益于 Fuwari 良好的架构设计和清晰的配置约定，我的二开过程总体顺利，没有遇到太大的困难。

主要做了以下几件事：

1. **清理示例内容** — 删掉默认的文章和示例，保留干净的空白博客框架
2. **个性化配置** — 修改 `src/config.ts` 中的站点信息、头像、社交链接
3. **调整部署配置** — 适配 GitHub Pages 的构建流程，修复了矩阵构建导致的 artifact 冲突问题
4. **本地调试** — 确认 `base` 路径配置正确后，本地预览正常

:::note
Fuwari 的配置高度集中化，几乎所有核心配置都在 `src/config.ts` 一个文件里完成，非常适合不想折腾过多文件的开发者。
:::

## Agent 全栈背景下的感受

作为 Agent 全栈开发者，我习惯于让 AI 辅助完成代码编写。但在这个博客项目中，我更倾向于把它当作一个**手工打磨的作品**来对待。

不是说 AI 不能帮我写，而是在这个过程中，我更在意的是**理解每一行配置背后的逻辑**。Fuwari 的代码结构清晰，组件化做得很好，这让定制变得非常自然。
