## 引言：第二个月，我开始「参与」了

如果说第一个月的关键词是「熟悉」，那么第二个月的关键词应该是「参与」。

第一个月，我像是一个站在场边的观众，看着团队忙碌，自己做些边边角角的辅助工作。但到了第二个月，我开始真正「参与」到开发流程中了——参与代码检视、协助接口联调、配合BUG定位。

这种感觉，就像是游戏里终于从「新手教程」进入了「正式关卡」。

这篇文章，我就来聊聊第二个月到底经历了什么。

## 第五周：Git Flow的真正理解

### 从「会用」到「用好」

还记得第一个月第一次接触Git Flow时，我的感觉是「懵」——这么多分支、这么多流程，什么时候该用哪个？

第二个月，我终于有了更深的理解。

mentor 给了我一个任务：从开发分支创建一个功能分支，完成辅助开发工作后合并回开发分支。这次我需要独立处理整个流程，包括中途遇到冲突时的解决。

```bash
# 1. 确保开发分支是最新的
git checkout develop
git pull origin develop

# 2. 创建功能分支
git checkout -b feature/assist-document-orginization

# 3. 在功能分支上完成工作
# ... 修改代码 ...

# 4. 提交遵循企业规范
git add .
git commit -m "feat: 协助整理xxx模块资料"

# 5. 推送前先拉取开发分支最新代码，检查是否有冲突
git fetch origin
git rebase origin/develop

# 6. 解决冲突后，强制推送（功能分支）
git push -f origin feature/assist-document-orginization
```

这里最让我收获的是**rebase操作**。之前只知道`merge`，但当分支多起来后，`merge`会让提交历史变得复杂。rebase能够保持一条干净的线性历史，更便于追溯。

### 第一次参与代码检视

这周还第一次正式参与了团队的代码检视。

说实话，一开始有些紧张——毕竟是要看别人的代码，还要提出意见。但mentor鼓励我说：「code review是很好的学习机会，你看别人的代码，能学到好的写法，也能看到一些容易踩的坑。」

我仔细阅读了前辈们提交的代码，发现了几个有意思的点：

| 发现点  | 具体内容                                         | 我的收获               |
| ---- | -------------------------------------------- | ------------------ |
| 命名规范 | `getUserById` vs `getUserInfo`，同一个项目内命名风格要统一 | 变量命名要顾全上下文         |
| 注释习惯 | 核心业务逻辑都有注释说明意图，而不只是描述「做什么」                   | 注释要写「为什么」，不只是「是什么」 |
| 异常处理 | 很多地方用了自定义异常类，而不是直接throw Exception            | 异常要分类处理，便于排查问题     |

:::note
第五周的核心感受：code review不仅是检查代码，更是学习他人经验、提升自己视野的好机会。
:::

## 第六周：接口联调的「苦与乐」

### 第一次协助接口联调

第六周，我第一次深度参与了接口联调工作。

联调的对象是一个第三方数据对接接口。甲方提供的文档写得很简洁，但真正对接起来才发现——文档和实际返回的数据格式有不小差异。

我的任务是协助正式开发人员排查这些差异：

- 参数名称不匹配：文档写的是`user_name`，实际返回的是`userName`
- 日期格式不一致：文档标注`YYYY-MM-DD`，实际返回的是时间戳
- 某些字段文档标注「必填」，实际返回却可能是null

这些看起来琐碎的问题，却花了我整整两天时间才排查清楚。

### BUG定位的「福尔摩斯」体验

这周还遇到了一个让我印象很深的BUG——前端调用接口时，偶尔会出现数据展示不正确的问题。

mentor让我协助复现和定位这个问题。我按照企业BUG处理流程操作：

```markdown
## BUG详情记录

**问题描述**：用户列表接口偶发性显示异常，部分用户名显示为「undefined」

**复现步骤**：
1. 打开用户管理页面
2. 快速切换筛选条件（省份、职业）
3. 约第3-5次切换时出现

**排查过程**：
1. 检查接口返回数据 → 数据正常
2. 检查前端取值逻辑 → 发现异步请求未完成时已渲染
3. 定位根因 → 竞态条件，前端未做请求取消处理

**解决方案**：建议前端增加请求取消机制或添加loading状态
```

这次BUG定位经历让我体会到了「福尔摩斯」的感觉——从表象出发，一步步追溯根因，最终找到真相。

:::note
第六周的核心感受：接口联调考验的是细心和耐心，而BUG定位考验的是逻辑推理能力。
:::

## 第七周：测试的世界

### 从「被测」到「测人」

第七周，我第一次协助测试人员完成接口功能测试。

之前在学校里，我的代码是被老师测试的对象。而在公司，我开始协助测试人员——这种视角的转换让我对「测试」有了全新的理解。

测试并不是在「挑刺」，而是在**保障质量**。一个好的测试用例，能够在发布前发现问题，而不是让用户来发现问题。

mentor给了我一个接口，让我协助补充简单的测试点：

```markdown
## 接口测试用例（辅助编写）

### 接口：GET /api/user/{id}

**用例1**：正常查询
- 输入：存在的用户ID
- 预期：返回用户完整信息，状态码200

**用例2**：用户不存在
- 输入：不存在的用户ID
- 预期：返回空数据，状态码404

**用例3**：参数格式错误
- 输入：ID为非数字字符串
- 预期：返回参数错误提示，状态码400

**用例4**：特殊字符
- 输入：ID包含特殊字符（如 %、&）
- 预期：返回参数校验失败，状态码400
```

### 测试反馈处理

编写测试用例只是第一步，更关键的是**处理测试反馈**。

测试人员测出一个问题后，会在Jira上创建BUG单。我需要协助查看这个问题，判断是接口问题还是测试理解偏差，并配合修复。

这个过程让我深刻理解了：**测试和开发是同一个目标的两个视角**，都是为了交付高质量的产品。

:::note
第七周的核心感受：测试不是挑刺，而是质量保障。学会从测试视角审视代码，能写出更健壮的程序。
:::

## 第八周：复盘与沉淀

### 开发日志的进阶

第八周，我开始更认真地对待开发日志。

mentor建议我不仅记录「做了什么」，还要记录「学到了什么」和「遇到了什么」：

```markdown
## 2025.04.13 工作日志

### 今日完成
- [x] 完成第三方接口数据格式差异排查（文档 vs 实际返回）
- [x] 协助复现用户列表显示异常BUG
- [x] 补充用户管理接口测试用例4条

### 今日学习
- 学会了通过Postman构造请求模拟接口调用
- 理解了「竞态条件」的概念和常见解决方案

### 遇到的问题
- 接口文档与实际返回不一致问题 → 通过实际调用验证替代盲目相信文档

### 明日计划
- [ ] 继续跟进BUG修复进展
- [ ] 学习接口鉴权相关知识
```

### 第一次参与团队复盘会

这周还第一次参加了团队的阶段性复盘会。

复盘会的内容让我印象深刻——不是流水账式的汇报，而是真正的「反思」：

1. **上周的目标完成了吗？**
2. **遇到了哪些问题？根因是什么？**
3. **有哪些经验可以沉淀成流程或规范？**

mentor说：「复盘不是为了追责，而是为了沉淀经验，避免下次再踩同一个坑。」

我默默记下了这句话。

:::note
第八周的核心感受：定期复盘、沉淀经验，比埋头赶路更重要。
:::

## 第二个月总结

### 技术能力提升

| 能力项     | 第一月末     | 第二月末                  |
| ------- | -------- | --------------------- |
| Git版本控制 | 掌握基础分支策略 | 熟练处理rebase、冲突解决       |
| 代码检视    | 未参与      | 理解code review的意义并积极参与 |
| 接口联调    | 未接触      | 掌握接口联调流程，能够排查数据格式差异   |
| BUG处理   | 只能定位简单问题 | 能够完整复现问题、记录详情、协助修复    |
| 测试配合    | 未参与      | 掌握测试用例编写方法，熟悉测试反馈处理流程 |

### 心态转变

第二个月最大的转变，是从「被动等待任务」到「主动发现问题」。

第一个月时，mentor给我分配什么我就做什么。但第二个月，我开始主动思考：「这个接口文档有没有可能补充更详细？」「这个场景有没有可能被遗漏？」

这种主动性的转变，让我从「旁观者」逐步变成了「参与者」。

### 不足与改进

当然，第二个月也暴露了一些问题：

- **技术深度不够**：能够发现问题，但解决问题还需要依赖前辈
- **业务理解有限**：接触的都是辅助性工作，对核心业务逻辑理解不足
- **沟通可以更主动**：有时候遇到问题还是会犹豫是否要请教前辈

***

*第二个月结束了，开始真正「参与」了。旅程还在继续。*
