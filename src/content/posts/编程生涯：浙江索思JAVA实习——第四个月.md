## 引言：第四个月，我开始「沉淀」了

如果说第一个月的关键词是「适应」，第二个月的关键词是「参与」，第三个月的关键词是「贡献」，那么第四个月的关键词应该是「沉淀」。

前三个月，我经历了从陌生到熟悉、从旁观到参与、从配合到贡献的过程。而第四个月，我开始真正把前三个月学到的内容串联起来，形成自己的知识体系。

这种感觉，就像是在大量摄入知识之后，终于开始有了消化的能力。

这篇文章，我就来聊聊第四个月到底经历了什么。

## 第十三周：代码检视的进阶

### 从「被检视」到「检视他人」

还记得第一次参与代码检视时，我还有些不知所措——看别人的代码，到底该看什么？该说什么？

mentor 给了我一个任务：独立对一份代码提交进行规范检查，并提出优化建议。这次不再是「参与」，而是真正「负责」。

### 代码检视清单

这次检视让我总结出了一套自己的检查清单：

| 检查维度  | 具体内容              | 发现的问题示例                               |
| ----- | ----------------- | ------------------------------------- |
| 命名规范  | 变量、方法、类名是否符合团队约定  | `getData()` 语义不明确，应改为 `getUserList()` |
| 注释完整度 | 核心逻辑是否有注释说明       | 批量查询逻辑缺少分页说明注释                        |
| 代码格式  | 缩进、空格、换行是否统一      | 部分行尾多了空格                              |
| 异常处理  | 是否对边界情况做了处理       | 空列表未做判空直接返回                           |
| 安全规范  | 敏感信息是否脱敏、SQL是否防注入 | 日志中打印了完整用户密码                          |

当我在检视意见中写下第一条建议时，mentor 看了后说：「对，就按这个思路来。code review 不是挑刺，而是帮助彼此写出更好的代码。」

### 一次有意义的讨论

这周检视时，我提了一个建议：某个方法可以抽取为公共工具类复用。

但前辈的回复让我印象深刻：「这个建议本身没错，但要看场景——如果这个方法只在这里用到，抽取反而会增加项目的复杂度。代码复用不是目的，可维护才是。」

这次讨论让我理解了一个道理：**技术选型要看实际场景，而不是追求「最优解」**。

:::note
第十三周的核心感受：代码检视不仅是检查规范，更是在讨论中沉淀出「什么是好代码」的理解。
:::

## 第十四周：BUG修复的独立闭环

### 第一次独立负责BUG修复

这次的BUG是之前遗留的一个偶发性问题：用户在批量导出数据时，偶尔会出现导出的文件内容与预期不符。

mentor 只给了一个提示：「这次你来独立完成，有问题可以讨论，但修复方案要你自己出。」

### 问题定位过程

按照企业BUG修复流程，我开始系统性地排查：

**第一步：收集信息**

```markdown
BUG-20250520-001

问题现象：批量导出用户数据时，偶发性出现导出内容与查询条件不匹配
影响范围：使用批量导出功能的测试人员
复现频率：约10次出现1次
```

**第二步：复现问题**

我在测试环境尝试复现这个BUG。通过反复操作，终于捕捉到了问题的规律——当导出数据量超过500条时，问题更容易出现。

**第三步：日志分析**

```bash
# 查看导出接口的调用日志
docker logs [container_id] --since "1h" | grep "batch-export"

# 发现关键日志
[2025-05-20 15:30:12] INFO: Starting batch export, total: 523
[2025-05-20 15:30:12] DEBUG: Query params: page=1, pageSize=500
[2025-05-20 15:30:13] DEBUG: Query result: 523 records
[2025-05-20 15:30:13] WARN: Result size mismatch, expected: 523, got: 500
```

**第四步：定位根因**

原来是分页逻辑的问题——导出接口在内部使用了分页查询，当总数超过单页限制时，查询的是第一页而非全量数据。

```java
// 修复前的问题代码
public List<User> batchExport(Long[] userIds) {
    // 直接查询，但内部逻辑会分页
    return userMapper.selectByIds(userIds);
}

// 根因：selectByIds 内部在数据量大时使用了分页
```

修复方案需要确保导出时不进行分页限制，或者改用流式查询。

### 修复与回归测试

最终修复后，我编写了详细的回归测试方案：

```markdown
## 回归测试方案

### 用例1：正常导出（小数据量）
- 输入：50个用户ID
- 预期：导出50条完整数据 ✓

### 用例2：边界值导出（500条）
- 输入：500个用户ID
- 预期：导出500条完整数据 ✓

### 用例3：超大数据量导出
- 输入：1000个用户ID
- 预期：导出1000条完整数据 ✓

### 用例4：偶发性测试
- 输入：随机批量查询，重复10次
- 预期：每次导出数量与查询条件一致 ✓
```

这次BUG修复，让我真正理解了什么叫「独立闭环」——从发现、定位、修复到验证，全流程自己负责。

:::note
第十四周的核心感受：独立负责BUG修复，是检验是否真正掌握流程的最佳方式。
:::

## 第十五周：资料归档的「最后一环」

### 项目资料整理

第十五周，我接到任务：整理项目开发过程中的各项资料，按照企业归档规范进行分类存储。

mentor 告诉我：「项目结束了不代表工作结束了，资料归档是开发流程的最后一环，也是后续维护的基础。」

### 企业资料归档规范

通过这次归档工作，我系统学习了一套资料分类方法：

| 资料类型  | 归档目录               | 文件命名规范          | 保管期限 |
| ----- | ------------------ | --------------- | ---- |
| 接口文档  | `/docs/api/`       | `模块名_v版本_日期.md` | 永久   |
| 开发日志  | `/docs/dev-log/`   | `日期_姓名_内容概述.md` | 1年   |
| BUG记录 | `/docs/bug-fix/`   | `BUG_编号_简述.md`  | 1年   |
| 测试用例  | `/docs/test-case/` | `模块名_用例编号.md`   | 6个月  |
| 版本说明  | `/docs/release/`   | `v版本号_日期.md`    | 永久   |

### 归档中的发现

整理过程中，我翻看了这四个月来所有的开发日志，有一种「时光胶囊」的感觉：

- 第一个月的日志：充满了「这是什么」「那个怎么用」的懵懂记录
- 第二个月的日志：开始有了一些技术细节的思考
- 第三个月的日志：有了独立解决问题的经验总结
- 第四个月的日志：更多是流程规范和团队协作的感悟

mentor 看过我的归档后说：「这些资料整理好，不仅是给项目留下的资产，更是给自己留下的成长记录。」

:::note
第十五周的核心感受：资料归档不是结束，而是沉淀。好的归档习惯让项目和个人都能持续受益。
:::

## 第十六周：版本更新的初体验

### 第一次参与版本更新

第十六周，项目进入版本更新阶段。这是我第一次深度参与版本发布流程。

mentor 给我安排的任务是：协助测试人员进行版本验收，并整理版本更新说明。

### 版本更新全流程

通过这次实践，我理解了一个版本的完整生命周期：

```markdown
## 版本更新流程

1. 需求确认 → 产品确认最终需求
2. 开发整合 → 所有功能分支合并到发布分支
3. 测试验收 → 测试人员进行全面回归
4. 问题修复 → 发现问题返回开发修复
5. 版本定稿 → 测试通过，锁定版本
6. 更新说明 → 编写版本更新文档
7. 正式发布 → 部署到生产环境
```

### 协助版本验收

我的主要工作是协助测试人员复现问题，并记录验收结果：

````markdown
## 版本 v2.3.0 验收记录

### 功能验收
| 功能模块 | 验收内容 | 结果 |
| -- | -- | -- |
| 用户管理 | 批量导出功能 | ✓ 通过 |
| 权限管理 | 角色分配接口 | ✓ 通过 |
| 数据统计 | 报表导出 | ✓ 通过 |

### 回归测试
| 测试项 | 测试结果 |
| -- | -- |
| 原有功能不受影响 | ✓ 通过 |
| 接口响应时间正常 | ✓ 通过 |
| 数据准确性验证 | ✓ 通过 |

### 版本更新说明（辅助编写）
```markdown
# v2.3.0 更新说明

## 新增功能
- 用户管理新增批量导出功能
- 优化了数据查询性能

## BUG修复
- 修复了批量查询偶发性空数据问题
- 修复了导出大数据量时数据不完整问题

## 注意事项
- 导出功能有频率限制，每分钟最多10次
```\
````

当看到自己的写的版本说明被正式发布时，有种说不出的成就感——这是第一次，自己的文字成为了项目文档的一部分。

:::note
第十六周的核心感受：版本更新是开发流程的收尾，也是对整个迭代的检验。参与其中，才能理解什么叫「交付」。
:::

## 第四个月总结

### 技术能力提升

| 能力项   | 第三月末      | 第四月末                |
| ----- | --------- | ------------------- |
| 代码检视  | 只能参与讨论    | 能够独立进行规范检查并提出建议     |
| BUG处理 | 独立修复简单BUG | 能够独立完成BUG定位到验证的全流程  |
| 文档管理  | 掌握基础编写    | 能够按规范进行资料分类归档       |
| 版本更新  | 未参与       | 掌握版本更新全流程，协助验收与文档编写 |

### 心态转变

第四个月最大的转变，是从「完成任务」到「沉淀经验」。

前三个月，我更多的是在吸收和配合。但第四个月，我开始主动思考：**「这次学到的，下次如何避免再踩同样的坑？」「这个规范为什么要这样定，背后是什么考量？」**

mentor 说过一句话让我印象很深：「经验不是经历，经历加上反思才是经验。只有沉淀下来的东西，才是真正属于你的。」

### 实习即将结束

第四个月接近尾声时，mentor 告诉我下周开始要做工作交接了。

说实话，心情有些复杂。有不舍——这四个月学到了太多课堂上永远学不到的东西。有感慨——四个月前那个忐忑不安的新人，现在能够独立完成各项辅助工作。有期待——带着这些积累，未来无论做什么都会更有底气。

***

*第四个月结束了，实习也接近尾声。旅程还在继续。*
