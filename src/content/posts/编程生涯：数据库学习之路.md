## 从「柳暗花明」开始的数据库之旅

一切要从一句SQL查询说起。

那是2023年4月初，我第一次在黑马的教程里看到`SELECT`语句的时候，内心其实是轻松的——比起之前折腾Java那些绕来绕去的框架，数据库的语法简直像是在说人话。

```sql
SELECT * FROM users WHERE id = 1;
```

这条语句的含义清晰得不需要解释：从中挑选出符合条件的数据。比起Java里那些层层嵌套的调用链，SQL简直是清流。

:::note
后来的学习告诉我，这种「轻松感」其实建立在我已经掌握了编程基础的前提下。没有变量、控制流这些概念打底，直接上手数据库，怕是另一番光景。
:::

## CRUD：数据库的「Hello World」

按照惯例，学习任何技术都要从最基础的CRUD开始，数据库也不例外。

### Create：数据的「出生证明」

插入数据是最直观的操作，毕竟你得先有数据，才能谈得上「使用」。

```sql
-- 插入单条数据
INSERT INTO users (name, email, created_at) VALUES ('张三', 'zhangsan@example.com', '2023-04-04');

-- 插入多条数据
INSERT INTO users (name, email, created_at) VALUES
    ('李四', 'lisi@example.com', '2023-04-04'),
    ('王五', 'wangwu@example.com', '2023-04-04');
```

### Read：数据的「读取」

查询是数据库最核心的操作，也是我花时间最多的地方。

```sql
-- 查询所有数据
SELECT * FROM users;

-- 条件查询
SELECT name, email FROM users WHERE id > 10 ORDER BY created_at DESC;

-- 分页查询
SELECT * FROM users LIMIT 10 OFFSET 20;
```

### Update：数据的「改写」

```sql
-- 更新数据
UPDATE users SET email = 'new@example.com' WHERE id = 1;
```

### Delete：数据的「消亡」

```sql
-- 删除数据
DELETE FROM users WHERE id = 1;
```

:::warning
Update和Delete操作务必带WHERE条件，否则就是「删库跑路」的节奏。虽然我们只是学习环境，但养成好习惯很重要。
:::

## 那些让我栽过跟头的「小陷阱」

学习数据库的过程并非一帆风顺，有些坑至今记忆犹新。

### 聚合函数后的WHERE vs HAVING

这是最容易混淆的地方：聚合函数后面的筛选条件，必须用`HAVING`，不能用`WHERE`。

```sql
-- 错误写法
SELECT COUNT(*) FROM users WHERE COUNT(*) > 5;

-- 正确写法
SELECT COUNT(*) as cnt FROM users GROUP BY name HAVING cnt > 5;
```

两者的区别在于：

| 子句     | 作用时机  | 是否能用于聚合函数 |
| ------ | ----- | --------- |
| WHERE  | 分组前筛选 | 否         |
| HAVING | 分组后筛选 | 是         |

### NULL的「幽灵」特性

数据库里的NULL是一个让无数新手头疼的存在。它不等于0，也不等于空字符串，更不等于另一个NULL。

```sql
-- 查询为NULL的记录，不能用 = NULL，要用 IS NULL
SELECT * FROM users WHERE email IS NULL;

-- 查询不为NULL的记录，要用 IS NOT NULL
SELECT * FROM users WHERE email IS NOT NULL;
```

这个坑我踩过，当时对着空结果百思不得其解，最后发现是NULL在作祟。

## 从表面到内部：数据库原理的敲门砖

如果只是停留在CRUD层面，那还算不上真正的「学习数据库」。老师的课程还涉及了一些底层原理，虽然当时听得云里雾里，但现在回看，这些概念确实为我后来深入理解打下了基础。

### 存储引擎：数据库的「心脏」

MySQL支持多种存储引擎，不同引擎有不同的特点：

| 引擎     | 特点          | 适用场景      |
| ------ | ----------- | --------- |
| InnoDB | 支持事务、行级锁、外键 | 通用场景，默认选择 |
| MyISAM | 查询快、不支持事务   | 只读场景      |
| Memory | 数据存内存，速度极快  | 临时表、缓存    |

当时觉得「反正用默认的就行」，现在想来，不同场景选对引擎很重要。

那 InnoDB 是如何存储数据的呢？它把数据组织成「页」（Page）的形式，每页默认 16KB。数据页就像图书馆的书架格子，里面存放着一条条数据记录。

```sql
-- 查看表使用的存储引擎
SHOW CREATE TABLE users;

-- 修改存储引擎
ALTER TABLE users ENGINE = InnoDB;
```

:::note
为什么是 16KB？这是一个经验值：太小会增加磁盘 IO 次数，太大则浪费空间且读取效率降低。现代磁盘通常以 4KB 为单位读写，16KB 是 4 的倍数，能较好地平衡 I/O 次数和单次读取量。
:::

#### InnoDB vs MyISAM：一场存储引擎的较量

| 对比维度 | InnoDB         | MyISAM               |
| ---- | -------------- | -------------------- |
| 事务支持 | 是              | 否                    |
| 行级锁  | 是              | 否（表级锁）               |
| 外键约束 | 支持             | 不支持                  |
| 全文索引 | 5.6+ 支持        | 原生支持                 |
| 崩溃恢复 | 自动恢复（redo log） | 需要手动修复               |
| 存储结构 | 表空间 + 数据字典     | 三个文件（.frm/.MYD/.MYI） |

### 索引：加速查询的「秘密武器」

索引是什么？老师给了一个形象的比喻：

> 如果把数据库比作一本书，那么索引就是书的目录。有了目录，查找内容就不需要一页一页翻，而是直接定位。

```sql
-- 创建索引
CREATE INDEX idx_email ON users(email);

-- 查看查询计划
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';

-- 查看表的索引
SHOW INDEX FROM users;
```

但索引究竟是怎么工作的？当时老师没讲太深，后来我才知道 InnoDB 的索引底层用的是 **B+ 树**结构。

#### B+ 树的秘密

如果说数据页是图书馆里实际存放书籍的书架，那么 B+ 树就是那个帮你快速找到书籍的索引系统。

| 层级        | 作用              | 通俗类比      |
| --------- | --------------- | --------- |
| 根节点/非叶子节点 | 存储索引键值和指向下一层的指针 | 图书馆楼层的指示牌 |
| 叶子节点      | 存储实际数据或数据指针     | 书架上的实际书籍  |

B+ 树的特点：

- 所有叶子节点在同一层，查询效率稳定
- 叶子节点之间用链表连接，范围查询高效
- 矮胖结构，深度通常只有 2-4 层，减少磁盘 IO

```
        [根节点：索引页]
         /    |    \
   [索引页] [索引页] [索引页]
     |       |       |
   数据 ←→ 数据 ←→ 数据 ←→ ...
   (叶子节点链表连接，支持范围扫描)
```

#### 聚簇索引 vs 辅助索引：数据的两种存放方式

这是理解 InnoDB 索引的关键：

| 类型         | 定义      | 数据存放            | 数量       |
| ---------- | ------- | --------------- | -------- |
| 聚簇索引       | 主键自动建立  | 叶子节点直接存储完整数据行   | 一个表只能有一个 |
| 辅助索引（二级索引） | 手动创建的索引 | 叶子节点存储主键值，而非数据行 | 可以有多个    |

一个形象的比喻：

> 聚簇索引就像按姓氏排序的字典——数据本身就是按索引顺序排列的；辅助索引像字典末尾的检字表——先查检字表找到页码，再翻到对应页读取数据。

这意味着，使用辅助索引查询时，需要**回表**：

```sql
-- 假设 name 上有辅助索引
SELECT * FROM users WHERE name = '张三';

-- 执行过程：
-- 1. 在 name 的辅助索引树中找到 '张三' 对应的主键值，比如 id=15
-- 2. 再用 id=15 去聚簇索引树中查找完整数据行
-- 3. 返回结果

-- 如果只查主键和 name，就不需要回表
SELECT id, name FROM users WHERE name = '张三';  -- 覆盖索引，无需回表
```

#### 覆盖索引：索引的「终极形态」

当查询所需要的数据都在索引树中可以直接返回时，就称为「覆盖索引」。这是性能优化的重要手段。

```sql
-- 覆盖索引示例：查询的数据和条件都在 idx_name_age 上
CREATE INDEX idx_name_age ON users(name, age);

SELECT name, age FROM users WHERE name = '张三';  -- 覆盖索引，无需回表
```

#### 最左前缀原则：联合索引的查找规则

对于联合索引，数据是按最左边的列依次排序的：

```sql
-- 创建联合索引
CREATE INDEX idx_name_age_email ON users(name, age, email);

-- 查找过程：
-- 1. 能命中索引的情况：
--    WHERE name = '张三'              -- 只用 name，最左前缀
--    WHERE name = '张三' AND age = 25  -- 使用 name 和 age
--    WHERE name = '张三' AND age = 25 AND email = '...'  -- 全部使用

-- 2. 无法命中索引的情况：
--    WHERE age = 25                    -- 跳过最左列
--    WHERE email = 'zhangsan@example.com'  -- 跳过前两列
```

:::warning
最左前缀原则很容易踩坑。比如有些同学觉得创建了联合索引 `(name, age, email)` 就万事大吉，结果 WHERE age = 25 的时候索引完全失效。
:::

#### 索引失效的「玄学」

学了索引之后，我发现一个让人困惑的现象：明明建了索引，查询却没用上？

常见导致索引失效的场景：

```sql
-- 场景1：使用函数或运算
SELECT * FROM users WHERE YEAR(created_at) = 2023;  -- 索引失效
-- 正确做法：改写为范围查询
SELECT * FROM users WHERE created_at >= '2023-01-01' AND created_at < '2024-01-01';

-- 场景2：左通配符模糊查询
SELECT * FROM users WHERE email LIKE '%@example.com';  -- 索引失效
-- 正确做法：右通配符可行
SELECT * FROM users WHERE email LIKE 'zhang%';  -- 可以使用索引

-- 场景3：类型转换
SELECT * FROM users WHERE id = '1';  -- id是INT，传入了字符串，可能导致全表扫描
-- 正确做法：使用正确类型
SELECT * FROM users WHERE id = 1;

-- 场景4：OR连接了未索引列
SELECT * FROM users WHERE name = '张三' OR age = 20;  -- 如果age没索引，整条都走全表扫描
-- 正确做法：为age也建立索引，或者用 UNION 改写

-- 场景5：不等于判断
SELECT * FROM users WHERE status != 1;  -- 通常不走索引
SELECT * FROM users WHERE status NOT IN (1, 2);  -- 同上
```

:::note
索引并非越多越好。索引会占用额外的存储空间（有时比数据本身还大），而且对写入操作有性能影响——每次 INSERT/UPDATE/DELETE 都要维护索引树。对于写入频繁的表，要谨慎建索引。
:::

### 事务：数据一致性的「保护伞」

事务的四大特性（ACID）当时背得滚瓜烂熟，但真正理解它们的含义，还是在后来做项目的时候：

- **原子性（Atomicity）**：要么全成功，要么全失败
- **一致性（Consistency）**：事务前后数据状态必须一致
- **隔离性（Isolation）**：并发事务相互隔离
- **持久性（Durability）**：事务提交后永久生效

#### 隔离级别：并发的「度」

隔离性听起来简单，但实现起来却是个难题——完全隔离会降低性能，不隔离又会引发各种问题。SQL 标准定义了四个隔离级别：

| 隔离级别                     | 脏读  | 可重复读 | 幻读  | 性能 |
| ------------------------ | --- | ---- | --- | -- |
| Read Uncommitted         | 可能  | 可能   | 可能  | 最高 |
| Read Committed           | 不可能 | 可能   | 可能  | 中等 |
| Repeatable Read（MySQL默认） | 不可能 | 不可能  | 可能  | 较低 |
| Serializable             | 不可能 | 不可能  | 不可能 | 最低 |

可能读意味着什么？举个例子：

```
脏读：事务A读取了事务B还未提交的数据，如果B回滚，A读到的就是脏数据
不可重复读：事务A两次读取同一行数据，结果不同（因为事务B修改并提交了）
幻读：事务A按条件查询，事务B插入或删除了一条满足条件的数据，A再次查询时多了或少了记录
```

MySQL 默认的 Repeatable Read 隔离级别，靠 **MVCC（多版本并发控制）** 来实现——每个事务看到的是某个时间点的快照，不受其他事务影响。

#### MVCC：快照读取的秘密

MVCC 的核心思想是：每行数据都有两个隐藏列——创建版本号和删除版本号。

| 操作     | 版本号变化                                     |
| ------ | ----------------------------------------- |
| INSERT | 新增记录的创建版本号 = 当前事务ID                       |
| DELETE | 删除记录的删除版本号 = 当前事务ID                       |
| UPDATE | 先标记旧记录删除（设置删除版本号），再新增一条新记录                |
| SELECT | 只读取创建版本号 <= 当前事务ID 且删除版本号为空/ > 当前事务ID 的记录 |

这样，读操作不加锁、不阻塞，效率大大提升。

```sql
-- MySQL 查看当前隔离级别
SHOW VARIABLES LIKE 'transaction_isolation';

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

#### 锁：事务的「排队机制」

当两个事务同时修改同一条数据时，数据库需要一种机制来保证数据不出问题，这就是锁。

| 锁类型           | 作用范围   | 作用                  |
| ------------- | ------ | ------------------- |
| 共享锁（S锁）       | 行      | 读取数据时加锁，多个事务可以同时持有  |
| 排他锁（X锁）       | 行      | 修改数据时加锁，持有后其他事务不能加锁 |
| 意向共享锁（IS）     | 表      | 表示事务想要在表上加S锁        |
| 意向排他锁（IX）     | 表      | 表示事务想要在表上加X锁        |
| 间隙锁           | 范围     | 锁定一个区间，防止幻读         |
| 临键锁（Next-Key） | 前开后闭区间 | 记录锁+间隙锁的组合          |

一个形象的比喻：

> 共享锁像阅览室的座位——大家可以同时坐着看书，但都不能挪动桌椅；排他锁像厕所——一个人进去后就把门锁上，别人必须等。

#### 快照读 vs 当前读：两种不同的读取方式

MVCC 解决的是快照读的并发问题，但有些操作必须读取最新数据，这就需要当前读。

```sql
-- 快照读：普通 SELECT，不加锁，读取历史快照
SELECT * FROM users;  -- 快照读

-- 当前读：读取最新数据，加锁
SELECT * FROM users FOR UPDATE;    -- 加排他锁
SELECT * FROM users LOCK IN SHARE MODE;  -- 加共享锁

-- INSERT/UPDATE/DELETE 也属于当前读
INSERT INTO users VALUES (...);
UPDATE users SET ...;
DELETE FROM users WHERE ...;
```

#### redo log 与 undo log：崩溃恢复的「双保险」

MySQL 实现持久性和原子性，靠的是两个重要的日志：

| 日志类型 | redo log（重做日志）     | undo log（回滚日志）     |
| ---- | ------------------ | ------------------ |
| 作用   | 保证持久性              | 保证原子性              |
| 记录内容 | 物理页的修改             | 事务的反向操作            |
| 写入时机 | 事务提交前              | 事务执行中              |
| 空间   | 循环使用，可配置大小         | 随事务完成而删除           |
| 崩溃恢复 | 已提交事务从 redo log 恢复 | 未提交事务用 undo log 回滚 |

一个具体的例子：

```sql
-- 执行 UPDATE users SET age = 26 WHERE id = 1;

-- redo log 记录：
-- <id=1, age=26, page_no=5>  -- 在哪个数据页修改了什么

-- undo log 记录：
-- <id=1, age=25>  -- 反向操作，用于回滚
```

### SQL执行流程：一条查询的「人生旅程」

当时一直好奇，一条 SELECT 语句到底是怎么执行的？大致流程如下：

```
SQL语句 → 解析器（生成语法树） → 优化器（选择最优执行计划） → 执行器（调用存储引擎）
                          ↓
                    存储引擎：从磁盘读取数据页到缓冲池 → 返回结果
```

#### 慢查询日志：定位性能问题的「听诊器」

MySQL 提供了慢查询日志，用来记录执行时间超过指定阈值的 SQL：

```sql
-- 查看慢查询是否开启
SHOW VARIABLES LIKE 'slow_query%';

-- 设置慢查询阈值（单位：秒）
SET GLOBAL slow_query_time = 2;

-- 查看最近的一条慢查询
SHOW GLOBAL STATUS LIKE 'Slow_queries';

-- 完整记录未使用索引的查询
SET GLOBAL log_queries_not_using_indexes = ON;
```

使用 `EXPLAIN` 分析查询计划是关键：

```sql
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';
```

| 列名    | 含义                                                    |
| ----- | ----------------------------------------------------- |
| type  | 访问类型，从好到差：const > eq\_ref > ref > range > index > ALL |
| key   | 实际使用的索引                                               |
| rows  | 预计扫描的行数，越少越好                                          |
| Extra | 额外信息，如 Using index（覆盖索引）、Using filesort（需额外排序）        |

关键概念：**缓冲池（Buffer Pool）**

这是 InnoDB 维护的一块内存区域（默认 128MB，可配置），用来缓存热数据页。

```
读取数据：
磁盘 → 读取16KB数据页 → 写入缓冲池 → 返回结果
                  ↓
         下次查询先查缓冲池，有则直接返回，无则再读磁盘

写入数据：
写入请求 → 写入缓冲池（标记为脏页） → 返回"成功"
                  ↓
         后台线程定期将脏页刷新到磁盘（Checkpoint机制）
```

:::tip
理解了这个，你就能明白为什么「把大表的热数据放内存」能大幅提升性能——减少磁盘 IO。生产环境中，可以通过调大 `innodb_buffer_pool_size` 来提升性能。
:::

## 力扣刷题：简单到怀疑人生

老师还给我们找了一些力扣的SQL题目作为练习。说实话，那些题目的难度让我一度怀疑：

> 这是数据库练习题？这不就是套模板吗？

确实，像「合并两个表」「查找第二高的薪水」这类题目，解法套路化严重，没有太多弯弯绕绕。

但后来我才明白，这些简单题目其实是打基础。真正的难点不在于「会不会」，而在于「熟不熟」。当你在真实项目中面对复杂的多表关联查询时，那些基础概念的熟练度就直接决定了你的解题速度。

## 越学越无知的感悟

回顾这段学习经历，有一个感受越来越强烈：

> **越学，越觉得自己无知。**

当时觉得CRUD就是数据库的全部，后来发现索引优化、慢查询分析、事务隔离级别、锁机制这些才是真正考验功力的地方。

当时觉得SQL语法简单没什么难点，后来才发现子查询、联表查询、视图、存储过程这些高级用法背后，是一整套关系代数理论在支撑。

这种感觉，大概就是编程学习路上的「常态」吧。每掌握一个知识点，就会发现这个知识点背后还有更深的一层。

***

*数据库的学习没有终点，只有里程碑。2023年4月的这段经历，为我后来理解数据存储、了解系统底层打下了基础。*
