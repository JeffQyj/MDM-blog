## 梦开始的地方

2022年9月，我拖着行李箱走进了大学校门。

那时的我，对代码怎么写、电脑是怎么听懂人的指令的——一无所知。

开学第一课，C语言。

老师站在讲台上说：「编程是门手艺，多敲就会了。先去PTA注册个账号，我们从刷题开始。」

于是，我的编程之路，从PTA开始了。

## 第一阶段：PTA上的「纯手敲」

现在回想起来，PTA简直是个神奇的地方。

没有代码提示，没有自动补全，没有智能纠错。写代码就像在原始森林里探险——每一步都得小心翼翼，因为随时可能踩坑。

### C语言：简洁的折磨

刚开始学C语言的时候，觉得这玩意儿还挺友好的。变量声明、循环语句、数组操作，语法也就那么些。

但真正上手刷题才知道什么叫「简洁的折磨」。

比如一道经典的「打印九九乘法表」：

```c
#include <stdio.h>
int main() {
    for (int i = 1; i <= 9; i++) {
        for (int j = 1; j <= i; j++) {
            printf("%dx%d=%d", j, i, i*j);
            if (j < i) printf(" ");
        }
        if (i < 9) printf("\n");
    }
    return 0;
}
```

看着短，但每一个字符都得亲手敲进去。敲完一运行——

```
1x1=1
1x2=2 2x2=4
...
```

咦，格式不对，少了个空格。再来。空格对了，但换行位置不对。再来。

一道题改个五六遍是常态，有时候一道题能磨上一两个小时，就为了那一点微妙的格式差异。

### Java：繁琐的噩梦

如果说C语言是「简洁的折磨」，那Java就是「繁琐的噩梦」。

学Java的时候，已经是第二学期了。老师说：「Java是企业级开发语言，语法严谨，适合大型项目。」

但彼时的我，只觉得Java好长。

同样是实现一个「判断闰年」的功能：

```c
// C语言
int isLeap(int year) {
    return (year % 4 == 0 && year % 100 != 0) || (year % 400 == 0);
}
```

```java
// Java
public class Main {
    public static boolean isLeap(int year) {
        return (year % 4 == 0 && year % 100 != 0) || (year % 400 == 0);
    }
}
```

C语言一行搞定的事情，Java要写一个类，还要声明public static。刷题的时候，每道题都得套一层类，一遍遍敲 `public class Main`、`public static void main`、`Scanner scanner = new Scanner(System.in)`……

敲久了手都酸。

:::note
回过头看，「纯手敲」时代虽然效率低，但让我对每一种语言的语法结构都印象深刻。当你能亲手敲出每一行代码的时候，你很难忘记它的细节。
:::

### PTA格式地狱

说到PTA刷题，最让人崩溃的不是算法，是**格式**。

「输出样例」摆在那儿，你信心满满写完代码，提交——

```
答案错误
```

你盯着屏幕看了半天，输出明明一模一样啊？再仔细看：

```
样例输出：Hello, World!
你的输出：Hello, World!
```

一模一样啊！但就是不过。

后来才发现，是换行符的问题。样例最后一行没有换行，你多打了一个回车。不行。

还有空格问题、逗号是中文还是英文、分号是全角还是半角……

PTA用一次次「答案错误」告诉你：**编程是门精细活，差一点都不行。**

## 第二阶段：CV工程师

大概在PTA上刷了两三百道题之后，我开始接触Java Web。

这时候，IDE换成了IDEA，代码不再是一个个孤立的「刷题文件」了，而是组成了一个完整的项目。

随之而来的问题是：**我不知道怎么写。**

课上学的那点东西，顶多实现个「Hello World」。图书管理系统？没学过。博客系统？没听说过。

怎么办？

**搜。**

CSDN、博客园、GitHub、Stack Overflow……全靠搜索。

「Java Web登录功能怎么实现」——搜。
「Spring Boot配置文件怎么写」——搜。
「MyBatis多表查询怎么做」——搜。

搜到一篇博客，cv下来，运行——成功了！

那种感觉，就像在黑暗中摸索了三天三夜，突然看到一盏灯。

### CV的快乐与悲哀

CV（Ctrl+C & Ctrl+V）的日子，痛并快乐着。

快乐的是：**问题总能解决。**不管什么功能，搜索一下，总能找到答案。cv下来，一般都能跑起来。

悲哀的是：**下次遇到同样的问题，还是不会。**

有一次我照着博客cv了一个文件上传功能，跑通了，交作业了。

三个月后做课设，需要文件上传，我又去搜索了。找到一篇看起来很眼熟的博客，cv下来，又跑通了。

这时候我才意识到：这不是「学会」，这是「存档」。代码虽然跑通了，但我不理解它的原理。下次遇到问题，还是得重新搜。

### PTA格式问题的延续

说到格式，PTA刷题的经历在CV阶段依然发挥着作用。

有时候搜到的代码，跑起来输出和样例一模一样，但就是过不了。

比如这道题：

```
输入：5
输出：1 2 3 4 5
```

你输出 `1 2 3 4 5`，不对。再看——原来是「数字之间用一个空格分隔，最后一个数字后面没有空格」。

这种细节，搜索来的代码不一定能注意得到。往往是你本地跑通了，提交到PTA——「答案错误」。

然后你开始一行行对比输出，发现某个地方多了个空格，或者少了个换行。

**CV一时爽，对格式火葬场。**

:::note
回头看，CV阶段让我学到了两件事：
1. 解决问题的能力很重要——会搜索的人，总能找到答案
2. 理解原理更重要——不理解原理，下次遇到问题还是得重新搜索
:::

## 第三阶段：Tab工程师

2024年，AI编程工具开始普及了。

最开始是GitHub Copilot，后来是国内的各种AI编程助手。

第一次用Copilot的时候，我只觉得：**这东西有点东西。**

写Java代码的时候，我敲了个注释：

```
// 实现用户登录功能
```

然后Copilot就给我补全了一整段代码——校验密码、生成Token、异常处理，全都有。

按一下Tab，代码就采纳了。

**原来还可以这样？**

### 从「自己写」到「审AI写」

Tab工程师的核心转变是：**从写代码，变成了审代码。**

以前是「这个功能怎么实现？我来写。」
现在是「这个功能怎么实现？AI你先写，我来看看对不对。」

效率确实提高了。以前一道题可能要自己琢磨半小时，现在AI补全一下，自己检查一下，五分钟搞定。

但问题也随之而来：**有时候，我审不出问题。**

AI补全的代码，看起来逻辑清晰、结构完整。但仔细一看——咦，这个边界条件好像没处理？咦，这个变量命名好像有点奇怪？

更可怕的是，有时候AI补全的代码有bug，但我看不出来。因为我自己的水平也有限，分不清「这段代码是对的还是错的」。

### 刷题场景下的Tab工程师

在PTA刷题的场景下，Tab工程师的体验是这样的：

看到一道题，脑子里先想思路。想出来后，在IDE里敲个注释：

```java
// 使用双指针，left从0开始，right从末尾开始
```

然后Copilot就开始补全。你一看——思路对了，逻辑也没问题。按Tab采纳。

提交，通过。

有时候一道hard级别的题目，AI补全 + 自己审核，十几分钟就做出来了。

**爽。**

但问题也有：有些题目AI会理解错题意，补全出来的东西牛头不对马嘴。这时候就得靠自己去分辨——AI理解对了吗？思路对了吗？

如果自己的水平不够，很可能看不出来问题所在。

## 第四阶段：Vibe Coding

2025年初，Andrej Karpathy提出了一个词——**Vibe Coding**。

翻译过来就是「氛围编程」或者「直觉编程」——你描述需求，AI来实现。

第一次尝试Vibe Coding，我做了一个「今天吃什么」的小工具。

就是那种：你点一下，它告诉你「今天吃火锅」「今天吃烧烤」「今天吃沙县小吃」的应用。

打开Cursor，输入：

「帮我做一个今天吃啥的小应用，点击按钮随机显示一种食物」

三分钟后，代码出来了。

运行一下——能跑。

再美化一下界面——「加个渐变色背景」「按钮改成圆角」「加个动画效果」

AI一一照做。

**整个过程不到十分钟。**

如果是以前的我，光是想「用什么框架」「怎么实现随机」「前端怎么写」，可能就要折腾一下午。

### Vibe Coding的正确打开方式

后来我用Vibe Coding做了很多小玩意儿：

- 「睡前故事生成器」——输入一个关键词，AI生成一个小故事
- 「代码片段收藏夹」——输入语言和功能描述，AI生成示例代码
- 「周报生成器」——输入这周做了什么，AI帮你生成周报

这些都是**小工具**。功能单一、逻辑简单、不需要复杂的架构设计。

Vibe Coding做这些，**真的爽**。

但如果是复杂项目呢？

我尝试用Vibe Coding做一个「校园论坛系统」——包含用户模块、帖子模块、评论模块、私信模块……

结果呢？

AI确实给我生成了一大堆代码。但这些代码之间关系混乱，很多地方逻辑重复，有些地方甚至自相矛盾。

我花了三天时间去梳理这些代码，最后发现——还不如自己从头写。

:::warning
Vibe Coding适合做「小玩具」，不适合做「复杂工程」。它的优势在于快速验证想法，而不是直接产出生产级代码。
:::

## 第五阶段：SDD

经历了Vibe Coding的尝试之后，我开始思考一个问题：

**AI写代码很快，但它怎么才能「听话」？**

Vibe Coding的时候，我描述需求，AI来理解。但人的描述往往是模糊的、遗漏的、不准确的。

AI理解错了，做出来的东西就不对。
AI自由发挥了，代码就乱了。

**怎么让AI「听话」？**

答案很简单：**给它一份明确的规格说明书。**

这就是**SDD**（Specification Driven Development，规格驱动开发）的由来。

### SDD是什么？

SDD的核心思想是：**在让AI写代码之前，先用结构化的方式把需求写清楚。**

这份规格文档，包含：
- 这个功能要做什么
- 输入是什么，输出是什么
- 有什么约束条件
- 怎么才算「完成」了

有了这份规格，AI就知道「该做什么」和「不该做什么」，而不是漫无边际地自由发挥。

### SDD怎么做？

SDD的开发流程分为四个阶段：

**1. Specify（规格定义）**

这是最关键的环节。你要像写合同一样，把需求写清楚。

```
## 用户登录模块规格

### 功能
- 支持用户名密码登录
- 支持第三方登录（GitHub）
- 登录成功后返回Token

### 输入
- username：字符串，4-20个字符
- password：字符串，6-20个字符

### 输出
- 成功：返回Token
- 用户不存在：返回错误码 USER_NOT_FOUND
- 密码错误：返回错误码 INVALID_PASSWORD
- 账号锁定：返回错误码 ACCOUNT_LOCKED

### 安全约束
- 密码错误5次，锁定账号15分钟
- Token有效期24小时

### 验收标准
1. 正确用户名+密码 → 登录成功
2. 错误密码（<5次） → 返回错误，错误次数+1
3. 错误密码（>=5次） → 账号锁定
4. 锁定期间登录 → 拒绝
```

**2. Plan（技术规划）**

根据现有技术栈，设计合理的架构。

**3. Tasks（任务拆分）**

把大需求拆成小任务。

**4. Implement（实现）**

把规格文档丢给AI，让它严格按照规格来写代码。

### SDD vs Vibe Coding

| 对比 | Vibe Coding | SDD |
|------|------------|-----|
| 需求表达 | 自然语言，模糊、易遗漏 | 结构化文档，清晰、无歧义 |
| AI行为 | 自由发挥，容易跑偏 | 按规格执行，不会跑偏 |
| 适用场景 | 小工具、快速原型 | 复杂工程、生产级代码 |
| 风险 | 需求理解偏差，返工多 | 规格即合同，按约执行 |

## AI时代的思考

经历了这五个阶段的演进，我最大的感悟是：

> **懂技术的人，才能更好地驾驭AI。**

Vibe Coding很爽，动动嘴就能做个小工具。但如果不懂技术：

- 你可能描述不清楚需求，AI理解偏差
- 你可能看不出AI代码里的问题
- 你可能在关键地方被AI带偏

而如果懂技术：

- 你知道怎么精准描述需求
- 你能看出代码里的潜在问题
- 你知道什么时候该信AI，什么时候该信自己

| 维度 | 不懂技术 | 懂技术 |
|------|---------|--------|
| 需求描述 | 模糊、遗漏多 | 精准、结构化 |
| 代码审查 | 看不出问题 | 能识别风险 |
| 问题排查 | 完全依赖AI | 自己定位+AI辅助 |
| AI驾驭 | 被AI牵着走 | 让AI按方向执行 |

:::warning
AI时代，「会用AI」已经不是优势，「能驾驭AI」才是。

那些基础扎实、理解原理的人，才能在AI时代真正发挥创造力。
:::

## 小结

2025年10月，回顾这段从PTA到SDD的编程历程。

从「纯手敲」的体力劳动，到「CV刷题」的信息搬运，再到「Tab补全」的效率提升，最后到「Vibe Coding」的直觉编程——每一个阶段都是编程方式的演变。

而SDD，是这些演变之后的沉淀：**在AI时代，用规格文档来驾驭AI，而不是被AI驾驭。**

工具在变，核心能力不变。

AI能帮你写代码，但帮你写「好」代码的前提是——**你知道什么是「好」**。

所以，别慌。学好基础，理解原理，保持好奇。

**新时代比的不是谁「会用AI」，而是谁能「更好地驾驭AI」。**
