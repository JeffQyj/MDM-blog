:::tip
2023年9月，学习 JavaWeb 的时候，我遇到了一个让我困惑了很久的问题：「为什么 JSP 里的代码一会儿是 Java，一会儿是 HTML？」

这个问题困扰了我整整一周。

后来接触了 Thymeleaf，我才明白——原来模板引擎可以这么「优雅」。

再后来了解了 FreeMarker 和 Velocity，才发现模板引擎的世界远比 JSP 要精彩得多。
:::

## 故事要从一个「奇怪的文件」说起

2023年9月，大二开学不久。

学 JavaWeb 的时候，老师丢给我们一个 `.jsp` 文件，说：「这是你们的登录页面。」

我打开一看，满脑子问号：

```jsp
<%@ page contentType="text/html;charset=UTF-8" language="java" %>
<%@ page import="java.util.List" %>
<!DOCTYPE html>
<html>
<head>
    <title>用户列表</title>
</head>
<body>
    <h1>用户列表</h1>

    <%
        List<User> users = (List<User>) request.getAttribute("users");
        for (User user : users) {
    %>

        <div class="user-item">
            <span>用户名：<%= user.getUsername() %></span>
            <span>邮箱：<%= user.getEmail() %></span>
        </div>

    <%
        }
    %>

    <p>当前时间：<%= new java.util.Date() %></p>
</body>
</html>
```

这是啥？

- `<%@ page %>` 是什么？
- `<% %>` 里面写的是 Java 代码？
- `<%= %>` 又是啥？
- `%>` 后面怎么又是 HTML？
- 这到底是 Java 文件还是 HTML 文件？

老师说：「这是 JSP，JavaServer Pages。在 HTML 里写 Java。」

我内心OS：**在 HTML 里写 Java？那为什么不直接写 Java？**

:::note
后来我才明白，JSP 的诞生是因为那个年代的 Web 开发还没有前后端分离——后端程序员既要写业务逻辑，又要写页面。

JSP 的出现，就是让后端程序员能在「看起来像 HTML」的文件里写 Java。

它是一种「妥协」，也是一种「进步」。
:::

## JSP：模板引擎的「远古版本」

### JSP 的核心原理

JSP 文件会被 Tomcat（或其他 Servlet 容器）「翻译」成 Java 文件，然后编译成 `.class` 文件执行。

```
login.jsp  ──翻译──▶  login_jsp.java  ──编译──▶  login_jsp.class
```

翻译后的 Java 大概长这样：

```java
public class login_jsp extends HttpJspBase {

    public void _jspService(HttpServletRequest request,
                           HttpServletResponse response) {
        // out 是输出流，用来生成 HTML
        out.write("<html>\n<head>...</head>\n<body>\n");
        out.write("<h1>用户列表</h1>\n");

        // 你写的 Java 代码会原封不动地出现在这里
        List<User> users = (List<User>) request.getAttribute("users");
        for (User user : users) {
            out.write("<div class=\"user-item\">");
            out.write("<span>用户名：" + user.getUsername() + "</span>");
            out.write("</div>\n");
        }

        out.write("</body>\n</html>");
    }
}
```

所以 JSP 的本质是：**一种特殊的 Servlet，专门用来生成 HTML。**

### JSP 的元素一览

| 元素 | 语法 | 说明 |
|------|------|------|
| 指令 | `<%@ page %>` | 设置编码、导入包、错误页面 |
| 声明 | `<%! %>` | 定义成员变量、方法（不推荐） |
| 代码块 | `<% %>` | 写 Java 逻辑 |
| 表达式 | `<%= %>` | 输出变量值到 HTML |
| 注释 | `<%-- --%>` | JSP 注释，客户端看不到 |
| 动作 | `<jsp:include>` | 包含其他页面 |

### JSP 的「坑」

1. **Java 和 HTML 混在一起**——一个 500 行的 JSP，前端看到 Java 代码会懵，后端看到 HTML 也会懵。

2. **难以调试**——JSP 报错只会告诉你「在第几行」，但那个「行」是翻译后的 Java 代码的行号，不是你 JSP 里的行号。

3. **难以测试**——JSP 需要服务器环境，没法像普通 Java 类那样写单元测试。

:::warning
JSP 还有一个致命问题：**所有业务逻辑都可以直接写在 JSP 里**。

这意味着，你可以在 JSP 里直接查询数据库、直接操作文件、直接调用任何 Java 代码。

没有分层，没有架构，一个 JSP 文件可以干任何事。

这就是「意大利面代码」的根源。
:::

## EL 表达式和 JSTL：JSP 的「补丁」

为了解决 JSP 的「混乱」问题，后来出现了 EL 表达式和 JSTL。

### EL 表达式：让变量输出更优雅

原来我们这样写：

```jsp
<%
    User user = (User) request.getAttribute("user");
    out.print(user.getUsername());
%>
```

用 EL 表达式：

```jsp
${user.username}
```

如果 `user` 为空，EL 表达式会输出空字符串，不会像 `<%= %>` 那样抛出空指针异常。

EL 还支持运算：

```jsp
${1 + 2}              <%-- 输出 3 --%>
${user.age >= 18 ? '成年' : '未成年'}  <%-- 条件表达式 --%>
${user.score * 2}     <%-- 算术运算 --%>
```

### JSTL：让逻辑处理更「标签化」

原来我们这样写循环：

```jsp
<%
    List<User> users = (List<User>) request.getAttribute("users");
    for (User user : users) {
%>
        <li><%= user.getUsername() %></li>
<%
    }
%>
```

用 JSTL：

```jsp
<c:forEach items="${users}" var="user">
    <li>${user.username}</li>
</c:forEach>
```

JSTL 提供了很多标签：

| 标签 | 作用 | 示例 |
|------|------|------|
| `<c:if>` | 条件判断 | `<c:if test="${user.age >= 18}">成年</c:if>` |
| `<c:forEach>` | 循环遍历 | `<c:forEach items="${list}" var="item">...</c:forEach>` |
| `<c:choose>` | 多条件选择 | `<c:choose><c:when>...</c:when><c:otherwise>...</c:otherwise></c:choose>` |
| `<c:set>` | 设置变量 | `<c:set var="count" value="${list.size()}"/>` |
| `<c:out>` | 输出（防 XSS） | `<c:out value="${user.content}"/>` |

:::note
EL + JSTL 的组合，让 JSP 代码「看起来」整洁了一些。

但本质上，它只是「看起来」整洁了——业务逻辑仍然可以写在 JSP 里，只是换了一种写法而已。

真正的解决方案，是 Thymeleaf。
:::

## FreeMarker：功能强大的「老将」

### FreeMarker 是什么？

在 JSP 之后，FreeMarker 是另一个非常流行的 Java 模板引擎。

它的设计理念是：**模板（FreeMarker Template，FTL）+ 数据模型（Data Model）= 输出（HTML/文本）**

```html
<!DOCTYPE html>
<html>
<head>
    <title>${title}</title>
</head>
<body>
    <h1>${heading}</h1>

    <#list users as user>
        <div class="user-item">
            <p>用户名：${user.username}</p>
            <p>邮箱：${user.email}</p>
        </div>
    </#list>

    <p>日期：${now?string("yyyy-MM-dd")}</p>
</body>
</html>
```

### FreeMarker 的特点

1. **完全不能写 Java 代码**——FreeMarker 明确禁止在模板里写 Java 代码，只能用它的指令
2. **强大的宏定义**——可以定义可复用的模板片段
3. **丰富的内置函数**——日期格式化、字符串处理、数字计算等

### FreeMarker 的核心语法

| 语法 | 作用 | 示例 |
|------|------|------|
| `${var}` | 输出变量 | `${user.username}` |
| `<#list>` | 循环遍历 | `<#list users as u>...</#list>` |
| `<#if>` | 条件判断 | `<#if cond>...</#if>` |
| `<#assign>` | 赋值变量 | `<#assign name="张三">` |
| `<#macro>` | 定义宏 | `<#macro greet>Hello</#macro>` |
| `<#include>` | 包含其他模板 | `<#include "header.ftl">` |
| `?string("格式")` | 格式化 | `${date?string("yyyy-MM-dd")}` |

### FreeMarker vs JSP

| 对比 | FreeMarker | JSP |
|------|------------|-----|
| Java 代码 | 禁止 | 允许（混乱根源） |
| 模板可预览 | 需要数据模型 | 需要服务器 |
| 宏支持 | 强大 | 弱（JSTL 勉强支持） |
| 性能 | 高 | 一般 |
| 学习曲线 | 较陡 | 较简单 |

:::note
FreeMarker 的「禁止写 Java」是一把双刃剑。

好处是：模板代码干净，不会变成「意大利面」。

坏处是：有时候想写点逻辑，发现 FreeMarker 的语法不如 Java 直接。

但总体来说，FreeMarker 比 JSP 更适合大型项目。
:::

## Velocity：简洁至上的「前辈」

### Velocity 是什么？

Velocity 是 Apache 基金会的项目，比 FreeMarker 还要早一些。

它的设计理念是：**简单、清晰、一致**。

```html
<!DOCTYPE html>
<html>
<head>
    <title>$!{title}</title>
</head>
<body>
    <h1>$!{heading}</h1>

    #foreach($user in $users)
        <div class="user-item">
            <p>用户名：$user.username</p>
            <p>邮箱：$user.email</p>
        </div>
    #end

    <p>日期：$date</p>
</body>
</html>
```

### Velocity 的特点

1. **语法简洁**——`#foreach`、`#if`、`#set` 都很直观
2. **通用性强**——不只用于 Web，还用于代码生成、邮件模板等
3. **已经停止维护**——最后版本是 2009 年的 1.7，目前不再更新

### Velocity 的核心语法

| 语法 | 作用 | 示例 |
|------|------|------|
| `$var` 或 `$!{var}` | 输出变量（后者更安全） | `$!{user.username}` |
| `#foreach` | 循环遍历 | `#foreach($item in $list)...#end` |
| `#if` | 条件判断 | `#if($cond)...#end` |
| `#set` | 赋值 | `#set($name = "张三")` |
| `#include` | 包含文件 | `#include("header.vm")` |
| `#parse` | 解析模板 | `#parse("footer.vm")` |

### Velocity 的衰落

Velocity 在 2009 年之后基本停止维护，原因有几个：

| 问题 | 说明 |
|------|------|
| 停止维护 | 2009 年后没有更新 |
| 功能受限 | 不如 FreeMarker 功能丰富 |
| 性能一般 | 略逊于 FreeMarker |
| 社区萎缩 | 开发者转向其他模板引擎 |

:::warning
虽然 Velocity 已经停止维护，但在某些老项目里你可能还会遇到它。

如果接手这样的项目，最好有心理准备——它的语法简单是简单，但找资料可就没那么容易了。

我当时就踩过这个坑——遇到一个 Velocity 问题，百度谷歌一起上，结果 Stack Overflow 的答案都是 10 年前的。
:::

## Thymeleaf：现代化的模板引擎

### 为什么需要 Thymeleaf？

JSP 的问题是：**它只能在服务器端运行**。

这意味着：
- 前端设计师没法直接打开 `.jsp` 文件预览效果（因为它包含 Java 代码）
- 测试人员没法直接测试页面（需要完整的服务器环境）
- 前后端开发必须耦合——后端写完 Java 才能看页面效果

Thymeleaf 的核心理念是：**模板即 HTML，HTML 即模板**。

### Thymeleaf 的「优雅」

Thymeleaf 允许你在 HTML 标签上添加特殊属性，这些属性在服务器端会被解析，在浏览器里完全看不到。

```html
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
    <meta charset="UTF-8">
    <title th:text="${title}">默认标题</title>
</head>
<body>
    <h1 th:text="${heading}">默认标题</h1>

    <div th:each="user : ${users}">
        <p th:text="${user.username}">用户名</p>
        <p th:text="${user.email}">邮箱</p>
    </div>

    <p th:text="${#dates.format(now, 'yyyy-MM-dd')}">日期</p>
</body>
</html>
```

这段代码：
- 在浏览器里直接打开，就是普通的 HTML，有「默认标题」「用户名」「邮箱」这些占位符
- 在服务器端渲染时，`${title}` 会被真实数据替换，`th:text` 会被解析执行

这就是 Thymeleaf 的「优雅」——**模板可以原生打开预览，不需要服务器**。

### Thymeleaf 的核心语法

| 语法 | 作用 | 示例 |
|------|------|------|
| `th:text` | 设置文本内容 | `<p th:text="${name}">默认</p>` |
| `th:utext` | 设置 HTML 内容（不转义） | `<div th:utext="${htmlContent}">` |
| `th:if` | 条件判断 | `<p th:if="${count > 0}">有数据</p>` |
| `th:each` | 循环遍历 | `<tr th:each="user : ${users}">` |
| `th:href` | 设置链接 | `<a th:href="@{/user/{id}(id=${user.id})}">` |
| `th:src` | 设置图片源 | `<img th:src="@{/images/${img}}">` |
| `th:class` | 动态class | `<div th:class="${active} ? 'active' : ''">` |
| `th:fragment` | 定义片段 | `<div th:fragment="footer">` |

### Thymeleaf 的片段包含

Thymeleaf 有一个强大的功能：**片段包含**。

定义一个公共片段（比如导航栏）：

```html
<!-- templates/common/navbar.html -->
<nav th:fragment="navbar">
    <a href="/">首页</a>
    <a href="/about">关于</a>
    <a href="/contact">联系</a>
</nav>
```

在其他页面引入：

```html
<!-- 方式一：th:insert（插入到标签内部） -->
<div th:insert="common/navbar :: navbar"></div>

<!-- 方式二：th:replace（替换当前标签） -->
<footer th:replace="common/navbar :: navbar"></footer>

<!-- 方式三：th:include（保留当前标签，只包含内容） -->
<div th:include="common/navbar :: navbar"></div>
```

### Thymeleaf 的表单处理

Thymeleaf 对表单的处理非常优雅：

```html
<form th:action="@{/user/save}" th:object="${user}" method="post">
    <input type="text" th:field="*{username}" placeholder="用户名">
    <input type="email" th:field="*{email}" placeholder="邮箱">
    <button type="submit">提交</button>
</form>
```

`*{...}` 是 `th:object` 的简写：
- `th:field="*{username}"` 等价于 `th:field="${user.username}"`
- Thymeleaf 会自动处理 `<input>` 的 `name`、`id`、`value` 属性

## 语法对比：JSP vs Thymeleaf

### 循环遍历

| 功能 | JSP | Thymeleaf |
|------|-----|-----------|
| 基本循环 | `<% for(User u : users) { %> <%= u.getUsername() %> <% } %>` | `<p th:each="u : ${users}" th:text="${u.username}">` |
| 循环状态 | 需要手动维护计数器 | 内置 `uStat` 对象：`${uStat.count}` |
| 空集合 | 手动判断 | `th:if` 配合 |

### 条件判断

| 功能 | JSP | Thymeleaf |
|------|-----|-----------|
| if | `<% if(cond) { %> content <% } %>` | `<p th:if="${cond}">content</p>` |
| if-else | `<% if(cond) { %> A <% } else { %> B <% } %>` | `<p th:if="${cond}">A</p><p th:unless="${cond}">B</p>` |
| switch | JSTL `<c:choose>` | Thymeleaf `th:switch` |

### 链接处理

| 功能 | JSP | Thymeleaf |
|------|-----|-----------|
| 动态链接 | `<%= request.getContextPath() %>/user/` + id | `@{/user/{id}(id=${user.id})}` |
| 参数传递 | 字符串拼接 | 表达式内联 |

### 表单处理

| 功能 | JSP | Thymeleaf |
|------|-----|-----------|
| 绑定对象 | `<%= user.getUsername() %>` | `th:field="*{username}"` |
| 错误提示 | 手动判断 | `th:errors="*{username}"` |
| 重复渲染 | 手动处理 | 自动处理 |

:::note
从上面的对比可以看出，Thymeleaf 的语法更「HTML 化」。

它不是在 HTML 里嵌入 Java，而是在 HTML 标签上添加「指令」，让服务器解析这些指令来动态生成内容。

这让前端设计师和后端开发者可以更好地协作——前端看模板能看到结构，后端看模板能看到数据流。
:::

## 渲染模式的演进：从混合到分离

### 模式一：前后端混合（JSP 时代）

```
┌──────────────────────────────────────────────────┐
│                 后端服务器                         │
│  ┌─────────────────────────────────────────────┐ │
│  │              Java 代码                       │ │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────────┐  │ │
│  │  │ Servlet │──│ Service │──│ JSP 视图    │  │ │
│  │  └─────────┘  └─────────┘  └─────────────┘  │ │
│  │       │             │              │        │ │
│  │       └─────────────┴──────────────┘        │ │
│  │                  │                          │ │
│  └──────────────────│──────────────────────────┘ │
│                     ▼                            │
│            ┌────────────────┐                    │
│            │  生成的 HTML   │                    │
│            └────────────────┘                    │
└──────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐
│     浏览器       │
│   接收 HTML     │
│   直接渲染       │
└─────────────────┘
```

特点：
- 后端生成完整的 HTML
- 浏览器只负责渲染
- 前端「存在感」极低——前端工程师的工作就是写 JSP，然后被 Java 代码「污染」

### 模式二：模板引擎渲染（Thymeleaf 时代）

```
┌──────────────────────────────────────────────────┐
│                 后端服务器                         │
│  ┌─────────────────────────────────────────────┐ │
│  │              Java 代码                       │ │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────────┐  │ │
│  │  │ Controller│──│ Service │──│ Thymeleaf  │  │ │
│  │  └─────────┘  └─────────┘  └─────────────┘  │ │
│  │       │             │              │        │ │
│  │       └─────────────┴──────────────┘        │ │
│  │                  │                          │ │
│  └──────────────────│──────────────────────────┘ │
│                     ▼                            │
│            ┌────────────────┐                    │
│            │  模板 + 数据   │                    │
│            │  → 渲染 → HTML │                    │
│            └────────────────┘                    │
└──────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐
│     浏览器       │
│   接收 HTML     │
│   直接渲染       │
└─────────────────┘
```

特点：
- 后端返回「模板 + 数据」
- 模板是 HTML，可以独立预览
- 但本质上还是「后端渲染」——浏览器收到的还是完整 HTML

### 模式三：前后端分离（REST API 时代）

```
┌─────────────────┐              ┌─────────────────────┐
│     浏览器       │   JSON     │      后端 API        │
│                 │◀───────────▶│                     │
│  ┌───────────┐  │   REST     │  ┌─────────────────┐ │
│  │ Vue/React │  │   API      │  │ Controller      │ │
│  │ 前端框架  │  │            │  │ 返回 JSON       │ │
│  └───────────┘  │            │  └─────────────────┘ │
│       │        │            │           │           │
│       ▼        │            │           ▼           │
│  ┌───────────┐ │            │  ┌─────────────────┐ │
│  │ JS 渲染   │ │            │  │ Service         │ │
│  │ HTML 生成 │ │            │  │ 业务逻辑         │ │
│  └───────────┘ │            │  └─────────────────┘ │
│       │        │            │           │           │
│       ▼        │            │           ▼           │
│   浏览器渲染    │            │  ┌─────────────────┐ │
│   SPA 单页应用  │            │  │ 数据库           │ │
└─────────────────┘            └─────────────────────┘
```

特点：
- 后端只返回 JSON 数据，不生成 HTML
- 前端用 Vue/React/Angular 构建完整 UI
- 前端后端完全分离，并行开发
- API 是「公共接口」——网页、App、小程序都可以调用同一套 API

:::note
现在回头看这三种模式，像是看到了 Web 开发的三种「文明形态」：

| 模式 | 文明 | 特点 |
|------|------|------|
| JSP 混合 | 农业文明 | 自给自足，一个人干所有活 |
| Thymeleaf | 工业文明 | 分工合作，后端渲染模板 |
| 前后端分离 | 信息文明 | API 接口，数据与展示分离 |

没有绝对的「好坏」——不同场景用不同模式。

但作为程序员，了解这段历史，是为了理解：为什么现在的项目大多是前后端分离？为什么 Java 开发者的岗位要求里越来越多的「Vue/React」？
:::

## 模板引擎的选型

### 什么时候用 JSP？

- 老项目，维护成本优先
- 小型项目，前后端分工不明确
- 对页面性能要求极高（服务端渲染减少 HTTP 请求）

### 什么时候用 Thymeleaf？

- 新项目，需要良好的代码可维护性
- 团队中有前端开发人员
- 需要「模板可预览」的特性

### 什么时候用前后端分离？

- 中大型项目，需要并行开发
- 移动端（App、小程序）需要复用后端接口
- 对用户体验要求高，需要「无刷新」交互
- 团队有专门的前端开发

## 回望：模板引擎教会我的

### 数据与展示的分离

学习 JSP 的时候，我以为「Java 代码 + HTML」就是 Web 开发。

学习 Thymeleaf 的时候，我开始理解「模板 + 数据 = HTML」的思维。

学习前后端分离的时候，我才真正明白：**数据是数据，展示是展示，它们本来就是两件事。**

### 技术没有优劣，只有适合

JSP 被很多人吐槽，但它养活了一代 Java Web 开发者。

Thymeleaf 很优雅，但它的性能不如 JSP。

前后端分离很潮流，但它增加了复杂度，需要更多的人力来维护。

:::note
选技术，不是选「最好的」，而是选「最合适的」。

这是我在学习模板引擎的过程中，慢慢理解到的道理。
:::

### 站在历史的角度看技术

JSP 诞生于 1999 年，那时候还没有「前端工程师」这个职位。

Thymeleaf 诞生于 2011 年，那时候 jQuery 还是主流。

前后端分离流行于 2015 年之后，那时候 Angular、React、Vue 相继崛起。

每一种技术的出现，都有它的时代背景。

理解历史，是为了更好地理解现在，也是为了更好地预测未来。
