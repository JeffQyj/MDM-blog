## 前情提要：虚拟机老师的"预告片"

在上篇文章（[虚拟机全认知](https://blog.madm葚至.top/posts/编程生涯：虚拟机全认知)）的结尾，我写道：

> 虚拟机虽好，但"每个虚拟机都要装一个操作系统"这件事，终究还是太重了。于是有了**容器**——容器不需要虚拟化整个操作系统，只需要隔离进程。

没想到，话音刚落，容器就来了。

2023年12月12日，老师在虚拟机课程之后，正式开始讲解Docker。当时我的心情是：**刚折腾完虚拟机，又要折腾新东西了？**

但这次，我没想到的是——Docker带来的震撼，比虚拟机还要大。

## Docker到底是什么？

### 官话版

> Docker是一个开源的容器化平台，让开发者可以打包应用及其依赖到容器中，实现"构建一次，到处运行"。

听起来很美好。但这句话对于当时的我来说，信息量太大了。

### 通俗类比：集装箱与货物

还记得虚拟机的集装箱比喻吗？Docker容器可以理解为**更轻量的集装箱**。

| 集装箱（虚拟机） | 快递箱（Docker容器） |
|----------------|-------------------|
| 体积大，需要卡车运输 | 体积小，快递即可送达 |
| 里面要装整套家具（操作系统） | 里面只装货物（应用） |
| 搬运一次耗时久 | 搬运一次秒级完成 |
| 标准尺寸，但仍然笨重 | 轻便灵活，随用随开 |

虚拟机的集装箱是在码头（物理服务器）上运货，而Docker的快递箱可以直接"飞到"任何地方。

### 核心概念三件套

学Docker，最核心的是理解三个概念：

| 概念 | 类比 | 作用 |
|------|------|------|
| **镜像（Image）** | 菜谱/模具 | 定义应用长什么样、如何运行 |
| **容器（Container）** | 做好的菜/用模具生产的产品 | 镜像的实例，运行中的进程 |
| **仓库（Registry）** | 菜市场/超市 | 存储和分发镜像的地方 |

```
镜像 → 容器
菜谱 → 菜品
```

**镜像**是只读的模板，**容器**是镜像运行起来的实例。可以把容器理解为"从菜谱做出来的那道菜"——你可以做很多份相同的菜，但每份都是独立的。

## 第一个坑：镜像源

如果要评选"Docker初学者最容易踩的坑"，**镜像源**绝对能排第一。

### 问题现象

课程第一个作业是：在Docker里运行一个Nginx。

按照老师的步骤，我执行：

```bash
docker pull nginx
```

然后……就没有然后了。

进度条一动不动，等了五分钟，还是不动。

### 根因分析

当时我不知道，Docker Hub（官方镜像仓库）在国外，直接访问速度极慢，有时候干脆就是超时。

这就好像你要从海外代购一个爆款产品，直邮的话可能一个月都到不了。

### 解决方案

后来我知道了两个解法：

**方案一：配置国内镜像加速器**

在Docker里，添加国内的镜像源：

| 镜像加速器 | 地址 |
|-----------|------|
| 阿里云 | https://xxxx.mirror.aliyununc.com（需要登录获取） |
| 网易 | https://hub-mirror.c.163.com |
| Docker中文社区 | https://registry.docker-cn.com |

```json
{
  "registry-mirrors": [
    "https://hub-mirror.c.163.com"
  ]
}
```

配置完成后，再执行 `docker pull nginx`，速度直接从"龟速"变成"光速"。

**方案二：科学上网**

这个不多说，懂得都懂。

:::note
镜像源这个坑，几乎每个国内的Docker初学者都会踩。我的经验是：**第一时间配置好国内镜像源**，不要等到要用的时候才想起来。
:::

### 经验总结

| 教训 | 启示 |
|------|------|
| 不了解网络状况就开始下载 | 做事前先了解环境，特别是网络环境 |
| 只知道一种下载渠道 | 多准备几个备选方案 |
| 不看错误信息 | 学会读懂错误信息，timeout就是网络问题 |

## 第一个容器：Hello-World

配置好镜像源后，老师让我们运行第一个容器：

```bash
docker run hello-world
```

这行命令的背后，发生了什么呢？

### docker run 做了什么？

```
1. 本地查找是否有 hello-world 镜像
2. 本地没有，去镜像仓库拉取
3. 拉取完成后，创建容器
4. 运行容器，输出 "Hello from Docker!"
```

### 好奇心驱使的探索

看到 "Hello from Docker!" 的输出，我很兴奋，但也很困惑：

- 容器在哪里？
- 怎么查看它？
- 它现在还在运行吗？

带着这些问题，我开始探索Docker命令。

### 基础命令一览

| 命令 | 作用 |
|------|------|
| `docker images` | 查看本地所有镜像 |
| `docker ps` | 查看正在运行的容器 |
| `docker ps -a` | 查看所有容器（包括已停止的） |
| `docker run` | 创建并运行容器 |
| `docker stop` | 停止容器 |
| `docker rm` | 删除容器 |
| `docker rmi` | 删除镜像 |

```bash
# 查看本地镜像
docker images
# 输出：
# REPOSITORY    TAG       IMAGE ID       CREATED        SIZE
# hello-world   latest    feb5d9fea6a5   3 months ago   13.3kB

# 查看运行中的容器
docker ps
# 输出：（空的，因为hello-world运行完就退出了）

# 查看所有容器
docker ps -a
# 输出：
# CONTAINER ID   IMAGE         COMMAND    CREATED        STATUS
# abc123         hello-world   "/hello"   5 minutes ago  Exited (0) 5 minutes ago
```

## nginx之旅：让Docker跑起来

Hello-World只是热身，真正的挑战是运行Nginx。

### 为什么是Nginx？

因为Nginx是一个**有实际作用的Web服务器**——它运行起来后，可以真正访问到一个网页。这比"输出一行文字"要有成就感得多。

### 运行Nginx

```bash
docker run -d -p 8080:80 nginx
```

这行命令有几个关键参数：

| 参数 | 含义 |
|------|------|
| `-d` | 后台运行（detached） |
| `-p 8080:80` | 端口映射，虚拟机的8080映射到容器的80 |
| `nginx` | 镜像名称 |

### 验证Nginx是否运行成功

```bash
# 查看运行中的容器
docker ps
# 输出：
# CONTAINER ID   IMAGE   COMMAND                  CREATED         STATUS         PORTS                  NAMES
# def456         nginx   "/docker-entrypoint.…"   5 seconds ago   Up 4 seconds   0.0.0.0:8080->80/tcp   modest_wescoff

# 在虚拟机里访问Nginx
curl http://localhost:8080
# 输出：HTML页面内容
```

在虚拟机里执行 `curl http://localhost:8080`，看到HTML页面内容的那一刻，我感受到了Docker的"真香"。

### 端口映射的原理

`-p 8080:80` 这个参数让我困惑了一会儿。后来我理解了这个原理：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         宿主机（本机 Windows）                          │
│                                                                      │
│    注：我们在 Windows 上通过 FinalShell 连接虚拟机进行操作              │
└─────────────────────────────────────────────────────────────────────┘
                                    │
                                    │ SSH 连接
                                    ▼
┌─────────────────────────────────────────────────────────────────────┐
│                        虚拟机（VMware 中的 Linux）                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                     Docker 守护进程                              │ │
│  │  ┌──────────────────────────────────────────────────────────┐  │ │
│  │  │                      容器（Container）                     │  │ │
│  │  │                                                          │  │ │
│  │  │   :80 (nginx)  ←─────  虚拟机 :8080  ←─────  宿主机访问    │  │ │
│  │  │                                                          │  │ │
│  │  └──────────────────────────────────────────────────────────┘  │ │
│  └────────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

简单来说：**虚拟机（Linux）的8080端口和容器里的80端口建立了映射关系**。

当时我的操作环境是：
- **宿主机**：Windows，通过FinalShell远程连接
- **虚拟机**：CentOS 7（VMware里跑的），Docker安装在此
- **容器**：跑在虚拟机里的Docker容器

:::note
当时我的理解是这样的：对于容器来说，**虚拟机就是它的"宿主机"**。所以 `-p 8080:80` 的意思是把容器的80端口映射到虚拟机的8080端口。如果要从Windows（真正的宿主机）访问，还需要额外配置虚拟机的端口转发。
:::

## 数据持久化：删了容器，数据还在

运行Nginx几天后，我遇到了一个新问题：**如果我删掉这个容器，里面的数据会丢失吗？**

### 问题场景

我想保存Nginx的日志文件，但不知道该怎么操作。

### 解决方案：数据挂载

Docker提供了三种数据管理方式，其中最常用的是**挂载（Mount）**：

| 方式 | 命令参数 | 说明 |
|------|---------|------|
| **绑定挂载** | `-v /host/path:/container/path` | 把虚拟机目录挂载到容器 |
| **匿名挂载** | `-v /container/path` | Docker自动创建虚拟机目录 |
| **命名挂载** | `-v my-volume:/container/path` | 给挂载起个名字，方便管理 |

```bash
# 绑定挂载：把虚拟机的目录挂载到容器
docker run -d -p 8080:80 -v /my-nginx/logs:/var/log/nginx nginx

# 这样日志文件就保存在虚拟机了
# 即使删除容器，日志也不会丢失
```

### 为什么要挂载？

| 不挂载 | 挂载 |
|--------|------|
| 容器删了，数据也没了 | 容器删了，数据还在虚拟机 |
| 数据和容器绑定，难以迁移 | 数据独立于容器，迁移方便 |
| 无法在容器外修改数据 | 可以在虚拟机直接编辑数据 |

这就好像**租房子 vs 买房子的区别**：

- 不挂载：住在酒店里，离开时什么都带不走
- 挂载：有了自己的仓库，东西放哪都行

:::warning
数据挂载是Docker使用的重中之重。**重要数据一定要挂载出来**，不要让数据"困在"容器里。
:::

## Docker-Compose：编排的艺术

如果说单个容器让我感受到了Docker的便捷，那Docker-Compose让我真正感受到了**编排多个容器的魅力**。

### 为什么需要Docker-Compose？

随着学习深入，我发现一个应用往往需要**多个容器**配合工作。比如一个典型的前后端项目：

| 服务 | 容器 | 作用 |
|------|------|------|
| MySQL | mysql容器 | 数据库 |
| Redis | redis容器 | 缓存 |
| 后端 | springboot容器 | API服务 |
| 前端 | nginx容器 | 静态资源 |

如果一个个手动启动：

```bash
docker run -d --name mysql -p 3306:3306 mysql:8.0
docker run -d --name redis -p 6379:6379 redis
docker run -d --name backend -p 8080:8080 my-backend
docker run -d --name frontend -p 80:80 my-frontend
```

太麻烦了。而且这些容器之间还有依赖关系——后端需要等MySQL和Redis启动好才能启动。

### Docker-Compose的解法

Docker-Compose通过一个YAML文件来定义多容器应用：

```yaml
version: '3.8'

services:
  mysql:
    image: mysql:8.0
    ports:
      - "3306:3306"
    environment:
      MYSQL_ROOT_PASSWORD: 123456
    volumes:
      - mysql-data:/var/lib/mysql

  redis:
    image: redis
    ports:
      - "6379:6379"

  backend:
    build: ./backend
    ports:
      - "8080:8080"
    depends_on:
      - mysql
      - redis

  frontend:
    build: ./frontend
    ports:
      - "80:80"
    depends_on:
      - backend

volumes:
  mysql-data:
```

然后一行命令启动全部：

```bash
docker-compose up -d
```

### docker-compose 命令一览

| 命令 | 作用 |
|------|------|
| `docker-compose up -d` | 启动所有服务（后台） |
| `docker-compose down` | 停止并删除所有服务 |
| `docker-compose ps` | 查看服务状态 |
| `docker-compose logs -f` | 查看日志 |
| `docker-compose restart` | 重启所有服务 |
| `docker-compose exec [服务名] [命令]` | 在服务里执行命令 |

:::note
Docker-Compose解决的不仅是"启动方便"的问题，更重要的是**环境一致性问题**。开发环境、测试环境、生产环境，用的都是同一个 docker-compose.yml，不存在"在我电脑上能跑"的问题。
:::

## 网络模式：容器之间怎么通信？

多个容器在一起工作，就涉及到网络通信问题。

### Docker网络模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **bridge** | 默认模式，容器在独立网络里 | 独立容器 |
| **host** | 容器直接使用虚拟机网络 | 性能敏感场景 |
| **overlay** | 跨Docker主机通信 | 集群 |
| **none** | 无网络 | 离线运行 |

### 容器间通信

同一个bridge网络里的容器，可以通过**容器名**互相访问：

```yaml
services:
  backend:
    networks:
      - my-network
    depends_on:
      - mysql
  mysql:
    networks:
      - my-network

networks:
  my-network:
    driver: bridge
```

在 `backend` 容器里，可以通过 `mysql:3306` 访问MySQL，而不需要知道MySQL的IP地址。

这就相当于给每个容器分配了一个**固定的主机名**。

## 从Docker到容器化：更大的图景

学完Docker基础后，我开始理解"容器化"这个更大的概念。

### 容器 vs 虚拟机

| 维度 | Docker容器 | 虚拟机 |
|------|-----------|--------|
| 启动速度 | 秒级 | 分钟级 |
| 体积 | MB级 | GB级 |
| 隔离性 | 进程级 | 系统级 |
| 性能损耗 | 几乎无 | 5-15% |
| 适用场景 | 应用级 | 系统级 |

### Kubernetes：容器编排之王

学完Docker-Compose后，我知道了一个词：**Kubernetes**（简称K8s）。

如果说Docker-Compose是"多容器编排"，那Kubernetes就是**大规模容器编排平台"。

| Docker | Kubernetes |
|--------|------------|
| 单机管理 | 集群管理 |
| 单机容器编排 | 跨主机容器编排 |
| 手动扩缩容 | 自动扩缩容 |
| 简单的负载均衡 | 复杂的Service Mesh |

当然，2023年12月的我还没有真正接触Kubernetes。但我知道，这是迟早的事。

:::note
学习技术要有层次感：先会用，再理解原理，最后了解生态。Docker是基础，Kubernetes是进阶，Service Mesh是高级。先把Docker搞定，其他的慢慢来。
:::

## 写在最后

2023年12月的这次Docker学习，给我的最大收获是**"隔离"这个概念**。

以前我总觉得程序之间"和平共处"是正常的。学了Docker才发现：**隔离才是问题，环境不应该是问题**。

更重要的是，Docker让我理解了一个道理：**好的工具要善用，但工具背后的原理更重要**。镜像、容器、卷、网络……这些概念不是凭空出现的，它们解决的是实际问题。

从虚拟机到Docker，2023年的这两次学习，让我对"虚拟化"这个概念有了更深的理解：

- 虚拟机：虚拟化硬件
- 容器：虚拟化操作系统
- 未来的方向：更高层次的抽象

最后，记住一句话：**Docker不是银弹，它只是工具。但当你学会用它，你会发现——原来部署可以这么简单。**
