# DM点餐系统 — 项目分析报告

## 一、项目概述

| 项目   | 说明                             |
| ---- | ------------------------------ |
| 项目名称 | DM点餐系统                         |
| 项目类型 | 餐厅堂食扫码点餐系统                     |
| 应用场景 | 餐厅堂食，顾客扫码点餐，服务员出餐              |
| 终端构成 | 微信小程序（uni-app） + 后台管理（Next.js） |
| 后端服务 | Next.js API Routes             |
| 数据库  | Supabase (PostgreSQL)          |
| 设计风格 | 鬼灭之刃水墨风                        |
| 当前版本 | v1.1（PRD）/ v1.0（技术设计）          |

***

## 二、产品价值与目标

### 2.1 业务驱动力

- 提升点餐效率，减少服务员工作量
- 优化顾客用餐体验，实现自助化点餐
- 为餐厅经营者提供数据化运营能力

### 2.2 项目目标

- 实现完整的**扫码点餐 → 订单处理 → 菜品/桌台管理 → 数据统计**闭环
- 为顾客提供清晰的菜单浏览和实时订单状态
- 为店长和收银员提供高效的后台管理工具

### 2.3 不包含范围（当前阶段）

- 在线支付、外卖配送、会员积分、多门店管理

***

## 三、用户与角色体系

| 角色  | 终端   | 核心需求               |
| --- | ---- | ------------------ |
| 顾客  | 小程序端 | 扫码点餐、查看订单、取餐       |
| 收银员 | 后台管理 | 处理订单、操作状态、结账       |
| 店长  | 后台管理 | 全权限管理 + 菜品/员工/数据统计 |

**权限控制模型**：RBAC（基于角色的访问控制），小程序端支持游客模式。

***

## 四、技术架构

```
小程序端 (uni-app) ──┐
                      ├── Next.js API Routes ── Supabase PostgreSQL
后台管理 (Next.js) ───┘
                              │
                      Supabase Realtime（订单状态实时推送）
```

### 4.1 核心技术栈

| 层级    | 技术选型                                        |
| ----- | ------------------------------------------- |
| 小程序   | uni-app (Vue 3) + Pinia + Supabase客户端       |
| 后台前端  | Next.js 14 App Router + Zustand + shadcn/ui |
| 后端API | Next.js API Routes                          |
| 数据库   | PostgreSQL 15 (Supabase)                    |
| 认证    | 微信静默登录 + JWT (后台)                           |
| 文件存储  | 阿里云OSS（菜品图片）                                |
| 实时通信  | Supabase Realtime                           |

### 4.2 接口协议规范

- 协议：HTTP/HTTPS + JSON
- 风格：RESTful
- 认证：Bearer JWT Token（后台）；微信openid（小程序）
- 统一错误码：分层体系（10000系统/20000业务/30000权限/40000参数错误）
- 分页规范：`page, page_size, sort_by, sort_order`

## 六、数据库设计分析

### 6.1 核心数据模型（7张表）

| 表名           | 说明              |
| ------------ | --------------- |
| users        | 小程序顾客（微信openid） |
| employees    | 后台员工（店长/收银员）    |
| categories   | 菜品分类            |
| dishes       | 菜品（规格/辣度/库存）    |
| tables       | 桌台（8个固定座位）      |
| orders       | 订单主表（状态流管理）     |
| order\_items | 订单明细（菜品快照）      |

### 6.2 设计特点

- **主键**：使用 `uuid` 防枚举
- **时间**：`timestamptz` 确保时区一致性
- **快照设计**：`order_items` 存储下单时的菜品名称，防止改名后历史异常
- **库存管理**：下单预扣，取消恢复，使用数据库行锁防并发
- **JSON字段**：`spec_prices`、`spice_levels` 使用 `jsonb` 存储半结构化数据

### 6.3 关键业务规则

- **订单状态流转**：`pending → preparing → completed`（不可逆）
- **取消规则**：`pending` 可主动取消；`pending/preparing` 后台可取消；15分钟超时自动取消
- **订单编号**：`DM + YYYYMMDD + 3位序号`，每日重置

## 七、UI/UX 设计分析

### 7.1 整体风格

- **“鬼灭之刃水墨风”**：融合浓厚墨、水波纹、流动渐变、火焰强调色
- 主色调：深海墨、水波蓝、烈焰橙、宣纸白
- 动效：水波扩散、墨迹晕染、火焰摇曳

### 7.2 页面结构设计

- **小程序端**：采用卡片式布局、浮层弹窗（规格选择）、固定底部购物车栏
- **后台管理**：侧边导航 + 顶部状态栏 + 数据卡片/图表布局

### 7.3 交互规范

- 按钮：大小统一，主按钮用烈焰橙渐变，配有水墨晕染过渡
- 反馈：Toast/Modal/加载动画统一设计语言
- 触摸友好：最小点击区域 44px，适配刘海屏安全区域

## 九、核心流程分析

### 9.1 顾客点餐流程

```
扫码 → 桌号识别 → 选择人数 → 浏览菜单 → 规格选择 → 加入购物车
  → 确认订单 → 提交（库存预扣） → 订单详情（实时状态） → 取餐
```

### 9.2 后台订单处理流程

```
新订单（pending）→ 接单（preparing）→ 完成出餐（completed）
```

### 9.3 库存流程

```
下单：stock - quantity（预扣）
取消/超时：stock + quantity（恢复）
```

## 十、风险与改进建议

### 10.1 潜在风险

| 风险项          | 分析                       | 缓解措施         |
| ------------ | ------------------------ | ------------ |
| 库存并发超卖       | 高并发下单可能导致库存不一致           | 数据库行锁 + 事务控制 |
| 微信API依赖      | 小程序码生成依赖ACCESS\_TOKEN稳定性 | Token缓存+过期刷新 |
| Realtime连接中断 | 网络波动可能导致实时通知丢失           | 增加降级轮询机制     |
| 游客模式数据归属     | 游客订单无法关联用户               | 后续可引导登录后绑定   |
| 技术栈复杂性       | 前后端分离 + 实时订阅增加开发复杂度      | 严格按照PRD分阶段实施 |

### 10.2 建议改进方向（后续迭代）

- 增加**支付功能**，完成点餐支付闭环
- 增加**语音/打印**等后台通知方式
- 完善**经营报表**和**菜品销售分析**
- 支持**多门店/分店**模式
